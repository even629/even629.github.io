This article introduces the core principles and working mechanisms of the dynamic binary translation engine TCG (Tiny Code Generator) in QEMU, discussing in detail the differences between Guest and Target architectures, the role of TCG IR, as well as the process of translating basic blocks into translation blocks, variable classification, caching mechanisms, and performance optimization techniques such as direct block linking.

Environment

Create .clangd

gdb

QEMU supports multiple accelerators, but they can be broadly divided into two categories: instruction emulation technology (TCG), virtualization technology (KVM, HVF), etc.

Common translation techniques:

  • Interpreter: parses and executes one Guest instruction at a time, in a loop.
  • Static Binary Translation: translation is performed before the program runs. There is no translation overhead at runtime, but the scope for optimization is limited.
  • Dynamic Binary Translation: dynamic translation during program execution. It generally translates based on program traces rather than translating everything, allowing for deep optimization of hot code.

TCG (Tiny Code Generator) was originally a compiler backend for the C language, and later evolved into QEMU’sdynamic binary compilation (translation) engine。

Target & Guest

  • Guest (virtual machine / emulated architecture)

    • refers to CPU architecture emulated by QEMU, which is the CPU type of the system you are running.

    • For example:

      • riscv64(RISC-V 64-bit)
      • aarch64(ARM 64-bit)
      • i386(x86 32-bit)

    • Specify the Guest architecture when starting QEMU, for example:

    • The Guest determines the instruction set, registers, etc., emulated by QEMU.

  • TCG Target (TCG target architecture)

    • TCG (Tiny Code Generator) is QEMU’s dynamic translator, used to convert Guest instructions into native instructions executable by the Host CPU。
    • TCG Target is the architecture of the host CPU, determining on which CPU the machine code generated by QEMU will run.
    • For example:
      • Host =x86_64
      • Guest =riscv64
      • TCG Target =x86_64(because QEMU ultimately generates x86_64 machine code to execute on the host)

TCG IR

Similar to LLVM, QEMU also defines its own IR, and the process is as follows:

Translate Guest instructions intoQEMU IR, this is the frontend, the backend translatesQEMU IRinto host instructions

Advantages：

  • Good extensibility: to support a new frontend (Guest), one only needs to implement source -> IR;
  • Easy to pipeline: similar to LLVM, various passes can be introduced to optimize different stages.

Disadvantages：

  • Generally low performance (relatively speaking)

TCG translation process

Basic process

TCG’s binary translation uses code blocks (Basic Blocks) as the basic unit, and the translation product is a Translation Block.

Basic Block division rules:Branch instructions；Privileged instructions / exceptions；Code segment crossing page boundaries。

  1. Guest PC points to a Basic Block
    • Basic Block（BB） which is a continuous, branch-free sequence of instructions.
    • For example, a sequence of sequentially executed instructions in x86/RISC-V.
  2. Check TB Cache, translation cache
    • TB（Translated Block） is the result after a Basic Block is translated by DBT into host executable code .
    • DBT maintains a hash table or cache, where the key is usually the Guest PC.
  3. TB Cache hit (Y)
    • If the TB corresponding to the Guest PC has already been generated, jump directly to Exec TB Execute.
    • No need to translate again, improving efficiency.
  4. TB Cache miss (N)
    • Call translation module, which converts the Guest’s Basic Block into a Translation Block (TB).
    • After translation, the TB is saved to the cache for fast execution next time.
  5. Direct Block Chaining
    • Each TB saves its corresponding Guest PC, and usually also records the next Guest PC(the jump target of the last instruction)
    • After executing a TB, if the next Guest PC already has a TB, it directly jumps to execute the next TB without returning to theCheck TB Cache。
    • This is achieved by generating a direct jump instruction at the end of the TB.
    • It optimizes execution efficiency by avoiding returning to the interpreter loop every time, especially for loop-intensive code.
  • Once a Basic Block is converted into a TB by DBT, the next time the same Basic Block is executed, the TB can be directly fetched from the cache for execution without needing to go through conversion again:

Translation Block

TCG’s binary translation uses code blocks (Basic Blocks) as the basic unit, and the translation product is a Translation Block.

TCG has three types of variables: temporary, local temporary, and global

  • global: Generally, system registers are global variables. Global variables are further divided into registers and memory. Memory generally defines the registers inside the CPU

    For memory type, during backend translation, the values of guest CPU registers are first loaded into host registers, and after calculation, they are stored back into the guest CPU structure. The simulation process involves memory access.

    For register type TCGv, during backend translation, it is directly mapped to host registers, allowing direct access each time. It is generally used to store the pointer to the guest CPU env.

  • temporary: The variable’s lifetime is only within one BB

  • local temporary: The variable’s lifetime is within one TB, and can span across BBs.

Basic Block division rules:Branch instructions；Privileged instructions/exceptions；Code segment crosses page。

BB starts from the end of the previous BB or a set_starts with a label instruction, BB ends with a branch instruction (brcond_xxx)、goto_tb and exit_tb ends,

  • Prologue：

    • Save Guest CPU state (registers, flags, etc.) to host registers or memory

    • Set up execution environment

  • Epilogue：

    • Restore Guest CPU state

    • Jump back to Dispatcher or directly to the next TB

Direct block chaining

Taking the x86_64 platform as an example, each context switch requires executing about 20 instructions (which also perform memory reads and writes), so one of the optimization measures for DBT isreducing context switches, implementing direct linking between TBs: (e.g., direct jump instructions can directly connect two TB blocks, but indirect jump instructions cannot because they depend on runtime calculation)

Code Buffer

  • code_buffer is the continuous memory area on the host machine where TCG (Tiny Code Generator) stores translated Guest instructions (TB) continuous memory area.
  • all Translation Block（TB） are all generated, stored, and executed in this buffer.
  • Subsequent Prologue / TB.code / Epilogue are all located in code_buffer.

tcg_code_gen_epilogue：

  • Generate the epilogue part of the TB, and updateTB.tc.ptr

tcg_qemu_tb_exec：

  • Get TB.code based on TB.struct, and jump to TB.code

TB.struct：

  • Record the metadata of the TB, for example:
    • Guest PC (entry address)
    • TB length
    • Pointer to TB.code
    • Next TB (Direct Block Chaining)

TB.code：

  • Host machine code generated by TCG translation
  • Actual execution starts from here

Return to QEMU world after epilogue execution

Early in QEMU startup, a function called tcg is executed_init_machine, completing the allocation and initialization of code_buffer.

  • All subsequent code translation and execution work revolves around code_buffer
  • The backend management of TCGContext also revolves around code_buffer

DecodeTree

Decodetree is a mechanism proposed by Bastian Koppelmann in 2017 when porting RISC-V QEMU. The main reason for proposing this mechanism is that past instruction decoders (such as ARM) used a bunch of switch-cases for judgment. They were not only hard to read but also difficult to maintain.

Therefore, Bastian Koppelmann proposed the Decodetree mechanism, where developers only need todefine the format of each instruction through Decodetree’s syntax, and then hand it over to Decodetree to dynamically generate the corresponding instruction decoder.c containing switch cases.

Decodetree is essentially a Python script that takes a file defining the architecture’s instruction format as input and outputs the instruction decoder source code file.

  • input: Instruction encoding format file defined by the architecture
  • output: Source code of the instruction decoder (participates in QEMU compilation)

Decodetree syntax

Decodetree syntax is divided into four parts: Fields, Argument Sets, Formats, and Patterns.

  • Fields, describing fields such as registers and immediates in the instruction encoding;
  • Argument Sets, describing the values used to hold the fields extracted from the instruction;
  • Formats, describing the format of the instruction and generating the corresponding decode function;
  • Pattern, describing the decode method of an instruction.

Decodetree Field

Field defines how to extract the values of each field (eg: rd, rs1, rs2, imm) from an instruction.

  • %identifier
    • The name of the field, defined by the developer
    • Example:%rd,%rs1,%imm
  • unnamed_field
    • Specify the bit position of a field in the instruction
    • Format:high_bit : low_bit
    • OptionalsIndicates sign extension
    • Example:7:5Indicates bit[7:5] of the instruction
    • Example:31:s20Indicates bit[31:20] and requires sign extension
  • !function=identifier
    • After extracting the field value from the instruction, call a function for further processing
    • For example, immediate values require sign extension, bit concatenation, or address translation

Decodetree Argument Sets

Argument Set definitionUsed to store the values of the fields extracted from the instruction。

  • &identifier
    • Define a name for the Argument Set, customized by the developer
    • Example:&regs,&loadstore
  • args_elt
    • Elements contained in the Argument Set, usually previously defined Fields
    • Example:rd,rs1,imm
    • Meaning: save the values extracted from these fields into this Argument Set
  • !extern
    • Indicates that the Argument Set has already been defined in another Decoder; if this field exists, the corresponding argument set struct will not be generated again
    • Avoid generating duplicate structs

Decodetree Pattern

Pattern actually defines the decode method of an instruction. Decodetree will dynamically generate the corresponding switch-case decode judgment branches based on the definition of Patterns.

  • The identifier can be customized by the developer, such as: addl_r, addli… etc.
  • pat_elt can adopt the following different syntaxes:
    • fixedbit_elt is the same as fixedbit in Format_definition.
    • field_elt is the same as field in Format_definition.
    • field_ref is the same as field in Format_definition.
    • args_ref is the same as args in Format_definition.
    • fmt_ref directly references a defined Format.
    • const_elt can directly specify the value of a certain argument.

TCG IR

Previously, we discussed how to use QEMU’s helper functions to simulate instruction functionality, but generally, helpers are mainly used when IR implementation is inconvenient.

If you want to achieve better performance, it is recommended to use IR for implementation.

The front end of TCG is responsible for converting target architecture instructions into TCG ops, while the back end of TCG is responsible for converting TCG ops into target architecture instructions.
