This article introduces the development history of Linux asynchronous I/O, discusses the principles and functions of the high-performance asynchronous I/O framework io_uring, summarizes its core advantages such as unified asynchronous framework, true asynchrony, and high scalability, and provides related program examples and performance test results.

io_uringis 2019 Linux 5.1 high-performance first introduced by the kernel asynchronous I/O framework, can significantly accelerate the performance of I/O-intensive applications. But if your applicationis already using traditional Linux AIO,and using it appropriately, thenio_uringit will not bring much performance improvement, according to the original test, even with advanced features enabled, it is only 5%. Unless you really need that extra 5% performance, otherwise switchingtoio_uringthe cost may also be significant, because you need to rewrite the applicationto adaptio_uring(or let the dependent platform or framework adapt, in short, code changes are needed).

Since the performance is similar to traditional AIO, why is it still calledio_uringa revolutionary technology?

  1. Its first and greatest contribution is:unified the Linux asynchronous I/O framework，
    • Linux AIO only supports direct I/O modestorage file (storage file), and is mainly used inthe subfield of databases；
    • io_uringsupports storage files and network files (network sockets), and also supports more asynchronous system calls (accept/openat/stat/...), rather than being limited toread/writesystem calls.
  2. BeforeIt is truly asynchronous I/O by design, in contrast, although Linux AIO is also asynchronous, it may still block, and its behavior is unpredictable in some cases;
  3. Flexibility and scalabilityVery good, even can be based onio_uringRewrite all system calls, while Linux AIO was not designed with scalability in mind.

eBPF is also an asynchronous framework (event-driven), but withio_uringno essential connection, they belong to different subsystems, and there is an essential difference in the model:

  1. eBPF is transparent to users, just upgrade the kernel (to a suitable version),the application does not need any modification；
  2. io_uringprovidesnew system calls and user-space APIs, thereforerequires application modification。

This article introduces the development history of Linux asynchronous I/O,io_uringits principles and functions, and provides someprogram examplesandperformance stress testresults.

Evolution of Linux I/O system calls

Blocking I/O based on fd:read()/write()

As the most familiar read/write method, the Linux kernel providessystem calls based on file descriptors, these descriptors may point tostorage file(storage file), or network sockets：

both are calledblocking system calls(blocking system calls), because when the program calls these functions, it enters sleep state and is then scheduled out (yields the processor) until the I/O operation completes:

  • If the data is in a file and the file contentis already cached in the page cache, the call willreturn immediately；
  • If the data is on another machine, it needs to be obtained over the network (e.g., TCP), which will block for a while;
  • If the data is on the hard disk, it will also block for a while.

But it is easy to think that as storagedevices become faster and programs become more complex, the blocking method, the simplest approach, is no longer applicable.

Non-blocking I/O:select()/poll()/epoll()

After blocking, some new non-blocking system calls emerged, such asselect()、poll()and the newerepoll(). When applications call these functions for reading and writing, they do not block, but insteadreturn immediately, returning a list of file descriptors that are ready。

select()

  • Cross-platform(supported by Linux, Windows, macOS)
  • Maximum number of fds is limited: usuallyFD_SETSIZE = 1024
  • using a bitmap (fd_set) to represent the set of fds to monitor
  • Each call requires:
    • Copy the user-space fd_set to the kernel;
    • The kernel traverses all fds to check status;
    • On return, copy the entire fd_set back to user space;
    • The user needs to traverse all fds to determine which are ready.
  • Only supports level-triggered (LT)
  • Disadvantages：
    • Low upper limit on the number of fds;
    • Each call requires full copy + full polling → O(n) time complexity;
    • Cumbersome programming (need to maintain three fd_sets: read, write, exception)

poll()

  • Linux/Unix support，Windows does not support
  • Usestruct pollfdArray replaces bitmap, breaking the 1024 limit (only limited by system ulimit)
  • Still need to pass the entire array to the kernel each time, the kernel traverses and checks, and after return the user still needs to traverse to find ready fds
  • Still an O(n) polling model
  • Only supports LT mode

✅ Advantage: No 1024 limit
❌ Disadvantage: Performance still worse than epoll (under large-scale connections)

epoll()

  • Linux-specific, not cross-platform

  • Based on event-driven + kernel callback mechanism

  • Three core functions：

  • Kernel data structure：

    • Red-black tree: Stores all registered fds (add, delete, modify O(log n))
    • Ready doubly linked list: When fd is ready, the kernel automatically adds it to the linked list

  • epoll_wait()Directly return the list of ready events, no need to traverse all fds → O(1) get ready events

  • Supports two trigger modes：

    • LT（Level Triggered）: Default mode, continuously notifies as long as there is data in the buffer (similar to poll)
    • ET（Edge Triggered）: Notifies only once when state changes (e.g., from unreadable to readable), requires non-blocking I/O, read all data at once

✅ Advantages:

  • Efficiently handle Tens of thousands or even millions of concurrent connections(e.g., Nginx, Redis, Kafka)
  • No hard limit on number of fds
  • Less memory copy (only passed once during registration, only returns ready items when event occurs)

❌ Disadvantages:

  • Linux only
  • ET mode programming is complex (easy to miss reading data)

epollMonitoring a regular file fd is meaningless— it will continuously trigger events, but cannot achieve the purpose of ‘asynchronously notifying data arrival’.

But this method has a fatal flaw:Only supports network sockets and pipes —— epoll() doesn’t even support storage files.

Thread pool approach

For storage I/O, the classic solution is thread pool: The main thread distributes I/O to worker threads, which perform blocking reads and writes on behalf of the main thread, so the main thread does not block.

The problem with this approach isthe overhead of thread context switching can be very large, as will be seen in later performance tests.

Direct I/O (database software): bypassing the page cache

Then a more flexible and powerful approach emerged:Database software(database software) sometimes does not want to use the operating system’s page cache, but instead wants to open a file andread and write this file directly from the device(direct access to the device). This approach is calledDirect Access(direct access) ordirect I/O（direct I/O），

  • Need to specify O_DIRECT flag；
  • needthe application manages its own cache —— which is exactly what database software wants;
  • is zero-copy I/O, because the application’s buffered data is sent directly to the device, or read directly from the device.

Asynchronous I/O (AIO)

As mentioned earlier, as storage devices become faster, the proportion of context switching overhead between the main thread and worker threads increases. Some devices on the market now, such as Intel Optane,have latency as low as the same order of magnitude as context switching(microsecondsus). To put it another way, it makes us feel this overhead more: Each time we switch context, we lose one opportunity to dispatch I/O。

dispatch I/O refers tosubmitting (initiating) an I/O requestthe action to the storage device.

In high-performance systems (such as databases, KV stores), the goal isto dispatch I/O concurrently as much as possible, to keep the device fully loaded (because devices support high queue depths, e.g., NVMe supports thousands of concurrent commands).

Therefore, Linux 2.6 the kernel introduced the asynchronous I/O interface, for convenience, abbreviated in this article aslinux-aio。AIO Principlesis very simple:

  • The user, throughio_submit()submits an I/O request,
  • and later callsio_getevents()to check which events are ready.
  • enables programmersto write fully asynchronous code。

Recently, Linux AIO even supportsepoll(): that is, not only can it submit storage I/O requests, but also network I/O requests. If this trend continues, linux-aio seems to become a king. But due to its poor evolution path, this wish is almost impossible to realize. We can see from Linus’s iconic fierce remarks：

Reply to: to support opening files asynchronously

So I think this is ridiculously ugly.

AIO is a horrible ad-hoc design, with the main excuse being “other, less gifted people, made that design, and we are implementing it for compatibility because database people — who seldom have any shred of taste — actually use it”.

— Linus Torvalds (on lwn.net)

Linux AIO is indeed plagued with problems:

  1. only supportsO_DIRECTfile, thereforefor regular non-database applications （normal, non-database applications）is almost useless；
  2. The interface wasnot designed with extensibility in mind. Although it can be extended — and indeed it was — adding anything is quite complex;
  3. Although froma technical standpoint the interface is non-blocking, in practice there are many possible reasons that can cause it to block, and the ways it triggers are unpredictable.

Summary

From the above, the evolution of Linux I/O can be clearly seen:

  • Initially, synchronous (blocking) system calls;
  • Then, asactual needs and specific scenarios, new asynchronous interfaces were continuously added, while maintaining compatibility and cooperation with old interfaces.

It is also seen that on the issue of non-blocking reads and writes,no unified solution was formed：

  1. In the network socket domain: add an asynchronous interface, then poll to check if the request is complete (readiness);
  2. In the storage I/O domain:only for a specific niche(database) needs at a particular time, a customized asynchronous interface was added.

This is the evolution history of Linux I/O — focusing only on the present, introducing a design whenever a problem arises, without much foresight — untilio_uringthe emergence of

io_uring

io_uring came from the idea of veteran kernel developer Jens Axboe, who has done extensive research in the Linux I/O stack. From the earliestpatch aio: support for IO pollingit can be seen that this work started from a very simple observation: as devices become faster, the interrupt-driven mode has become less efficient than the polling mode （polling for completions) — This is also one of the most common topics in the high-performance field.

  • io_uringofThe basic logic is similar to linux-aio.: Provides two interfaces, one to submit I/O requests to the kernel, and one to receive completion events from the kernel.
  • But as development progressed, it gradually became a completely different interface: the designers began to think from the source. How to support fully asynchronous operations.。

Differences from Linux AIO

io_uringandlinux-aiohas essential differences:

  1. It is truly asynchronous in design(truly asynchronous). As long as the appropriate flag is set, itin the system call context just puts the request into a queue, and does nothing else extra,ensuring that the application never blocks。

  2. Supports any type of I/O: cached files, direct-access files, and even blocking sockets.

    Due to its async-by-design nature, thereforeno need for poll+read/write to handle sockets. Just submit a blocking read, and after the request completes, it will appear in the completion ring.

  3. Flexible and scalable: Based onio_uringIt can even re-implement every system call of Linux.

Principle and core data structures: SQ/CQ/SQE/CQE

Each io_uring instance hastwo ring queues(rings), shared between the kernel and the application:

  • Submission queue：submission queue (SQ)
  • Completion queue：completion queue (CQ)

These two queues:

  • are bothsingle producer, single consumer, size is a power of 2;
  • providelock-free interface(lock-less access interface), internally use memory barriersfor synchronization (coordinated with memory barriers).

Usage：

  • Submission
    • Application creates SQ entries (SQE), updates SQ tail;
    • Kernel consumes SQEs, updates SQ head.
  • Completion
    • Kernel creates CQ entries (CQE) for one or more completed requests, updates CQ tail;
    • Application consumes CQEs, updates CQ head.
    • Completion events may arrive in any order, but are always associated with a specific SQE.
    • Consuming CQEs does not require switching to kernel mode.

Benefits

io_uringAnother benefit of this submission method is that what originally required multiple system calls (read or write) now becomes a batch submission.io_uringThis batch processing capability is brought to system calls other than storage I/O some other system calls, including:

  • read
  • write
  • send
  • recv
  • accept
  • openat
  • stat
  • some dedicated system calls, such asfallocate

In addition,io_uringmaking the use cases of asynchronous I/O no longer limited to database applications,ordinary non-database applications can also use it. This point is worth repeating:

Althoughio_uringandaiothere are some similarities, itsscalability and architecture are revolutionary: itbrings the power of asynchronous operations to all applications(and their developers), and is no longer limited to the niche of database applications。

Avi Kivity at the Core C++ 2019 event gave a talk about async. Key points include:In terms of latency,，

  1. modern multi-core, multi-CPU devices are themselves a basic network;
  2. between CPUsis another network;
  3. between CPU and disk I/Ois yet another network.

Therefore, it is wise to use asynchronous in network programming, and now developing your own applications should also consider asynchronous. Thisfundamentally changes the design approach of Linux applications：

  • Previously, it was a sequential code flow, executing system calls only when needed,
  • Now you need to think about whether a file is ready, thus naturally introducing an event-loop, continuously submitting requests and receiving results through shared buffers.

Three working modes

An io_uring instance can work in three modes:

  1. Interrupt-driven mode（interrupt driven）

    Default mode. It can use io_uring_enter() to submit I/O requests, then directly check the CQ status to determine if they are complete.

  2. Polling mode（polled）

    Busy-waiting for an I/O completion, instead of receiving notification via asynchronous IRQ (Interrupt Request).

    This mode requires the filesystem (if any) and block device to support polling. Compared to interrupt-driven mode, this mode has lower latency (even saving system calls), but may consume more CPU resources.

    Currently, only file descriptors that have theO_DIRECTflag set can use this mode. After a read or write request is submitted to a polled context, the application must callio_uring_enter()to poll the CQ to determine if the request has completed.

    For an io_uring instance,mixing polling and non-polling modes is not supported。

  3. Kernel polling mode（kernel polled）

    In this mode, a kernel thread is createdto perform the SQ polling work.

    For an io_uring instance using this mode, the application does not need to enter kernel space to issue I/O operations. By submitting SQEs through the SQ and monitoring the completion status of the CQ, the application can submit and reap I/Os without any system calls.

    If the kernel thread’s idle time exceeds the user-configured value, it will notify the application and then enter idle state. In this case, the application must callio_uring_enter()to wake up the kernel thread. If I/O is consistently busy, the kernel thread will not sleep.

io_uring system call API

There are three:

  • io_uring_setup(2)
  • io_uring_register(2)
  • io_uring_enter(2)

Detailed introduction below. See the full documentation at manpage。

io_uring_setup()

To perform asynchronous I/O, you first need toset up the context：

This system call

  • creates an SQ and a CQ，
  • queue size at leastentrieselements,
  • returns a file descriptor, which is then used to perform operations on this io_uring instance.

The SQ and CQ are shared between the application and the kernel, avoiding data copying when initiating and completing I/O.

For parameter p:

  • For the application, it is used to configure io_uring,
  • and the SQ/CQ configuration information returned by the kernel is also brought back through it.

io_uring_setup()On success, it returns a file descriptor (fd). The application can then pass this fd to the mmap(2) system call to map the submission and completion queues, or pass it toio_uring_register()orio_uring_enter()system calls.

io_uring_register()

Register for asynchronous I/Ofiles or user buffers（files or user buffers）：

Registering files or user buffers allows the kernel tohold long-term references to the internal kernel data structures associated with the files(internal kernel data structures associated with the files), or createlong-term mappings of application memory(long term mappings of application memory associated with the buffers), this operation is performed only once during registration, not for every I/O request, thus reducing per-I/O overhead.

Nature of registered buffers

  • Registered buffers willbe locked in memory(be locked in memory), andcount against the user’s RLIMIT_MEMLOCK resource limit.
  • Additionally, each buffer has a size limit of 1GB。
  • Currently, buffers must beanonymous, non-file-backed memory(anonymous, non-file-backed memory), such as memory returned by malloc(3) or mmap(2) with the MAP_ANONYMOUS flag set.
  • Huge pages are also supported. The entire huge page will be pinned to the kernel, even if only part of it is used.
  • A registered buffer cannot be resized; to resize, you must first unregister it and then register a new one.

Througheventfd()Subscribe to completion events

can useeventfd(2)Subscribe to completion events of an io_uring instance. Just register the eventfd descriptor via this system call.

The credentials of the running application can be registered with io_uring which returns an id associated with those credentials. Applications wishing to share a ring between separate users/processes can pass in this credential id in the SQE personality field. If set, that particular SQE will be issued with these credentials.

io_uring_enter()

This system call is used to initiate and complete I/O, using shared SQ and CQ. A single call simultaneously performs:

  1. Submit new I/O requests
  2. Wait for I/O completion

Parameters:

  1. fdisio_uring_setup()The returned file descriptor;
  2. to_submitSpecifies the number of I/O submissions in the SQ;
  3. Depending on the mode:
    • Default mode, if specifiedmin_complete, it will wait for this number of I/O events to complete before returning;
    • If io_uring is in polling mode, this parameter indicates:
      • 0: Ask the kernel to return all current and completed events, non-blocking;
      • Non-zero: If there are completed events, the kernel still returns immediately; if there are no completed events, the kernel will poll, waiting for the specified number of completions, or until the process’s time slice is exhausted.

Note: For interrupt-driven I/O,the application can check CQ event completions without entering the kernel。

io_uring_enter()Supports many operations, including:

  • Open, close, and stat files
  • Read and write into multiple buffers or pre-mapped buffers
  • Socket I/O operations
  • Synchronize file state
  • Asynchronously monitor a set of file descriptors
  • Create a timeout linked to a specific operation in the ring
  • Attempt to cancel an operation that is currently in flight
  • Create I/O chains
  • Ordered execution within a chain
  • Parallel execution of multiple chains

When this system call returns, it indicates that a certain number of SQEs have been consumed and submitted, and it is safe to reuse the SQEs in the queue. At this point, the I/O submission may still be in an asynchronous context, meaning the SQE may not have actually been submitted yet — but the user does not need to worry about these details — because when the kernel later needs to use a specific SQE, it has already made a copy.

Advanced Features

io_uringProvides some advanced features for special scenarios:

  1. File registration(File Registration): Each time an operation on a specified file descriptor is initiated, the kernel needs tospend some clock cycles（cycles）to map the file descriptor to an internal representation. For thoserepeated operations on the same filescenarios,io_uringsupportsregistering these files in advance, so that later direct lookup can be done.
  2. Buffer registration(Buffer Registration): Similar to file registration, in direct I/O scenarios, the kernel needs to map/unmap memory areas.io_uringSupports registering these buffers in advance.
  3. Poll ring(Polling Ring Buffer): For very fast devices, the overhead of handling interrupts is relatively large.io_uringAllows users to disable interrupts and use polling mode. This was also mentioned in the previous ‘Three Working Modes’ section.
  4. Linked operations(Linked Operations): Allows users to send chained requests. Both requests are submitted simultaneously, but the latter will wait for the former to complete before executing.

link-cp

link-cp uses the io_uring advanced feature SQE chaining to copy files.

I/O chain

io_uring supports creating I/O chains. I/O within a chain is executed sequentially, and multiple I/O chains can be executed in parallel.

io_uring_enter()The manpage hasIOSQE_IO_LINKa detailed explanation：

When this flag is specified, it forms a link with the next SQE in the submission ring. That next SQE will not be started before this one completes. This, in effect, forms a chain of SQEs, which can be arbitrarily long. The tail of the chain is denoted by the first SQE that does not have this flag set. This flag has no effect on previous SQE submissions, nor does it impact SQEs that are outside of the chain tail. This means that multiple chains can be executing in parallel, or chains and individual SQEs. Only members inside the chain are serialized. A chain of SQEs will be broken, if any request in that chain ends in error. io_uring considers any unexpected result an error. This means that, eg, a short read will also terminate the remainder of the chain. If a chain of SQE links is broken, the remaining unstarted part of the chain will be terminated and completed with -ECANCELED as the error code. Available since 5.3.

To implement the file copy function, link-cp creates an SQE chain of length 2.

  • The first SQE is a read request that reads data from the input file into a buffer;
  • The second request, linked to the first request, is a write request that writes data from the buffer to the output file.

Three functions are implemented in the code:

  2. posix-aioThe worst performance is directly due totoo many context switches, which is also related to the scenario: in this kind of CPU-saturated situation, its thread pool becomes a burden and completely slows down performance.

  3. linux-aioandis not designed for buffered I/O, in this scenario where page cache returns directly, itsasynchronous interface actually causes performance loss — splitting the operation into dispatch and consume steps not only yields no performance benefit but also incurs extra overhead.

Performance test summary

Finally, a reminder: this section is an extreme application/scenario (100% CPU + 100% cache miss/hit) test. The behavior of real applications usually lies between synchronous and asynchronous reads: sometimes some blocking operations, sometimes some non-blocking operations. But regardless, after using io_uring, users no longer need to worry about the proportion of synchronous and asynchronous operations, because itperforms well in any scenario。

  1. If the operation is non-blocking,io_uringthere is no extra overhead;
  2. if the operation is blocking, it’s also fine,io_uringit is fully asynchronous and does not rely on thread pools or expensive context switches to achieve this asynchronous capability;

This article tests random reads, but forother types of operations，io_uringthe performance is also very good. For example:

  1. opening/closing files
  2. setting timers
  3. transferring data through network sockets

Moreover,the same set of io_uring interfaces is used。

ScyllaDB and io_uring

Scylla heavily relies on direct I/O, and has been using from the beginninglinux-aio. When we switched toio_uringduring the process, initial tests showed a performance improvement of over 50% for some workloads. Butafter in-depth research, we foundthat this is because wehad not used linux-aio well enough before. This also reveals aoften overlooked fact: achieving high performance is not that difficult (provided you get it right). After comparingio_uringandlinux-aiothe applications, wequickly improved a version, and the performance gap between the two disappeared. But frankly, solving this problemrequires some effortbecause it involves modifying an interface that has been used for many years based onlinux-aio. Forio_uringapplications, making similar changes is easy.

The above is just one scenario,io_uringIn comparison,linux-aioofadvantageis that it can be applied to scenarios beyond file I/O. Additionally, it comes with special high-performance interfaces, such as buffer registration, file registration, polling mode, etc.

Enablingio_uringAfter enabling advanced features, we indeed see performance improvements: on Intel Optane devices, reading 512 bytes with a single CPU, we observed a 5% performance improvement.
