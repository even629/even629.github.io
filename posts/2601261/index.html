This article introduces the development history of Linux asynchronous I/O, discusses the principles and functions of the high-performance asynchronous I/O framework io_uring, summarizes its core advantages such as unified asynchronous framework, true asynchronicity, and high scalability, and provides related program examples and performance benchmark results.

io_uringwas first introduced in 2019Linux 5.1by the kernel as a high-performanceasynchronous I/O framework, which can significantly accelerate the performance of I/O-intensive applications. But if your applicationis already usingtraditional Linux AIO,and using it appropriately, thenio_uringit will not bring significant performance improvement. According to the original test, even with advanced features enabled, it’s only 5%. Unless you really need that extra 5% performance, otherwiseswitchingHowever,io_uringthe cost may also be significant, because you need torewrite the applicationto adapt toio_uring(or have the dependent platform or framework adapt, in short, code changes are needed).

Since the performance is similar to traditional AIO, why is it still calledio_uringa revolutionary technology?

  1. Its first and greatest contribution is:unifying the Linux asynchronous I/O framework，
    • Linux AIO only supports direct I/Omodestorage files(storage files), and is mainly used inthe niche field of databases；
    • io_uringsupports storage files and network sockets, as well as more asynchronous system calls (accept/openat/stat/...), rather than being limited toread/writesystem calls.
  2. Indesign, it is truly asynchronous I/O, in contrast, Linux AIO, although also asynchronous, can still block, and its behavior is unpredictable in some cases;
  3. Flexibility and scalabilityare excellent, even allowing all system calls to be rewritten based onio_uring, whereas Linux AIO was not designed with scalability in mind.

eBPF is also considered an asynchronous framework (event-driven), but it has no essential connection withio_uring; they belong to different subsystems and have a fundamental difference in their models:

  1. eBPF is transparent to users, requiring only a kernel upgrade (to a suitable version),and applications need no modification；
  2. io_uringprovidesNew system calls and user-space APIs, thereforeapplications need to be modified。

This article introduces the development history of Linux asynchronous I/O,io_uringits principles and functions, and provides someprogram examplesandperformance benchmarksresults.

Evolution of Linux I/O system calls

Blocking I/O based on file descriptors:read()/write()

As the most familiar read/write method, the Linux kernel providessystem calls based on file descriptors, which may point tostorage files(storage file), or it could benetwork sockets：

the two are calledblocking system calls(blocking system calls), because when a program calls these functions, it enters a sleep state and is then scheduled out (yielding the processor) until the I/O operation completes:

  • If the data is in a file and the file contentis already cached in the page cache, the call willreturn immediately；
  • If the data is on another machine, it needs to be obtained over the network (e.g., TCP), which will block for a period of time;
  • If the data is on a hard disk, it will also block for a period of time.

But it is easy to see that as storagedevices become faster and programs become more complex, blocking, the simplest method, is no longer suitable.

Non-blocking I/O:select()/poll()/epoll()

After blocking, some new, non-blocking system calls emerged, such asselect()、poll()and newerepoll(). When the application calls these functions for reading and writing, it does not block, but insteadreturns immediately, returning alist of file descriptors that are already ready。

select()

  • Cross-platform(supported on Linux, Windows, and macOS)
  • Maximum number of fds is limited: typicallyFD_SETSIZE = 1024
  • **Uses a bitmap (fd_set)**to represent the set of fds to monitor
  • Each call requires:
    • Copying the fd_set from user space to the kernel;
    • The kernel iterates through all fds to check their status;
    • Then copy the entire fd_set back to user space upon return;
    • The user must traverse all fds to determine which are ready.
  • Only supports level-triggered (LT) mode
  • Disadvantages：
    • Low upper limit on the number of fds;
    • Each call requires full copy + full polling → O(n) time complexity;
    • Cumbersome programming (requires maintaining three fd_sets: read, write, exception)

poll()

  • Supported on Linux/Unix，Not supported on Windows
  • Usingstruct pollfdarray instead of bitmap, breaking the 1024 limit (only limited by system ulimit)
  • Still requires passing the entire array to the kernel each time, the kernel traverses and checks, and after return the user still needs to traverse to find ready fds
  • Still an O(n) polling model
  • Only supports LT mode

✅ Advantage: No 1024 limit
❌ Disadvantage: Performance still worse than epoll (under large-scale connections)

epoll()

  • Linux-specific, not cross-platform

  • Based on event-driven + kernel callback mechanism

  • Three core functions：

  • Kernel data structures：

    • Red-black tree: Stores all registered fds (add/delete/modify O(log n))
    • Ready doubly linked list: When fd is ready, the kernel automatically adds it to the linked list

  • epoll_wait()Directly returns the list of ready events, no need to traverse all fds →O(1) retrieval of ready events

  • Supports two trigger modes：

    • LT（Level Triggered）: Default mode, continuously notifies as long as the buffer has data (similar to poll)
    • ET（Edge Triggered）: Notifies only once on state change (e.g., from unreadable to readable), requires non-blocking I/O to read all data at once

✅ Advantages:

  • Efficiently handlestens of thousands or even millions of concurrent connections(e.g., Nginx, Redis, Kafka)
  • No hard limit on the number of fds
  • Minimal memory copy (only passed once during registration, only ready items returned on event)

❌ Disadvantages:

  • Linux only
  • ET mode programming is complex (data can be easily missed)

epollMonitoring a regular file fdis meaningless—it will continuously trigger events, but cannot achieve the purpose of ‘asynchronously notifying data arrival’.

But this approach has a fatal drawback:only supports network sockets and pipes— epoll() does not even support storage files.

Thread pool approach

For storage I/O, the classic solution isthread pool: the main thread distributes I/O to worker threads, which perform blocking reads and writes on behalf of the main thread, so the main thread does not block.

The problem with this approach isthe overhead of thread context switching can be very large, as will be seen in later performance stress tests.

Direct I/O (database software): bypassing the page cache

Subsequently, more flexible and powerful methods emerged:database software(database software) sometimes does not want to use the operating system’s page cache, but instead hopes that after opening a file,it reads and writes the file directly from the device(direct access to the device). This method is calleddirect access(direct access) ordirect I/O（direct I/O），

  • needs to specify O_DIRECT flag；
  • needsthe application to manage its own cache— which is exactly what database software wants;
  • iszero-copy I/O, because the application’s buffered data is sent directly to the device or read directly from the device.

Asynchronous I/O (AIO)

As mentioned earlier, as storage devices become faster, the proportion of context switching overhead between the main thread and worker threads increases. Some devices on the market now, such as Intel Optane,Latency has dropped to the same order of magnitude as a context switch(microsecondsus). To put it another way, we can better appreciate this overhead:Each time a context switch occurs, we lose one opportunity to dispatch I/O。

dispatch I/O refers tosubmitting (initiating) an I/O requestto the storage device.

In high-performance systems (such as databases, KV stores), the goal isto dispatch I/O concurrently as much as possible, keeping the device fully loaded (because devices support high queue depths, e.g., NVMe supports thousands of concurrent commands).

Therefore, the Linux2.6kernel introduced an asynchronous I/O interface, which for convenience is abbreviated in this article aslinux-aio。AIO principleis very simple:

  • The user, throughio_submit()Submit I/O requests,
  • and call laterio_getevents()to check which events are ready.
  • Enables programmersto write fully asynchronous code。

Recently, Linux AIO even supportsepoll(): meaning it can submit not only storage I/O requests but also network I/O requests. With this development, linux-aioseems poised to become a king. However, due to its poor evolution path, this wish is almost impossible to realize. We can get a glimpse fromLinus’s characteristic sharp remarks：

Reply to: to support opening files asynchronously

So I think this is ridiculously ugly.

AIO is a horrible ad-hoc design, with the main excuse being “other, less gifted people, made that design, and we are implementing it for compatibility because database people — who seldom have any shred of taste — actually use it”.

— Linus Torvalds (on lwn.net)

Linux AIO is indeed plagued with problems:

  1. Only supportsO_DIRECTfiles, thereforefor regular non-database applications （normal, non-database applications）is almost useless；
  2. The interface wasnot designed with extensibility in mind. Although it can be extended — and indeed it has been — adding each new thing is quite complex;
  3. Although from atechnical standpoint the interface is non-blocking, in practice there are many possible reasons that can cause it to block, and the ways it triggers are unpredictable.

Summary

From the above, the evolution of Linux I/O can be clearly seen:

  • It started with synchronous (blocking) system calls;
  • Then, asactual needs and specific scenariosarose, new asynchronous interfaces were continuously added, while maintaining compatibility and cooperation with older interfaces.

Additionally, it can be seen that on the issue of non-blocking reads and writes,no unified solution has been formed：

  1. In the network socket domain: add an asynchronous interface, then poll to check if the request is complete (readiness);
  2. In the storage I/O domain:Only targeting a specific niche area(Database) added a customized asynchronous interface to meet the demands of a specific period.

This is the evolution history of Linux I/O— focusing only on the present, introducing a design whenever a problem arises, without much foresight — untilio_uringemerged.

io_uring

io_uring came from the idea of veteran kernel developer Jens Axboe, who has extensive research in the Linux I/O stack field. From the earliestpatch aio: support for IO pollingit can be seen that this work began with a very simple observation: as devices become faster,the interrupt-driven mode has become less efficient than the polling mode （polling for completions) — this is also one of the most common themes in the high-performance domain.

  • io_uring’sbasic logic is similar to linux-aio: providing two interfaces, one to submit I/O requests to the kernel, and one to receive completion events from the kernel.
  • But as development deepened, it gradually became a completely different interface: the designers began to think from the sourceHow to support fully asynchronous operations。

Differences from Linux AIO

io_uringis fundamentally different fromlinux-aio:

  1. It is truly asynchronous by design(truly asynchronous). As long as the appropriate flag is set, itonly places the request into a queue within the system call context, and does nothing else extra,ensuring the application never blocks。

  2. Supports any type of I/O: cached files, direct-access files, and even blocking sockets.

Due to its async-by-design nature,there is no need for poll+read/write to handle sockets. Simply submit a blocking read, and once the request completes, it will appear in the completion ring.

  1. Flexible and scalable: Based onio_uringcan even re-implement every system call of Linux.

Principle and Core Data Structures: SQ/CQ/SQE/CQE

Each io_uring instance hastwo ring queues(rings), shared between the kernel and the application:

  • Submission Queue：submission queue (SQ)
  • Completion Queue：completion queue (CQ)

These two queues:

  • are bothsingle-producer, single-consumer, with size being a power of 2;
  • providelock-less access interface(lock-less access interface), internally usingMemory barrierSynchronization (coordinated with memory barriers).

Usage：

  • Submission
    • The application creates SQ entries (SQE) and updates the SQ tail;
    • The kernel consumes SQEs and updates the SQ head.
  • Completion
    • The kernel creates CQ entries (CQE) for one or more completed requests and updates the CQ tail;
    • The application consumes CQEs and updates the CQ head.
    • Completion events may arrive in any order, but are always associated with a specific SQE.
    • Consuming CQEs does not require switching to kernel mode.

Benefits

io_uringAnother benefit of this submission method is that what previously required multiple system calls (read or write) is now submitted as a batch in one go.io_uringThis brings batch processing capability toareas beyond storage I/O system callsSome other system calls, including:

  • read
  • write
  • send
  • recv
  • accept
  • openat
  • stat
  • some specialized system calls, such asfallocate

Furthermore,io_uringmakes the use cases for asynchronous I/O no longer limited to database applications,ordinary non-database applications can also use it. This point is worth repeating:

Althoughio_uringandaiohave some similarities, itsscalability and architecture are revolutionary: Itbrings the power of asynchronous operations to all applications(and their developers), whileis no longer limited to the niche of database applications。

Avi Kivity, at the Core C++ 2019 event,gave a talk about async.Key points include:In terms of latency,，

  1. modern multi-core, multi-CPU devices are inherently a basic network internally;
  2. between CPUsis another network;
  3. between CPU and disk I/Ois yet another network.

Therefore, it is wise to use async in network programming, and now developing your own applications should also consider async. Thisfundamentally changes the design approach of Linux applications.：

  • Previously, it was a sequential code flow, executing system calls only when needed,
  • now you need to consider whether a file is ready, thus naturally introducing an event-loop, continuously submitting requests and receiving results through shared buffers.

Three working modes

An io_uring instance can work in three modes:

  1. Interrupt-driven mode（interrupt driven）

    Default mode. I/O requests can be submitted via io_uring_enter(), and then directly check the CQ status to determine if they are complete.

  2. Polling mode（polled）

Busy-waiting for an I/O completion, rather than receiving notification via asynchronous IRQ (Interrupt Request).

This mode requires the filesystem (if any) and block device to support polling. Compared to interrupt-driven mode, this approach has lower latency (even eliminating system calls), but may consume more CPU resources.

Currently, only file descriptors opened with theO_DIRECTflag can use this mode. After a read or write request is submitted to a polled context, the application must callio_uring_enter()to poll the CQ queue to determine if the request has completed.

For an io_uring instance,mixing polling and non-polling modes is not supported。

  1. Kernel polling mode（kernel polled）

In this mode, akernel thread is created(kernel thread) to perform the polling work of the SQ.

An io_uring instance using this mode,the application does not need to switch to kernel modeto issue I/O operations. By submitting SQEs through the SQ and monitoring the completion status of the CQ, the application can submit and reap I/Os without any system calls.

If the kernel thread’s idle time exceeds the user-configured value, it notifies the application and then enters an idle state. In this case, the application must callio_uring_enter()to wake up the kernel thread. If I/O remains busy, the kernel thread will not sleep.

io_uring system call API

There are three:

  • io_uring_setup(2)
  • io_uring_register(2)
  • io_uring_enter(2)

Detailed introduction below. See the full documentation atmanpage。

io_uring_setup()

To perform asynchronous I/O, you first need toset up the context：

This system call

  • creates an SQ and a CQ，
  • queue size at leastentrieselements,
  • Returns a file descriptor, which is subsequently used to perform operations on this io_uring instance.

The SQ and CQ are shared between the application and the kernel, avoiding data copying when initiating and completing I/O.

For the parameter p:

  • For the application, it is used to configure io_uring,
  • The SQ/CQ configuration information returned by the kernel is also brought back through it.

io_uring_setup()On success, returns a file descriptor (fd). The application can then pass this fd to the mmap(2) system call to map the submission and completion queues, or pass it toio_uring_register()orio_uring_enter()system calls.

io_uring_register()

register for asynchronous I/Ofiles or user buffers（files or user buffers）：

Register files or user buffers so that the kernel canhold long-term references to the internal kernel data structures associated with the files(internal kernel data structures associated with the files), or createlong-term mappings of application memory(long-term mappings of application memory associated with the buffers). This operation is performed only once during registration, rather than for each I/O request, thus reducing per-I/O overhead.

Nature of registered buffers

  • Registered buffers willbe locked in memory(be locked in memory), andcount towards the user’s RLIMIT_MEMLOCKresource limit.
  • Additionally, each buffer has asize limit of 1GB。
  • Currently, buffers must beanonymous, non-file-backed memory(anonymous, non-file-backed memory), such as memory returned by malloc(3) or mmap(2) with the MAP_ANONYMOUS flag set.
  • Huge pages are also supported. The entire huge page will be pinned to the kernel, even if only a portion of it is used.
  • A registered buffer cannot be resized; to resize, you must first unregister it and then register a new one.

Byeventfd()subscribing to completion events

you can useeventfd(2)Subscribe to completion events of the io_uring instance. Simply register the eventfd descriptor via this system call.

The credentials of the running application can be registered with io_uring which returns an id associated with those credentials. Applications wishing to share a ring between separate users/processes can pass in this credential id in the SQE personality field. If set, that particular SQE will be issued with these credentials.

io_uring_enter()

This system call is used to initiate and complete I/O, using shared SQ and CQ. A single call simultaneously performs:

  1. Submit new I/O requests
  2. Wait for I/O completion

Parameters:

  1. fdisio_uring_setup()the returned file descriptor;
  2. to_submitspecifies the number of I/O submissions in the SQ;
  3. Depending on the mode:
    • Default mode, ifmin_completeis specified, it will wait for this number of I/O events to complete before returning;
    • If io_uring is in polling mode, this parameter indicates:
      • 0: Requests the kernel to return all current and completed events, non-blocking;
      • Non-zero: If events are completed, the kernel still returns immediately; if no events are completed, the kernel will poll, waiting for the specified number to complete, or until the process’s time slice is exhausted.

Note: For interrupt-driven I/O,Applications can check CQ event completions without entering the kernel.。

io_uring_enter()Supports many operations, including:

  • Open, close, and stat files
  • Read and write into multiple buffers or pre-mapped buffers
  • Socket I/O operations
  • Synchronize file state
  • Asynchronously monitor a set of file descriptors
  • Create a timeout linked to a specific operation in the ring
  • Attempt to cancel an operation that is currently in flight
  • Create I/O chains
  • Ordered execution within a chain
  • Parallel execution of multiple chains

When this system call returns, it indicates that a certain number of SEQs have been consumed and submitted, and the SEQs in the queue can be safely reused. At this point, the IO submission may still be in an asynchronous context, meaning the SQEs may not have been actually submitted yet — however, users do not need to worry about these details — when the kernel later needs to use a specific SQE, it has already made a copy.

Advanced Features

io_uringProvides some advanced features for special scenarios:

  1. File registration(File Registration): Each time an operation on a specified file descriptor is initiated, the kernel needs tospend some clock cycles（cycles）to map the file descriptor to an internal representation. For thosescenarios involving repeated operations on the same file,io_uringsupportsregistering these files in advance, and then directly look them up later.
  2. Buffer registration(Buffer Registration): Similar to file registration, in direct I/O scenarios, the kernel needs to map/unmap memory areas.io_uringSupports pre-registering these buffers.
  3. Poll ring(Polling ring buffer): For very fast devices, the overhead of handling interrupts is relatively high.io_uringAllows users to disable interrupts and use polling mode. This was also mentioned in the previous ‘Three Working Modes’ section.
  4. Linked operations(Linked operations): Allows users to send chained requests. Both requests are submitted together, but the latter waits for the former to complete before executing.

link-cp

link-cp uses the io_uring advanced feature SQE chaining to copy files.

I/O chain

io_uring supports creating I/O chains. I/O within a chain is executed sequentially, while multiple I/O chains can run in parallel.

io_uring_enter()In the manpage forIOSQE_IO_LINKhavedetailed explanation：

When this flag is specified, it forms a link with the next SQE in the submission ring. That next SQE will not be started before this one completes. This, in effect, forms a chain of SQEs, which can be arbitrarily long. The tail of the chain is denoted by the first SQE that does not have this flag set. This flag has no effect on previous SQE submissions, nor does it impact SQEs that are outside of the chain tail. This means that multiple chains can be executing in parallel, or chains and individual SQEs. Only members inside the chain are serialized. A chain of SQEs will be broken, if any request in that chain ends in error. io_uring considers any unexpected result an error. This means that, eg, a short read will also terminate the remainder of the chain. If a chain of SQE links is broken, the remaining unstarted part of the chain will be terminated and completed with -ECANCELED as the error code. Available since 5.3.

To implement the file copy functionality, link-cp creates an SQE chain of length 2.

  • The first SQE is a read request, reading data from the input file into a buffer;
  • The second request, linked to the first, is a write request, writing data from the buffer to the output file.

Additional Notes

Three functions are implemented in the code:

  1. copy_file(): High-level copy loop logic; it callsqueue_rw_pair(ring, this_size, offset)to construct SQE pairs; and through a singleio_uring_submit()call, submits all constructed SQE pairs.

This function maintains a maximum number of DQ inflight SQEs as long as data copying is still in progress; otherwise, once all data has been read, it starts waiting for and harvesting all CQEs.

  1. queue_rw_pair()Construct a read-write SQE pair.

The read SQE’sIOSQE_IO_LINKThe flag indicates the start of a chain; the write SQE does not need to set this flag, marking the end of the chain. The user data field is set to the same data descriptor, which will be used in subsequent completion processing.

  1. handle_cqe()Extract from the CQE the previously saved data descriptor byqueue_rw_pair()the saved data descriptor, and record the processing progress (index) in the descriptor.

If the previous request was canceled, it will also resubmit the read-write pair.

After both members of a CQE pair have been processed (index==2), release the shared data descriptor. Finally, notify the kernel that this CQE has been consumed.

  1. posix-aioThe worst performance is directly due totoo many context switches, which is also related to the scenario: in thisCPU-saturated situation, its thread pool becomes a burden, completely slowing down performance.

  2. linux-aioandis not designed for buffered I/O, in this scenario where the page cache returns directly, itsasynchronous interface actually causes performance loss— splitting operations into dispatch and consume steps not only yields no performance benefit but also incurs additional overhead.

Performance Test Summary

Finally, a reminder: this section is for extreme applications/scenarios (100% CPU + 100% cache miss/hit) testing. Real application behavior typically falls between synchronous and asynchronous reads: sometimes blocking operations, sometimes non-blocking operations. Regardless, after using io_uring, users no longer need to worry about the proportion of synchronous and asynchronous operations, because itperforms well in any scenario。

  1. If the operation is non-blocking,io_uringthere is no additional overhead;
  2. if the operation is blocking, that’s fine too,io_uringit is fully asynchronous and does not rely on thread pools or expensive context switches to achieve this asynchronous capability;

This article tests random reads, but forother types of operations，io_uringthe performance is also very good. For example:

  1. opening/closing files
  2. Set a timer
  3. Transfer data through network sockets
