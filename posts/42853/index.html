This article introduces the complete process of setting up a UEFI development environment based on edk2 on Linux, including basic environment configuration, source code download, and compilation tool installation. Through the HelloWorld and HelloStd examples, it explains in detail the writing of DSC and INF configuration files, the implementation of compilation scripts for x64 and aarch64 platforms, and demonstrates how to run programs using Emulator and QEMU and debug with GDB.

HelloWorld.inf

The INF file is the configuration file for the edk2 app, where

  • [Defines] This section defines some basic information of the module.
    • BASE_NAME: the name of the app
    • FILE_GUID can be generated using the uuidgen command; UEFI distinguishes different modules by GUID.
    • MODULE_TYPE: Fill in UEFI here_APPLICATION
    • ENTRY_POINT: The name of the main function in C code
  • [Sources]: The source code of the module, typically .c and .h files
  • [Packages]: Packages that need to be used
  • [LibraryClasses]: Libraries that need to be used

For a complete explanation of the INF file, refer to the following link:

INF-Specification

https://tianocore-docs.github.io/edk2-InfSpecification/release-1.27/

qemu run

First, compile and install qemu. Here I choose version 8.1.5. If yours does not achieve the expected results, consider using this version of qemu.

Next, write a script to run with qemu. Some parameters here are for debugging with gdb in the next section, but it doesn’t matter if you just want to run it with qemu.

debug-x64.sh

This script first compiles OVMF (Open Virtual Machine Firmware), which is an EDKII-based firmware that can run under the qemu x86-64 virtual machine. This makes debugging and experimenting with UEFI firmware easier, whether for testing OS booting or using the (built-in) EFI shell.

The OVMF firmware (UEFI implementation for QEMU) is split into two files:

  • OVMF_CODE.fd: contains the actual UEFI firmware.
  • OVMF_VARS.fd: serves as a ‘template’ for emulating persistent NVRAM storage.
    All virtual machine instances can share the system-wide read-only OVMF from the ovmf package._CODE.fd file, but each instance requires a private, writable OVMF_VARS.fd copy.
    In qemu, you can specify OVMF separately_CODE.fd and OVMF_VARS.fd, or use a simplified notation:

Run debug-x64.sh, and if all goes well, the following interface will appear, which is the UEFI Shell

In this shell, type fs0: (note the English colon), then type HelloWorld.efi to run our program, expecting the output “Hello World!!!”

In the Shell, if BackSpace does not respond, you can press Ctrl+H instead

To exit qemu, pressCTRL+A - X

Debugging

Debugging UEFI programs with gdb is a bit troublesome, but you can automate some operations with scripts. The overall process is as follows:

  1. Run debug.sh, then enter the UEFI Shell and run the following code (same operation as running with qemu in the previous section, mainly to obtain the driver’s startup address in the _ovmf_dbg/debug.log)
  2. Open another terminal and run the following script addr.sh
  3. In the _ovmf_dbg/UEFI_Disk directory, run gdb -x gdb_commands.txt
  4. Set breakpoints in gdb, for example break UefiMain
  5. Add gdb debug target remote localhost:1234
  6. Run, enter c to jump to the first breakpoint
  7. Run your code in the UEFI Shell
