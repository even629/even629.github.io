    groups and function

    In the pinctrl subsystem, there are two key concepts:Pin group (groups) and Function (function)

    • Pin group (Groups): A pin group is a collection of pins with similar functions, constraints, or that work together. Each pin group is typically associated with a specific hardware function or peripheral. For example, one pin group can be used to control serial communication interfaces (such as UART or SPI), and another pin group can be used to drive GPIO.
    • Function (Function): Defines the functions with peripheral capabilities on the chip. Each function node corresponds to the configuration information of one or more IO groups. These functions can be peripheral functions such as UART, SPI, I2C, etc.

    struct pinctrl_desc

    • const char *name: Pin controller name, used to identify the pin controller uniquely.

    • const struct pinctrl_pin_desc *pins: Pin descriptor array, which is a pointer to a pin descriptor, used to describe the pin’s attributes and configuration. Each pin descriptor contains information such as the pin’s name, number, and mode.

    • unsigned int npins: indicatesthe number of elements in the pin descriptor array, used to determine the length of the pin descriptor array.

    • const struct pinctrl_ops *pctlops: A pointer to the pin control operation functions, used to define the pin controller’s operation interface. Through these operation functions, you canconfigure, enable, and disable pinsand other operations.

    • const struct pinmux_ops *pmxops: A pointer to the pin multiplexing operation functions, used to define the pin’s multiplexing functions. The multiplexing function allowsswitching the function of the pin to different modes, to accommodate different device requirements

    • const struct pinconf_ops *confops: A pointer to the pin configuration operation function, used to define other configuration options for the pin. These configuration options can includepin pull-up, pull-down configurations, electrical characteristics, etc.。

    • struct module *owner: A pointer to the module that owns the pin controller structure. This field is used to track the owner of the pin controller structure.

    • unsigned int num_custom_params: Represents the number of custom configuration parameters, used to describe the custom configuration parameters of the pin controller.

    • const struct pinconf_generic_params *custom_params: A pointer to the custom configuration parameters, used to describe the attributes of the custom configuration parameters of the pin controller. Custom configuration parameters can be defined according to specific needs, used to expand the configuration options of the pin controller.

    • const struct pin_config_item *custom_conf_items: A pointer to the custom configuration items, used to describe the attributes of the custom configuration items of the pin controller. Custom configuration items can be defined according to specific needs, used to expand the configuration options of the pin controller.

    struct rockchip_pinctrl

    Rockchip, in order to adapt to the specific needs and functions of Rockchip chips, madestruct pinctrl_descRe-encapsulated. The re-encapsulatedstruct rockchip_pinctrlstruct is based onstruct pinctrl_descadds fields and pointers related to Rockchip chips. This encapsulation provides better integration, ease of use, and scalability, while maintaining compatibility with the generic pin controller framework.

    • struct regmap *regmap_base: Pointer to the basic register map (regmap). The basic register map is an interface used to access chip registers, providing read and write operations for chip registers.

    • int reg_size: Represents the byte size of the register, used to determine the address range of the register.

    • struct regmap *regmap_pull: Pointer to the pull register map. The pull register map is used to control the pull-up and pull-down functions on the pins.

    • struct regmap *regmap_pmu: Pointer to the Power Management Unit (PMU) register map. The PMU register map is used to control the power management functions of the pins

    • struct device *dev: Pointer to the device structure. The device structure is used to represent hardware-related devices, including the device’s physical address, interrupts, and other information

    • struct rockchip_pin_ctrl *ctrl: Pointer to the Rockchip chip pin controller. This structure stores the relevant information and operations of the Rockchip chip-specific pin controller.

    • struct pinctrl_desc pctl: Contains an instance of the struct pinctrl_desc structure. Used to describe the properties and operations of the pin controller, including the pin controller’s name, pin descriptor array, function pointers, etc.

    • struct pinctrl_dev *pctl_dev: Pointer to the pin controller device structure. The pin controller device structure is used to represent the device instance of the pin controller in the system, containing device information and operation interfaces related to the pin controller.

    • struct rockchip_pin_group *groups: Pointer to the Rockchip chip pin group. A pin group is a set of related pins that can be configured and managed together.

    • unsigned int ngroups: Represents the size of the pin group array, used to determine the length of the pin group array.

    • struct rockchip_pmx_func *functions: Pointer to the Rockchip chip pin function. Pin functions define the different functions that a pin can assume, such as UART, SPI, I2C, etc.

    • unsigned int nfunctions: The number of pin functions. It represents the size of the pin function array, used to determine
      the length of the pin function array.

    According to the device treecompatible = "rockchip,rk3568-pinctrl"Matched the pin description data for RK3568, and dynamically calculated the actual offset addresses of the multiplexing (iomux) and drive strength (drv) registers in each GPIO bank, preparing for subsequent pin configuration.

    syscon

    syscon= abbreviation for “system controller”.

    It is essentially a:

    • lightweight platform driver
    • Used to encapsulate a section ofmemory-mapped control register region(such as GRF, PMU, CRU, etc.)
    • into a struct regmap object
    • for other drivers to safely read and write these registers through standard interfaces (such asregmap_read/write)

    Compared to direct ioremap, why is it necessary tosyscon

    methodProblemsyscon advantage
    Each driver itselfioremapmaps the same physical address multiple times, wasting resourcesGlobally unique mapping
    No synchronization mechanismMultiple drivers writing to registers simultaneously → race conditionregmapBuilt-in lock
    Code duplicationEvery driver has to handle offsets and bit operationsUnified abstract interface
    Strong device tree couplingDriver needs to know the specific addressDecoupled via phandle

    struct regmap

    Common system controllers in Rockchip SoCs include:

    ControllerFull nameFunction
    GRFGeneral Register FilesControls GPIO multiplexing, voltage domains, peripheral routing, etc.
    PMUPower Management UnitLow-power pin control, power management

    These registers typically:

    • by shared by multiple subsystems(e.g., pinctrl, USB, I2C all need to configure GRF)
    • cannot be exclusively occupied by a single driver
    • requires atomic operations, bit operations, lock protection

    sysconexactly solves these problems.

    syscon device tree

    • GRF：General Register Files —— main system controller,core dependency of pinctrl！
    • The multiplexing, enabling, and voltage domains of almost all peripherals are configured here.
    • Sub-functions:
      • io_domains: Main IO voltage domain control (e.g., levels of HDMI, MIPI, SDIO)
      • lvds0 / lvds1: LVDS display output controller (connecting the screen)
      • rgb: RGB parallel display interface (old-style LCD)

    rockchip_dt_node_to_map()

    rockchip_dt_node_to_mapThe function creates pin maps based on the device node’s information, including mux maps and config maps.

    • Mux maps are used to associate the functions of the pin group with the functions of the parent node
    • Config maps are used to associate the config info of the pin with the name of the pin

    These maps will be used to configure the pin controller to ensure that pins are correctly configured and used in the system. This function is called during device tree parsing to create corresponding pin maps for each device node.

    pinconf_ops

    struct pinconf_opsis the core operation interface used for pin electrical characteristic configuration (pin configuration) in the pinctrl subsystem.

    Ifpinmux_opsis responsible for ‘which functional module the pin is connected to’ (such as I2C, UART), thenpinconf_opsis responsible for ‘how the electrical behavior of this pin is configured’ (such as pull-up resistors, drive strength, Schmitt trigger, etc.).

    Structstruct pinconf_ops, used todefine pin configuration operationsfunction pointers. Each function pointer corresponds to a specific operation, such as getting pin configuration, setting pin configuration, getting pin group configuration, etc. These functions are implemented in the driver to configure and control hardware pins.

    pinmux_ops

    struct pinmux_opsis a structure describing pin multiplexing operations, a set of callback functions that the pinctrl driver must implement (if pin multiplexing is supported), allowing the pinctrl subsystem to:

    • Query which functions the SoC supports (e.g., I2C, UART, SPI…)
    • Query which pin groups can be used for each function
    • Dynamically switch pin functions at runtime (e.g., switching GPIO1_B0 from a regular GPIO to I2C_SCL）
