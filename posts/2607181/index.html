Compilation

Key points:

  1. ARCH=arm64 for Kbuild, TARGET_ARCH=aarch64 for NVIDIA’s utils.mk (the two cannot be mixed).
  2. The toolchain is doubly secured via PATH + explicit CC/CXX/LD/AR, because utils.mk does not read CROSS_COMPILE.
  3. SYSSRC/SYSOUT point to linux-5.10.238 configured with the same toolchain.

To clean up:

Description

  1. Native compilation does not require CROSS_COMPILE, the host gcc is sufficient for tools. Still need to explicitly pass CC/CXX/LD/AR, because utils.mk does not read CROSS_COMPILE。
  2. The difference from arm64 is only in ARCH/TARGET_ARCH: for arm64 it is ARCH=arm64 + TARGET_ARCH=aarch64；x86_64 both are x86_64. Note that there is no TARGET_ARCH called arm64 in utils.mk, only aarch64.

Directory structure

1. Top-level directory structure

Core mechanism: Two-layer build model

The key to understanding this repository lies in its two-layer build model. This repository is not a single Linux kernel module, but rather an assembly of the NVIDIA OS-independent core and the Linux kernel adaptation layer, ultimately generating 5 .ko modules.

  • First layer output: Generated insrc/nvidia/_out/Linux_<ARCH>/nv-kernel.o. Code uses-ffreestandingand-fno-stack-protectorcompiled, allowing the core logic to be reused across different operating systems (such as Linux, FreeBSD, etc.).

  • Second layer: Links the .o files generated by the first layer as “black box binaries” into the final .ko module, wrapping them with Linux-specific hook code.

2. src: OS-independent core

(approx. 2.11 million lines)

3.src/nvidia/src/kernel/gpu/Subdirectory (GPU IP engine)

This is the most core hardware control part of the driver, divided by hardware engine:

  • gr/: Graphics (rasterization/shading/rendering core)
  • fifo/: Command submission queue management
  • ce/: Copy Engine (high-performance memory copy engine)
  • intr/: Interrupt handling
  • sec2/: Security engine
  • falcon/: Falcon microcontroller core framework
  • gsp/: ★ GSP (GPU System Processor) core
  • fsp/：Firmware Security Processor
  • gsplite/: Lightweight GSP logic
  • pmu/: Performance monitoring unit
  • nvdec/ nvenc/ nvjpg/ ofa/: Video codec and optical flow acceleration engine
  • bif/ bus/: PCIe bus interface and bus control
  • mem_mgr/ mem_sys/ mmu/ mig_mgr/: Video memory management, MMU page tables, MIG instance partitioning
  • disp/: Display output control
  • perf/ hwpm/ rc/ timer/ …: performance, hardware monitoring, reset recovery, and timer

Architectural Evolution Background: The core of the RM (Resource Manager) of modern NVIDIA GPUs mainly runs on the GSP (a RISC-V microprocessor) inside the GPU. A large part of the code in src/nvidia is actually the source code of the GSP firmware itself—this is also the key underlying evolution of NVIDIA’s gradual move towards open source after 2020.

4. kernel-open/ Linux Adaptation Layer

(approx. 220,000 lines)

conftest.shMechanism: This is the core secret of how NVIDIA drivers can “use one source code to be compatible with countless kernel versions”. Before compilation, it performs a large number of fragment compilation tests on the target kernel (e.g., detecting whether kmem_cache_create_usercopy、pgprot_t, etc.). The test results are written into conftest/headers.h, and then in the source code, through#if NV_IS_ENABLED(...)macros to conditionally compile the corresponding compatibility code.
