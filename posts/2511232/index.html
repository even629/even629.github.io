This article introduces the core mechanisms and implementation flow of QEMU hardware modeling, detailing the six stages of virtual machine creation, the peripheral modeling process using PL011 UART and edu device as examples (covering the QOM type system, MMIO callbacks, DMA transfers, and asynchronous computation), and provides an in-depth analysis of the complete interrupt routing and handling flow from device triggering to Guest OS response.

Machine Modeling

The core process of QEMU creating a virtual machine can be divided into 6 stages, each involving specific components and operations

Command Line Parsing

Entry function: main() Key operations:

  • Parse the -M parameter to specify the machine type (e.g., virt)
  • Initialize the QOM (QEMU Object Model) type system
  • Create the global state machine MachineState

system/main.c

system/vl.c

Machine type selection and registration

Core component: MachineClass (machine class description)

Key operations:

  • Match user-specified machine type via select_machine()
  • Look up the corresponding TypeImpl from the global type table type_table

Machine object instantiation

Core component: MachineState (machine runtime state)

Key operations:

  • Via QOM’s object_new_with_class() to instantiate the machine
  • Initialize basic properties such as memory and CPU topology

Hardware device initialization

Core phase: machine_run_board_init()

Key operations:

  • Call the machine-specific init() method (e.g., RISC-V virt’s virt_machine_init()）
  • Initialize core components such as CPU, memory controller, and bus

Accelerator and CPU initialization

Core components: AccelState and CPUState

Key operations:

  • Select KVM/TCG accelerator based on -accel parameter
  • Create vCPU threads and bind them to physical CPUs

Start virtual machine

Final stage: qemu_main_loop()

system/runstate.c

Peripheral modeling (PL011 UART)

The modeling of the PL011 UART in QEMU fully demonstrates the core process of hardware peripheral emulation:

  • Type system integration: Implement device class inheritance hierarchy via QOM
  • Precise state modeling: Full-cycle management of registers, FIFO, and interrupt states
  • Hardware interface implementation: MMIO callbacks handle CPU access, character backend interfaces with host I/O
  • Interrupt system integration: Works in coordination with the platform interrupt controller (PLIC)
  • Device lifecycle: implements complete lifecycle including reset, migration, and destruction

Device type definition

Core mechanism: QEMU Object Model (QOM)

Key operations:

  • Define TypeInfo structure to describe device type
  • Implement class initialization (class_init) and instance initialization (instance_init)
  • Register into the global type system

Device state structure design

Core component: PL011State

Key fields:

  • Register state (lcr, imsc, etc.)
  • FIFO buffer
  • Interrupt signal line
  • Character backend (CharBackend)

Device class initialization

Core tasks:

  • Set device properties

  • Bind realize and reset methods

  • Define device migration support

Device instantiation and hardware connection

Core operations:

  • Initialize Memory Region
  • Register MMIO Callback Functions
  • Connect Interrupt Signal
  • Bind Character Device Backend

Implement Device Function Logic

Core Components:

  • MMIO Read/Write Callbacks

Key implementation points

Precise register modeling:

  • Implement precise behavior of all 32 registers
  • Handle special bits (e.g., FIFO enable bit, interrupt mask bit)
  • Support baud rate calculation (IBRD/FBRD)

FIFO state machine:

Clock domain handling:

  • Implement pl011_clock_Update callback
  • Handle baud rate change events
  • Support dynamic clock adjustment

Migration support:

Performance Optimization:

  • Use bit operations instead of division to calculate baud rate
  • Use masks instead of modulo operations for FIFO operations
  • Cache interrupt status to reduce computation

This modeling approach allows PL011 to accurately simulate real hardware behavior in QEMU, supporting full-stack development from bare-metal programs to the Linux kernel.

Peripheral Modeling (edu Device)

In a computer system, the interaction between the CPU and peripherals essentially involves address access and interrupt notification:

  • Address Access: The CPU reads and writes device registers via MMIO (Memory-Mapped I/O) or PIO (Port I/O)
  • Interrupt Notification: The peripheral sends event signals to the CPU via the interrupt controller

Introduction to the edu Device

edu is a teaching virtual device provided by QEMU, with a streamlined yet complete functional design:

  • Basic Function: Implements factorial calculation (fact register)
  • Advanced features: DMA transfer, interrupt triggering
  • Typical application: Demonstrating full lifecycle management of PCI devices

Device interaction from the guest perspective

When the guest program’s driver accesses the edu device:

  1. Probe device: identified by PCI ID 0x1234:11e8
  2. Map MMIO: ioremap() to obtain register virtual addresses
  3. Calculate factorial
  4. Handle interrupt: clear status flag in the interrupt service routine

Key points summary

edu encapsulates the 5 core elements of peripheral modeling:

Mainly involves:

  • State machine model: represents device state via register bits (e.g., computing/completed)
  • Asynchronous processing: offload time-consuming operations to background threads to avoid blocking VCPU
  • Address isolation: each device has its own independent MMIO space
  • Interrupt abstraction: supports both traditional INTx and modern MSI modes
  • DMA security: address verification to prevent virtual machine escape

Interrupt modeling

  • Device layer: PL011 triggers interrupt signal based on hardware status
  • Interrupt controller layer: PLIC implements priority arbitration and interrupt distribution
  • CPU layer: RISC-V architecture responds to external interrupts and switches context
  • Guest layer: Guest program or OS interrupt handler completes device service
