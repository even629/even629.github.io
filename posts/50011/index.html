    This article introduces the basic concepts, core features, and cluster architecture components of Kubernetes, summarizing the functions of Master and Node nodes and core concepts such as Pod, Controller, and Service. In addition, this article also discusses methods for building clusters using kubeadm and binary packages, as well as the kubectl command-line tool, YAML file configuration specifications, and the implementation mechanism and resource management strategies of Pods.

    Kubernetes Overview and Features

    k8s Overview

    • k8s is an open-source containerized cluster management system released by Google in 2014
    • Using k8s for containerized application deployment
    • Using k8s facilitates application scaling
    • The goal of k8s is to make deploying containerized applications simpler and more efficient

    k8s Features

    1. Automatic bin packing
    2. Automatic repair (self-healing capability)
    3. Horizontal scaling
    4. Service discovery
    5. Rolling updates
    6. Version rollback
    7. Secret and configuration management (similar to hot deployment)
    8. Storage orchestration
    9. Batch processing

    k8s Cluster Architecture Components

    Master (control node) and Node (worker node)

    (1) master components

    • apiserver

    Unified entry point of the cluster, stored in etcd in a RESTful manner

    • scheduler

    Node scheduling, selecting nodes for application deployment

    • controller-manager

    Handles routine background tasks in the cluster; one resource corresponds to one controller

    • etcd

    Storage system, used to save cluster-related data

    (2) node components

    • kubelet

    Agent dispatched by the master to the node, managing local containers

    • kube-proxy

    Provides network proxy, implementing load balancing and other operations

    Kubernetes core concepts

    1. Pod
    • Smallest deployment unit
    • A collection of containers
    • Shared network
    • Lifecycle is ephemeral
    1. Controller
    • Ensures the expected number of pod replicas
    • Stateless application deployment
    • Stateful application deployment
    • Ensures all nodes run the same pod
    • One-time tasks and scheduled tasks
    1. Service

    • Defines access rules for a group of pods

    • Master (Master Node): The machine that controls Kubernetes nodes, and also the place where job tasks are created.

    • Node: These machines execute assigned tasks under the control of the Kubernetes master node.

    • Pod： A collection consisting of one or more containers, deployed as a whole to a single node. Containers in the same pod share IP addresses, inter-process communication (IPC), hostnames, and other resources. Pod abstracts the network and storage of the underlying containers, making container migration within the cluster more convenient.

    • Replication Controller: Controls the number of instances of a pod running on the cluster.

    • Service: Decouples service content from specific pods. The Kubernetes service proxy is responsible for automatically distributing service requests to the correct pods, regardless of where the pods move in the cluster, or even if they are replaced.

    • Kubelet： This daemon runs on each worker node, responsible for getting the list of containers and ensuring that the declared containers have started and are running normally.

    • kubectl： This is the command-line configuration tool for Kubernetes.

    Kubernetes core technology

    Pod

    Pod overview

    1. Pod is the smallest unit that can be created (deployed) and managed in the k8s system

    2. k8s does not directly handle containers, but Pods. A Pod can contain multiple containers (a collection of containers)

    3. Share network namespace in a pod

    4. Each Pod has a special Pause container known as the “root container”. The image corresponding to the Pause container is part of the k8s platform. In addition to the Pause container, each Pod contains one or more closely related user business containers.

    5. Pods are ephemeral

    The purpose of Pod’s existence

    1. Containers are created using Docker; one Docker corresponds to one container, one container has a process, and one container runs one application
    2. Pod is designed for multi-process, capable of running multiple applications; one Pod has multiple containers, and one container runs one application
    3. Pod exists for close interaction:
      • Interaction between two applications
      • Calls between networks
      • Two applications need to call each other frequently

    Pod implementation mechanism

    1. Shared network

    Containers are inherently isolated from each other. Kubernetes uses the Pod’s Pause container (infra container) to add other business containers into the Pause container, allowing all business containers to be in the same namespace, which enables network sharing

    1. Shared storage

    Pod persistent data: log data, business data

    Use Volume data volumes for persistent storage

    Pod image pull policy

    imagePullPolicy

    IfNotPresent: Default value, pull the image only when it does not exist on the host

    Always: Pull the image every time a Pod is created

    Never: Pod will never actively pull this image; it needs to be pulled manually

    Pod resource limits

    1c = 1000m (1 core CPU)

    Pod restart mechanism

    restartPolicy

    Always: Always restart the container when it terminates, default policy

    OnFailure: Restart the container only when it exits abnormally (exit code is not 0)

    Never: Never restart the container when it terminates

    Pod health check

    Container check:

    Cannot detect Java heap memory overflow (status is still running)

    Application-level health check:

    echo $? indicates whether the previous command in Linux executed successfully

    Pod creation process

    • Master node

    createpod – apiserver – etcd

    scheduler – apiserver – etcd – scheduling algorithm, schedules the pod to a certain node

    • Node

    kubelet – apiserver – reads etcd to get the pod assigned to the current node – docker creates the container

    Attributes affecting scheduling

    1. Pod resource limits: resources

    2. Node selector labels affect Pod scheduling

    Need to label the node first

    1. Node affinity affects Pod scheduling

    Node affinity nodeAffinity is basically the same as the previous nodeSelector, it decides which nodes the Pod is scheduled to based on the label constraints on the nodes

    (1) Hard affinity (requireDuringSchedulingIgnoreDuringExecution)

    Constraints must be satisfied

    (2) Soft affinity (preferredDuringSchedulingIgnoredDuringExecution)

    Try to satisfy, not guaranteed

    Common operators:

    In NotIn Exists Gt Lt DoesNotExists

    Anti-affinity: Use NotIn and DoesNotExists

    • Basic Introduction:

    nodeSelector and nodeAffinity: Schedule Pods to certain nodes, Pod attributes, implemented during scheduling

    Taint: Nodes are not scheduled for ordinary allocation, it is a node attribute

    Eviction based on Taint

    There are three taint values:

    NoSchedule: Definitely not scheduled

    PreferNoSchedule: Try not to be scheduled

    NoExecute: Will not be scheduled, and will also evict existing Pods on the Node

    Differences between stateless and stateful

    1. Stateless
      • Considering Pods to be identical (replicas are identical)
      • No ordering requirements
      • No need to consider which node to run on
      • Freely scale and expand
    2. Stateful
      • All the above factors need to be considered
      • Make each pod independent, maintain pod startup order and uniqueness (unique network identifier, persistent storage, ordered, such as MySQL master-slave)

    k8s cluster installation mechanism

    Overview

    1. Accessing the k8s cluster requires three steps to complete specific operations

      (1) Authentication

      (2) Authorization

      (3) Admission control

    2. When accessing, the process needs to go through apiserver, apiserver does unified coordination, like a guard, the access process requires certificate, token, or username+password, if accessing pod requires serviceAccount

    Step 1 Authentication

    • Transmission security: Do not expose port 8080 to the outside, only internal access, use port 6443 externally
    • Authentication: Common methods of client identity authentication:
      • https certificate authentication, based on ca certificate
      • http token authentication, identify users through token
      • http basic authentication, username+password authentication

    Step 2 Authorization

    • Perform authorization operations based on RBAC
    • Role-based access control

    Step 3 Admission control

    • It is the list of admission controllers, if the list has content, it passes

    RBAC Role-based access control

    Role Based Access Control

    1. Role

    • role: Role, access permissions for a specific namespace

    • clusterRole: Access permissions for all namespaces

    1. Role binding
    • roleBinding: Binds a role to a subject
    • ClusterRoleBinding: Binds a cluster role to a subject
    1. Subject
    • user: User
    • group: User group
    • serviceAccount: Service account

    RBAC implements authorization

    1. Use certificate

    Ingress

    Overview

    1. Expose the port externally, access via IP+port, implemented using NodePort in Service
    2. NodePort drawbacks
    • Ports are opened on every node, access is achieved through any node via node IP + exposed port
    • means that each port can only be used once, with one port corresponding to one application
    • In actual access, domain names are used, and requests are routed to different port services based on different domain names

    Relationship between Ingress and Pod

    • pod and ingress are associated through service
    • ingress serves as a unified entry point, associated with a group of pods through service
