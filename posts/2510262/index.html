    This article introduces the weakly-ordered memory model adopted by the ARMv8 architecture to improve performance, and discusses in detail the memory barrier mechanisms required to ensure memory access ordering in scenarios such as multi-core shared data. The article provides an in-depth analysis of the working principles, core differences, and parameter configurations of the Data Memory Barrier (DMB), Data Synchronization Barrier (DSB), and Instruction Synchronization Barrier (ISB).

    Causes of Memory Barriers

    The ARMv8 architecture adopts a weakly-ordered memory model. Generally, this means that the order of memory accesses does not need to be the same as the program order of load and store operations. The processor is able to reorder memory read operations relative to each other. Writes can also be reordered (e.g., write combining). Therefore, hardware optimizations (such as the use of caches and write buffers) act in a way that improves processor performance, meaning the required bandwidth can reduce the latency between the processor and external memory, and hide the long latencies associated with such external memory accesses.

    Reads and writes to normal memory can be reordered by hardware, subject only to data dependencies and explicit memory barrier instructions. Certain situations require stricter ordering rules.

    • The processor adopts superscalar technology: out-of-order issue, out-of-order execution, improving instruction parallelism

    The order of these two statements is indeterminate because there is no dependency.

    Weak consistency memory model

    • In 1986, Dubois et al. published a paper describing the definition of the weak consistency memory model
      • The weak consistency memory model requires that synchronization accesses (accessing global synchronization variables) be sequentially consistent; before a synchronization access can be executed, all previous data accesses must be completed
      • Before a normal data access can be executed, all previous synchronization accesses must be completed
      • Processors use memory barrier instructions to implement the functionality of the entire synchronization access
    • The basic principles of memory barrier instructions are as follows:
      • All data accesses before the memory barrier instruction must be completed before the memory barrier instruction。
      • All data accesses after the memory barrier instruction must wait for the memory barrier instruction to complete。
      • Multiple memory barrier instructions are executed in order
    • Memory barrier instructions can be parameterized by scope (shareability domain)

    • The sequence of memory accesses may differ from the sequence in the code

    • ARMv8 architecture supports speculative accesses

      • Prefetching data or instructions from memory

      • Speculative cache line fills

    Under what circumstances do we need to consider memory barrier instructions?

    • Sharing data between multiple different CPU cores (threads), such as mailboxes, etc.
    • Sharing data with peripherals, such as DMA operations
    • Modifying memory management policies, such as context switching, page faults, modifying page tables
    • Modifying memory areas storing instructions (instruction memory): such as self-modifying code, loading a program into RAM

    DMBPoints to note about the instruction

    • The DMB instruction focuses onthe sequence of memory accesses, and does not care when data access instructions complete execution
    • data access instructions before the DMB must be observed by data access instructions after the DMB

    • Data/unified cache instructions before the DMB must complete execution (be observed) before the memory access instructions after the DMB

    DSB instruction

    Completion of Load/Store instructions

    • The DSB instruction is much stricter than the DMB instruction
    • Inany instruction after the DSB instruction, must wait until the following is completed,before it can start execution：
      • Before the DSB instruction,all data accessesmust complete execution
      • Instructions such as cache, branch predictor, TLB, etc. before the DSB instruction must complete execution

    Points to note about the DSB instruction

    • DMB instructiononly focuses on the sequence of data accesses, while the DSB instructionstart to focus on when instructions must complete execution
    • Instructions after the DSB instruction must wait until:
      • All data access instructions before the DSB have completed execution
      • All Cache, TLB, and other instructions before the DSB have completed execution

    The dsb instruction is more like a barrier

    • In amulti-core system, cache and TLB instructions are broadcast to other CPU cores, so the DSB instruction will wait until these instructions are broadcast and acknowledgment signals from all cores are received before it is considered complete

    Parameters of DMB and DSB instructions

    • Parameters of two dimensions can be specified, one isShareability domain, and the other isbefore-afteraccess

    • Shareability domain

      • Full System
      • Outer Shareable, prefixed withOSH
      • Inner Shareable, prefixed withISH
      • Non-shareable, prefixed withNSH

    • before-after access (i.e., before and after the memory barrier instruction, further refined to read/write memory barrier)

      • Read barrier: Load-Load/Store: suffixed withLD

        This means that the barrier requires all loads to complete before the barrier but
        does not require stores to complete. Both loads and stores that appear after the
        barrier in program order must wait for the barrier to complete.

      • Write barrier: Store-Store: suffixed withST

        This means that the barrier only affects store accesses and that loads can still be
        freely re-ordered around the barrier.

      • Read-write barrier: Any-Any: suffixed withSY

        This means that both loads and stores must complete before the barrier. Both
        loads and stores that appear after the barrier in program order must wait for the
        barrier to complete.

    One-way memory barriers (one way barriers)

    Load-Acquire (LDAR)

    ​ All loads and stores that are after an LDAR in program order, and that match the shareability domain of the target address, must be observed after the LDAR.

    Store-Release (STLR)

    ​ All loads and stores preceding an STLR that match the shareability domain of the target address, must be observed before the STLR.

    There are also exclusive versions of the above, LDAXR and STLXR, available.

    • DMB and DSB are both bidirectional memory barrier instructions; ARMv8 supports “one-way” memory barrier primitives

    • Acquire primitive: refers toRead and write operations after this barrier primitive cannot be reordered before this barrier primitive, usually this barrier primitiveis combined with load instructions

    • Release primitive: refers toRead and write operations before this barrier primitive cannot be reordered after this barrier primitive., usually this barrier primitivecombined with store instructions

    • Load-Acquire barrier primitive: ordinary read and write operations can pass this barrier instruction backward, but subsequent read and write operations cannot pass this barrier instruction forward.

      • used to protect a critical section of data

      • has better performance than the full-featured DMB instruction
      • has no effect on data cache maintenance instructions, because it does not wait for cache broadcasts

    Instruction-level memory barrier instruction: ISB instruction

    Context synchronization

    • The ISB instruction is powerful, it willflush the pipeline, and thenrefetch instructions from the instruction cache or memory。

    • ISB instruction guarantees

      • Instructions after the ISB are re-fetched from the instruction cache or memory
      • Context-changing operations before the ISB instruction have been completed (here context refers to system register states, etc.)

    • Context-changing operations include:

      • Maintenance operations for Cache, TLB, and branch predictor, etc.
      • Changing system registers, such as TTBR0_EL1

      The ARMv8 architecture defines context as the state of the system registers and context-changing operations as things like cache, TLB, and branch predictor maintenance operations, or changes to system control registers, for example, SCTLR_EL1, TCR_EL1, and TTBRn_EL1. The effect of such a context-changing operation is only guaranteed to be seen after a context synchronization event.

    • The effect of context-changing operations: only aftera context synchronization eventcan be observed

    • Context synchronization event (context synchronization event）

      • An exception occurs
      • Returning from an exception
      • ISB instruction

    In practice, an ISB instruction is generally required after modifying system registers, especially when modifying system control registers (such as SCTLR_EL1、TCR_EL1, TTBRn_EL1), but not all system register modifications require ISB; modifications to some system registers (such as the stack pointer SP) take effect on subsequent instructions without ISB

    • No memory barrier instruction is used between updating the new code content (str x11,[x1]) and the clean data cache instruction
      • Updating the code content and cleaning the data cache both operate on the same address, and both are data-related operations. There is a data dependency between them, which can be understood as the same observer
      • The program order between them can be guaranteed
    • A memory barrier is required between cleaning the data cache and invalidating the instruction cache
      • Although these two cache instructions operate on the same address, they are different observers (one is the data side, the other is the instruction side
      • The DSB here ensures that the instruction cache is invalidated only after the data cache is cleaned
    • In a multi-core coherent system, the DSB instruction ensures that the cache maintenance instruction completes execution, meaning other CPUs can observe the completion of the cache maintenance instruction
    • The ISB instruction does not broadcast, so CPU1 also needs to execute the ISB instruction

    Non-temporal load and store pair

    LDNP and STNP

    LDNP and STNP are instructions that read or write a pair of register values. They also hint to the memory system that caching this data is not useful. This hint does not prohibit memory system activities such as address caching, prefetching, or gathering, but merely indicates that caching is unlikely to improve performance. A typical use case might be streaming data, but it should be noted that effective use of these instructions requires a microarchitecture-specific approach.

    Summary: Memory barrier instructions and cache/TLB maintenance instructions

    • Data cache or unified cache maintenance instructions
      • A DMB instruction can be used to ensure that the cache maintenance instruction completes execution within the specified shareable domain
      • Load-acquire and store-release barriers have no effect on data cache maintenance instructions (because they do not wait for the cache to broadcast)
    • Instruction cache maintenance instructions
      • From the perspective of memory observers, the instruction cache and the data cache are two different observers
      • After the instruction cache maintenance operation completes, execute a DSB instruction to ensure that all CPU cores in the inner shareable domain can see the completion of the instruction cache maintenance operation
    • TLB maintenance instructions
      • The page table walking unit and the data access hardware unit are actually two different observers of the memory system
      • A DSB instruction needs to be executed after the TLB maintenance instruction to ensure that all CPUs in the inner shareable domain can complete
    • The ISB instruction does not broadcast, if each CPU core needs to call the ISB instruction individually

    CPU Stall Phenomenon Caused by Cache Coherence Protocol

    • The MESI protocol is a bus snooping (snoopy) based protocol, and its bus transmission bandwidth is related to the load between CPUs and the number of CPU cores
    • The change of cache line state heavily depends on the response signals of other cache lines, that is, it must receive the response signals of all other CPUs’ cache lines before proceeding to the next state transition. In a scenario where the bus is busy or the bus bandwidth is tight, the CPU may need a long time to wait for the response signals from other CPUs, which greatly affects system performance. This phenomenon is called CPU stall (CPU stall）

    Example Analysis: CPU Stall Caused by MESI Protocol

    In a 4-core CPU system, data A is shared on CPU1, CPU2, and CPU3, and the state of their corresponding cache lines is S (Shared), with the initial value of A being 0. However, data A is not cached in CPU0’s cache, and its state is I (Invalid)

    At this point, CPU0 writes a new value (e.g., 1) into data A. So, how will the state of these cache lines change?

    Time T1

    CPU0 sends a local write operation signal to the bus

    Time T2

    CPU1, CPU2, and CPU3 all receive the BusRdX signal from the bus

    Time T3

    CPU1 checks whether it has a copy of cached data A in its local cache. CPU1 replies with aFlushoptsignal and sends the data to the bus, then sets its own cache line state to Invalid, changing the state to I, and finally broadcasts an acknowledge signal.

    Time T4

    CPU2 and CPU3 both check their local caches, and the state changes from S to I

    Time T5

    CPU0 can only write data A from the store buffer into the cache line after receiving acknowledgment signals from all other CPUs and confirming that there are no valid cached copies of the data on other CPUs. Finally, the state of CPU0’s cache line changes to M.

    Summary

    CPU0 has a waiting process; it needs to wait for acknowledgment signals from all other CPUs

    Optimization Method 1: Store Buffer

    • No need to wait for acknowledgment signals from other CPUs, canfirst write the data into the store buffer, and continue executing the next instruction
    • After CPU0 receives the acknowledgment signals replied from all other CPUs, CPU0 then writes the latest value of data A from the store buffer into the local cache line, and modifies the state of the cache line to M

    Side effects brought by the store buffer

    Data a has a cached copy in CPU1’s cache, and its state is E (Exclusive)

    Data b has a cached copy in CPU0’s cache, and its state is E (Exclusive)

    So, in a system with a store buffer, can an assert failure occur?

    Time T1 CPU0 executes the statement “a=1”

    Time T2 CPU0 executes the statement “b=1”

    Time T3 CPU1 executes the statement “while(b==0)”

    Time T4 CPU0 receives the bus read signal.

    Time T5 CPU1 gets the latest value of b

    Solution for side effects: use write memory barrier instructions

    • store bufferoptimizes the performance degradation caused by long waits for acknowledgment signals between multi-core processors, but stillcannot guarantee that the write order between multi-core CPUs is visible to other cores
    • write memory barrier instruction(e.g., smp_wmb()) is to mark all writes in the current store buffer that have not yet been flushed into the cache, and thenflush the store buffer, ensuring that these marked writes are updated to the cache before subsequent write operations can be executed

    Optimization Method 2: Invalidate Queue

    • The store buffer in Optimization Method 1 is small and can easily be filled up.
    • One reason for CPU stalls is:Waiting for other CPUs to perform “invalidate” operations, which is time-consuming.
    • Invalidate Queue:Cache the “invalidate operation”, first send an acknowledgment signal to the requester, and then slowly perform the invalidate operation,, so that other CPUs do not have to wait for a long time.

    The CPU that replies to the invalidate operation itself does not need this data.

    • When a CPU receives a bus request and needs to invalidate a local cache line, it adds the request to the invalidate queue and immediately sends an acknowledgment signal to the other party, without waiting to invalidate the cache line before replying.
    • If a CPU adds a request to the invalidate queue, then before the invalidate operation corresponding to that request is completed, the CPU cannot send any bus messages related to the cache line corresponding to that request to the bus.

    Side Effects of the Invalidate Queue

    Assume the initial values of data a and data b are 0, data a has copies in both CPU0 and CPU1 with state S, and data b has a cached copy in CPU0 with state E. Will the assert succeed?

    Side effect solution: use read memory barrier instruction

    read memory barrier instructioncanensure that all invalidation operations in the invalidation queue are executed before the read operation following the read barrier instruction can be executed。

    Summary of memory barriers and cache coherence

    • In SMP, a simple load and store instruction is not simple; its behavior needs to be analyzed in combination with the MESI cache coherence protocol

    • Memory barriers need to be analyzed in combination with MESI cache coherence

    • Store buffers and invalidation queues are hardware optimization techniques, but they also bring some side effects

    • Read memory barrier instructions act on the invalidation queue, ensuring that all pending invalidation operations in the queue are executed before subsequent read operations can be executed

    • Write memory barrier instructions act on the store buffer, ensuring that data in the store buffer is written to the cache before subsequent write operations can be executed
