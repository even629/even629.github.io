hotplug

Hotplug refers to the ability to safely insert or remove hardware devices while the device is running, without needing to shut down or restart the system. This means you can insert or remove hardware components (such as USB devices, expansion cards, hardware drives, etc.) on a computer or other electronic device without shutting down or interrupting ongoing operations.

The main purpose of hotplug is to provide convenience and flexibility. With hotplug, you can quickly replace or add hardware devices without stopping ongoing tasks. This is very useful in many scenarios, such as:

  1. USB devices: You can insert or remove USB devices at any time, such as mice, keyboards, printers, storage devices, etc., without needing to restart the system.
  2. Hard drives: In some servers or storage systems, you can add or replace hard drives at runtime to expand storage capacity or replace faulty drives.
  3. Expansion cards: You can insert or remove expansion cards such as graphics cards, network cards, or sound cards on a computer to meet different needs or upgrade hardware performance.

To support hot-plug functionality, both the hardware device and the system must have corresponding support. On the hardware side, the device interface must be designed so that it can be inserted and removed without damaging the device or the system. The system needs to provide corresponding drivers and management functions to properly configure and identify the device when it is inserted or removed.

Hot-plug mechanism

Hot-plug is the interaction between the kernel and user space by calling user space programs (such ashotplug、udevandmdev). When the kernel needs to notify the user that a hot-plug event has occurred, it calls this user space program to achieve the interaction.

In the Linux kernel, the hot-plug mechanism supports the dynamic insertion and removal of components such as USB devices, PCI devices, and even CPUs. This mechanism implements the connection between the underlying hardware, kernel space, and user space programs, and has been continuously evolving and improving. A device file system is a mechanism used to manage device files. In Linux, there are three common device file systems: devfs, mdev, and udev.

  • devfs: devfs is a kernel-based dynamic device file system that first appeared in the Linux 2.3.46 kernel. It handles device files by dynamically creating and managing device nodes. However, devfs had some limitations and performance issues, and was removed starting from Linux version 2.6.13.
  • mdev: mdev is alightweight hot-plug device user space program, typically used in embedded Linux systems. It is a simplified version of udev, using theuevent_helpermechanism to handle device insertion and removal events. mdev calls the corresponding user program to create device nodes when a device is inserted.
  • udev: udev is currently thewidely used hot-plug device user space program on PCs. It is based on thenetlinkmechanism, listening for uevents sent by the kernel to handle device insertion and removal. udev can dynamically create and manage device nodes, and load appropriate drivers when a device is inserted. It provides rich configuration options, allowing users to flexibly manage device files.

Kernel sending events to user space

kobject_uevent()

kobject_ueventis a function in the Linux kernel used to generate and send uevent events. It is a way for udev and other device management tools to communicate with the kernel.

Parameter Description:

  • kobj: the kernel object (kobject) to send the uevent event
  • action: indicates the action that triggers the uevent, which can be device insertion, removal, attribute change, etc. Below are some common action parameter values. These action types are used to describe different events that occur on a device. By passing the corresponding action type as the action parameter tokobject_ueventfunction, the corresponding uevent event can be triggered to notify the user-space udev to perform the corresponding operation.
    • KOBJ_ADD: indicatesdevice addition or insertion operation, indicating adding an object to the kernel object system.
    • KOBJ_REMOVE: indicatesdevice removal or ejection operation, indicating deleting an object from the kernel object system.
    • KOBJ_CHANGE: indicatesdevice attribute modification operation, indicating changes to the kernel object, such as attribute modification, etc.
    • KOBJ_MOVE: indicatesdevice move operation, that is, the device moves from one location to another.
    • KOBJ_ONLINE: indicatesbringing the device online, that is, the device transitions from an offline state to an online state, making it accessible.
    • KOBJ_OFFLINE: indicatestaking the device offline, that is, the device transitions from an online state to an offline state, making it inaccessible.
    • KOBJ_BIND: indicatesconnecting a device to a kernel object
    • KOBJ_UNBIND: indicatesunbinding a device from a kernel object
    • KOBJ_MAX: indicates the maximum value of the enumeration type, typically used to represent no operation behavior.

kobject_ueventThe primary function of this function is to generate uevent events in the kernel and send them to the user-space udev via the netlink mechanism. When this function is called, the kernel encapsulates the relevant device information and event type into a uevent message and sends the message to user space through the netlink socket.

The user-space udev receives these uevent messages and performs corresponding operations based on the device information and event type in the messages, such as creating or deleting device nodes, loading or unloading drivers, etc.

udevadm command

udevadmis a command-line tool used to interact with the udev device manager. It provides a series of subcommands for querying and managing devices, triggering uevent events, and performing other udev-related operations. Some commonudevadmsubcommands and their functions are as follows:

  1. udevadm info: Used to get detailed information about the device, including device path, properties, driver, etc.
  2. udevadm monitor: Used to monitor and display uevent events in the current system. It displays device insertion, removal, and other related events in real-time.
  3. udevadm trigger: Used to manually trigger uevent events for devices. This command can be used to simulate device insertion, removal, and other operations to trigger corresponding event handling.
  4. udevadm settle: Used to wait for udev to process all queued uevent events. It blocks until udev completes all current device processing operations.
  5. udevadm control: Used to interact with the udev daemon and control its behavior. For example, this command can be used to reload udev rules, set log levels, etc.
  6. udevadm test: Used to test the matching and execution process of udev rules. This command can be used to test whether a specific device can correctly trigger the corresponding rules.

netlink

netlink listens for broadcast information

Netlink is in the Linux kernela mechanism for duplex communication between the kernel and user space. Itbased on the socket communication mechanism, and provides a reliable, asynchronous, multicast, and ordered communication method.

The main features of the Netlink mechanism include:

  1. Duplex communication: Netlink allows bidirectional communication between the kernel and user space, enabling the kernel to send messages to user space and also receive messages from user space.
  2. Reliability: Netlink provides a reliable message delivery mechanism, ensuring the integrity and reliability of messages. It uses acknowledgment and retransmission mechanisms to ensure reliable message transmission.
  3. Asynchronous communication: Netlink supports asynchronous communication, meaning the kernel and user space can independently send and receive messages without synchronously waiting for a response from the other party.
  4. Multicast support: Netlink allows broadcasting messages to multiple processes or sockets to achieve one-to-many communication.
  5. Ordered transmission: Netlink guarantees ordered transmission of messages, meaning sent messages are received in the order they were sent at the receiving end.

Netlink has a wide range of applications, common applications include:

  1. System management tools: Tools such as ifconfig and ip use Netlink to communicate with the kernel to obtain and configure network interface information.
  2. Inter-process communication: Processes can use Netlink for cross-process communication to achieve data exchange and coordination between processes.
  3. Communication between kernel modules and user space applications: Kernel modules can send notifications to user space applications or receive instructions from user space applications via Netlink.

Receive data

Netlink sockets do not need to call the listen function when receiving data, but can directly use the recv function to receive.

Required header filesFunction prototype
#include<sys/types.h>
#include<sys/socket.h>		ssize_t recv(int sockfd, void *buf, size_t len, int flags);

Function parameters:

  • sockfd: Specifies the socket descriptor, i.e., the Netlink socket to receive data.
  • buf: A pointer to the data receive buffer, used to store the received data.
  • len: Specifies the byte size of the data to be read.
  • flags: Specifies some flags, used to control the data reception method. In normal cases, it can be set to 0.

Return value:

  • In case of success, returns the actual number of bytes read.
  • If the return value is 0, it means the other party has closed the connection.
  • If the return value is -1, it means an error has occurred, and the specific error code can be obtained by checking the errno variable.

Using the recv function, you can receive data from the specified Netlink socket and store it in the provided buffer. The return value of the function indicates the actual number of bytes read, and you can determine whether the data was successfully received based on the return value.

  • @/etc/mdev/usb_insert.shindicates that when a device matching the rule is inserted, mdev will execute/etc/mdev/usb_insert.shscript.@symbol indicates that a shell command is being executed.
  • $/etc/mdev/usb_remove.shindicates that when a device matching the rule is removed, mdev will execute/etc/mdev/usb_remove.shscript.$symbol indicates that an internal command is being executed.

Fix the device node of the USB device

Requirements

In Linux systems, when using multiple USB-to-serial devices, it is often encountered that the unstable device nodes are caused by changes in the plugging and unplugging order of the USB-to-serial adapters. To solve this problem, we can use udev to fix the device nodes, ensuring that the device nodes are not affected by the plugging and unplugging order of the USB-to-serial adapters.

Solution

Insert the USB device:

udevadm info -a -n /dev/ttyUSB0is a command used to obtain detailed information about a specific device node, including device attributes, drivers, device paths, etc. The meanings of the parameters of these commands are as follows:

  • -a: Displays all attributes associated with the specified device node.
  • -n: Specifies the path or name of the device node.

The output of this command may include the following information:

  1. Device path (DEVPATH): The path of the device in the system, for example/devices/pci0000:00/0000:00:1d.0/usb2/2-1/2-1.2/2-1.2:1.0/ttyUSB0/tty/ttyUSB0。
  2. Device node (DEVNAME): The node of the device in the file system, i.e./dev/ttyUSB0。
  3. Device attributes: Including the device’s vendor ID, product ID, serial number, and other information.
  4. Device driver: The name and path of the driver used by the device.
  5. Device type (SUBSYSTEM):usb

Next, in the development board’setc/udev/rules.d/directory, create a rule file named 001.rules:

  1. KERNELS=="5-1:1.0"

    • Match the device’s Kernel Device Name

    • 5-1:1.0is the path identifier of the USB device in the kernel:

      • 5: USB bus number (Bus)
      • 1: USB device number (Device)
      • 1.0: Interface number (Interface), representing the 0th endpoint of the 1st interface
      • This value comes fromudevadm infothe output’sKERNELSfield

  2. SUBSYSTEMS=="usb"

    • Match the device’s Subsystem

    • indicating that this is a USB type device

    • Can be used to filter all USB devices

  3. MODE:="0777"

    • Set the device file’s permissions (mode)
  • 0777means:
    • All users (root, regular users, others) have read, write, and execute permissions
    • i.e.:rwxrwxrwx, so that any program can directly access the device (such as/dev/ttyUSB0）

⚠️ Note:MODEThe preceding is:=, not==。:=means ‘assignment’, while==is ‘match’.

  1. SYMLINK+="myusb"

    • Create a symbolic link for the device

    • For example, the original device is/dev/ttyUSB0, now there will be an additional soft link:/dev/myusb -> /dev/ttyUSB0

    • can usemyusbto replacettyUSB0, more convenient for naming and script invocation
