This article introduces the architectural principles and programming interfaces of ARM SMMUv3 and its controller MMU-600, discussing the role of SMMU/IOMMU in addressing security and performance issues of device DMA access in bare-metal and virtualization scenarios. The article summarizes in detail the internal composition of MMU-600, the two-stage address translation mechanism, the page table traversal and configuration process based on StreamID and SubstreamID, and briefly mentions the composition and core functions of the Linux IOMMU driver framework.

Reference documents:

Arm Architecture Reference Manual Armv8

https://developer.arm.com/documentation/ddi0487/fc/

Technical Manual:

  1. System Memory Management Unit Architecture Specification, SMMU architecture version 3
  2. MMU-600 System Memory Management Unit, Technical Reference Manual

What is SMMU/IOMMU

  • SMMU is an IOMMU (Input/Output Memory Management Unit) implemented by ARM, which converts the virtual addresses accessed by devices into physical addresses.
  • AMD IOMMU
  • Intel VT-d

Bare-metal OS scenario

There may be issues with DMA access for IO devices:

  1. IO devices are given physical addresses, allowing them to access all memory addresses. Therefore, DMA can corrupt other devices or system memory, such as from a malicious IO device.
  2. There is no protection for device drivers either, such as malicious drivers.
  3. Information leakage between IO devices is easy: side-channel attack.

Virtualization scenario

Disadvantages：

Every DMA operation needs to trap into the VMM to allocate physical memory for the IO device, resulting in a performance loss (about 20~30%).

Solution: SMMU

Each IO device has its own page table.

Scatter-gather means that an IO device can use a contiguous virtual address, but the final translated physical addresses do not necessarily need to be contiguous.

SMMU Development History

Usage of SMMU

For CPU accessing devices, the MMU is sufficient; for devices accessing Memory, it needs to go through the SMMU.

MMU-600 Controller

  • SMMUv3 is the spec, and MMU-600 is a controller implemented based on the SMMUv3.1 specification.
  • MMU-600 translates device IOVA to PA, supporting two-stage address translation or Bypass mode.
    • Stage1: Convert IOVA->PA or IOVA->IPA
    • Stage2：IPA->PA
    • Both: IOVA->IPA->PA
    • Bypass mode (skip SMMU)
  • Features
    • Compatible with SMMUv3.1
    • Supports ARMv8 AArch32 and AArch64 page table formats
    • Supports 4KB, 16KB, 64KB pages
    • Supports PCIe, supports ATS and PASIDs
    • Supports PRI (Page Request Interface)
    • Supports ACE5-Lite atomic operations
    • Supports translation fault, software can implement demand paging
    • Supports GICv3 integration, supports message-based interrupts

MMU-600 block diagram

MMU-600 internal composition

  • TBU (Translation Buffer Unit)
    • Contains TLB to cache translation results
    • Each connected Master has at least one TBU
    • If TBU does not find a TLB entry, then send a request to TCU
  • TCU（Translation Control Unit）
    • Hardware unit for address translation
    • MMU-600 has only one TCU
    • Manage memory requests
    • Page table walk
    • Configure page tables
    • Implement backup caching structures
    • SMMU programming interface
  • DTI
    • Used to connect TBU to TCU
    • Using AXI Stream protocol

StreamID

  • A DMA transfer includes: target address, size, read/write attributes, security attributes, shareability attributes, cacheability attributes, and StreamID
  • StreamID is used to associate a device (function)
  • StreamID is used to index the Stream Table, which contains per-SMMU related page table information
  • For PCIe devices, StreamID[15:0] == RequesterID[15:0] equals BDF
    • Bits[15:8] Bus number
    • Bits[7:3] Device number
    • Bits[2:0] Function number
  • For non-PCIe devices, StreamID is obtained through DTS (therefore, StreamID is fixedly allocated by the platform)

Traversing the Stream Table

  • Each entry STE in the Stream Table contains:
    • Page table base address for Stage2 translation
    • Points to a Context Descriptor, which contains the translation page table base address for stage1
  • Use StreamID to index the Stream Table
  • The Stream Table needs to be created and populated by OS software
  • If the number of STEs (Stream Table Entries) is greater than 64, a 2-level table must be used
  • StreamID[n:x] indexes the first-level table, StreamID[x-1:0] indexes the second-level table
    • N represents the most significant bit of StreamID, usually 15
    • X is the split point, STRTAB_BASE_CFG.SPLIT
    • For example: n=15, x=8, then StreamID[15:8] indexes the first-level table, StreamID[7:0] indexes the second-level table

Two-level table example

Assume StreamID is only 10 bits, i.e., StreamID[9:0], assume X is 8, then use StreamID[9:8] to index the first-level table, and use StreamID[7:0] to index the second-level table

The first-level table has 4 entries, and each second-level table has 256 entries

StreamID is 10 bits in total, 2 to the power of 10 is 1024, which means it can index up to 0~1023 STEs

Context Descriptors table

  • CD contains the page table base address for stage1, ASID, page table attributes, etc.

  • S1ContextPtr in STE points to CD

  • The page table base address for Stage2 is in the STE

  • Software management:

    • Virtualization: Hypervisor manages Stream Table and stage 2 page tables, Guest OS manages CD and stage1 page tables
    • Bare-metal OS manages Stream Table and CD

    SubstreamID

  • S1ContextPtr in STE points to a CD (Context Descriptor) table, SubstreamID is used to index this table

  • Each CD contains the page table base address to be used in stage1

  • A device may be used by multiple processes, SMMU distinguishes them through SubstreamID, one SubstreamID can be assigned per process

  • For PCIe devices, PASID is used as SubstreamID

Summary

Command and Event queues

  • Command queue is used for input, event queue is used for output. Each queue has a producer and a consumer
  • The output queue contains data produced by SMMU, which is then consumed by software. The input queue contains data produced by software, which is then consumed by SMMU

For specific descriptions of each CMD, see Chapter 4 of the SMMUv3 manual

Linux IOMMU driver framework

Two major roles of the IOMMU driver:

  1. Provide DMA interfaces and functions for IO devices
  2. Provide SVA (Shared Virtual Addressing) functionality for IO devices, sharing the virtual address space between the IO device and a process

IOMMU includes:

  1. IOMMU framework
  2. IOMMU controller driver
  3. IOMMU dma-mapping
  4. IOMMU interfaces provided to IO devices

  • IOMMU Domain is mainly used to provide the ability to access the IOMMU controller
  • iommu_group is the minimum resource isolation unit
    • Generally, one device occupies one iommu_group. Multiple devices based on certain hardware topology relationships and mutual trust, or point-to-point protected transmission devices, can be added to one iommu group
  • The server system may consist of multiple IOMMU controllers.

From the driver’s perspective, what new API interfaces are added?

Summary

IO page fault

  • CPU-side page fault, handled by the CPU’s page fault handler_mm_fault
  • Device-side page fault:
    • PCIe devices: PRI extension (Page Request Interface)
    • Platform devices: utilizing SMMU’s stall mode
  • Stall mode: When an IO device triggers an exception, the transfer transaction is suspended, and this event is recorded in the event queue. OS software needs to handle it, then send a CMD_RESUME command to resume the transfer transaction.

Question: Why is the CPU-side handler called when an IO device-side page fault occurs here?_mm_fault() is used to establish VA->PA page table entries, but what about the IO device’s page table? Who establishes it?

SVA, shares a single set of page tables

TLB invalidation operation

  • When the CPU side modifies the mm, or releases memory
    • Call flush_tlb() to invalidate the CPU-side TLB
    • Invalidate the corresponding IOTLB on the IO device side and the IOTLB of the PCIe ATC through the callback function registered by mmu_notifier

  • Is it possible for the IO device to actively modify the VA->PA mapping relationship?
    • In non-SVA cases, through iommu_map and iommu_unmap interfaces to allocate and release DMA buffers. iommu_unmap()->iotlb_sync()
    • In SVA cases, the virtual address allocated by the CPU side can be used as IOVA. When an IO device triggers a page fault, it directly goes through the IO page fault handling process.

Newly added ATS (Address Translation Services) for PCIe

  • ATS mechanism in PCIe: The device caches the VA-to-PA mapping. When the device uses PA for memory access, it does not need to go through IOMMU page table translation
  • The ATC (Address Translation Cache) of a PCIe device has its own TLB
  • Before the device performs DMA, it queries the ATC for the entry corresponding to the VA
    • If present: directly use PA to access memory
    • If absent: sends an ATS request to the SMMU. After the SMMU finds the PA, it replies with an ATS completion
  • The PCIe ATS specification defines the format of ATS request and completion message packets
    • Address Translation Services Revision 1.1
  • SMMU driver:
    • Enable ATS
    • ATS invalidation operation. When the SMMU changes the VA->PA mapping, the SMMU needs to send an ATS invalidation request to the PCIe ATC
    • SMMU provides CMD_ATC_INV command

PRI (Page Request Interface) support

  • The advantage of the PRI mechanism is that when preparing to initiate a DMA operation, there is no need to prepare (pin) the DMA buffer in advance
  • Use case: High-speed network cards. In burst scenarios, the Host does not need to reserve and occupy a large amount of buffer in advance
  • PRI mechanism:
    • When ATC lookup results in a TLB miss, it sends a Page Request Message
    • The RC sends the message to the SMMU
    • The SMMU writes the page request into the PRI queue, triggering a PRI interrupt
    • OS software allocates physical memory
    • OS software replies with CMD_PRI_RESP command
