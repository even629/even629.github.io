    Integer overflow

    For example, a u8 can hold values from 0 to 255. So what happens when you change it to 256? This is called “integer overflow” (integer overflow ), which leads to one of two behaviors. Whencompiled in debug mode, Rust checks for these issues and causes the program to panic, a term Rust uses to indicate that the program exited due to an error.

    In release builds, Rust does not check for overflow; instead, it performs an operation called two’s complement wrapping. In short, values larger than the maximum the type can hold wrap around to the minimum value; 256 becomes 0, 257 becomes 1, and so on. Relying on integer wrapping is considered an error, even if the behavior might occur. If you explicitly need this behavior, there is a type in the standard library that provides it explicitly: Wrapping. To explicitly handle the possibility of overflow, you can use the following methods provided by the standard library on primitive numeric types:

    • Available in all modes wrapping_ methods for wrapping*, such as wrapping_add
    • If checked_ methods*returns a None value if overflow occurs
    • Use overflowing_ methods*returns the value and a boolean indicating whether overflow occurred
    • Use saturating_ methods*saturates at the minimum or maximum value

    The Stack and the Heap

    The stack and the heap are both parts of memory available to your code at runtime, but they are structured in different ways. The stack stores values in the order it gets them and removes values in the opposite order. This is known as last in, first out（last in, first out）。

    Adding data is called pushing onto the stack（pushing onto the stack), and removing data is called popping off the stack（popping off the stack). All data stored on the stack must have a known, fixed size.

    Data with an unknown size at compile time or a size that might change must be stored on the heap. The heap is less organized: when you put data on the heap, you request a certain amount of space. The memory allocator finds an empty spot in the heap that is big enough, marks it as being in use, and returns a pointer, which is the address of that location. pointer（pointer). This process is called allocating on the heap（allocating on the heap), often abbreviated as just “allocating”. (Pushing data onto the stack is not considered allocating).Because the pointer to the data on the heap is known and has a fixed size, you can store the pointer on the stack, but when you need the actual data, you must follow the pointer.

    **Pushing to the stack is faster than allocating on the heap because the allocator never has to search for a place to store new data; that place is always at the top of the stack.**Allocating space on the heap, on the other hand, requires more work because the allocator must first find a big enough space to hold the data and then perform bookkeeping to prepare for the next allocation.

    Accessing data in the heap is slower than accessing data on the stack because you have to follow a pointer to get there. Modern processors are faster if they jump around less in memory (caching). Continuing the analogy, consider a waiter at a restaurant taking orders from many tables. It is most efficient to get all the orders from one table before moving on to the next table. Taking an order from table A, then an order from table B, then one from A again, and then one from B again would be a much slower process. By the same token, a processor can do its job better if it works on data that’s close to other data (as it is on the stack) rather than farther away (as it can be on the heap).

    When your code calls a function, the values passed into the function (including, potentially, pointers to data on the heap) and the function’s local variables get pushed onto the stack. When the function is over, those values get popped off the stack.

    Keeping track of what parts of code are using what data on the heap, minimizing the amount of duplicate data on the heap, and cleaning up unused data on the heap so you don’t run out of space are all problems that ownership addresses. Once you understand ownership, you won’t need to think about the stack and the heap very often, but knowing that the main purpose of ownership is to manage heap data can help explain why it works the way it does.

    A test in Rust is a function annotated with the test attribute. Attributes are metadata about pieces of Rust code

    To turn a function into a test function, you need to add before the fn line #[test]

    Running tests

    • Use the cargo test command to run all test functions

    Rust will build a Test Runner executable that runs the functions annotated with test and reports whether they ran successfully

    • When creating a library project with cargo, a test module is generated with a test function inside

    You can add any number of test modules or functions

    Test failures

    • A test function panicking indicates failure
    • Each test runs in a new thread
    • When the main thread sees that a test thread has died, that test is marked as failed

    Assert

    Use the assert! macro to check test results

    The assert! macro, from the standard library, is used to determine if a state is true

    • true, the test passes
    • false, calls panic!, the test fails

    Using assert_eq! and assert_ne! macros to test equality

    • Both come from the standard library
    • Determine whether two parameters are equal or not equal
    • In fact, they use the assert! macro with the == and != operators
    • When an assertion fails, the values of the two parameters are automatically printed

    Print parameters using debug format: requires the parameters to implement the PartialEq and Debug traits (all primitive types and most standard library types implement them)

    Custom error message

    You can pass an optional failure message to the assert!, assert_eq! and assert_ne! macros, which can print the custom failure message along with the test failure.

    • The first parameter of the assert! macro is required, and the custom message is passed as the second parameter
    • assert_eq! and assert_ne! requires the first two parameters, and the custom message is passed as the 3rd parameter
    • The custom message parameter is passed to the format! macro, and you can use the {} placeholder

    Can verify whether the code panics under specific circumstances

    should_panic Attribute:

    • Function panics: test passes
    • Function does not panic: test fails

    Making should_panic more precise

    You can add an optional expected parameter to the should_panic attribute. The testing tool will ensure that the error message contains the text it provides

    Using Result<T, E> in tests

    Without needing to panic, you can write tests using Result<T, E> as the return type

    • Returns Ok: test passes
    • Returns Err: test fails

    Note:You cannot use the #[should_panic] annotation on these tests that use Result<T, E>。

    Test Organization

    The Rust community tends to think about tests in terms of two main categories:

    unit tests（unit tests) and integration tests（integration tests）

    • Unit tests tend to be more small and focused, testing one module in isolation at a time, or testing private interfaces.
    • Integration tests, on the other hand, are entirely external to your library. They use your code in the same way any other external code would, exercising the public API and potentially testing multiple modules per test.

    Annotating the test module with #[cfg(test)]

    • Running cargo build will not
    • Integration tests are in a different directory, and they don’t need the #[cfg(test)] annotation

    Separation of Concerns for Binary Projects

    The problem of the main function being responsible for organizing multiple tasks is common in many binary projects. Therefore, the Rust community has developed a guiding process for separating concerns in binary programs when the main function starts to become large. This process has the following steps:

    • Split the program into main.rs and lib.rs and put the program’s logic into lib.rs。
    • When the command line parsing logic is small, it can be kept in main.rs .
    • When command line parsing starts to become complex, also extract it from main.rs extract it into lib.rs .

    After these processes, the responsibilities retained in the main function should be limited to:

    • Calling the command-line parsing logic with the parameter values
    • Setting up any other configuration
    • Call lib.rs the run function in
    • Handling the error if run returns an error

    Everything about this pattern is about separation of concerns:main.rs handles program execution, while lib.rs handles all the real task logic. Because the main function cannot be tested directly, this structure allows us to test them by moving all the program logic into lib.rs functions. The code remaining only in main.rs will be small enough that its correctness can be verified by reading it.

    TDD (Test-Driven Development)

    Following the Test-Driven Development (TDD) pattern to incrementally add the search logic to minigrep. This is a software development technique that follows these steps:

    1. Write a failing test and run it to ensure it fails for the reason you expect.
    2. Write or modify just enough code to make the new test pass.
    3. Refactor the code you just added or modified, and ensure the tests still pass.
    4. Repeat from step 1!

    This is just one of many ways to write software, but TDD helps drive the design of the code. Writing tests before writing the code that makes them pass helps maintain high test coverage throughout the development process.
