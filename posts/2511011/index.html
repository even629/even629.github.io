lseek()

All open files have acurrent file offset, hereinafter referred to as cfo. cfo is usually anon-negative integer, used to indicate the number of bytes from the beginning of the file to the current file position. Read and write operations usually start at cfo and increase cfo by the number of bytes read or written. When a file is opened, cfo is initialized to 0, unless O_APPEND is used. The lseek function can be used to change the file’s cfo.

Among the three parameters above, the most important is the second parameter, cmd, which is of type unsigned int. To efficiently use the cmd parameter to pass more control information, an unsigned int cmd is split into 4 segments, each with its own meaning. The bitfield breakdown of the unsigned int cmd is as follows:

  • cmd[31:30]Data (args) transfer direction (read/write)
  • cmd[29:16]Data (args) size
  • cmd[15:8] **Command type, which can be understood as the command’s key,**usually an ASCII code (a character from 0-255, some characters are already occupied, and the number field for each character may be partially occupied)
  • cmd[7:0]Command number, which is an 8-bit number (number, between 0-255)

Library

A library is an executable binary file, which is compiled code。

Using libraries can improve development efficiency. In Linux, there are static libraries and dynamic libraries.

  • Static libraries are linked into the target code when the program is compiled. So the program no longer needs the static library when it is running. Therefore, the compiled volume is relatively large.Starts with lib, ends with .a。

  • Dynamic libraries (also known as shared libraries) are not linked into the target code when the program is compiled, but are loaded when the program is running. So the program needs the dynamic library when it is running. Therefore, the compiled volume is relatively small.Starts with lib, ends with .so。

Static library

  • Steps to create a static library:

    1. Write or prepare the source code of the library

    2. Compile the source .c file to generate a .o file

    3. Use the ar command to create a static library

    4. Test the library file

Orphan process

Orphan process：After the parent process ends, the child process has not yet ended, this child process is called an orphan process.

Orphan processes are adopted by the init process(/sbin/upstart in ubuntu), and the init process is responsible for reclaiming its child process resources

Zombie process

Zombie process：After the child process ends, the parent process is still running but has not called wait() to reclaim the child process, this child process is called a zombie process

wait

The wait() function is generally used in the parent process to wait for the reclamation of child process resources, thereby preventing the generation of zombie processes.

ItemDescription
functionpid_t wait(int *status)
Header file#include <sys/wait.h>
Return valueReturns the PID of the reclaimed child process on success, and -1 on failure
Function and interpretationwaitis a system call in Unix/Linux systems used by a parent process to wait for a child process to terminate. It blocks the parent process until a child process exits, then reclaims the child process’s resources and returns its PID.statusThe pointer is used to store the exit status of the child process, which can be parsed throughWIFEXITED、WEXITSTATUSmacros to determine whether the child process exited normally or terminated abnormally and its specific exit code. This function is a key tool for process synchronization and resource reclamation, often used in scenarios where the parent process needs to wait for the child process to complete its task before continuing execution, such as in multiprocess programming where the parent process uniformly manages the lifecycle of child processes.

Two macro definitions related to the parameters of the wait function:
WIFEXITED(status): Ifthe child process exits normally, this macro evaluates to true
WEXITSTATUS(status): Ifthe child process exits normally, the value of this macro is the exit value of the child process。

Daemon process

A daemon process is a special type of process that runs in the background, used to perform specific system tasks. Many daemons start at system boot and run continuously until the system shuts down. Others start only when needed and end automatically after completing their tasks.

Daemons have no controlling terminal, so when certain situations occur, whether it’s general reporting information or urgent information that needs to be handled by an administrator, it needs to be output in some way. The Syslog function is the standard method for outputting this information, itsends the information to the syslogd daemon。

Basic requirements for a daemon process：

  1. Must be a child process of our init process (making the child process an orphan process)
  2. Do not interact with the controlling terminal

Manually creating a daemon process

Steps:

  1. First timefork(): Detach from the parent process
  • Purpose: Let the child process run in the background, allowing the parent process to return to the shell immediately.
  • The child process is no longer the process group leader (to prepare for the next stepsetsid).
  1. Callsetsid(): Create a new session
  • Function：
    • Detach from the controlling terminal;
    • No longer receive signals from the terminal (such asCtrl+C、SIGHUP）；
    • Become the new session leader and process group leader。

⚠️ Must ensure that the process callingsetsid()is not a process group leader, so the first stepfork()is necessary.

  1. Secondfork(): Ensure that the terminal cannot be reopened
  • Purpose: The newly forked child process is no longer the session leader。
  • According to POSIX regulations,only the session leader can reacquire the controlling terminal。
  • Therefore, the secondfork()the process after can never open a tty, completely becoming an ‘orphan’.
  1. Change the working directory to the root directory/
  • Reason: to prevent the daemon from occupying a mount point (such as/home), causing the file system to be unable to be unmounted (umountfailure).

Optional: You can also switch to a specific log or data directory, but you must ensure that the directory will not be unmounted.

  1. Close and redirect standard file descriptors (0,1,2)
  • Reason：
    • Inherited from the parent processstdin/stdout/stderrmay point to a terminal or pipe;
    • if not closed, it may cause resource leaks or unexpected output;
    • Redirect to/dev/nullcan avoid errors when library functions write (such asprintfwill not crash).
  1. Set the file permission maskumask(0)
  • Function: Clear the file creation mask, so that the permissions of subsequently created files/directories are completely specified by the program (such as0644、0755）。
  • making it easy to precisely control the permissions of logs, PID files, etc.

Inter-process communication

Inter-process communication is also widely used, such as data transfer between background processes and GUI interfaces, sending signals to shut down, terminating running programs with Ctrl+C, etc.

Anonymous pipe

Anonymous pipes are the oldest method of inter-process communication, with the following two characteristics:

  1. Can only be used for data interaction between related processes, such as parent-child processes, sibling processes, and descendant processes. The file node cannot be seen in the directory, and the read/write file descriptors exist in an int array.
  2. Can only transmit data in one direction, that is, after the pipe is created, one process can only perform read operations, and the other process can only perform write operations, and the byte order read out is the same as the order written.
ItemDescription
functionint pipe(int pipefd[2])
Header file#include <unistd.h>
Parameter pipefd [2]Integer array,pipefd[0]is the read-end file descriptor,pipefd[1]is the write-end file descriptor
Return valueReturns 0 on success, -1 on failure
Function and interpretationpipeis a system call used to create anonymous pipes in Unix/Linux systems. Anonymous pipes are a method of inter-process communication, suitable only for bidirectional communication between related processes (such as parent-child processes). After creation, one process can write data through the write end (pipefd[1]), and the other process reads data through the read end (pipefd[0]) reads data, achieving unidirectional data transmission (if bidirectional communication is required, two pipes need to be created). It is commonly used for command passing and data sharing between parent and child processes. For example, the parent process sends commands to the child process via the pipe, and the child process returns the results to the parent process via the pipe after execution.

Step：

  1. Call pipe() to create an unnamed pipe；
  2. fork() creates a child process, one process reads using read(), and one process writes using write()。

Note:The unnamed pipe must be created before the fork function!

Shared memory

Shared memory, as the name implies, allows two unrelated processes to access the same logical memory. Shared memory is a very effective way to share and pass data between two running processes. The memory shared between different processes is usually the same segment of physical memory. Processes can connect the same segment of physical memory to their own address spaces, and all processes can access the addresses in the shared memory. If a process writes data to the shared memory, the changes made will immediately affect any other process that can access the same segment of shared memory.

Characteristics：

  1. Fast speed, because shared memory does not require kernel control, so there are no system calls. Moreover, there is no process of copying data to the kernel, so the efficiency is the fastest compared to the previous ones, and it can be used for batch data transmission, such as images.
  2. No synchronization mechanism, requires the use of other tools provided by Linux for synchronization, usually using semaphores

Steps to use shared memory：

  1. Call shmget() to create shared memory segment id,
  2. Call shmat() to attach the shared memory segment identified by id to the process’s virtual address space,
  3. Access the mapped address space portion attached to the process, which can be read and written using IO operations.

Message queue

Three ipcs

The steps for the application layer to use IPC communication are

  1. Obtain the key value, the kernel will map the key value into an IPC identifier, common methods to obtain the key value:

    • Use the IPC_PRIVATE constant as the key value in the get call.
    • Use ftok() to generate a key.

  2. Execute the IPC get call, obtain the integer IPC identifier id through the key, each id represents an IPC object.

    • Message queue: msgget()
    • Shared memory: shmget()
    • Semaphore: semget()

  3. Access the IPC object through the id.

Characteristics of message queues：

  1. Sent messages are stored in the form of a linked list, equivalent to a list, processes can add and retrieve messages to/from the corresponding “list” based on the id.
  2. When a process receives data, it can retrieve data from the queue according to type.

Steps to use message queues：

  1. Create key;
  2. msgget() creates (or opens) a message queue object id using a key;
  3. Use msgsnd()/msgrcv() to send and receive;
  4. Delete the IPC object via msgctl()

After obtaining the id via the msgget() call, the message queue can be used to access the IPC object. Common message queue APIs are as follows:

Semaphore

To prevent a series of problems caused by multiple programs accessing a shared resource simultaneously, we need a method that authorizes access by generating and using tokens, allowing only one execution thread to access a critical section of code at any given time. A critical section refers to code that updates data and must be executed exclusively. Semaphores provide such an access mechanism, ensuring that only one thread accesses a critical section at a time. In other words, semaphores are used to coordinate processes’ access to shared resources.

Semaphores can only perform two operations: wait and signal, namely P(sv) and V(sv),

  • P(sv): If the value of sv is greater than zero, decrement it by 1; if its value is zero, suspend the execution of the process

  • V(sv): If other processes are suspended waiting for sv, resume their execution; if no processes are suspended waiting for sv, increment it by 1.
