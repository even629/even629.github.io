Experiment Content

bitXor

Problem:

Requires using only ~ (bitwise NOT) and & (bitwise AND) to implement XOR. You may use equivalence calculus from discrete mathematics. XOR means ‘same is 0, different is 1’:

x⊕y=¬(¬x∧¬y)∧¬(x∧y)x \oplus y = \lnot (\lnot x \land \lnot y) \land \lnot ( x \land y) x⊕y=¬(¬x∧¬y)∧¬(x∧y)

tmin

Problem:

Return the smallest two’s complement integer. The definitions of sign-magnitude, ones’ complement, and two’s complement are as follows:

RepresentationDefinition
Sign-MagnitudeThe highest bit is the sign bit (0 for positive, 1 for negative), and the remaining bits represent the absolute value of the number.
Ones’ ComplementThe ones’ complement of a positive number is the same as its sign-magnitude; for a negative number, the sign bit remains unchanged, and the remaining bits are inverted.
Two’s ComplementThe two’s complement of a positive number is the same as its sign-magnitude; for a negative number, it is its ones’ complement plus 1.

Two’s complement encoding is asymmetric; except for the smallest negative number, every negative number has a corresponding positive number. In a 64-bit system, the range of integer two’s complement representation is:

  • Minimum negative number: 0x8000000000000000 (i.e., -2^63).
  • Maximum positive number: 0x7FFFFFFFFFFFFFFF (i.e., 2^63 - 1)

isTmax

Problem:

  • Return 1 if x is the maximum two’s complement number, otherwise return 0.
  • The maximum value is 0x7FFFFFFF, characterized by the highest bit being 0 and the rest being 1. Then 0x7FFFFFFF + 0x1 becomes 0x80000000, with the highest bit being 1 and the rest being 0, which is exactly the opposite.
  • Here we need to use the property of XOR. For example, x ^ target: if x is inverted and XORed with target, since XOR yields 1 for different bits and 0 for same bits, only when x and target are exactly the same will x ^ target be 0x0. Then adding the ! operator gives !(x ^ target), which is 1 when x and target are the same, and 0 otherwise.
    For 0x7FFFFFFF, adding 1 gives 0x80000000. Since the characteristics are opposite, inverting it gives 0x7FFFFFFF again. So we can use XOR to check if they are the same: !(~(x+1) ^ x).
  • Continuing to think, is 0x7FFFFFFF the only number that satisfies the property that adding 1 is equivalent to bitwise inversion? Obviously, 0xFFFFFFFF also satisfies this property because when overflow occurs, the highest bit is discarded and becomes 0x0. So we need to construct an expression to AND with !(~(x+1) ^ x), call it expression1, i.e., !(~(x+1) ^ x) & expression1. When x is 0x7FFFFFFF, expression1 is 1; when x is 0xFFFFFFFF, it is 0.
  • The method to exclude must use a characteristic that 0x7FFFFFFF has but 0xFFFFFFFF does not, i.e., the difference between these two numbers. The simplest difference is that 0x7FFFFFFF + 0x1 becomes 0x80000000, a non-zero number, while 0xFFFFFFFF + 0x1 becomes 0. So !(x+1) is 0 when x is 0x7FFFFFFF and 1 when x is 0xFFFFFFFF. !(x+1) is exactly the opposite of what we need, so we add another ! operation, i.e.,

!( ~(x + 1) ^ x) & !!(x + 1)

allOddBits

Problem:

  • Return 1 if all odd-numbered bits are 1, otherwise return 0. That is, if a number has bits 1, 3, 5, 7… all set to 1 (regardless of other bits), return 1; otherwise return 0.
  • We can remove the influence of all even-numbered bits and look at the odd-numbered bits. For example, AND x with 0xAAAAAAAA, then all even-numbered bits become 0, and check if it equals 0xAAAAAAAA (using the method from isTmax of checking equality via bitwise AND). Alternatively, OR x with 0x55555555, then all even-numbered bits become 1, and check if it equals 0xFFFFFFFF. That is:

!((x|0x55555555)^0xFFFFFFFF)

!((x&0xAAAAAAAA)^(0xAAAAAAAA))

  • However, due to the constraints of the experiment, we cannot directly write 0xAAAAAAAA; we can only write 0xAA and obtain 0xAAAAAAAA through bitwise operations.

Note: According to the C standard, the precedence of shift operations is lower than addition and subtraction.

negate

Problem:

  • Return the negative of x
  • We know:

x+(-x)=0

x+(~x)=-1

  • Subtracting the two expressions gives:

-x=~x+1

isAsciiDigit

Problem:

  • Check if x is an ASCII digit, simply put, determine whether x is between 0x30 and 0x39.
  • In assembly, comparing sizes is done by subtraction and setting flags. Therefore, we can subtract x from 0x30 and 0x39 respectively and check the sign bit: 0x30 - x should be less than 0 with sign bit 1, and 0x39 - x > 0 with sign bit 0. However, if x is at the boundary, e.g., x == 0x30, then 0x30 - x is 0 with sign bit 0, which also satisfies the requirement. So we can change <= to <:

0x30<= x <= 0x39

The above expression is equivalent to:

0x2F< x <= 0x39

Then, does the right side x < 0x39 need an equivalent transformation? 0x39 - x should be greater than or equal to 0, with sign bit 0; 0x39 - x < 0 has sign bit 1. After analysis, the boundary condition is included, so no equivalent transformation is needed (transformation would exclude the boundary condition).
Let the sign bit of 0x2F - x be 1, and the sign bit of 0x39 - x be 0; both conditions must be satisfied simultaneously. First, consider 0x2F < x. How to extract the sign bit after subtraction? This can be done via arithmetic right shift, shifting 31 bits so the sign bit fills the entire 32 bits. That is, if 0x2F - x < 0, then (0x2F + (~x + 1)) >> 31 is 0xFFFFFFFF; if 0x2F - x >= 0, (0x2F + (~x + 1)) >> 31 is 0;

(0x2F+(~x+1))>>31

Now consider x <= 0x39. Using the same method, if 0x39 - x < 0, then (0x39 + (~x + 1)) >> 31 is 0xFFFFFFFF; if 0x39 - x >= 0, (0x39 + (~x + 1)) >> 31 is 0; we can negate it to satisfy the condition.

~((0x39+(~x+1))>>31)

Both conditions must be satisfied simultaneously, so

(0x2F + (~x + 1)) >> 31 & ~((0x39 + (~x + 1)) >> 31)

Since the final return is 1 or 0, we just take the least significant bit:

(0x2F + (~x + 1)) >> 31 & ~((0x39 + (~x + 1)) >> 31) & 0x1

conditional

Problem:

  • The expression for conditional is the same as x ? y : z
  • To achieve this kind of decision of returning y or z based on the value of x, using only bitwise operations is very difficult, because although we can easily determine whether x is 0, it is hard to associate x with both y and z simultaneously. So we turn to addition operations, assuming we return an expression like:

a+y+z

When x!=0, let a=-z=~z+1; when x==0, let a=-y=~y+1 to meet the requirement. But this requirement is also hard to satisfy because a is related to both y and z, and a is determined by whether x is 0. Therefore, we consider splitting a:

b+c+y+z

When x is non-zero, let b=0, c=-z=~z+1; when x is 0, let b=-y=~y+1, c=0. This way, b is only related to y, and c is only related to z.
First, we need to determine whether x is 0. This is not difficult; we can use the ! operator. When x is non-zero, !x is 0; when x is 0, !x is 1. Let’s first focus on b. When !x==0, b should be 0; when !x!=0, b should be ~z+1. Obviously, the AND operation can satisfy this:

b = !x&(~y+1)

But when !x!=0, !x1, and its bitwise AND with ~z+1 only retains the lowest bit, which is not necessarily ~z+1. Therefore, we need to get 0xFFFFFFFF when x!=0, and 0 when x0. So we define a:

a = ~(!x)+1

When x is 0, !x1, ~(!x)0xFFFFFFFE, a0xFFFFFFFF; when x is non-zero, !x0, ~(!x)0xFFFFFFFF, a0. Therefore, the expression for b should be:

b = a&(~y+1)

Now consider c. The situation for c is exactly opposite to b, so we just need to add another ~ to a.

c = ~a&(~z+1)

There is no need to worry about addition overflow here.

isLessOrEqual

Problem:

  • If x<=y, return 1; otherwise, return 0.
  • Comparison uses subtraction. Let y-x. If y-x>=0, meaning the sign bit is 0, return 1; if y-x<0, meaning the sign bit is 1, return 0. y-x = y + ~x + 1. Then shift right by 31 bits to get the sign bit. We need a negation operation so that when the sign bit is 0, we return 1, and when it is 1, we return 0:

~((y+(~x+1))>>31) & 0x1

logicalNeg

Problem:

  • Implement the ! operator, i.e., for !x, return 1 when x is 0, and return 0 when x is non-zero.
  • The most intuitive difference between 0 and non-zero is that every bit of 0 is 0, while non-zero has at least one bit that is 1. But since we cannot use cyclic shift operations to determine this, we have to find another way.
  • Another difference between 0 and non-zero is that the negative of 0 is still 0, i.e., ~0+1=0, while the negative of a non-zero number is not 0. We can use this by checking whether the negative of x is the same as x itself. Earlier, we used a trick with XOR to determine if two numbers are the same: this expression returns 0 when x equals 0, and returns non-zero when x is non-zero, but that essentially does nothing.

(~x+1)^x

We can use another point: the sign bit of a non-zero number’s negation is definitely opposite to that of the original number, one must be 1 and the other 0, while the sign bit of 0’s negation and the sign bit of 0 are both 0. Therefore, we only need to AND the two to get: when non-zero, the sign bit of the expression is 1; when zero, the sign bit of the expression is 0.

(~x+1)|x

Next is to extract the sign bit and invert it, because we need to return 1 when it is 0 and return 0 when it is 1. We used this trick in the previous problem: that is, arithmetic right shift it by 31 bits and invert, then AND with 0x1 to get the lowest bit:

~(((~x+1)|x)>>31) & 0x1

howManyBits

Problem:

  • Return the number of bits needed to represent x in two’s complement form. Essentially, it is to find the position of the highest 1 bit in the number, then add 1 for the sign bit. This problem is a bit difficult, mainly because it is hard to think of. Below is someone else’s implementation method.
  • Here we use the binary search method:
    First consider positive numbers and 0
  1. High 16 bits: Check if the high 16 bits have a 1; if so, at least 16 bits are needed.
  2. High 8 bits: In the remaining 16 bits, check if the high 8 bits have a 1.
  3. High 4 bits: In the remaining 8 bits, check if the high 4 bits have a 1.
  4. High 2 bits: In the remaining 4 bits, check if the high 2 bits have a 1.
  5. High 1 bit: In the remaining 2 bits, check if the high 1 bit has a 1.
  6. Lowest bit: The last remaining 1 bit.
    Finally, add up the bits from all parts and add 1 (the sign bit).
  • Handling negative numbers:
    flag is x >> 31, i.e., the sign bit. If x is negative, flag is 1; otherwise, it is 0.
  1. If x is negative (flag == 1), then x is inverted: x = ~x.
  2. If x is non-negative (flag == 0), then x remains unchanged.
    In the two’s complement representation of a negative number, the sign bit and the magnitude part are mixed together. If you directly calculate the number of significant bits of a negative number, the sign bit will cause an incorrect result. For example:
    The two’s complement of -1 is 11111111 11111111 11111111 11111111. If you directly calculate the number of bits, you would get 32 bits, but in reality we only care about its significant bits.
    Through the inversion operation, the two’s complement representation of a negative number is converted to a positive number, and the number of significant bits in its binary representation is the same as that of the original negative number. For example:
  • The two’s complement of -1 is 11111111 11111111 11111111 11111111. After inversion, it becomes 00000000 00000000 00000000 00000000, with 1 significant bit.
  • The two’s complement of -2 is 11111111 11111111 11111111 11111110. After inversion, it becomes 00000000 00000000 00000000 00000001, with 2 significant bits.

floatScale2

Problem:

  • Multiply a single-precision floating-point number (float) by 2 and return the binary representation of the result. The function’s input and output are both of type unsigned int, but they actually represent the bit pattern of a single-precision floating-point number. If the input is NaN, simply return the input. Also, finally we can use if and while 😭😭😭

  • First, let’s reviewIEEE 754Knowledge about floating-point numbers:

For floating-point numbers represented by IEEE 754, there are three categories in total

Denormalized numbers are used to represent extremely small values near zero, preventing floating-point underflow and ensuring continuity of floating-point operations.

First, we extract the sign bit, mantissa frac, and exponent exp:

sign=uf>>31&0x1

frac=uf&0x7FFFFF

exp = (uf&0x7f800000)>>23

Determine whether the floating-point number is a normalized number, denormalized number, or a special case based on the exponent.

floatFloat2Int

Problem:

  • Represent the input IEEE 754 floating-point number as an int; if it exceeds the representable range, return 0x80000000u.
  • If it is a denormalized number, return 0 directly; if it is a special case, return 0x80000000u directly.
  • If it is a normalized number, we first need to determine whether it exceeds the int representation range. An int has 32 bits, with one bit for the sign, a maximum of 2^31, and a minimum of -(2^31+1). Therefore, the exponent E cannot exceed 30 (when E is 31, shifting left by E bits would overwrite the sign bit). When the exponent E is less than 0, it indicates a number less than 1, so return 0 directly. Also:
  • sign 1 bit
  • exp 8 bits
  • frac 23 bits

V=(−1)sign×2exp−127×(1+frac)V = (-1)^{sign} \times 2^{exp - 127} \times (1 + frac) V=(−1)sign×2exp−127×(1+frac)

We can view it as1frac(The 24th bit is 1, bits 1~23 form frac) At this point, it is equivalent to1.frac(The integer part is 1, the fractional part is frac) shifted left by 23 bits. At this point, the exponent is E, meaning multiply by 2^E, i.e., shift left by E bits. Therefore, when E is less than 23, we should shift right by (23-E) bits to truncate the (23-E) bits after frac; when E is greater than 23, since we have already shifted left by 23 bits, we only need to shift left by (E-23) bits.

floatPower2

Problem:

  • Compute 2.0^x and return the unsigned representation under IEEE 754. If the result is too small to be represented even as a denormalized number, return 0; if the result is too large, return +INF. Clearly, this problem mainly tests the representation range of IEEE 754.

ForSingle-precision float

  • Sign bit: 1 bit
  • Exponent exp: 8 bits
  • Mantissa frac: 23 bits

Representation range

  • Normalized numbers: exponent exp is neither all 0s nor all 1s,

V=(−1)S×2E×M=(−1)sign×2exp−bias×(1+frac)V = (-1)^S \times 2^E \times M = (-1)^{sign} \times 2^{exp - bias} \times (1 + frac) V=(−1)S×2E×M=(−1)sign×2exp−bias×(1+frac)

exp range is 1 to 254, E = Exp - Bias = Exp - 127, E range is -126 >= E >= 127; for frac, minimum is 0, maximum is 1 - 2^(-23), therefore, without considering S

Vmax=(−1)S×2127×(2−2−23)=(−1)S×2127×(2−2−23)V_{max} = (-1)^S \times 2^{127} \times (2-2^{-23}) = (-1)^S \times 2^{127} \times (2-2^{-23}) Vmax​=(−1)S×2127×(2−2−23)=(−1)S×2127×(2−2−23)

Vmin=(−1)S×2−126×(1+0)=(−1)S×2−126V_{min} = (-1)^S \times 2^{-126} \times (1+0) = (-1)^S \times 2^{-126} Vmin​=(−1)S×2−126×(1+0)=(−1)S×2−126

  • Denormalized numbers: exponent is all 0s

V=(−1)S×2E×M=(−1)sign×21−Bias×(0+frac)V = (-1)^S \times 2^E \times M = (-1)^{sign} \times 2^{1-Bias} \times (0 + frac) V=(−1)S×2E×M=(−1)sign×21−Bias×(0+frac)

Exponent exp is all 0s, E = 1 - Bias = 1 - 127 = -126, therefore:

V=(−1)sign×2−126×(0+frac)V = (-1)^{sign} \times 2^{- 126} \times (0 + frac) V=(−1)sign×2−126×(0+frac)

For frac, minimum is 0, but the smallest non-zero representable is 2^(-23), values from 0 to 2^(-23) cannot be represented, maximum is 1 - 2^(-23), therefore, when frac is 0, value is 0, but to calculate the smallest non-zero representable, set frac = 2^(-23), without considering S

Vmax=(−1)S×2−126×(0+1−2−23)=(−1)S×2−126×(1−2−23)V_{max} = (-1)^S \times 2^{-126} \times (0+1-2^{-23}) = (-1)^S \times 2^{-126} \times (1-2^{-23}) Vmax​=(−1)S×2−126×(0+1−2−23)=(−1)S×2−126×(1−2−23)

  • Special values
  1. +infinity: all exponent bits are 1, frac is 0, S is 0
  2. -infinity: all exponent bits are 1, frac is 0, S is 1
  3. NaN: all exponent bits are 1, frac is not 0

Therefore, based on the above analysis:

  • Return +NaN when x > 127
  • Normalized number when -126 <= x <= 127
  • Denormalized number when -149 <= x < -126
  • When x < -149, too small to represent, return 0
