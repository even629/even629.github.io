SoC boot process

This article introduces the complete boot process of an SoC from power-on to loading the operating system, detailing the mechanism by which BootROM initializes hardware and loads programs into internal SRAM via Bootstrap Pins. To address the limited capacity of internal SRAM, the article further discusses the uboot relocation and SPL scheme, and takes Rockchip as an example to analyze the division of responsibilities: TPL is responsible for system hardware initialization, and SPL is responsible for loading uboot and trust images.

BootROM

Generally speaking, SoC manufacturers build a ROM inside the SoC. This ROM is very small and contains the power-on boot code (once solidified, it can never be changed, as it is built into the chip during manufacturing). This part of the code is called BootROM, also known asfirst-stage bootloader。

After power-on, the first to take over the system is the SoC manufacturer’s BootROM, which initializes the hardware and then loads the program into SRAM.

Initialize hardware

  • CPU configuration
  • Disable watchdog
  • Initialize clock
  • Initialize some peripherals (such as USB Controller, MMC Controller, NAND Controller, etc.);

Load program into SRAM

A new SoC generally willexternally attach some storage devices(eMMC, NAND, NOR, SDCard, etc.) and memory (SDRAM, DDR, etc.).

Then we need to burn a program for the CPU to execute. Burning a program is essentially writingexecutable binary filetoexternal storage deviceon (eMMC、Nand、SDetc.). When the system powers on and starts, it willread them into memory and execute。

After power-on, the first to take over the system is the SoC manufacturer’s BootROM. Other executable programs (u-boot, Kernel) are placed (burned) onto external storage. Therefore, in addition to initializing the hardware environment, the BootROM code also needs to read the next executable program from external storage into memory for execution.

Since it is read into memory for execution, can this memory be our onboard DDR?

Theoretically yes, but the DDR controller designed by the SoC manufacturer generally supports many types of DDR devices and provides a compatibility list. How could the SoC manufacturer know exactly which memory the user has used on the PCB?

Therefore, directly reading the external executable program into DDR is obviously not very friendly, so generally,SoCs typically include a small internal SRAM ，BootROM reads the external executable program from external storage and places it in SRAM for execution；

So from which specific memory does BootROM read the binary file?

SoC manufacturers generally support multiple boot modes, such as:

  • Read from eMMC
  • Read from SDCard
  • Read from Nand Flash, etc.

When powering on, it needs to be told which peripheral to use to read the subsequent boot binary file;

A common design approach is to have a set of Bootstrap Pins. When powered on, BootROM samples the levels of these IOs to determine which external memory to load the subsequent executable from. For example, with 2 IOs, 2’b00 means boot from Nand, 2’b01 means boot from eMMC, 2’b10 means boot from SDCard, etc.;

After BootROM reads these values, it initializes the corresponding peripheral and then reads the code to be executed. These IOs are typically implemented as onboard DIP switches to adjust the chip’s boot mode;

Here, when reading the burned binary, some details need attention. For example, the SoC manufacturer tells you that you need to initialize the SDCard with a certain file system, then put the data in for it to work; becauseA file system is a way to organize files, not a raw partition; youput it in according to file system A, and then the SoC’s BootROM reads it according to file system A, so that they are consistent；

SPL

After the chip is powered on, BootROM determines from which memory to read the executable binary file into SRAM and execute it based on the Bootstrap Pins. In theory, this binary file could be ouru-boot.binfile; that is, BootROM directly loadsu-boot.bin；

In theory, yes, but there is a problem: SRAM is expensive. Generally, the on-chip SRAM of SoCs is not large, typically 4KB, 8KB, 16KB… up to 256KB. However, u-boot compiled is very large, several hundred KB, and cannot fit.

What to do if it doesn’t fit? There are two methods:

  • Assume the on-chip SRAM is 4KB,The first 4KB of u-boot implements u-boot’s relocation, that is,Copy u-boot to SDRAM to run；
  • Make a smaller boot program,First let BootROM load this small program, then this small boot loads u-boot;

Method 1: u-boot relocation

For example, our u-boot is 400KB, SRAM is 4KB, external SDRAM is 64MB;

If using the first scheme, the first 4KB of uboot is loaded into SRAM for execution, uboot is truncated, we need toensure that in the first 4KB code of uboot, the onboard SDRAM is initialized, the entire uboot is copied to SDRAM, and then jump to SDRAM for execution；

Scheme 2: SPL

For the second scheme, we make a small uboot, this uboot is calledSPL（Secondary Program Loader), it is very very small (smaller than SRAM size), itis first loaded into SRAM by BootROM to run, the main thing SPL does is toinitialize the memory controller, then read the real large u-boot from external storage into SDRAM, and then jump to the large uboot。

As shown in the figure above:

  1. After power-on, BootROM starts executing, initializes the clock, disables the watchdog, disables Cache, disables interrupts, etc., determines the boot device according to the Bootstrap Pin, and initializes peripherals;
  2. Use peripheral drivers to read SPL from memory;

---------------- The above part is the responsibility of the SoC manufacturer, the following is what the user needs to do ----------------

  1. SPL is read into SRAM for execution, at this point, control is transferred to our SPL;
  2. SPL initializes external SDRAM;
  3. SPL uses drivers to read uboot from external storage and place it in SDRAM;
  4. Jump to uboot in SDRAM for execution;
  5. Load the kernel;

In actual situations, many issues need to be noted:

  • Link address during compilation, does it need to be position-independent?
  • Is there any overlap between SPL code and uboot code? If so, does it mean that what SPL has executed will be executed again when jumping to uboot?
  • In specific cases, which hardware needs to be configured? How to configure?

Rockchip Boot Process

For different solutions, Rockchip provides two different bootloader methods, and their steps and generated image files are completely different.

  • TPL/SPL Loading: Using the TPL/SPL provided by Rockchip official is the small uboot we mentioned above), this method is completely open source;
  • Official firmware loading: Using Rockchip idbLoader, it consists ofRockchip rkbin projectofRockchip ddr init binandminiloader bincombined, this method is not open source;

Above we introduced SPL, so what is TPL? Actually, what we said aboveThe part of SPL that initializes hardware such as SDRAM is separated out, that is TPL. So let’s summarize:

  • TPLisTarget Program Loader, that ischip-level initialization process, at this time the code is based on the chip platform part, it performs DDR initialization and some other system configurations during the boot process, so that the subsequent SPL can run correctly;
  • SPLisSecondary Program Loader, it reads from the storage devicereads trust (such as ATF/OP-TEE) and uboot binary files, loads them into system memory and runs them, thereby starting the complete operating system；

The difference between TPL and SPL is that their responsibilities are different. TPL is mainly responsible for initializing system hardware, while SPL is responsible for loading and running other software components, such as trust and uboot.

In addition, in some special cases, such as encrypted boot or secure boot mode, TPL may also perform other additional tasks.

Boot stages

The boot stage involves multiple image files:

  • BootROM in stage one is provided by the SoC manufacturer;
  • Stage two method needs to provide aidbloader.img；
  • Stage three is actually the uboot image file, there are two types: uboot.img (also needs to be paired with trust.img)andu-boot.itb (this is because it has already packaged ATF into it); besides both containing the original u-boot.bin binary file, these two files also contain some other things, which can beidbloader.imgrecognized, then loaded;
  • Stage four and stage five are the kernel image and root file system;

When we discuss booting from eMMC/SD/USB drive/network, they involve different concepts:

  • The first stage is always in BootROM, it loads the second stage and possibly the third stage (when SPL is enabled_BACK_when the TO_BROM option is selected);
  • Booting from SPI flash means the second and third stage firmware (only SPL and U-Boot) are in SPI flash, and the fourth/fifth stages are elsewhere;
  • Booting from eMMC means all firmware (including second, third, fourth, and fifth stages) are in eMMC;
  • Booting from SD Card means all firmware (including second, third, fourth, and fifth stages) are in SD Card;
  • Booting from USB drive means the fourth and fifth stage firmware (excluding SPL and U-Boot) are on the disk, optionally only the fifth stage;
  • Booting from Net/TFTP means the fourth and fifth stage firmware (excluding SPL and U-Boot) are on the network.

idbloader.img

The idbloader.img file is a Rockchip-format preloader that works when the SoC boots. It contains:

  • IDBlock header known by the Rockchip BootROM;
  • DDR initialization program, loaded by BootROM into SRAM, running inside SRAM;
  • Next-stage loader, loaded by BootROM and running on DDR;

u-boot.img

u-boot.bin is the raw binary image generated after compiling the U-Boot source code, which can be directly flashed into the device’s flash memory. u-boot.img is generated bymkimagethe tool adding a header to u-boot.bin. This header may also include some additional data, such as boot parameters and kernel image addresses.

Therefore, byusing u-boot.img instead of u-boot.bin, the boot ROM can more easily recognize the U-Boot image and better guide U-Boot to boot correctly on the device.

u-boot.itb

u-boot.itb is actually another variant of u-boot.img, also built by mkimage. In addition to the two files compiled from U-Boot source code, u-boot.dtb and u-boot-nodtb.bin, it also contains ARM Trusted Firmware such as bl31.elf, bl32.bin, tee.bin. Among them, bl31.elf is mandatory, while bl32.bin and tee.bin are optional and can be omitted.

trust.img

ARM64 SoCs also need to compileATF (ARM Trust Firmware)，ATF is mainly responsible for switching the CPU from secure EL3 to EL2 before booting U-Boot，then jumps to U-Boot, and after the kernel starts, it is responsible for starting other CPUs。

ATF completely and uniformly divides the system boot from the lowest level, placing the secure monitor functionality in bl31. Thus, after the system fully boots, when an SMC or other interrupt is triggered in the CA or TEE OS, it first traverses the corresponding services registered in bl31 to determine the specific handler. Note 1: If loader1 has multiple stages, the program will return to BootROM, which will load and run the next stage. For example, if loader1 is TPL and SPL, BootROM will first run to TPL, TPL initializes DDR and returns to BootROM, then BootROM will load and run to SPL.

Note 2: If trust is enabled, in secure mode (EL3 in armv8), loader1 needs to load both trust and U-Boot, then run to trust, which initializes in non-secure mode (EL2 in armv8) and runs to U-Boot.

Note 4: In boot.img, the content can be Linux zImage and its dtb, optionally grub.efi, or it can be AOSP boot.img, with ramdisk optional.
