This article introduces the exception model of the ARM64 architecture, detailing the exception levels, terminology definitions, and the classification of synchronous and asynchronous exceptions. It also summarizes the hardware and OS processing mechanisms during exception entry and return, and explains the exception routing rules based on system configuration registers.

Reference documents:

Arm Architecture Reference Manual Armv8

https://developer.arm.com/documentation/ddi0487/fc/

ARM64 Exception Levels (Exception Level)

  • EL0 Unprivileged mode, e.g., application
  • EL1 Privileged mode, e.g., OS kernel
  • EL2 Hypervisor, e.g., hypervisor
  • EL3 Secure mode, e.g., secure monitor

Terminology (Exception terminology)

  • Taking an exception: Taking an exception
  • Returning from an exception: Returning from an exception
  • Exception levels: Exception level
  • Precise exception: Precise exception
  • Synchronous and asynchronous exception: Synchronous and asynchronous exceptions

Exception types

  • Synchronous exception
    • System calls, SVC, HVC, SMC, etc.
    • MMU-induced exception
    • SP and PC alignment check
    • Unallocated instruction
      • Unallocated instruction opcode
      • Instructions requiring a higher privilege level than the current exception level.
      • Disabled instructions.
      • Any instruction when the PSTATE.IL field is set
    • Debug exception
  • Asynchronous exception
    • IRQ interrupt
    • FIQ interrupt
    • SError(System error)

A synchronous exception is an expected exception, while an asynchronous exception is an unexpected exception

Exception Entry

Process state or PSTATE is an abstraction of process state information

  • When an exception occurs,CPU hardwarewhat does it do
    • PSTATE is saved to SPSR_ELx (The Saved Program Status Register)
    • The return address is saved to ELR_ELx
    • The DAIF fields in the PSTATE register are all set to 1, which is equivalent to disabling debug exceptions, system errors (SError), IRQ interrupts, and FIQ interrupts
    • The ESR_ELx (Exception Syndrome Register) is updated, which contains the cause of the synchronous exception or SError
    • SP is switched to SP_ELx
    • Switches to the corresponding EL and then jumps to the exception vector table for execution
  • After an exception occurs,the operating systemwhat does it need to do

The operating system needs to set up the exception vector table so that the CPU, based on the type of exception that occurred, will jump to the appropriate entry in the exception vector table

Each entry in the exception vector table stores a jump function for exception handling, then jumps to the appropriate exception handler to process the exception

The Exception Syndrome Register (ESR_ELn) contains information that allows the exception handler to determine the cause of the exception. It is updated only for synchronous exceptions and SError, not for IRQ or FIQ, because these interrupt handlers typically obtain status information from the registers of the Generic Interrupt Controller (GIC).

Exception return

  • The operating system executes aneretinstruction
    • Restore the PC pointer from the ELR_ELx (Exception Link Register)
    • Restore the processor state from the SPSR_ELx (Saved Program Status Register)

Exception return address

  • Two registers for the return address
    • x30: Return address of the subroutine. Use the ret instruction to return
    • ELR_ELx: Exception return address. Use the eret instruction to return
  • ELR_ELxThe register saves the exception return address
    • For asynchronous exceptions, the return address is the next instruction at the time the interrupt occurred, or the first instruction that was not executed
    • For synchronous exceptions other than system calls, the return address is the instruction that triggered the synchronous exception
    • For a system call, it returns the instruction following the svc/hvc instruction

Asynchronous exception (interrupt)

The hardware has set the ELR to the next instruction to be executed，eretjust go back and continue execution, no additional modification needed

Synchronous exception (non-system call)

The hardware sets the ELR to the instruction that triggered the exception
If you fix the cause of the exception (e.g., page fault), just eretit will re-execute this instruction
If you want to skip it (no retry), you need to manuallyELR_ELx += 4

system call（svc）

The hardware sets the ELR to the instruction after the svc, soeretit will directly return to the next one

Exception handling routing

When an exception occurs at a specific ‘Exception Level (EL)’, which Exception Level should the CPU jump to handle it

  • When an exception occurs, it can be handled atthe current EL or at a higher EL
    • EL0 cannot be used to handle exceptions
    • Synchronous exceptions can be handled at the current EL, for example, when a synchronous exception occurs at EL1
    • Forasynchronous exceptions, they can be routed to EL1, EL2, or EL3 for handling, configuration is requiredHCR(Hypervisor Configuration Register) andSCR(Secure Configuration Register) related registers
    • If the exception is generated due to an instruction fetch at EL0, it is treated as an exception of EL1, unless HCR_EL2.TGE is set to 1 and the processor is in the Non-secure state, in which case it is treated as an exception of EL2.

Refer to table D1-10 of armv8.6

  • SCR_EL3: Secure Configuration Register
  • HCR_EL2: Hypervisor Configuration Register

The table in the image (Table D1-10) isthe interrupt/exception routing rule table when EL3 (Secure Monitor mode) and EL2 (Hypervisor mode) are implemented simultaneously in the ARM architecture, which is primarily used to define the target privilege level (EL0~EL3) that the processor ultimately jumps to after an interrupt (IRQ/FIQ) or exception (such as Abort) is triggered under different system states (register configurations).

  1. Row 1: SCR=0, NS EEL2a=0, EA IRQ FIQ=0, RW=0
  • Input conditions: Security configuration is default (SCR=0), Non-secure EL2 exception entry is disabled (NS EEL2a=0), no active interrupt/exception (EA IRQ FIQ=0), read/write permission is 0; HCR TGE, E2H are don’t care (X).
  • Routing result：
    • Triggered from EL0: Jumps to “FIQ IRQ Abt” (i.e., the native processing entry that triggers the corresponding interrupt/exception, not intercepted by a higher EL);
    • Triggered from EL1: Same as EL0, jumps to “FIQ IRQ Abt”;
    • Triggered from EL2:n/a(Not applicable, as EL2 is not enabled for this scenario when NS EEL2a=0);
    • Triggered from EL3: Jumps to “C” (usually referring to the “exception handler entry in Secure state”, in the ARM architecture “C” often represents the default target related to security).
  1. Row 2: SCR=0, NS EEL2a=0, EA IRQ FIQ=0, RW=1
  • Input conditions: Only RW (read/write permission) changes from 0 to 1, the rest is the same as Row 1.
  • Routing result：
    • Triggered from EL0/EL1: Jump to “EL1” (i.e., the exception is intercepted and handled by EL1 (kernel mode), rather than the native interrupt entry, because RW=1 corresponds to the kernel processing logic with higher privileges);
    • Triggered from EL2:n/a(Same as row 1, EL2 is not enabled);
    • Triggered from EL3: Jump to “C” (the secure state entry remains unchanged).
  1. Row 3: SCR=0, NS EEL2a=0, EA IRQ FIQ=1
  • Input conditions: EA IRQ FIQ=1 (active interrupt/exception triggered), other configurations are the same as the previous two rows; HCR TGE, E2H are arbitrary values.
  • Routing result：
    • Triggered from EL0/EL1: Jump to “EL3” (active interrupt/exception is intercepted by the highest security level EL3, which conforms to the logic in the ARM security architecture that “high-priority exceptions are handled by EL3”);
    • Triggered from EL2:n/a(EL2 is not enabled);
    • Triggered from EL3: Jump to “EL3” (exceptions triggered by EL3 itself are handled within EL3, without jumping to other levels).
  1. Row 4: SCR=0, NS EEL2a=1, EA IRQ FIQ=0, HCR TGE=0, E2H=0, RW=0
  • Input conditions: NS EEL2a=1 (Non-secure EL2 exception entry enabled), no active interrupts/exceptions (EA IRQ FIQ=0), virtualization mode off (HCR TGE=0), EL2 does not trap EL1 exceptions (E2H=0).
  • Routing result：
    • Triggered from EL0: jump to “FIQ IRQ Abt” (native interrupt entry);
    • Triggered from EL1: jump to “FIQ IRQ Abt” (EL2 does not trap, because E2H=0);
    • Triggered from EL2: jump to “C” (exception triggered by Non-secure EL2, routed to Secure state entry “C”);
    • Triggered from EL3: jump to “C” (processed within Secure state, target remains unchanged).

Stack selection

  • Each exception level EL has a corresponding stack pointer register SP_ELx
    • SP_EL0, SP_EL1, SP_EL2, SP_EL3
  • The stack must be 16-byte aligned. Hardware can detect whether the stack pointer is aligned
  • When an exception occurs and jumps to the target exception level, the hardware automatically selects SP_ELx
  • The operating system is responsible for allocatingand ensuring that the stack corresponding to each exception level EL is available

Execution mode

Execution mode of exception handling

  • When an exception occurs, it switches to a higher-level EL. Which mode does this EL run in? AArch64 or AArch32
    • HCR_EL2.RW（Hypervisor Configuration Register） Records which mode EL1 is to run in
      • 1 indicates aarch64
      • 0 indicates aarch32

  • After an exception occurs, the execution mode can change
    • An aarch32 application is running, and an interrupt comes in; it might jump to EL1 in aarch64 execution state to handle this interrupt

Execution mode for exception return

  • When returning from an exception, the SPSR register (Saved Program Status Register) records:
    • Return to which EL? SPSR.M[3:0]
    • Execution mode of the target EL for return?SPSR.M[4]
      • 0 indicates aarch64
      • 1 indicates aarch32

Exception Vector Table (Exception vectors)

  • Each exception level EL has its own exception vector table, except for EL0

  • The base address of the exception vector table needs to be set in the VBAR_ELx（Vector Base Address Register） register

  • VBAR_EL1 register:

    Bits 0~10 are reserved, therefore theaddress is 2K aligned

  1. Row (Exception taken from)

    Defines the “current execution state” when an exception is triggered, divided into 4 scenarios:

    • Current Exception level with SP_EL0: The current exception level uses theSP_EL0stack pointer (e.g., when an exception is triggered while using the user-level stack pointer in EL1).
    • Current Exception level with SP_ELx, x>0: The current exception level uses theSP_ELx(x≥1, e.g., for EL1SP_EL1, for EL2SP_EL2and other kernel-level stack pointers) an exception is triggered.
    • Lower Exception level（AArch64）: Triggered from a lower exception level, and the “lower level” is in the AArch64 execution state (e.g., when switching from EL0/EL1 to a higher EL, and the original level was in 64-bit mode).
    • Lower Exception level（AArch32）: Triggered from a lower exception level, and the “lower level” is in the AArch32 execution state (e.g., when switching from EL0/EL1 in 32-bit mode to a higher EL).

  2. Column (Offset for exception type)

    Classified by exception type, corresponding to different offsets:

    • Synchronous: Synchronous exception (e.g., instruction execution errors, software-triggered exceptions, etc., occurring synchronously with the instruction stream).
    • IRQ or vIRQ: Normal interrupt (IRQ) or virtual interrupt (vIRQ, in virtualization scenarios).
    • FIQ or vFIQ: Fast interrupt (FIQ, usually with higher priority) or virtual fast interrupt (vFIQ, in virtualization scenarios).
    • SError or vSError: System error (SError, such as bus errors and other severe errors) or virtual system error (vSError, in virtualization scenarios).

  3. Offset values (e.g., 0x000, 0x080, etc.): Based on the combination of row (trigger state) and column (exception type), it provides the offset relative to the vector table base address. The processor can find the entry address of the exception handler via base address + offset.

  • Overall structure of the exception vector table

In AArch64 EL1 and above execution levels, the exception vector table (Vector Table) is stored at VBAR_ELx (Vector Base Address Register) register.

This table contains a total of 4 “blocks”, corresponding to different exception sources:

  1. From the current SP (SP0) triggered exceptions
  2. From the current SP (SPx) triggered exceptions (x≠0, e.g., SP_EL1)
  3. From a lower privilege level using AArch64 (e.g., EL0) exceptions
  4. From a lower privilege level using AArch32 (e.g., EL0) exceptions

Each block further contains 4 exception types：

  • Synchronous exception (synchronous exceptions, such as SVC instruction, data abort)
  • IRQ (normal interrupt)
  • FIQ (fast interrupt)
  • SError (system error, usually hardware exception)

So in total:
4 blocks × 4 exceptions = 16 table entries

Size of each table entry

ARMv8-A specifies:

  • The size of each table entry is fixed at 128 bytes(0x80) (size of 32 instructions).
  • 16 table entries × 128 bytes = 2048 bytes = 2KB。
  • So the size of the entire vector table is 2KB。

In other words, the exception vector table must be 2KB alignedarea.

The exception vector table actually consists of 4 groups of 4 entries

FAR Fault Address Register (at EL1)

Its function is:

  • When a synchronous exception occurs , the processor will write the virtual address that caused the exceptioninto theFAR_ELxcorresponding register.
  • EL1corresponds to the kernel mode (OS layer), soFAR_EL1what is saved is the virtual address when the exception occurred under EL1。

Common scenarios

  1. Page Fault
    If the CPU triggers an exception when accessing a non-existent or illegal virtual address,FAR_EL1It will save this address, and the kernel can decide based on it whether to allocate a new page or kill the process.
  2. Alignment Fault (Unaligned Access Exception)
    If the access address does not comply with the alignment rules,FAR_EL1that address will also be saved.
  3. Watchpoint/Breakpoint Exception
    FAR_EL1It may also be used to store the address that triggered the exception.

Related registers

  • ESR_EL1(Exception Syndrome Register)
    Stores the exception’s cause code(such as page fault, permission error, unalignment, etc.).
  • FAR_EL1
    Stores the specific virtual address。

Exception parsing

ESR_ELx (Exception Syndrome Register)

ESR_ELx is only updated for synchronous exceptions and SError, not for IRQ or FIQ, because these interrupt handlers typically obtain status information from the registers of the Generic Interrupt Controller (GIC).

  • The ESR register contains a total of four fields
    • Bits 32~63 Reserved bits
    • Bits 26~31, areException Class (abbreviated as EC), this fieldindicates the type of exception that occurred, and also used toindex the ISS field
    • Bit 25, IL, indicates the instruction length of the synchronous exception instruction, Instruction Length
    • Bit 0~24，ISS (Instruction Specific Syndrome) specific exception instruction encoding. This exception instruction encoding table depends on different exception types,different exception types have different encoding formats

Corresponding encoding of common exception types

The first indicates an instruction exception from a lower exception level, the second indicates an instruction exception at the current exception level
