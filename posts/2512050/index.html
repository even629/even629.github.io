Introduction to Virtualization

Virtualization Technology

As a resource management technology, virtualization technology can abstract and combine various physical resources of a computer (such as CPU, memory, I/O devices, etc.) and allocate them to multiple virtual machines.

  • Platform Virtualization（platform virtualization): Virtualization targeting computers and operating systems, such as KVM, etc.
  • Resource Virtualization（resource virtualization): Virtualization targeting specific system resources, including memory, storage, network resources, etc., such as container technology.
  • Application Virtualization（application virtualization): Includes emulation, simulation, interpretation techniques, etc., such as the Java Virtual Machine.

Virtualization Application Scenarios

  • Desktop Virtualization
  • Server Virtualization
  • Embedded Virtualization

Important Theoretical Foundations of Virtualization Technology

1974: Paper “Formal Requirements for Virtualizable Third Generation Architectures”

3 necessary elements for implementing virtualization:

  • Resource control. The VMM must be able to manage all system resources.
  • Equivalence. The behavior of the guest machine is consistent with running on bare metal.
  • Efficiency. Programs running on the guest machine are not interfered with by the VMM.

The x86 architecture encountered some challenges in implementing virtualization, especially in failing to meet the second condition mentioned above. Computer architectures contain two types of instructions.

  • Sensitive instructions: Instructions that operate on certain privileged resources, such as instructions that access or modify virtual machine modes or machine states.
  • Privileged instructions: Instructions with special privileges. These instructions are only used by operating systems or other system software and are generally not provided directly to users.

Hardware-assisted virtualization technology

  • In 2005, Intel began introducing hardware virtualization technology into CPUs, a technology called VT (virtualization technology).
  • The basic idea of VT is to create containers that can run virtual machines. In CPUs with VT enabled, there are two operating modes—root (VMX root) mode and non-root (VMX non-root) mode. Both operating modes support the 4 privilege levels of Ring 0 to Ring 3, so the hypervisor and virtual machines can freely choose their desired operating levels.

Full virtualization and paravirtualization

Currently, hardware-assisted virtualization technology is mainly used, and these two technologies are rarely used.

Classification of virtual machines

  • Hypervisor: Hypervisor, also known as VMM (Virtual Machine Manager), is located between the computer hardware and virtual machines, responsible for managing and allocating computer resources to each virtual machine.

    • Type1: The first type of hypervisor is like a small operating system, the purpose of which is to manage all virtual machines. Common virtualization software includes Xen, ACRN, etc.

    • Type2: The second type of hypervisor relies on operating systems such as Windows and Linux to allocate and manage scheduling resources. Common virtualization software includes VMware Workstation, KVM, and VirtualBox, etc.

Memory virtualization

  • Software simulation: Shadow Page Table

    • Low efficiency

  • Hardware memory virtualization technology—

    • Intel: Extended Page Table (EPT) technology
    • ARM: stage1 & stage2 page tables

  • 4 types of addresses used in virtualization

    • GVA (Guest Virtual Address): Guest virtual address.
    • GPA (Guest Physical Address): Guest physical address.
    • HVA (Host Virtual Address): Host virtual address.
    • HPA (Host Physical Address): Host physical address

I/O Virtualization

  • Software-emulated devicesTaking a disk as an example, the hypervisor can create a file or a region on the actual disk to emulate a virtual disk and pass it to the guest machine.

  • Device passthrough(Device Pass Through). The hypervisor directly assigns a physical device to a specific virtual machine.

  • SR-IOV(Single Root I/O Virtualization) technology

CPU Virtualization

vCPU and VM Concepts

  • The CPU virtualization technology of ARMv8/v9 is built on different exception levels

Creating multiple vCPUs

A physical CPU may create multiple vCPUs, utilizing the OS’s multi-process/multi-threading mechanism

A physical CPU can run multiple processes/threads, and the hypervisor can schedule them to run in a time-shared manner. When a process switches from EL2 to EL1, the vCPU is created.

ARMv8/v9 Exception Handling

  • Non-virtualized scenario: Exception handling is concentrated at EL1, such as exception vector tables, exception handling, interrupt handling, system calls, etc.

  • Virtualization scenario：

    • For the hypervisor: It needs to handle exceptions from the hypervisor itself, exceptions from the VM, system calls from the VM, interrupts from the VM, etc.
    • For the VM: The VM itself also needs to handle exceptions and system calls from the VM (GuestOS handles exceptions and system calls from apps)

Hypervisor entering VM

  1. Set HCR_EL2.RW to make the CPU run in AArch64
  2. Temporarily disable the MMU function of the vCPU (optional)
  3. Set SPSR_EL2 so that after the eret instruction, the CPU jumps to EL1
  4. Set the ELR_EL2 register so that after executing eret, it jumps to the vm_entry function to execute the eret instruction, transitioning the processor mode to make the CPU enter EL1
  5. Set the SP_EL1 stack to make the vCPU point to a new stack space

Exit from VM to Hypervisor

  • The VM’s Guest OS actively calls HVC
  • An exception occurred in the VM (cannot be handled in EL1, such as a GPA page fault exception)
  • A hardware interrupt occurred

HVC system call

  • SVC system call: An app running at EL0 requests to enter the OS at EL1.

  • HVC system call: A VM running at EL1 requests to enter the hypervisor at EL2.

  • SMC system call: Software running in the hypervisor or VM requests to enter the secure firmware at EL3

Follow the ARM64 standard function calling convention

  • x0: Return value / First parameter
  • x1,x2: Parameter
  • x8: HVC function number (i.e.which）

Exception Syndrome Register ESR_EL2

The ESR register contains a total of 4 fields (domains), where:

  • Bits 32~63 are reserved bits.
  • Bit 26~31，EC（Exception Class), this field indicates the type of exception that occurred, and is also used to index the ISS domain.
  • Bit 25，IL（Instruction Length for synchronous exceptions), indicates the instruction length for synchronous exceptions.
  • Bit 0~24，ISS（Instruction Specific Syndrome) the specific exception instruction encoding. This exception instruction encoding table depends on different exception types, and different exception types have different encoding formats.

Memory virtualization

ARM64 page table mapping

Only one-stage address mapping Stage 1, converting virtual addresses to physical addresses

Two-stage address mapping:

  • Stage1: GVA -> GPA in the VM
  • Stage2: HVA to HPA in the hypervisor

Each stage of mapping goes through a 3-level or 4-level page table

From a system perspective

Two-stage address translation for virtualization

  1. VA (Virtual Address) -> IPA (Intermediate Physical Address)(ARM manual terminology)
  2. IPA-> PA(Intermediate Physical Address)

The hypervisor itself also has memory access requirements, using TTBR0_EL2, which is another set of page tables, meaning there are three sets of page tables in total.

2-Stage page table translation

  • 2-Stage requires more memory consumption
  • VA->IPA->PA translation: may require up to 16 memory accesses
  • Both Stage 1 and Stage 2 can be cached in the TLB

Stage 1 page table mapping

  • Non-virtualization: VA -> PA

  • Virtualization: VA -> IPA

Major factors affecting page table mapping

  • Input address size: determined by TCR_ELx.TxSZ, specifying the VA range and the number of page table levels used. Usually 48 bits.
  • Output address size: determined by ID_AA64MMFR0_EL1.PARange, the maximum physical memory supported by the system, filled into TCR_ELx.{I}PS.
  • Page granularity: specified by TG0 and TG1 in TCR_ELx

Stage 2 page table mapping

  • Virtualization: IPA(GPA) -> PA

Major factors affecting page table mapping

  • Input address size: the maximum value of IPA is constrained by the system PA, PA is determined by ID_AA64MMFR0_EL1.PARange.

The GPA in the VM cannot exceed the physical memory size.

  • Output address size: determined by ID_AA64MMFR0_Determined by EL1.PARange, the maximum physical memory supported by the system is filled into VTCR_EL2.PS
  • Page granularity: specified by TG0 in VTCR_EL2

Stage 2 page table mapping supports 4-level or 3-level page table mapping

  • Input address size （PA size）
  • Page table granularity (4KB? 16KB? 64KB?)
  • Which level to start traversing the page table from? (VTCR_EL2.SL0)

How to know PA?

Taking the Cortex-A72 processor as an example, by consulting the Cortex-A72 Technical Reference Manual, we can knowPhysical Address range supports a maximum of 44bits，16TB。
Newer IPs such as Cortex-A77, A78, X2, etc., PA supports a maximum of 40bits, 1TB.

T0SZ is 64 -Support PA size

Taking 4KB page granularity as an example

  • When PA is 44bits, it supports 4-level page table mapping
  • When PA is 42bits, 40bits, 36bits, 32bits, it supports 3-level page table mapping

When PA is 42bits and 40bits, S2 concatenated page tables can be used to compress 4-level page tables to 3-level

S2 concatenated page tables

Taking 40bits+4KB as an example

There is an extra bit 39, used for the L0 page table, which has two page table entries, each pointing to an L1 page table

After using concatenated page tables, the two L1 page tables pointed to by the two page table entries of the original L0 page table are concatenated together, so the L1 page table is equivalent to 2 times the original

  • The L1 index range also becomes 2 times the original
  • The MMU starts indexing directly from the L1 page table, so the 4-level page table becomes a 3-level page table

Benefits: can avoid the overhead caused by an additional level of translation

  • The maximum number of concatenated tables supported is 16.
  • The VTCR_EL2.SL0 field indicates: which level of the page table to start indexing from

  • IA: Input Address, IA[47:12]-IA[39:12] indicates the use of a 40~48 bit virtual address
  • The first row “Tables” indicates the number of tables that can be concatenated, with a maximum of 16. 1 means no concatenation, and 2 means two page tables are concatenated
  • The first column “Initial lookup level” (SL0 value) indicates which level of the page table the hardware starts traversing from, configured by setting VTCR_EL2.SL0

Taking 43 bits + 4KB page granularity as an example, if S2 concatenated page tables are used, the L1 page table will be formed by concatenating 16 tables

IO device virtualization

Three mainstream methods

  1. Full software virtualization: Uses the “trap & emulate” approach to completely simulate the behavior of the device.

  2. Paravirtualization: Uses efficient front-end and back-end drivers to reduce VM trap events and improve I/O device virtualization performance, such as VirtIO technology.

  3. Hardware-assisted virtualization: Hardware direct passthrough, IOMMU, and SR-IOV (Single Root I/O Virtualization)

Parse the triggering instruction

  • VM accesses MMIO, triggering “Data Abort from a lower Exception level” exception.
  • No need to manually parse the exception instruction encoding; the hardware has already parsed it, just read the ISS:

  • ISV: Indicates that the exception instruction parsing contained in ISS[23:14] is valid.
  • SAS: The size of the memory access when the exception occurs.
  • SSE: Whether the data requires signed extension.
  • SRT: Destination register Rt
  • SF: Whether it is a 32-bit or 64-bit load/store instruction
  • AR: Whether the instruction contains acquire/release barrier semantics
  • VNCR: Whether the access is from nested virtualization
  • SET: Synchronous error type.
  • FnV: Whether the far register is valid.
  • WnR: Exception triggered by a read or write operation

Timer virtualization

  • System counter: Provides a system-wide, fixed-frequency system counter.
  • The System counter is broadcast to all cores
    • The CNTPCT_EL0 register returns the value of the system counter
    • The CNTFRQ_EL0 register is used to set the frequency corresponding to the system counter.
  • Each CPU core has a set of timers

generic timer

Each ARMv8/ARMv9 CPU core has a set of generic timers

Taking Cortex-A72 as an example

Difference between physical timer and virtual timer

The virtual counter canMeasure the passage of time on a virtual machine。

Time passage on a virtual machine != Time passage on a real physical machine

Assume two vCPUs run alternately, each vCPU runs for 1ms

If vCPU0 sets a timer at T=0 to trigger an interrupt after 3ms, has the interrupt been triggered?

Physical time vs Virtual time: Physical time refers to wall-clock time, i.e., the actual elapsed time. Virtual time refers to the time experienced by the virtual processor (vCPU)

vCPU0 has only run for 2 milliseconds at physical time 4 milliseconds. Based on the passage of physical time, the time set by vCPU0’s comparator has not yet reached 3 milliseconds, so the interrupt will not be triggered at this time.

  • Virtual counter = physical counter - offset

  • CNTVCT_EL0 (Virtual Counter) = CNTPCT_EL0 (Physical Counter) - CNTVOFF_EL2 (Virtual Offset)

Method 1 for calculating virtual time in a VM

Virtual time offset

offset=∑(Time_sched_in−Time_sched_out)\text{offset} = \sum(\text{Time\_sched\_in} - \text{Time\_sched\_out}) offset=∑(Time_sched_in−Time_sched_out)

Where Time_sched_in is the time when the vCPU is scheduled to run, Time_sched_out is the time when the vCPU is scheduled to be suspended.

Method 1: Each time a vCPU is scheduled to run, the hypervisor updates the CNTVOFF_EL2 register.

Method 2 for calculating virtual time in VM

Method 2:

  • Let the virtual offset be a constant value, virtual counter = physical counter - virtual offset
  • When the vCPU is scheduled out, the hypervisor disables the virtual timer interrupt, meaning the jiffies of that vCPU will temporarily stop incrementing.
  • When the vCPU is scheduled to run, the hypervisor enables the virtual timer interrupt, and the jiffies of that vCPU will resume incrementing.

Two ways for Timer to trigger interrupts

  • Using the TimeValue method, which assigns an initial value to the timer and lets it decrement. When it decrements to 0, an interrupt is triggered, and the timer is reassigned in the interrupt handler.

  • Using the CmpValue method: which assigns a comparison value CValue to the timer, and triggers an interrupt when the timer increments to CValue.

Interrupt virtualization

Routing of interrupt handling

For exceptions and interrupts, they can be routed to EL1, EL2, EL3 for processing, requiring the configuration of HCR and SCR related registers.

The HCR_EL2 register is the virtual hypervisor configuration register, which contains the following fields related to exception handling routing.

Virtual interrupt

  • There are two ways to send virtual interrupts to a vCPU.
    • Use system registers (VI and VF fields in HCR_EL2) to configure vIRQ and vFIQ.
    • Use GICv2 and GICv3 controllers.
  • Using vIRQ and vFIQ requires emulating the interrupt controller in the hypervisor.
    • Low efficiency, trapping and emulating.
    • The GIC interrupt controller supports vCPUs and does not require software emulation.

GICv2 interrupt controller

The Distributor registers (GICD_) contains interrupt settings and configuration.

The CPU Interface registers (GICC_) contains CPU-related special settings.

GIC interrupt types

  • SGI: Software Generated Interrupt, a software interrupt generated by software, used to send interrupt signals to other CPU cores.
  • PPI: Private Peripheral Interrupt, a private peripheral interrupt that is exclusive to a specific designated CPU.
  • SPI: Shared Peripheral Interrupt, a shared peripheral interrupt that can be accessed by all CPUs.
  • LPI: Locality-specific Peripheral Interrupt, a new interrupt type added in GICv3. A message-based interrupt type.

Note: Interrupt numbers 1020 ~ 1023 are reserved.

GICv2 registers

GICv2 registers are divided into two groups:

  • D: Represents Distributor registers.
  • C: Represents CPU interface registers.

Some registers are described according to interrupt numbers, for example, using a few bits to describe the related attributes of an interrupt number, and there can be n of the same register.

Virtualization support on GIC v2

GIC v2 to support virtualization:

  • Addedvirtual CPU interface, sending VFIQ/vIRQ to the vCPU.
  • AddGIC virtual interface control registers。
  • AddGIC virtual CPU interface registers

There is only one Distributor, the hypervisor needs to emulate a vDistributor for the VM

GIC v2 interrupt virtualization flow

  1. Ensure all interrupts are routed to the EL2 hypervisor for processing。
  2. The hypervisor emulates a vDistributor for the VM to use, meaning that when the GuestOS accesses the MMIO registers of the GIC distributor, it traps to the hypervisor。
  3. Establish a mapping in the hypervisor, meaning that when the GuestOS accesses the MMIO range of the GIC CPU interface, it is mapped to the GIC virtual CPU interface。
  4. When the hypervisor receives a physical interrupt, it must decide whether the interrupt is processed by the hypervisor or injected into the VM for processing。
  5. If an interrupt needs to be injected into the VM, update the GICH_LRn register to fill in the interrupt-related information. When the CPU returns to the vCPU, the interrupt injection is completed。
  6. When the VM is running, reading the GIC_IAR register can obtain the interrupt number

The peripheral interrupt flow is as follows:

  1. Enters the Distributor hardware unit inside the GIC controller, and then dispatches it to a physical CPU interface.
  2. The CPU will respond to this interrupt.
  3. This interrupt will be routed to EL2. The hypervisor reads the IAR register to get the physical interrupt number hwirq. The hypervisor writes GICC_EOIR to tell the GIC that the physical interrupt processing is complete.
  4. The hypervisor sets the GICH_LRn register to register a virtual interrupt.
  5. After the Virtual CPU interface receives this interrupt, it determines whether this virtual interrupt can be sent to the vCPU.
  6. This interrupt enters the virtual exception vector table of the vCPU. The Guest OS reads the GICC_IAR register, then obtains the vINTID virtual interrupt number, and then jumps to the interrupt handler for processing.
  7. After the Guest OS finishes processing the interrupt, it writes the GICC_EOIR register to tell the GIC that this virtual interrupt is complete.

The process is as follows:

  1. A peripheral triggers an IRQ interrupt, and the interrupt reaches the GIC
  2. GIC sends the interrupt to the CPU. Because HCR_EL2.IMQ=1 is set, the CPU is routed to EL2
  3. GIC configures the GIC List Register (GICH_LRn) to register a vIRQ.
  4. The Hypervisor switches to run the Guest OS, EL2 switches to EL1/EL0, and the CPU runs on the vCPU.
  5. The vCPU receives the vIRQ interrupt from the GIC.

Two sets of registers added for GIC v2 virtualization

MMIO for GIC virtualization

According to one of the three necessary conditions for virtualization: equivalence. That is, the virtual machine is unaware that it is running in a virtual environment.

  1. The VM accesses the GIC CPU interface as if it were real. The Hypervisor needs to map the GICC_interface to the GICV_interface. Base + 0x2000 —> (mapped to) Base + 0x6000

  2. GICC_interface and GICV_interface are two sets of registers with the same format.

  3. Since there is only one Distributor inside GICv2, the Distributor seen by the VM is simulated by the hypervisor.

vDistributor and Distributor ultimately access the same hardware unit, so access conflicts need to be avoided.

  • GICH_HCR Registers for overall GICH control
  • GICH_ELSR0 GICH_LR register status, determining whether the LR register is empty, etc.
  • GICH_LR0 Used to inject interrupts into the VM

GICH_LRn register implements interrupt injection

Writing to the GICH_LRn register can implement interrupt injection into the VM

Experiment 10: Injecting interrupts into the VM

Experiment Objective: Familiarize with how GIC v2 injects interrupts

Experiment Requirement: Implement the interrupt injection function in BenOS to inject virtual timer interrupts into the VM. The steps are as follows:

  • Enable GIC virtualization, GICH_HCR.EN = 1
  • Map the GICC in the VM_interface access address to the hypervisor’s GICV_interface
  • Simulate the Distributor in the hypervisor
  • When the virtual timer in the VM triggers an interrupt, route it to the EL2 hypervisor for processing
  • The hypervisor writes the interrupt information into the GICH_LRn register to inject the interrupt into the VM
  • EOI mode, handled by Hypervisor or inject vm
  • The GIC driver in the VM’s GuestOS reads the interrupt number and processes the interrupt

Points to note when simulating the vDistributor

  • The Distributor seen by the VM is simulated by the hypervisor.
  • When the hypervisor simulates the vDistributor, it needs to consider the issue of concurrent access between the VM and the hypervisor.
  • Some registers in the Distributor are described by interrupt number. For example, a few bits are used to describe the attributes of an interrupt number, so the same register can describe n interrupt sources.

If the vDistributor recklessly writes to such registers directly via trap and emulate, it will corrupt the hypervisor’s original state.
