Interrupt Concept

An interrupt is a mechanism triggered by external or internal events during the normal operation of a CPU. When an interrupt occurs, the CPU stops the currently executing program and switches to executing the interrupt handler that triggered the interrupt. After processing the interrupt handler, the CPU returns to the point where the interrupt occurred and continues executing the interrupted program. The interrupt mechanism allows the CPU to respond to external or internal events in real-time while maintaining the ability to process other tasks.

The interrupt mechanism gives us the ability to handle unexpected situations, and if we can make full use of this mechanism, we can accomplish multiple tasks simultaneously.

The interrupt mechanism enables us to handle multiple tasks simultaneously in an orderly manner, thereby improving concurrent processing capabilities. Similarly, computer systems also use interrupt mechanisms to cope with various external events. For example, when a keyboard input occurs, an interrupt signal is sent to the CPU to respond to the user’s operation in a timely manner. In this way, the CPU does not have to constantly poll the state of the keyboard and can focus on other tasks. The interrupt mechanism can also be used to handle events such as hard disk read/write completion and network data packet reception, improving the system’s resource utilization and concurrent processing capabilities.

Top and Bottom Halves of Interrupts

The execution of interrupts requires a fast response, but not all interrupts can be completed quickly. If the interrupt handling time is too long, it will cause problems.

In order to allow the system to better handle interrupt events and improve real-time performance and responsiveness, the interrupt service routine is divided into two parts:

  • Interrupt Top Half is the first part of the interrupt service routine, which mainlyhandles urgent tasks that require a fast response。
    • It is characterized by a short execution time and aims to complete the interrupt processing as soon as possible. These tasks may include saving register states, updating counters, etc., so that after the interrupt processing is completed, it can correctly return to the execution position before the interrupt.
  • Interrupt Bottom Half is the second part of the interrupt service routine, which mainlyhandles relatively time-consuming tasks。
    • Since the interrupt top half needs to be completed as soon as possible, the interrupt bottom half is responsible for handling tasks that cannot be completed immediately and require more time. These tasks may include complex calculations, accessing external devices, or performing long-duration data processing.

Interrupt Subsystem Framework

A complete interrupt subsystem framework can be divided into four layers, from top to bottom: the user layer, generic layer, hardware-related layer, and hardware layer. The introduction related to each layer is as follows:

  • User Layer: The user layer is the consumer of interrupts, mainly including various device drivers.

    These drivers apply for and register interrupts through interrupt-related interfaces. When a peripheral triggers an interrupt, the user layer driver performs corresponding callback processing and executes specific operations.

  • Generic Layer: The generic layer can also be called the framework layer, which is hardware-independent.
    The code in the generic layer is common across all hardware platforms and does not depend on specific hardware architectures or interrupt controllers. The generic layer provides unified interfaces and functions for managing and handling interrupts, enabling drivers to be reused across different hardware platforms.

  • Hardware-dependent layer: The hardware-dependent layer contains two parts of code.
    One part is code related to specific processor architectures, such as the interrupt handling code for ARM64 processors. This code is responsible for handling the interrupt mechanisms of specific architectures, including interrupt vector tables, interrupt handlers, etc. The other part is the driver code for the interrupt controller, used to communicate with and configure the interrupt controller. This code is related to the specific interrupt controller hardware.

  • Hardware layer: The hardware layer is located at the bottommost level and is related to specific hardware connections.

    It includes the physical connection part between peripherals and the SoC (System on Chip). Interrupt signals are passed from peripherals to the interrupt controller, which uniformly manages and routes them to the processor. The design and implementation of the hardware layer determine the transmission method of interrupt signals and the hardware’s interrupt handling capability.

Interrupt number

In the Linux kernel, weuse two IDs, IRQ number and HW interrupt ID, to identify an interrupt from a peripheral：

  • IRQ number: The CPU needs to number each peripheral interrupt, which we call IRQ Number. This IRQ number is a virtual interrupt ID, independent of hardware, and is merely used by the CPU to identify a peripheral interrupt。
  • HW interrupt ID: For the GIC interrupt controller, it collects the interrupt request lines of multiple peripherals and passes them upward, therefore, the GIC interrupt controller needs to encode the peripheral interrupts.The GIC interrupt controller uses HW interrupt ID to identify peripheral interrupts。If there is only one GIC interrupt controller, the IRQ number and HW interrupt ID can correspond one-to-one。

However, in the case of cascaded GIC interrupt controllers, using only the HW interrupt ID cannot uniquely identify a peripheral interrupt; it is also necessary to know which GIC interrupt controller the HW interrupt ID belongs to (HW interrupt IDs are encoded repeatedly across different interrupt controllers)

For driver engineers, our perspective is the same as the CPU’s; we only want to get an IRQ number and do not care about which specific HW interrupt ID on which specific GIC interrupt controller. One advantage of this is that when the interrupt-related hardware changes, the driver software does not need to be modified. Therefore,the interrupt subsystem in the Linux kernel needs to provide a mechanism to map HW interrupt ID to IRQ number, which is the irq domain。

      • IRQ_WAKE_THREAD: indicates that the interrupt service routine has handled the interrupt and requests to wake up a kernel thread to continue further processing. This is used in some interrupt situations that require long processing times

Interrupt handlers and locks

Interrupt handlers run in an atomic context and can only use spinlocks to control concurrency. Whenever there is global data accessible by both user code (user tasks, i.e., system calls) and interrupt code, this shared data should be protected byspin_lock_irqsave()in the user code.

The priority of an interrupt handler is always higher than that of a user task. Even if the task holds a spinlock, simply disabling IRQs is not enough; an interrupt may occur on another CPU. If a user task updating
data is interrupted by an interrupt handler trying to access the same data, it would be a disaster. Usingspin_lock_irqsave()will disable all interrupts on the local CPU, preventing the system call from being interrupted by any type of interrupt:

When sharing data between two different interrupt handlers (i.e., the same driver manages two or more devices, each with its own interrupt line), should also be used in these handlersspin_lock_irqsave()to protect shared data to prevent other IRQ triggers and useless spinning.

Work deferral mechanism

Deferral is a method of scheduling work to be executed in the future, which postpones the issuing of operations. It allows the delayed invocation and execution of any type of function.

  • SoftIRQ: Execution in atomic context.
  • Tasklet: Execution in atomic context.
  • WorkQueue: Execution in process context.

Softirq (software interrupt) this deferral mechanism is only used for fast processing because itruns with the scheduler disabled (in interrupt context). Softirq is rarely (almost never) used directly; only the network and block device subsystems use Softirq.

Tasklet is an instance of Softirq; for almost every situation where Softirq is needed, Tasklet is sufficient.

In most cases, Softirq is scheduled in hardware interrupts, which occur faster than they can be serviced, and the kernel queues them for later processing. Ksoftirqd is responsible for the later execution (process context). Ksoftirqd is a single-CPU kernel thread used to handle unserviced software interrupts. If CPU resources are heavily consumed by Ksoftirqd, it indicates that the system is overloaded or under an interrupt storm.

tasklet

In the Linux kernel,tasklet is a special software interrupt mechanism, widely used to handle tasks related to the interrupt bottom half. The Tasklet deferral mechanism is mostly used in DMA, network, and block device drivers.

Tasklets are essentially non-reentrant. If code can be interrupted anywhere during its execution and can be safely called again afterwards, it is called reentrant. Tasklets are designed to run on only one CPU (even on an SMP system), which is the CPU that scheduled them. Different Tasklets can run simultaneously on different CPUs.

It is a common and effective method to avoid concurrency issues on multi-core processing systems.The function bound to a Tasklet can only run on one CPU at a time, so there will be no concurrency conflicts。

Note

  • Calling on a Tasklet that has been scheduled but has not yet started executingtasklet_schedulewill not perform any operation, and the Tasklet will ultimately only execute once.

  • can be called within a Tasklettasklet_schedule, which means a Tasklet can reschedule itself.

  • High-priority Tasklets are always executed before normal-priority Tasklets. Abusing high-priority tasks will increase system latency. Be sure to use them only when fast execution is truly needed.

  • Functions that may cause sleep cannot be called within the function bound to a tasklet, otherwise it may cause a kernel exception.

This API is deprecated.

Tasklet advantages

  1. Simplified interface and programming model: Tasklet provides a simple interface and programming model, making it easier to handle deferred work in the kernel. Compared to adding softirqs manually, Tasklet offers a higher-level abstraction.

  2. Low latency: Tasklets execute in the softirq context, avoiding the context-switching overhead of kernel threads, thus having low latency. This is very important for latency-sensitive tasks that require fast response.

  3. Adaptive scheduling: Tasklets have the characteristic of adaptive scheduling; when multiple Tasklets are in a waiting state, the kernel merges them to reduce unnecessary context switching. This scheduling mechanism can improve system efficiency.

Tasklet disadvantages

  1. Unable to handle long-running tasks:Tasklets are suitable for short-running deferred work, if long-running tasks need to be handled, it may block the execution of other tasks. For longer operations, workqueues or kernel threads may need to be used.
  2. Lack of flexibility: The execution of Taskletsis limited to the softirq contextand is not suitable for all types of deferred work. In some cases, more flexible scheduling and execution mechanisms may be required, in which case custom softirqs might be more appropriate.
  3. Resource limitations: The number of Tasklets is limited,the number of available Tasklets in the system depends on the architecture and kernel configuration. If a large amount of deferred work processing is required, it may be limited by the number of Tasklets

Workqueue

Workqueue is one of the mechanisms for implementing the bottom half of interrupts, which is a data structure or mechanism used to manage tasks.

The basic principle of a workqueue is to arrange tasks that need to be executed in order in a queue, and provide a set of worker threads or worker processes to handle the tasks in the queue.

When new tasks arrive, they are added to the end of the queue, and the worker threads or worker processes retrieve tasks from the head of the queue and perform the corresponding processing operations.

Tasklet is also one of the mechanisms for implementing the bottom half of interrupts. If sleeping is required in the interrupt bottom half, workqueue is the only choice, because tasklets cannot sleep, while workqueues can sleep, so tasklets can be used to handle time-consuming tasks, whileWork queues can handle more time-consuming tasks。

After the work queue defers the work, it is handed over to a kernel thread for execution.During the boot process, Linux creates a worker kernel thread, which is in a sleep state after creation. When there is work to be processed, this thread is awakened to handle the work.。

In the kernel, work queues includeShared work queuesandCustom work queuesThese two types. These two types of work queues have different characteristics and uses.

  • Shared work queuesThey are processed by a set of kernel threads, each running on a CPU. Once a work task needs to be scheduled, it is queued in the global work queue and will be executed at the appropriate time.
  • Custom work queuesare work queues that run within dedicated kernel threads. This means that whenever a work queue handler needs to be executed, the dedicated kernel thread is awakened to handle it, rather than one of the default predefined threads.

Shared work queues

Shared queues are global work queues managed by the kernel，used to handle some system-level tasks in the kernel. Unless there is no other choice, or critical performance is required, or control is needed
over every detail from work queue initialization to work scheduling, if tasks are only submitted occasionally, the shared work queue provided by the kernel should be used. This queue is shared by the entire system, and it can but should not be monopolized for a long time.

The shared work queue is a default work queue in the kernel that can be shared by multiple kernel components and drivers.

Since pending tasks on the queue are executed serially on each CPU, tasks should not sleep for a long time. Because other tasks on the queue cannot run before it wakes up, a task may not even know which tasks it shares the work queue with, so it may take a long time for a task to get CPU time.

Work in the shared work queue is executed by kernel-createdevents/nthreads on each CPU

Delayed work

Delayed work is atechnique that delays the execution of work to a later point in time for processing.

Typically,When a task takes a long time, does not need to be executed immediately, or needs to be executed on time, delayed work will come in handy.

The basic idea of delayed work is to put tasks into a queue, and then the background work process or task scheduler will process the tasks in the queue. Tasks can be executed after a specified delay time, or sorted and processed based on priority, task type, or other conditions.

  1. Delayed work is often used to handle tasks that take a long time, such as sending emails, processing images, etc. By putting these tasks into a queue and executing them with a delay, the main thread of the application can be prevented from being blocked, improving the system’s response speed.
  2. Delayed work can be used to execute scheduled tasks, such as regularly backing up a database. By setting tasks to execute at a certain point in the future, the reliability and efficiency of the system can be improved.

In actual situations:

  • At time t1, the key is pressed, but due to bouncing, it does not stabilize until time t2. The period from t1 to t2 is the bouncing.

Timer debouncing:

Start a timer with a period of 10ms in the key interrupt. When the timing time is up, the timer interrupt is triggered. Finally, read the key value in the timer interrupt handler. If the key value is still in the pressed state, it means this is a valid key press.

In the figure above, the period from t1 to t3 is the key bouncing, which needs to be eliminated. The key is set to trigger on the falling edge, so the key interrupt will be triggered at times t1, t2, and t3. Each time the interrupt handler is entered, the timer interrupt is restarted, so the timer interrupt is started at times t1, t2, and t3.

However, the two periods t1~t2 and t2~t3 are shorter than the timer interrupt period we set (i.e., the debounce time, such as 10ms). Therefore, although the timer is started at t1, the timer is reset at time t2 before the timing time is up. Ultimately, only the timer started at t3 can complete the entire timing cycle and trigger an interrupt. Then we can process the key in the interrupt handler. This is the principle of using a timer to achieve key debouncing, and this is the principle used in key drivers in Linux!

Concurrency-managed workqueue

When using a work queue, we first define a work structure, then add the work to the workqueue, and finally the worker thread executes the workqueue.

When new work is generated in the work queue, the worker thread will execute each work in the work queue. When the execution is finished, the worker thread will sleep until a new interrupt occurs, work continues to be added to the work queue, and then the worker thread executes each work, repeating the cycle.

Single-core system

In a single-core threaded system, a work thread is usually initialized for each CPU (core) and associated with a work queue. This default setting ensures that each CPU has a dedicated thread to handle work items on its bound work queue.

multi-core system

In multi-core threaded systems, the design of the work queue differs from that of single-core threaded systems.In multi-core threaded systems, there are usually multiple work queues, each bound to a worker thread. This allows for full utilization of the parallel processing capabilities of multiple cores.

When a new work item is generated, the system needs to decide which work queue to assign it to. A common strategy is to useLoad balancing algorithm, balancing the assignment of work items based on the load of the work queues to avoid overloading a specific work queue which could lead to performance degradation.

Each work queue independently manages its own work items. When a new work item is added to a work queue, the worker thread retrieves the pending work item from its associated work queue and executes the corresponding processing function.

In multi-core threaded systems, multiple worker threads can simultaneously execute work items in their respective bound work queues. This enables parallel processing, improving the overall performance and responsiveness of the system.

Disadvantages of work queues

  1. While work item w0 is working or even sleeping, work items w1 and w2 are queued and waiting. In a busy system, the work queue may accumulate a large number of pending work items, leading to delays in task scheduling, which may affect the system’s responsiveness and increase the processing time of work items.
  2. In a work queue, different work items may have different processing times and resource requirements. If the processing times of work items vary significantly, some worker threads may be constantly busy processing long-duration work items while other worker threads remain idle, leading to unbalanced resource utilization.
  3. In a multi-threaded environment, multiple worker threads accessing and modifying the work queue simultaneously may lead to race conditions. To ensure data consistency and correctness, appropriate synchronization mechanisms, such as locks or atomic operations, are needed to protect shared data, but this may introduce additional synchronization overhead.
  4. Work queues typically process work items in a First-In-First-Out (FIFO) manner, lacking fine-grained control over the priority of work items. In some scenarios, priority scheduling may be required based on the importance or urgency of the work items, which the work queue itself cannot provide at this level of priority control.
  5. When worker threads fetch and execute work items from the work queue, frequent context switching may be required, switching the processor’s execution context from one thread to another. This context switching overhead may affect system performance and efficiency.

Concurrency Managed Workqueue

CMWQ The full name is Concurrency Managed Workqueue, which means concurrency managed workqueue. A concurrency managed workqueue is a concurrent programming pattern used to effectively manage and schedule tasks or work items to be executed. It is typically used in multi-threaded or multi-process environments to achieve concurrent execution and improve system performance.

When we need to handle multiple tasks or jobs simultaneously in a system, using a concurrency managed workqueue is an effective approach.

Threaded IRQ Technology

Threaded IRQ is an optimization technique used to improve the performance of multi-threaded programs. The primary goal isto minimize the time interrupts are disabled。

In multi-threaded programs, hardware or other events sometimes send interrupt signals that interrupt the currently executing thread, requiring a switch to the interrupt handler to process these events. Thisfrequent interrupt switching causes additional overhead and latency, affecting the performance of the program.

To solve this problem, threaded IRQ proposes an optimization scheme. It separates the interrupt handler from the main thread, creating a dedicated thread to handle these interrupt events. This way, the main thread is no longer disturbed by interrupts, can focus on its own work, and is no longer frequently interrupted.

The core idea of threaded interrupts isto separate interrupt handling from the work of the main thread, allowing them to execute in parallel. The interrupt thread is responsible for handling interrupt events, while the main thread is responsible for executing the main work tasks. In this way, not only can the overhead of switching be reduced, but the response speed and performance of the entire program can also be improved.

The handling of threaded interrupts can still be viewed as dividing the original interrupt into the top half and the bottom half.

The top half is still used to handle urgent matters, and the bottom half handles more time-consuming operations, but the bottom half is handed over to a dedicated kernel thread for processing. This kernel thread is only used for this interrupt.

When an interrupt occurs, this kernel thread is woken up, and then this kernel thread executes the interrupt bottom half function。

It should be noted that threaded interrupts also need to handle synchronization and data sharing issues between threads. Because the interrupt thread and the main thread may access and modify shared data simultaneously, reasonable synchronization operations are required to ensure data consistency and correctness.

Function definitionint request_threaded_irq(unsigned int irq, irq_handler_t handler, irq_handler_t thread_fn, unsigned long irqflags, const char *devname, void *dev_id);
Header file#include <linux/interrupt.h>
Parameter irqInterrupt number (IRQ line number)
Parameter handlerTop half handler function: the fast-handling function executed first when an interrupt occurs.

Can be set toNULLto use the system default handler.
Parameter thread_fnThreaded interrupt handler function (bottom half), executed in a kernel thread. If it isNULLthen interrupt threading is not used.
Parameter irqflagsInterrupt attribute flags, such asIRQF_TRIGGER_RISING、IRQF_SHAREDetc.
Parameter devnameInterrupt name, used to identify the device
Parameter dev_idDevice identifier passed to the handler function (usually a device structure pointer)
FunctionRegister a threaded interrupt, including top half and bottom half thread processing, to improve the real-time performance and schedulability of interrupt handling.
Return valueSuccess: returns 0;
Failure: returns a negative error code
  • handlerfunction: This is the same as the one used whenrequest_irq()registering. It represents the top half function, running in an atomic context (or hard interrupt). If it can handle the interrupt quickly enough, the bottom half may not be needed at all, and it should returnIRQ_HANDLED. However, if the interrupt handling requires100μsor more, as mentioned earlier, the bottom half should be used. In this case, it should returnIRQ_WAKE_THREAD, which causes the scheduling of thethread_fnfunction (which must be provided).
  • thread_fnfunction: This represents the bottom half, scheduled by the top half. When the hard interrupt handler (handler function) returnsIRQ_WAKE_THREAD, the kernel thread associated with this bottom half will be scheduled, and the thread_fn function will be called when the kernel thread runs. The thread_fn function must returnIRQ_HANDLEDafter execution. After execution, the interrupt is re-triggered, and before the hard interrupt returnsIRQ_WAKE_THREAD, the kernel thread will not be scheduled again.

Threaded interrupts can be used anywhere a workqueue can be used to schedule the bottom half. A true threaded interrupt must define both handler and thread_fn.

If handler is NULL and thread_fn is not NULL, the kernel will install a default hard interrupt handler, which will simply return IRQ_WAKE_THREAD to schedule the bottom half.

While the interrupt handler is executing, the interrupt it serves is always disabled on all CPUs, and is re-enabled when the hard interrupt (top half) completes.

However, if for some reason the interrupt line should not be re-enabled after the top half and should remain disabled until the threaded handler runs, a threaded interrupt with the IRQF_ONESHOT flag should be requested. This way, the interrupt line will not be re-enabled until the interrupt bottom half is complete.
