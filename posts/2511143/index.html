Interrupt Concept

An interrupt is a mechanism triggered by external or internal events during the normal operation of the CPU. When an interrupt occurs, the CPU stops the currently executing program and instead executes the interrupt handler that triggered the interrupt. After the interrupt handler finishes, the CPU returns to the point where the interrupt occurred and continues executing the interrupted program. The interrupt mechanism allows the CPU to respond to external or internal events in real time while maintaining the ability to handle other tasks.

The interrupt mechanism gives us the ability to handle unexpected situations, and if we can make full use of this mechanism, we can accomplish multiple tasks simultaneously.

The interrupt mechanism enables us to handle multiple tasks in an orderly manner simultaneously, thereby improving concurrency. Similarly, computer systems also use the interrupt mechanism to handle various external events. For example, when a keyboard input occurs, an interrupt signal is sent to the CPU to promptly respond to the user’s operation. In this way, the CPU does not have to constantly poll the keyboard status and can focus on other tasks. The interrupt mechanism can also be used to handle events such as hard disk read/write completion and network packet reception, improving system resource utilization and concurrency.

Top and Bottom Halves of Interrupts

Interrupt execution requires fast response, but not all interrupts can be completed quickly. If the interrupt processing time is too long, it will cause problems.

In order to allow the system to better handle interrupt events and improve real-time performance and responsiveness, the interrupt service routine is divided into two parts: the top half and the bottom half.

  • Interrupt Top Half is the first part of the interrupt service routine, which mainlyhandles urgent tasks that require fast response。
    • It is characterized by short execution time, aiming to complete interrupt processing as quickly as possible. These tasks may include saving register states, updating counters, etc., so that after the interrupt processing is completed, it can correctly return to the execution position before the interrupt.
  • Interrupt Bottom Half is the second part of the interrupt service routine, which mainlyhandles relatively time-consuming tasks。
    • Since the top half needs to be completed as quickly as possible, the bottom half is responsible for handling tasks that cannot be completed immediately and require more time. These tasks may include complex calculations, accessing external devices, or performing long data processing.

Interrupt Subsystem Framework

A complete interrupt subsystem framework can be divided into four layers, from top to bottom: the user layer, generic layer, hardware-related layer, and hardware layer. The introduction of each layer is as follows:

  • User Layer: The user layer is the user of interrupts, mainly including various device drivers.

    These drivers apply for and register interrupts through interrupt-related interfaces. When a peripheral triggers an interrupt, the user-layer driver performs corresponding callback processing and executes specific operations.

  • Generic Layer: The generic layer can also be called the framework layer, which is hardware-independent.
    The code of the generic layer is common across all hardware platforms and does not depend on specific hardware architectures or interrupt controllers. The generic layer provides unified interfaces and functions for managing and handling interrupts, allowing drivers to be reused on different hardware platforms.

  • Hardware-related layer: The hardware-related layer consists of two parts of code.
    One part is code related to a specific processor architecture, such as the interrupt handling code for ARM64 processors. This code is responsible for handling the interrupt mechanism of the specific architecture, including the interrupt vector table, interrupt handlers, etc. The other part is the driver code for the interrupt controller, used to communicate with and configure the interrupt controller. This code is specific to the interrupt controller hardware.

  • Hardware layer: The hardware layer is at the bottommost level and is related to specific hardware connections.

    It includes the physical connection part between peripherals and the SoC (System on Chip). Interrupt signals are transmitted from peripherals to the interrupt controller, which manages and routes them to the processor. The design and implementation of the hardware layer determine the transmission method of interrupt signals and the hardware’s interrupt handling capability.

Interrupt Number

In the Linux kernel, weuse two IDs, IRQ number and HW interrupt ID, to identify an interrupt from a peripheral：

  • IRQ number: The CPU needs to number each peripheral interrupt, which we call the IRQ Number. This IRQ number is a virtual interrupt ID, independent of hardware, used only by the CPU to identify a peripheral interrupt。
  • HW interrupt ID: For the GIC interrupt controller, it collects interrupt request lines from multiple peripherals and passes them upward. Therefore, the GIC interrupt controller needs to encode peripheral interrupts.The GIC interrupt controller uses HW interrupt ID to identify peripheral interrupts。If there is only one GIC interrupt controller, then IRQ number and HW interrupt ID can correspond one-to-one。

But in the case of cascaded GIC interrupt controllers, using only the HW interrupt ID cannot uniquely identify a peripheral interrupt; it is also necessary to know the GIC interrupt controller to which the HW interrupt ID belongs (HW interrupt IDs can be repeated across different interrupt controllers).

For driver engineers, our perspective is the same as the CPU’s. We only want to get an IRQ number, without caring about which HW interrupt ID on which GIC interrupt controller it is. The advantage of this is that when the interrupt-related hardware changes, the driver software does not need to be modified. Therefore,The interrupt subsystem in the Linux kernel needs to provide a mechanism to map HW interrupt IDs to IRQ numbers, that is, irq domain.。

      • IRQ_WAKE_THREAD: Indicates that the interrupt service function has handled the interrupt and requests to wake up a kernel thread to continue further processing. This is used in some interrupt cases that require long processing time.

Interrupt handlers and locks

Interrupt handlers run in atomic context and can only use spinlocks to control concurrency. Whenever there is global data accessible by user code (user tasks, i.e., system calls) and interrupt code, this shared data should be protected in user code byspin_lock_irqsave()protection.

The priority of interrupt handlers is always higher than user tasks. Even if the task holds a spinlock, merely disabling IRQs is not enough, as interrupts may occur on another CPU. If updating
the data user task is interrupted by an interrupt handler trying to access the same data, that would be a disaster. Usingspin_lock_irqsave()will disable all interrupts on the local CPU, preventing system calls from being interrupted by any type of interrupt:

When sharing data between two different interrupt handlers (that is, when the same driver manages two or more devices, each with its own interrupt line), these handlers should also usespin_lock_irqsave()to protect shared data, to prevent other IRQ triggers and useless spinning.

Work deferral mechanism

Deferral is a method of scheduling work to be executed in the future, which postpones the release operation. It allows deferred invocation and execution of any type of function.

  • SoftIRQ: Executed in atomic context.
  • Tasklet: Executed in atomic context.
  • WorkQueue: Executed in process context.

Softirq (soft interrupt or software interrupt) this deferred mechanism is only used for fast processing because itruns with the scheduler disabled (in interrupt context). Softirq is rarely (almost never) used directly; only the network and block device subsystems use Softirq.

Tasklet is an instance of Softirq. In almost every case where Softirq is needed, Tasklet is sufficient.

In most cases, Softirq is scheduled in hardware interrupts. These interrupts occur faster than they can be serviced, so the kernel queues them for later processing. Ksoftirqd is responsible for later execution (process context). Ksoftirqd is a per-CPU kernel thread that handles unserviced software interrupts. If CPU resources are heavily consumed by Ksoftirqd, it indicates that the system is overloaded or under an interrupt storm.

tasklet

In the Linux kernel,tasklet is a special soft interrupt mechanism, widely used to handle tasks related to the bottom half of interrupts. The Tasklet deferred mechanism is mostly used in DMA, network, and block device drivers.

Tasklet is inherently non-reentrant. If code can be interrupted at any point during execution and then safely called again, it is called reentrant. Tasklet is designed to run only on one CPU (even on SMP systems), the CPU that scheduled it. Different Tasklets can run simultaneously on different CPUs.

It is a common and effective method to avoid concurrency issues on multi-core systems.The function bound to a Tasklet can only run on one CPU at a time, so there is no concurrency conflict.。

Note

  • Calling on a Tasklet that has been scheduled but not yet started executiontasklet_schedulewill do nothing, and the Tasklet will eventually execute only once.

  • It can be called within a Tasklettasklet_schedule, meaning a Tasklet can reschedule itself.

  • High-priority Tasklets always execute before normal-priority Tasklets. Abusing high-priority tasks increases system latency. Only use them when truly fast execution is needed.

  • The function bound to a tasklet must not call functions that may cause sleep, otherwise it may cause kernel exceptions.

This API is deprecated.

Advantages of tasklet

  1. Simplified interface and programming model: Tasklet provides a simple interface and programming model, making it easier to handle deferred work in the kernel. Compared to adding softirqs manually, Tasklet offers a higher-level abstraction.

  2. Low latency: Tasklet executes in the softirq context, avoiding the context switch overhead of kernel threads, thus having lower latency. This is very important for latency-sensitive tasks that require fast response.

  3. Adaptive scheduling: Tasklet has the feature of adaptive scheduling. When multiple Tasklets are in a waiting state, the kernel merges them to reduce unnecessary context switches. This scheduling mechanism can improve system efficiency.

Disadvantages of tasklet

  1. Cannot handle long-running tasks:Tasklet is suitable for short-duration deferred work, if long-running tasks need to be processed, it may block the execution of other tasks. For longer operations, it may be necessary to use work queues or kernel threads.
  2. Lack of flexibility: The execution of Taskletis limited to the context of softirqs, and is not suitable for all types of deferred work. In some cases, a more flexible scheduling and execution mechanism may be needed, and custom softirqs may be more appropriate.
  3. Resource limitation: The number of Tasklets is limited,the number of Tasklets available in the system depends on the architecture and kernel configuration. If a large amount of deferred work needs to be processed, it may be limited by the number of Tasklets

Work queue

Work queue is one of the mechanisms for implementing the bottom half of interrupts, and is a data structure or mechanism for managing tasks.

The basic principle of a work queue is to arrange tasks to be executed in order in a queue, and provide a set of worker threads or worker processes to handle the tasks in the queue.

When new tasks arrive, they are added to the end of the queue, and worker threads or worker processes fetch tasks from the head of the queue and perform the corresponding processing operations.

Tasklet is also one of the mechanisms for implementing the bottom half of interrupts. If you need to sleep in the bottom half of an interrupt, the work queue is the only choice because tasklet cannot sleep, while work queue can sleep, so tasklet can be used to handle relatively time-consuming tasks, whilework queue can handle more time-consuming tasks。

After the work queue defers work, it hands it over to kernel threads for execution.During startup, Linux creates a worker kernel thread. After creation, this thread is in a sleep state. When there is work to be processed, it wakes up the thread to handle the work.。

In the kernel, work queues includeshared work queueandcustom work queueThese two types. These two types of work queues have different characteristics and purposes.

  • Shared work queueHandled by a set of kernel threads, each running on a CPU. Once a work task needs to be scheduled, the work is queued in the global work queue, and it will be executed at an appropriate time.
  • Custom work queueIt runs the work queue within a dedicated kernel thread. This means that whenever a work queue handler needs to be executed, a dedicated kernel thread is awakened to handle it, rather than one of the default predefined threads.

Shared work queue

The shared queue is a global work queue managed by the kernel.，Used to handle some system-level tasks in the kernel.. Unless there is no other choice, or critical performance is needed, or control
control every detail from work queue initialization to work scheduling, otherwise, if tasks are submitted only occasionally, the shared work queue provided by the kernel should be used. This queue is shared by the entire system, and it can be used but should not be exclusively occupied for a long time.

The shared work queue is a default work queue in the kernel that can be shared and used by multiple kernel components and drivers.

Since tasks pending on the queue are executed serially on each CPU, tasks should not sleep for a long time. Because before it wakes up, other tasks on the queue cannot run, and a task does not even know which tasks it shares the work queue with, so a task may take a long time to get the CPU.

Deferred work

Deferred work is atechnique of delaying the execution of work to a later point in time.

Typically,when a task takes a long time, does not need to be executed immediately, or needs to be executed on timeDelayed work will come in handy.

The basic idea of delayed work is to put tasks into a queue, and then the background worker process or task scheduler processes the tasks in the queue. Tasks can be executed after a specified delay time, or they can be sorted and processed according to priority, task type, or other conditions.

  1. Delayed work is often used to handle tasks that take a long time, such as sending emails, processing images, etc. By putting these tasks into a queue and delaying their execution, it can avoid blocking the main thread of the application and improve system responsiveness.
  2. Delayed work can be used to execute scheduled tasks, such as regularly backing up the database. By setting tasks to execute at a future point in time, it improves system reliability and efficiency.

Timer debouncing:

In the key interrupt, start a timer with a period of 10ms. When the timer expires, it triggers the timer interrupt. Finally, in the timer interrupt handler, read the key value. If the key value is still pressed, it indicates a valid key press.

Each time the interrupt handler is entered, the timer interrupt is restarted, so the timer interrupt is started at times t1, t2, and t3.

So although the timer is started at t1, the timer is reset at t2 before its time expires. Eventually, only the timer started at t3 can complete the entire timing period and trigger the interrupt. Then we can handle the key in the interrupt handler. This is the principle of using a timer for key debouncing. The key driver in Linux uses this principle!

Concurrency managed work queues

When using work queues, we first define a work structure, then add the work to the workqueue, and finally the worker thread executes the workqueue.

When new work is generated in the work queue, the worker thread executes each work in the work queue. When execution is complete, the worker thread sleeps, waiting for a new interrupt to occur, then work is added to the work queue again, and the worker thread executes each work, repeating the cycle.

Single-core system

In a single-core system, a worker thread is usually initialized for each CPU (core) and associated with a work queue. This default setting ensures that each CPU has a dedicated thread to handle work items on its bound work queue.

Multi-core system

In a multi-core system, the design of work queues differs from that in a single-core system.In a multi-core system, there are usually multiple work queues, each bound to a worker thread.. This allows full utilization of the parallel processing capabilities of multiple cores.

When a new work item is generated, the system needs to decide which work queue to assign it to. A common strategy is to useload balancing algorithm, which balances the allocation of work items based on the load of the work queues to avoid overloading a particular work queue and causing performance degradation.

Each work queue independently manages its own work items. When a new work item is added to the work queue, the worker thread retrieves the pending work item from its associated work queue and executes the corresponding handler function.

In a multi-core thread system, multiple worker threads can simultaneously execute work items from their respective bound work queues. This enables parallel processing, improving overall system performance and responsiveness.

Disadvantages of work queues

  1. When work item w0 is working or even sleeping, work items w1 and w2 are queued and waiting. In a busy system, the work queue may accumulate a large number of pending work items, causing task scheduling delays, which may affect system responsiveness and increase processing time for work items.
  2. In a work queue, different work items may have different processing times and resource requirements. If the processing times of work items vary greatly, some worker threads may be constantly busy processing long-duration work items while others remain idle, leading to unbalanced resource utilization.
  3. In a multi-threaded environment, multiple worker threads accessing and modifying the work queue simultaneously can lead to race conditions. To ensure data consistency and correctness, appropriate synchronization mechanisms such as locks or atomic operations are needed to protect shared data, but this may introduce additional synchronization overhead.
  4. Work queues typically process work items in a first-in-first-out (FIFO) manner, lacking fine-grained control over work item priority. In some scenarios, priority scheduling based on the importance or urgency of work items may be required, but the work queue itself cannot provide this level of priority control.
  5. When a worker thread retrieves a work item from the work queue and executes it, frequent context switches may be required, switching the processor’s execution context from one thread to another. This context switch overhead can affect system performance and efficiency.

Concurrency Managed Workqueue

CMWQ The full name is Concurrency Managed Workqueue, meaning concurrency-managed work queue. A concurrency-managed work queue is a concurrent programming pattern used to efficiently manage and schedule pending tasks or work items. It is commonly used in multi-threaded or multi-process environments to achieve concurrent execution and improve system performance.

When we need to process multiple tasks or jobs simultaneously in a system, using a concurrency-managed work queue is an effective approach.

Interrupt threading technique

Interrupt threading (threaded IRQ) is an optimization technique used to improve the performance of multi-threaded programs. The main goal isto minimize the time interrupts are disabled.。

In multi-threaded programs, sometimes hardware or other events issue interrupt signals, interrupting the currently executing thread, requiring a switch to the interrupt handler to process these events. This kind ofFrequent interrupt switching leads to additional overhead and latency, affecting program performance.

To solve this problem, interrupt threading proposes an optimization scheme. It separates the interrupt handler from the main thread and creates a dedicated thread to handle these interrupt events. In this way, the main thread is no longer disturbed by interrupts, can focus on its own work, and is no longer frequently interrupted.

The core idea of interrupt threading isto separate interrupt handling from the main thread’s work, allowing them to execute in parallel. The interrupt thread is responsible for handling interrupt events, while the main thread is responsible for executing the main work tasks. In this way, not only can the switching overhead be reduced, but also the overall program’s response speed and performance can be improved.

The processing of interrupt threading can still be seen as the original top half and bottom half of the interrupt.

The top half is still used to handle urgent matters, and the bottom half also handles relatively time-consuming operations, but the bottom half is handed over to a dedicated kernel thread for processing. This kernel thread is only used for this interrupt.

When an interrupt occurs, this kernel thread is awakened, and then this kernel thread executes the function of the bottom half of the interrupt.。

It should be noted that interrupt threading also needs to handle synchronization and data sharing between threads. Because the interrupt thread and the main thread may simultaneously access and modify shared data, it is necessary to perform synchronization operations properly to ensure data consistency and correctness.

  • handlerFunction: This is the same as usingrequest_irq()The function used when registering. It represents the top half function, running in atomic context (or hard interrupt). If it can handle the interrupt quickly, it may not need the bottom half at all, and it should returnIRQ_HANDLED. However, if interrupt handling requires100μsor more, as mentioned, then the bottom half should be used. In this case, it should returnIRQ_WAKE_THREAD, which causes scheduling ofthread_fnfunction (must be provided).
  • thread_fnFunction: This represents the bottom half, scheduled by the top half. When the hard interrupt handler (handler function) returnsIRQ_WAKE_THREAD, the kernel thread associated with this bottom half will be scheduled, and when the kernel thread runs, it calls thread_fn function. thread_fn function must return when completedIRQ_HANDLED. After execution, the interrupt is re-triggered, and before the hard interrupt returnsIRQ_WAKE_THREAD, the kernel thread will not be scheduled again.

Threaded interrupts can be used anywhere that work queues can schedule bottom halves. A true threaded interrupt must define handler and thread_fn.

If handler is NULL, and thread_fn is not NULL, the kernel will install a default hard interrupt handler that simply returns IRQ_WAKE_THREAD to schedule the bottom half.

When the interrupt handler executes, the interrupt it services is always disabled on all CPUs, and is re-enabled when the hard interrupt (top half) completes.

However, if for some reason it is necessary to not re-enable the interrupt line after the top half and keep it disabled until the threaded handler runs, then a threaded interrupt with the IRQF_ONESHOT flag should be requested. This way, the interrupt line will wait until the bottom half completes before being re-enabled.
