This article introduces the core mechanisms of Linux memory management, including the concepts of segmentation and paging, address concepts, and process address space layout. The article further discusses virtual memory area management, physical memory initialization, and common data structures, and summarizes the principles of multi-level page table mapping and the address translation process under ARM32 and ARM64 architectures.

About Addresses

Address space not isolated

  1. Malicious processes can arbitrarily modify the memory of other processes

  2. The efficiency of memory usage is very low, for example, when memory is short, all data in the process needs to be swapped to the swap partition

Early memory usage methods

  • Segmentation(Main idea: Map the virtual address space to the physical address space one-to-one; the program needs to be fully loaded into memory to run)
  • Paging(Main idea: When a program runs, allocate a page only when it is needed, i.e., allocate on demand; pages that are not needed for a long time can be swapped to disk)

Address Concepts：

  • Logical Address(Intel-specific terminology, the part of the offset address generated that is related to the segment)

  • Linear Address(Intel-specific terminology, an intermediate layer in the conversion from logical address to physical address; in segmentation, the logical address is the segment offset address, and adding the base address turns it into a linear address)

  • virtual address (Logical addresses and linear addresses are collectively referred to as virtual addresses)

  • Physical Address(The address required for the CPU to access physical memory via the external bus)

  • Process Address Space

TLB, also known as the translation lookaside buffer, caches the page table entries for page table translation

Memory Management Overview

Memory management from the process perspective:

  1. Typical 32-bit Linux system virtual address space layout:

    • User space (0-3G)：
      • Code segment & Data segment: Stores the program’s executable instructions and initialized data.
      • Heap space: Used for dynamic memory allocation (such asmalloc), grows towards higher addresses.
      • mmap space: Used for file mapping, shared memory, etc., grows towards lower addresses.
      • Stack space: Used for function calls and local variables, grows towards lower addresses.
    • Kernel space (3G-4G)：
      • All processes share this part of the address space, used for kernel code, data, and temporary mappings.

  2. Physical memory mapping：

    • Linear mapping: The kernel directly and linearly maps low-end physical memory (such asZONE_NORMAL) to a fixed area in the kernel address space, providing fast access.

    • High-end mapping: For high-end physical memory (such asZONE_HIGHMEM), the kernel temporarily maps it to the high-end part of the kernel address space only when needed.

  3. Process level:task_structandmm_struct

    • task_struct: Each process in the kernel is described by atask_structstructure, which contains all the information of the process, and itsmmmember points to the process’s memory descriptor.

    • mm_struct: This is the memory descriptor of the process, the core data structure for managing the process’s virtual address space. It mainly contains two key members:

      • mmap: Points to the root of the linked list or red-black tree of the process’s virtual memory areas (VMA).
      • pgd: Points to the process’s Page Global Directory (PGD), which is the top-level entry of the page table.

  4. Virtual Memory Area (VMA) Management

    • VMA（Virtual Memory Area）: The kernel usesvm_area_structstructure to describe a contiguous virtual address space. Each VMA represents a piece of virtual memory with the same attributes (such as readable, writable, executable), such as code segment, data segment, heap, stack, mmap mapping area, etc.

    • mm_structThrough themmapmember to organize all VMAs, making it convenient for the kernel to search, manage, and operate.

  5. mem_map[]array: The kernel uses a globalmem_maparray to manage all physical memory pages, where each element in the array is astruct pagestructure, representing a physical page frame.

  6. ZONE Management: Physical memory is divided into different memory management zones (ZONE), such asZONE_DMA、ZONE_NORMAL、ZONE_HIGHMEM, etc., each zone has different uses and management methods.

Page allocation mechanism

Buddy system

  • The buddy algorithm allocates memory blocks in sizes that are powers of 2, and these memory blocks are called buddies.
  • What is a buddy
    • Two blocks are of the same size
    • Two blocks have contiguous addresses
    • Two blocks must be split from the same large block

Migration type

  • To solve the problem of fragmentation, migration types were introduced in the buddy system.
    • MIGRATE_UNMOVABLE: Has a fixed location in memory and cannot be moved arbitrarily, such as memory allocated by the kernel.
    • MIGRATE_MOVABLE: Can be moved arbitrarily, memory allocated by user-space apps.
    • MIGRATE_RECLAIMABLE: Cannot be moved but can be deleted and reclaimed, such as file mappings.
  • Generation of memory fragmentation

page3 is used by functions allocated by the kernel itself, such as alloc_page(GPF_KERNEL), so it belongs to unmovable pages. Even though the other pages are free, page0 to page7 cannot be merged into a large memory block.

VMA operation

malloc

  • malloc stands for memory allocation, commonly known as dynamic allocation, and is the API mainly used in user space to allocate virtual memory.

  • calloc automatically initializes the allocated memory space to 0 after dynamic allocation, whereas malloc does not initialize it, and the data inside is random garbage data.

  • realloc modifies the size of a previously allocated memory block, allowing a block of memory to be expanded or shrunk.

  • malloc is an API encapsulated by the C library, which ultimately calls the brk system call.

  • glibc maintains a small warehouse; the implementation of the malloc function maintains a local small warehouse for the user process. When the process needs more memory, it requests goods from this small warehouse. When the small warehouse’s stock is insufficient, it wholesales from the kernel through the agent brk.

  • If malloc() is imagined as retail, then brk is the agent.

  • brk system call

  • The above figure searches for a free space in the heap space of the process’s address space, then creates a VMA, and then returns.
  • On-demand allocation (on-demand page)
    • The memory allocated by malloc may not be used for a long time, so allocating physical space immediately when the process allocates memory is actually not a good approach.
    • Solution: Page fault interrupt. When the process really needs to access these virtual pages, it is forced to create physical memory.

mlock system call

  • The mlock system call allows a program to lock part or all of its address space in physical memory. Calling mlock immediately allocates physical memory for the virtual memory.
  • This will prevent Linux from swapping this memory page to swap space.
  • In the implementation of the brk function, when finally returning the virtual memory address, it checks a variable VM_LOCKED, this VM_LOCKED is usually set from the mlock system call.
    • If there is, then mm_populate() needs to be called to immediately allocate physical memory and establish the mapping.
    • Generally, the work of allocating physical pages is delayed until the user process needs to access these virtual pages; physical memory is only allocated and mapped to the virtual address when a page fault occurs.

get_user_pages() function

  • A very important interface function for allocating physical memory. Many drivers use this API to allocate physical memory for user-mode programs.

follow_page() function

  • Returns the struct page data structure of the normal mapping pages that have already been mapped in the VMA of the user process address space.

vm_normal_page() function

  • This function divides page pages into two camps: one is normal page, and the other is special page.
    • normal page usually refers to normally mapped pages, such as anonymous pages, page cache, and shared memory pages.
    • special page usually refers to abnormally mapped pages. These pages do not want to participate in the reclamation or merging functions of memory management, such as pages mapped with the following characteristics:
      • VM_IO: Mapping memory for I/O devices
      • VM_PEN_MAP: Pure PFN mapping
      • VM_MIXEDMAP: Fixed mapping

Summary

  • The malloc function actually allocates process address space for user space. In kernel terms, it allocates a VMA, which is equivalent to an empty cardboard box. So when do you put things in the box? Two ways:
    • One is to put things in the box only when you actually use it.
    • The other is that the box is filled with what you want when it is allocated.
  • If two processes allocate the same virtual address with malloc, will they conflict?
    • Each user process has its own page table. Each process has an mm_struct data structure, which contains a page table belonging to the process itself, a red-black tree and a linked list for managing VMAs.
    • Even if process A and process B use malloc to allocate memory and return the same virtual address, they are actually two different VMAs, managed by two different sets of page tables respectively.

mmap

  • addr: Used to specify the starting address mapped to the process address space. For application portability, it is generally set to NULL, allowing the kernel to choose a suitable address.
  • length: Indicates the size mapped to the process address space.
  • prot: Used to set the read/write attributes of the memory mapping region.
    • PROT_EXEC: Indicates that the mapped page is executable.
    • PROT_READ: Indicates that the mapped page is readable.
    • PROT_WRITE: Indicates that the mapped page is writable.
    • PROT_NONE: Indicates that the mapped pages are inaccessible
  • flags: Used to set the properties of the memory mapping, such as shared mapping, private mapping, etc.
    • MAP_SHARED: Creates a shared mapping region. Multiple processes can map a file through shared mapping, so that other processes can also see the changes to the mapped content, and the modified content will be synced to the disk file
    • MAP_PRIVATE: Creates a private copy-on-write mapping. Multiple processes can map a file through private mapping, so that other processes will not see the changes to the mapped content, and the modified content will not be synced to the disk file
    • MAP_ANONYMOUS: Creates an anonymous mapping, i.e., a mapping not associated with a file
    • MAP_FIXED: Creates a mapping using the addr parameter. If the kernel cannot map the specified address addr, mmap will return failure. The addr parameter must be page-aligned. If the process address space specified by addr and length overlaps with an existing VMA region, the kernel will call do_The munmap() function destroys this overlapping region and then remaps the new content
    • MAP_POPULATE: For file mappings, it pre-reads the file content into the mapping region in advance. This feature only supports private mappings.
  • fd: Indicates that this is a file mapping, fd is the open file handle
  • offset: In file mapping, indicates the file offset

File mapping and anonymous mapping

  • Anonymous mapping: No corresponding file is mapped, the content of this mapped memory region will be initialized to 0
  • File mapping: The mapping is associated with an actual file, usually mapping the file’s content into the address space, so that the application can read and write the file just like the operating system process space

Private mapping and shared mapping

  • Private mapping: The created mapping can only be seen by itself, other processes cannot see the changes to the mapped content
  • Shared mapping: Other processes can see the changes to the mapped content. If a file is opened in a shared manner, the modified content will be synced to the disk file

  • The ARMv7 assembly processing flow is vectors_start -> vector_dabt -> datb_user / dabt_svc -> dabt_helpper -> v7_early_abort

  • Question 1: What is an anonymous page?

    • In the Linux kernel, pages that are not associated with a file mapping are called anonymous pages (Anonymous Page, abbreviated as anon page). For example, pages allocated bymalloc

  • Question 2: Why is a page fault interrupt needed for anonymous pages?

    • Without the anonymous page page fault interrupt, if you use themalloc()API in user space to allocate virtual memory, the kernel would have to actually allocate physical memory for it. The main problem is waste, because often applications allocate virtual memory and do not use it immediately, so there is no need to satisfy the application’s request right away

  • Question 3: What are the conditions for an anonymous page page fault interrupt to occur?

    • When the PRESENT bit is not set in the PTE page table entry
    • The PTE content is empty
    • Thevma->vm_ops->fault()function pointer is not specified

    When the above three conditions are met, we determine that it is an anonymous page page fault interrupt, and the handling function is: do_anonymous_page

  • Question 1: What is a file mapping?
    • A mapping associated with a file is a file mapping, which is the counterpart to anonymous mapping. It maps the contents of a file into the process address space
  • Question 2: Which types of mappings produce file mapping page fault interrupts?
    • The corresponding file mappings are mainly mappings of regular files, such as a video player reading a video file
    • Another case is when a device driver maps the device DMA buffer into the process address space viammap

Write page fault interrupt

  • The write page fault interrupt is the most complex scenario, and also the most commonly used one
  • Copy-on-write (COW) technology
    • When forking a child process, the kernel does not need to copy the entire user space content of the parent process to the child; instead, it lets the parent and child processes share the parent’s address space, thereby eliminating the very time-consuming copying operation and only needing to copy the page tables.
    • Data is only copied when one of the parent or child processes needs to write, at which point the parent and child processes will have their own copies. Therefore, the do_wp_page function here handles this sharing issue.
  • do_wp_What is the judgment condition of page()?
    • When the PTE_PRESENT bit is set and the page fault handling flags indicate a write protection error (this write protection error is usually reported by the hardware).

  • Overall, this process is quite complex and needs to handle many different situations:
    • normal mapping pages and special mapping pages
    • Pure anonymous pages, a made-up term, which refers to anonymous pages excluding KSM.
    • Single anonymous pages, also a made-up term, meaning that only one process’s virtual page maps this anonymous page, i.e., mapcount=0.
    • Non-single anonymous pages
    • Writable shared anonymous pages, which usually refer to pages with writable attributes and are shared, typically page cache.

Summary

  • After a page fault occurs, based on conditions such as the PRESENT bit of the PTE page table entry, whether the PTE content is empty (pte_none() macro), and whether it is a file mapping, the corresponding handling functions are as follows:
    • Anonymous page page fault do_anonymous_page()
      • Judgment condition
        • PRESENT is not set in the PTE page table entry
        • PTE content is empty
        • vma->vm_ops->fault() function pointer is not specified
      • Application scenario: malloc() allocates memory
    • File mapping page fault do_fault()
      • Judgment condition
        • PRESENT is not set in the PTE page table
        • PTE content is empty and vma->vm_ops->fault() function pointer is specified
        • do_fault() belongs to the case of page faults occurring in file mappings
        • If only a read error occurs, then call do_read_fault() function to read this page
        • If a write protection fault occurs in a private mapping VMA, Copy-on-Write (COW) is triggered. A new page (new_page) is allocated, the contents of the old page are copied to the new page, and a PTE entry is generated using the new page and set into the hardware page table entry. This is the so-called Copy-on-Write (COW).
        • If a write protection fault occurs in a shared mapping VMA, a dirty page is generated, and the system’s writeback mechanism is invoked to write back this dirty page.
      • Application scenarios:
        • Using mmap to read file contents, for example, using mmap in drivers to map device memory to user space.
        • Dynamic library mapping, for example, different processes can share a dynamic library through file mapping.
    • swap page fault handling_swap_page()
    • Copy-on-Write (COW) page fault handling_wp_page()
      • do_wp_The final handling of page() has two cases:
        • reuse old_page: single anonymous pages and writable shared pages
        • gotten Copy-on-Write: non-single anonymous pages, read-only or non-shared file-mapped pages
      • Judgment condition: the PRESENT bit in the PTE page table entry is set and a write fault page fault occurs.
      • Application scenario: fork, the parent process forks a child process, both parent and child processes share the parent’s anonymous pages. When one of them needs to modify the content, COW will occur.

pages data structure

After the MMU is enabled, the minimum unit of memory access by the CPU is a page (Page).

  • flag flag: PG_* flag bits
    • Defined in include/linux/page-flags.h

arm vexpress platform

  • _count field
    • When the value of _count is 0, it indicates that the page is free or about to be released.
    • When the value of _count is 0, it indicates that the page has been allocated and is being used by the kernel, and will not be released for the time being.

  • _mapcount field

    _The mapcount reference count indicates the number of times this page is mapped by processes, i.e., how many user PTE page tables it has been mapped to. In the 32-bit Linux kernel, each user process has 3GB of virtual space and an independent page table, so it is possible for multiple user process address spaces to map to the same physical page simultaneously. The RMAP reverse mapping system utilizes this feature to implement itself._The mapcount reference count is primarily used in the RMAP reverse mapping system.

    • _mapcount == -1 indicates that no PTE is mapped to the page.
    • _mapcount == 0 indicates the initial state of an anonymous page right after allocation (not yet mapped to a user process).

Usage:

  • The kernel code does not directly check the values of count and mapcount, but instead uses two macros.

  • static inline int page_mapcount(struct page *page)

  • static inline int page_count(struct page *page)

  • mapping field

    • When this page is used as a file cache, i.e., page cache, mapping points to the address_space object associated with this page cache, this address_space object belongs to a memory object (such as the page collection of an inode).
    • When this page is used as an anonymous page, mapping points to an anon_vma data structure, which is primarily used for Reverse Mapping.

  • lru field

    • Used in the LRU linked list algorithm for page reclaim; the LRU linked list algorithm defines multiple linked lists.
    • Used to add a slab to the slab full list, slab free list, and slab partial list.

  • virtual field

    • A pointer to the virtual address corresponding to the page.

  • The significance of struct page

    • The kernel knows the current state of this page (via the flags field).
    • The kernel needs to know whether a page is free, i.e., whether it has been allocated, and how many processes or memory paths are using this page (using the count and mapcount reference counts).
    • The kernel knows who is using this page, for example, is the user an anonymous page of a user space process or a page cache? (via the mapping field).
    • The kernel knows whether this page is used by the slab mechanism (via fields like lru, s_mem, etc.).
    • The kernel knows whether this page is linearly mapped (via the virtual field).

  • Page lock PG_Locked
    • The flags member of the struct page data structure defines a flag bit PG_Locked, the kernel usually utilizes PG_Use lock to set a page lock
      • The lock_page() function is used to request a page lock. If the page lock is occupied by another process, it will sleep and wait.
      • If trylock_page() returns false, it indicates that the lock acquisition failed; if it returns true, it indicates that the lock acquisition succeeded.
      • trylock_page() does not sleep and wait; it is only used to make a check.

Data structures used in reverse mapping:

RMAP four-step process (taking Linux 2.6.11 as an example)

  1. Parent process allocates an anonymous page

    1. do_anonymous_page()->page_add_anon_rmap()
    2. page_add_anon_rmap()

  2. Parent process creates a child process

    2. anon_vma_link(): add the child process’s VMA to the parent process’s vma->anon_vma->head linked list

  3. COW occurs in the child process

    1. When the parent and child processes share an anonymous page, COW occurs in the child process’s VMA

    Page fault -> handle_pte_fault()->do_wp_page()->allocate a new anonymous page -> page_add_anon_rmap()

    The newly allocated anonymous page’s page->mapping points to the parent process’s vma->anon_vma

  4. RMAP application

    1. Page reclaim: the page can only be reclaimed after all mapped user PTEs are disconnected
    2. Page migration: disconnect all mapped user PTEs

  • LRU is implemented based on zones. Each zone has a complete set of LRU linked lists

  • Second Chance Algorithm

    • Disadvantage of LRU Algorithm: It does not consider the frequent use of a page, and the page will still be evicted from the LRU linked list
    • Second Chance Algorithm:
      • Set an access status bit (hardware-controlled bit)
      • To check the page’s access bit
        • If it is 0, immediately evict it from the LRU list
        • If it is 1, it means it has been accessed again during this period. At this time, clear this access status bit, and then re-add this page to the head of the list, just like a newly added page.
    • The Linux kernel uses the following status bits to implement the second chance algorithm:
      • PTE_YONG hardware bit, when a page is accessed, the hardware automatically sets this bit
      • PG_active software bit
      • PG_referenced software bit

Page migration

NUMA:Non-Uniform Memory Access

  • Each processor hasits own local memory (Node Local Memory)。

  • can also access the memory of other processors (Remote Memory), but at a slower speed.

  • Commonly found inmulti-way servers, large multi-core systems。

UMA:Uniform Memory Access

  • All processors share a single physical memory。

  • Consistent access speed, which means the latency to access any memory is the same。

  • Commonly found insmall multi-core systems、Symmetric Multiprocessor systems (SMP)。

Causes of memory fragmentation

  • The physical memory of the Linux kernel is managed by the buddy system.
  • The buddy system consists of 11 linked lists of 2 to the power of order, with order ranging from 0 to 10.
  • The buddy system has a habit of cutting the cake.
  • The buddy system can have a magical repair function.

How to use KSM

  • Enable KSM:
    • madvise(addr,length,MADV_MERGEA_BLE)
  • Disable KSM:
    • madvise(addr,length,MADV_UNMERGEABLE)
  • The mmap function in Android’s bionic library enables KSM by default.

KSM statistics counters

  • run: Writing 1 to this node indicates starting the ksmd kernel thread, and writing 0 indicates stopping the ksmd kernel thread.
  • pages_to_scan: The number of pages scanned in a single scan; how many pages the ksmd kernel thread will scan each time it is woken up.
  • sleep_millisecs: How long ksmd sleeps before the next scan, in milliseconds.
  • pages_shared: The number of shared pages. If 1000 pages have the same content and are merged into one page, here pages_shared equals 1
  • pages_sharing: The number of sharable pages. If 1000 pages are merged into one page, then pages_sharing is 1000.
  • pages_unshared: The number of pages currently not merged, usually the number of unstable pages.
  • full_scans: The number of scans from beginning to end.
