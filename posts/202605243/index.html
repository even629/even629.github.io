    This article introduces the core data structures frequently used in the Linux 6.x kernel, detailing the definitions, core APIs, and typical usage scenarios of structures such as doubly circular linked lists, hash lists, red-black trees, radix trees, extensible arrays, and priority lists.

    Overview

    The Linux kernel defines a large number of general-purpose data structures that permeate various subsystems such as process scheduling, memory management, file systems, and device drivers. This article organizes them byactual usage frequencyfrom high to low, covering data structure definitions, core APIs, and typical usage scenarios.

    This article is based on theLinux 6.xkernel; some APIs may vary slightly between versions.

    hlist — hash linked list

    hlistis specifically designed forhash tablea doubly linked list variant. That is, when hashing the data to be stored, if a conflict occurs, usethe linked list methodto store the conflicting data. Typically, the order of element usage in a hash table is: data storage —> data retrieval —> data deletion. Compared tolist_headthe difference is that the head node uses only onestruct hlist_head(single pointer), saving memory in the hash table array.

    Its core design motivation is to solve the standard doubly circular linked listlist_headwhen used as a hash bucket, the existingmemory wasteandsemantic mismatchproblems.

    rbtree — Red-Black Tree

    the most used in the kernelself-balancing binary search tree, providing O(log n) search, insertion, and deletion. Each node of a red-black tree has a storage bit indicating the node’s color, which can be red or black. Properties of red-black trees:

    • Each node is either black or red.
    • The root node is black.
    • Every leaf node (NIL) is black. [Note: Here, leaf nodes refer to empty (NIL or NULL) leaf nodes!]
    • If a node is red, then its children must be black.
    • All paths from a node to its descendant nodes contain the same number of black nodes. This property ensures that no path is more than twice as long as any other, thus the red-black tree is a relatively balanced binary tree.

    All operations on a red-black tree must preserve its properties. Red-black trees are widely used, mainly for storing ordered data, with a time complexity of O(log n), making them highly efficient. cfs_rq uses a red-black tree to store tasks.

    Header file:<linux/rbtree.h>/<linux/rbtree_augmented.h>

    Data structure:

    At first glance, there seems to be no field for color, but this is a clever aspect of the red-black tree implementation here.__rb_parent_colorThis field actually contains both the color information and the parent node pointer. Since the field is of type long and requires alignment of sizeof(long), on typical 32-bit machines, the last two bits are always 0, so one of those bits can be used to represent the color.

    The key lies inmemory alignment：

    • struct rb_nodeis forced to be aligned tosizeof(long)alignment.
    • On a 32-bit system,sizeof(long) == 4, meaning any validrb_nodepointer address must be amultiple of 4, i.e., the lowest2 bitsare always00。
    • On a 64-bit system,sizeof(long) == 8, the lowest3 bitsare always000。

    In fact, the lowest bits are used here to represent color information. The following operations on the parent pointer and color information are essentially operations on therb_parent_colorbits.

    Linux’s red-black tree implementation is optimized for speed, thus having one less indirection layer (better cache locality) than traditional implementations. Eachstruct rb_nodeinstance of the structure is embedded in the data structure it manages, so there is no need to separate them with pointersrb_nodeand the data structure it manages.

    • Users should write their own tree search and insertion functions to call the provided red-black tree functions, rather than using a comparison callback function pointer.

    • Locking code is also left for the red-black tree user to write。

    Per-CPU variablesallocate independent memory copies for each CPU, allowing CPUs to access their own copies without locking and achieving high cache efficiency (variables are in the CPU’s local cache).

    APIDescription

    per_cpu(var, cpu)Access a specific CPU’s copy
    get_cpu_var(var)Get the local CPU’s copy (preemption disabled)
    put_cpu_var(var)Used with get_cpu_var, restore preemption
    this_cpu_ptr(ptr)Get pointer to this CPU’s copy
    per_cpu_ptr(ptr, cpu)Get pointer to specified CPU’s copy
    for_each_possible_cpu(cpu)Iterate over all CPUs
