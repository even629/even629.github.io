Debug core dump files

Linux core dump：

Generally referred to as core dump or kernel dump, we collectively call it a dump file. It is a mapping of a process’s memory information at a certain moment, meaning it contains the entire memory information and register information of the process at the time the dump file was generated. A dump file can be of a specific process or the entire system. It can be generated while the process is alive, or automatically when a process or the system crashes.

Creating a core dump file for a living process can generally be done through gdb. After attaching the process with gdb, execute the generate-core-file or gcore command to generate the core dump file.

More often, we analyze core dump files generated by crashes.

A watchpoint is a special breakpoint that interrupts when the value of an expression changes. The expression can be the value of a variable, or the values of one or more variables combined with operators, such as ‘a+b’. Sometimes called a data breakpoint.

When the program is paused, use the print command (abbreviated p) or the synonymous command inspect to view the current program’s runtime data, the format is:

  • is an expression in the program language being debugged
  • means format, for example, outputting in hexadecimal is /x

print(p) output format

Generally, GDB outputs the value of a variable based on its type. But you can also customize GDB’s output format. For example, you want to output an integer in hexadecimal, or binary to view the bits of this integer
type variable. To do this, you can use GDB’s data display format:

  • x Display variable in hexadecimal format. (hex)
  • d Display variable in decimal format. (decimal)
  • u Display unsigned integer in hexadecimal format. (unsigned hex)
  • o Display variable in octal format. (octal)
  • t Display variable in binary format. (two)
  • a Display variable in hexadecimal format. (address)
  • c Display variable in character format. (char)
  • f Display variable in floating-point format. (float)

expression

Expressions can be const constants, variables, functions, etc. in the current running program, but cannot be macros defined in the program

Program variables

In GDB, you can view the values of the following three types of variables at any time:

  1. Global variables (visible to all files)
  2. Static global variables (visible to the current file)
  3. Local variables (visible in the current Scope)

The value of the variable displayed by print will be the value of the local variable in the function. If you want to view the value of the global variable at this time, you can use the “::” operator:

example

Note: If your program was compiled with optimization options enabled, when debugging the optimized program with GDB, some variables may be inaccessible or have incorrect values. This is normal because the optimizer modifies your program, rearranges the order of statements, and removes meaningless variables. Therefore, when debugging such a program with GDB, the runtime instructions will differ from the instructions you wrote, leading to unexpected results. To deal with this situation, you need to disable compilation optimization when compiling the program. Generally speaking, almost all compilers support compilation optimization switches. For example, for the GNU C/C++ compiler GCC, you can use the “-gstabs” option to solve this problem.
