Debug core dump file

Linux core dump：

Generally referred to as core dump or kernel dump, collectively called dump files. It is a memory information map of a process at a certain moment, containing the entire memory information and registers of the process at the time the dump file was generated. Dump files can be for a single process or the entire system. They can be generated while the process is alive or automatically when the process or system crashes.

Creating a core dump file for a live process can generally be done using gdb. After attaching the process with gdb, execute the generate-core-file or gcore command to generate the core dump file.

More often, we analyze core dump files generated from crashes.

View runtime data

When the program is paused, use the print command (abbreviated p) or the synonymous command inspect to view the current program’s runtime data. The format is:

  • is an expression in the language of the program being debugged
  • means format, for example, output in hexadecimal is /x

print(p) output format

Generally, GDB outputs the value of a variable based on its type. But you can also customize GDB’s output format. For example, you want to output an integer in hexadecimal or binary to view this integer
the bits in a type variable. To do this, you can use GDB’s data display format:

  • x Display variable in hexadecimal format. (hex)
  • d Display variable in decimal format. (decimal)
  • u Display unsigned integer in hexadecimal format. (unsigned hex)
  • o Display variable in octal format. (octal)
  • t Display variable in binary format. (two)
  • a Display variable in hexadecimal format. (address)
  • c Display variable in character format. (char)
  • f Display variable in floating-point format. (float)

expression

Expressions can be const constants, variables, functions, etc. in the current program run, but cannot be macros defined in the program.

Program variables

In GDB, you can view the values of the following three types of variables at any time:

  1. Global variables (visible to all files)
  2. Static global variables (visible to the current file)
  3. Local variables (visible to the current scope)

The value of a variable displayed with print will be the value of the local variable in the function. If you want to view the value of a global variable at this point, you can use the ‘::’ operator:

example

Note: If your program is compiled with optimization options enabled, when debugging the optimized program with GDB, some variables may not be accessible or may return incorrect values. This is normal because the optimizer modifies your program, rearranges the order of statements, removes meaningless variables, etc. Therefore, when debugging such a program with GDB, the runtime instructions differ from the instructions you wrote, leading to unexpected results. To deal with this, you need to disable compilation optimization when compiling the program. Generally, almost all compilers support compilation optimization switches. For example, for GNU’s C/C++ compiler GCC, you can use the ‘-gstabs’ option to solve this problem.

examine (x) view memory

Use the examine command (abbreviated as x) to view the value at a memory address. The syntax of the x command is as follows:

  • n: number of units to display, starting from the memory address start, display several units (default is 1).

  • f: display format, for example:

    • x hexadecimal
    • d decimal
    • t binary
    • c character
    • f floating-point
    • s string
    • i instruction

  • u: size of read unit, determines how many bytes to read each time:

    • b = 1 byte
    • h = 2 bytes (half word)
    • w = 4 bytes (word, default)
    • g = 8 bytes (giant/quad word)

Basic concepts

inferior

gdb uses inferior to represent the state of a debugged process. Usually, an inferior represents a process. This is an internal concept and object of gdb. We can attach a running process to an inferior.

set schedule-multiple on/off

Allow multiple processes to execute simultaneously

Debug child process

By default, cannot debug child process

set follow-fork-mode child/parent

By default, follow-fork-mode is parent, meaning debugging follows the parent process. Only when set to child will it follow the child process.

set detach-on-fork on/off

By default, detach-on-fork is on, meaning when debugging the child process, the parent process continues to execute. The parent process may finish execution directly, making it impossible to debug the parent process. Therefore, if you want to debug both parent and child processes simultaneously, you need to
