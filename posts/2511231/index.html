This article introduces various debugging techniques for QEMU, including source-level debugging with gdb and vscode, remote debugging using the built-in gdbserver, and observing guest state through the logging system. Additionally, it discusses in detail the usage of tracing tools, the declaration and addition process of trace-events, and the tracing architecture supporting multiple backends, providing comprehensive guidance for tracing internal function execution and performance tuning in QEMU.

Source Debugging

Debugging QEMU source code does not require starting it first and then attaching with gdb like debugging the kernel; QEMU itself can be executed directly locally. (Note: enable-debug must be set during compilation.)

Remote Debugging

QEMU has a built-in gdbserver that can control the guest processor. Users can connect via any gdb client.

The specific operation commands are given below:

  • -S: Make QEMU stop at the first instruction of the guest, waiting for a gdb client connection.

Adding a new trace-event

Adding a new trace-event only requires two steps:

  • Declare the trace-event in the trace-events file of the corresponding directory
  • Add the function call for this event in the target source code that needs debugging.

Taking tracing QEMU memory allocation and release as an example, the format of the trace-event is as follows:

Each event declaration starts with the event name, followed by parameters, and finally a format string for pretty printing。

The format string should reflect the types defined in the trace event. Tracing only supports basic scalar types (char, int, long), not floating-point types (float, double)

Pay special attention to int64_t and uint64_t types use PRId64 and PRIu64 respectively, which ensures portability between 32-bit and 64-bit platforms.

The format string must not end with a newline character. The backend is responsible for adjusting line endings for proper logging.

The following scenarios are very suitable for using tracing debugging:

  1. Track state changes in code. Key points in code often involve state changes, such as start, stop, allocation, release, etc. State changes are ideal tracing events because they help understand the system execution process.
  2. Track guest operations. The guest’s I/O accesses (such as reading device registers) are good tracing events that can be used to analyze guest interaction behavior.
  3. Use correlation fields to understand the context of a single line of trace output. For example, track the pointer returned by malloc and its use as a parameter to free, so that malloc and free operations can be matched. Tracing events lacking context have limited practical value.

Introduction to trace backends

QEMU’s tracing adopts a front-end and back-end separation design, supporting multiple backends. In addition to the log mentioned above, it also supports the lighter-weight simple, as well as ftrace and dtrace.

We can also add more backend support through the tracetool script.

To enable different backends, you can use the following QEMU compilation command:
