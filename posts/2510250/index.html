This article introduces the basic architecture and core concepts of ARM Cache, discussing in detail the components of cache lines, cache types, mapping methods, and the advantages and disadvantages of physical and virtual caches. At the same time, this article summarizes the working principles and alias issues of cache architectures such as VIVT, PIPT, and VIPT, and briefly describes the hierarchical structure of multi-level cache systems.

Reference documents:

Arm Architecture Reference Manual Armv8

https://developer.arm.com/documentation/ddi0487/fc/

Classic cache architecture：

Cache internal architecture diagram

  • Cache line: The smallest unit of access in the cache
  • Index field: Used to index and find which line in the cache
  • Tag: Part of the cache address encoding, usually the high-order part of the cache address, used to determine whether the data address cached in the cache line matches the processor addressing address
  • Offset: The offset within the cache line. The processor can address the contents of the cache line by word or byte
  • Set：Same index field’s cache lines form a set
  • Way: In a set-associative cache, the cache is divided into several blocks of the same size

The main function of a set is to prevent cache “thrashing”

Cache types

On the ARM64 architecture, there are mainly:

  • Instruction Cache (I-Cache)
    Specifically caches instruction streams to speed up instruction fetching.
  • Data Cache (D-Cache)
    Caches data reads and writes (Load/Store).
  • Unified Cache
    Some levels (such as L2, L3) are often unified caches (storing both instructions and data), unlike L1 which strictly separates instructions/data.

Sometimes, a cache with both I-Cache and D-Cache is called a Separate Cache. It is a split cache structure: it has both an independent I-Cache and an independent D-Cache. Typically in L1 Cache。

Cache mapping methods

Direct mapping

When each set has only one cache line, it is called a direct-mapped cache (direct-mapping).

Example:

0x00, 0x40, and 0x80 are all mapped to the same cache line, causing frequent cache replacements and lower performance.

Fully associative

When the cache has only one set, meaning only one address in main memory corresponds to n cache lines, it is called fully associative.

Set associative

  • Taking a two-way set-associative cache as an example, where each way includes 4 cache lines, then for every two sets, there are two cache lines available for cache line replacement.
  • Reducing cache thrashing

Example:

  • The total size of the cache is 32KB, and it is 4-way, so the size of each way is 8KB: way_size = 32/4 = 8(KB)
  • The size of a cache line is 32 bytes, so the number of cache lines contained in each way is: num_cache_line = 8KB/32B=256

From this, the structure diagram of the cache can be drawn:

The index value is 12-5+1 = 8 bits, totaling 2^8=256, which can index 256 cache lines.

Physical cache

After the processor queries the MMU and TLB to obtain the physical address, it uses the physical address to query the cache.

Disadvantages: The processor can only access the cache after querying the MMU and TLB, which increases pipeline latency.

Virtual cache

The processor uses virtual addresses to address the cache

Disadvantages: It introduces many problems:

  • Aliasing problem
  • Homonyms problem

Aliasing problem

Also known as the aliasing problem

  • InIn an operating system, multiple different virtual addresses may map to the same physical address. Due to the virtual cache architecture, these different virtual addresses will occupy different cache lines in the cache, but they correspond to the same physical address
  • For example: Both VA1 and VA2 map to PA, and there are two cache lines in the cache storing VA1 and VA2
    • When a program writes data to VA1, the cache line corresponding to VA1 and the content of PA will be changed, but VA2 still holds the old data. Thus, a physical address stores two copies of data in the virtual cache, which causes ambiguity

Homonyms problem

  • The same virtual address corresponds to different physical addresses, because different processes in an operating system may have many identical virtual addresses, and these identical virtual addresses are translated into different physical addresses by the MMU, which creates the homonyms problem
  • The most common occurrence of the homonyms problem is during process switching. When a process switches to another process, if the new process uses virtual addresses to access the cache, it will access the cache left over by the old process, which is incorrect and useless for the new process. The solution is to invalidate the cache left over by the old process during process switching, ensuring that the new process gets a clean virtual cache when it executes.

Cache classification

  • VIVT (Virtual Index Virtual Tag): Usesthe index field and the tag field of the virtual address, which is equivalent tovirtual cache

    • When the CPU accesses the cache, it does not need to perform address translation (virtual address → physical address) first; it directly uses the virtual address to determine the cache line (index) and match the tag.
    • There is an aliasing problem (Synonym/Aliasing): When different virtual addresses map to the same physical address, multiple copies of data will appear in the cache, which may lead to data inconsistency.

  • PIPT (Physical Index Physical Tag): UsesPhysical address index field and physical address tag field, which is equivalent tophysical cache

    • The CPU first converts the virtual address to a physical address through the MMU, and then uses the physical address for cache indexing and matching.
    • No aliasing problem exists
    • Common L2 Cache

  • VIPT (Virtual Index Physical Tag): Usesvirtual address index field and physical address tag field

    • CPU Uses the lower bits of the virtual address as the cache index。Uses the physical address as the tag, for matching and determination. Becausecache line alignment (usually 64 bytes or 128 bytes), the lower bits of the virtual address are consistent with the lower bits of the physical address, so it is safe to use the virtual address for indexing.
    • avoids the aliasing problem, because the physical address is used as the tag
    • Limited by Page Size Limitation: The virtual index length must be ≤ the number of page offset bits, otherwise the same virtual index from different pages will conflict. (VIPT uses the lower bits of the virtual address to index the cache line)
    • Common L1 Cache

VIPT working process

From a general perspective:

From a VIPT perspectivePODivided into:

  • The VIPT cache’sindex bits come only from the page offset part of the virtual address(rather than the VPN), this is to prevent the index bits from being affected by virtual address translation (the PO is consistent in both virtual and physical addresses).
  • Cache line offset (Line Offset) is used to locate specific bytes within a cache line

Assume L1 Cache is 64KB with a cache line size of 64B:

  • Number of index bits = log₂(64KB / 64B) = log₂(1024) = 10 bits

  • Therefore, the cache index uses the virtual address’s 10 bits

  • Page offset = 12 bits (4KB page)

  • The page offset in virtual and physical addresses isexactly the same(paging only translates the VPN, not the PO).

  • If the index bits fall entirely within the page offset (i.e., index bit length ≤ page offset bit length), then the cache index uses “virtually identical” address bits, which can avoid the cache alias problem (Alias Problem) caused by different VPNs but the same physical page.

Therefore, the requirement is: index bits ≤ page offset bits

The left and right steps proceed simultaneously

VIPT alias problem

When two virtual pages are mapped to the same physical page simultaneously, both virtual pages fill up one way of the Cache together.

As shown in the figure, after Virtual Page1 is overwritten, Virtual Page2 still accesses the original data.

This alias problem can be avoided bycache layout , that is, making the VIPT cache’sindex bits come only from the page offset part of the virtual address(rather than the VPN)

Cache Policies

  • Cache-related policies areConfigured in the MMU page table。Only Normal memory can be cacheable

  • Cache policies include:

    • Cacheable/non-cacheable
    • Cacheable subdivisions
      • Read/write-allocate
      • Write-Back cacheable, write-through cacheable
      • Shareability

  • Cache allocation policy

    • Write allocation（WA）: Allocate a new cache line only on a write miss
    • Read allocation（RA）: Allocate a new cache line only on a read miss

  • Cache write-back policy:

    • Write-back（WB）: Write-back operation only updates the cache and does not immediately update the memory (cache line is marked as dirty)
    • Write through（WT）: Write-back operation directly updates both the cache and memory

Write Back and Write Through

  • WT Write-Through mode
    • During a write operation, data is written simultaneously to the current cache, the next-level cache, or main memory
    • Write-through mode can reduce the difficulty of implementing cache coherence, but its biggest drawback is consuming more bus bandwidth
    • ARM Cortex-A series processors treat WT mode as Non-cacheable
      • The Cortex-A72 processor memory system treats all Write-Through pages as Non-cacheable

  • WB Write-Back mode

    • Writes only update the cache and mark the cache line as dirty. External memory is updated only when the cache line is flushed or explicitly cleaned.
    • Cache line becomes Dirty data

Inner and Outer Shareability

  • Normal memoryInner or outer shareability can be set
  • How to distinguish between inner or outer, different designs have different distinctions
    • inner attributeUsuallyCPU IPIntegrated caches
    • outer attribute are exported on the bus

  • The inner attribute is the internally integrated cache
  • The outer attribute is the external cache mounted on the external bus

Point of Unification (PoU) and Point of Coherency (PoC)

  • PoU: indicates that in a CPU, theinstruction cache，data cacheandMMU，TLBetc., see the same copy of memory
    • PoU for a PE, means guaranteeing thatPEwhat is seenI/D cacheandMMUis the same copy. In most cases,PoU is observed from the perspective of a single-core system
    • PoU for inner share, meaning that ininner shareall insidePEcan see the same copy
  • PoC: all observers in the system, such asDSP, GPU, CPU, DMAetc., can see the same memory copy

Cache maintenance

  • Cache management operations
    • Invalidate (Invalidate) the entire cache or a specific cache line. Data on the cache will be discarded.
    • Clean (Clean) the entire cache or a specific cache line. The corresponding cache line will be marked as dirty, and the datawill be written back to the next level cache or main memory(also known as flush)
    • Zero (Zero) operation
  • Cache management objects
    • ALL : Entire cache block
    • VA: A certain virtual address, sometimes referred to as MVA (Modified Virtual Address, which is the cache line containing the specific virtual address)
    • Set/Way: Specific cache line or set and way
  • Scope of cache management
    • PoC
    • PoU
  • Shareability
    • inner

Cache instruction format

Instructions that accept address parameters use a 64-bit register that holds the virtual address to be maintained. This address has no alignment restrictions.

The AArch64 data cache invalidate by address instruction DC IVAC requires write permission; otherwise, a permission error will be generated.

Example 1

Traverse all CPU data cache (Data Cache) levels (L1/L2/L3…), and perform a clean by set/way on each cache level to ensure that all data in the cache is written back to memory:

CLIDR_EL1

Bit fieldMeaning
bits [26:24]LoC = Level of Coherency (Cache coherency level × 2)
bits [2:0]L1 cache type
bits [5:3]L2 cache type
……

Overall code flow:

StepActionExplanation
1Read CLIDR_EL1Get the number of Cache levels and types in the system
2Parse LoCDetermine how many levels of cache need to be traversed
3Iterate through each levelProcess each cache level sequentially from L1 → L2 → L3…
4Select the cache level using CSSELR_EL1Tell the CPU: which level of Cache I am going to access next
5Read CCSIDR_EL1Query the detailed configuration of this Cache (number of sets, number of ways, line size)
6Double loopIterate through Set and Way, cleaning the cache line by line
7DC CSWExecute Clean by Set/Way
8DSB+ISBData Synchronization Barrier, ensuring the cleaning is complete before continuing execution

Under normal circumstances, cleaning or invalidating the entire cache is something only firmware should do, as part of the kernel power-up or power-down sequence. It can also take a lot of time, the number of lines in the L2 cache can be very large, and it is necessary to loop through them one by one. Therefore, this kind of cleaning is absolutely only suitable for special occasions!

This code is standard Self-modifying code / JIT compilation synchronizationSteps:

Cache discovery

  • When we do cache instruction management, you need to know the following information:
    • How many levels of cache does the system support?
    • What is the Cache line size
    • For each level of cache, what are its set and way counts
    • For the zero operation, we need to know how much data can be zeroed?

  • Cache Level ID Register (CLIDR, CLIDR_EL1): Lists how many levels of cache there are; can read the number of cache levels
  • Cache Type Register(CTR, CTR_EL0): cache line size; can read the size of the cache line
  • If this needs to be accessed by user code running at execution level EL0, this can be done by setting the system control register (SCTLR/SCTLR_EL1) 's UCT bit.
  • sets and ways: requires accessing two registers to obtain
    • tells the Cache Size Selection Register (CSSELR, CSSELR_EL1) which cache to query
    • read relevant information from the Cache Size ID Register (CCSIDR, CCSIDR_EL1)

Other notes on Cache Discovery

  • The Data Cache Zero ID Register (DCZID_EL0) contains the block size to be zeroed by the zeroing operation.

    • DC ZVA (Data Cache Zero by Virtual Address)

      • Zeroes the memory corresponding to a cache line
      • DCZID_EL0 Stores the block size that can be zeroed
      • Access permissions: Only privileged levels (EL1 and above) can access DCZID_EL0

    • Control bits：

      • SCTLR_EL1.DZE: Controls whether DC ZVA is allowed at EL0
      • HCR_EL2.TDZ: Controls whether DC ZVA is allowed at EL0/EL1 in the Non-secure world

  • SCTLR/SCTLR_The [DZE] bit of EL1 and the Hypervisor Configuration Register (HCR/HCR_EL2) control which execution levels and which worlds can access DCZID_EL0。CLIDR_EL1、CSSELR_EL1 and CCSIDR_EL1 can only be accessed by privileged code, that is, PL1 or higher in AArch32, or EL1 or higher in AArch64.

  • If the Data Cache Zero by Virtual Address (DC ZVA) instruction is prohibited at an exception level, EL0 is controlled by SCTLR_EL1.DZE bit, and the Non-secure execution in EL1 and EL0 is controlled by HCR_EL2’s .TDZ bit; reading this register then returns a value indicating that the instruction is not supported.

  • The CLIDR register only knows how many levels of cache are integrated into the processor itself. It cannot provide information about any caches in the external memory system. For example, if only L1 and L2 are integrated, CLIDR/CLIDR_EL1 identifies two levels of cache, and the processor is unaware of any external L3 cache. When executing cache maintenance or code that maintains coherency with integrated caches, non-integrated caches may need to be considered.

Cache Writeback Granule

Typically,CWG = maximum cache line size, because write-back is done in units of entire lines (cache lines).

However, the ARM specification allows for microarchitectural differences:

  1. CWG ≥ cache line size: Some implementations might merge multiple cache lines into a larger burst write during write-back, such as 128B.
  2. CWG = cache line size: Common case, e.g., line=64B → CWG=0b0100 (16 words).
  3. CWG not provided (0b0000): must assumethe maximum write-back granularity is 2KB, or calculate it yourself by reading the Cache Size ID Registers.
