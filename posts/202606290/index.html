This article introduces the implementation mechanism of the Rpmsg Char Driver character device driver based on Linux 5.10.x. It discusses in detail the classic architecture design where the control device and endpoint device are separated, and summarizes the driver registration, delayed endpoint creation, lock and wait queue synchronization mechanisms, as well as the specific flow and key code logic for user-space message sending and receiving.

The current code analysis is based on Linux 5.10.x

This separation of ‘control device + data device’ is a classic pattern in Linux:The control node is used to instantiate the data node, and the data node carries the actual IO。

Lock and wait queue:

  • struct mutex ept_lock;: Protect ept pointer. ept becomes NULL in three cases:release、destroy ioctl, parent deviceremove. The read/write path must first acquire this lock to confirm ept is still alive.
  • spinlock_t queue_lock;: Protect queue (skb linked list). Callbackrpmsg_ept_cbEnqueues in interrupt/atomic context, read_iter dequeues in process context, so a spinlock is used here and in cb usespin_lock, in read usespin_lock_irqsave。
  • wait_queue_head_t readq;：rpmsg_ept_cbAfter enqueuingwake_up_interruptible，read_iterWhen the queue is emptywait_event_interruptible. This is a typical ‘producer-consumer + wait queue’.

rpmsg_chrdev_probe()

When the rpmsg bus assigns arpmsg_devicematch torpmsg_chrdev_drivercalled whenrpmsg_chrdev_probe. Process:

  1. kzalloc a ctrldev,ctrldev->rpdev = rpdev。
  2. device_initialize+ setparent/class，cdev_init(&ctrldev->cdev, &rpmsg_ctrldev_fops)。
  3. twiceida_simple_get: fromrpmsg_minor_idaget the secondary device number, fromrpmsg_ctrl_idaget id (forrpmsg_ctrl%d）。
  4. cdev_device_add— register cdev and device together,/dev/rpmsg_ctrlNappear.
  5. Key point:dev->release = rpmsg_ctrldev_release_deviceis incdev_device_addonly assign afterwards. The comment says “We can now rely on the release function for cleanup” — meaningcdev_device_addIf it fails before success, go through manual error label release; after success, hand over to the device model’s release callback as a fallback (kfree when the last reference goes to zero). This is a very standard refcount + release pattern in the Linux device model.
  6. dev_set_drvdata(&rpdev->dev, ctrldev)— so that during remove, ctrldev can be retrieved from rpdev.

rpmsg_eptdev_create()

  1. kzalloc eptdev, fill rpdev, chinfo.
  2. Initialize lock, skb queue, wait queue head.
  3. device_initialize + setclass/parent/groups/drvdata。groups = rpmsg_eptdev_groupsprovide name/src/dst read-only attributes to sysfs
  4. cdev_init usesrpmsg_eptdev_fops。
  5. twice ida_simple_get minor and ept id,dev_set_name(dev, "rpmsg%d", ret) →/dev/rpmsgN。
  6. cdev_device_add, after success set dev->release.
  7. Note: at this timeeptdev->ept == NULL! The real endpoint has not been created yet.The endpoint is created only when open. This is lazy creation to avoid occupying an address that no one uses.

The goto order of the error handling path is strictly opposite to the order of resource acquisition, which is the standard unwind style. Note free_put in eptdev_after device then kfree — there is actually a detail here:device_initializeafter that should use put_device lets the release callback free it, but because release is in cdev_device_set only after add succeeds, not set in this path, so put_device will not actually kfree, need to manually kfree. This is exactly the cost of delayed assignment of release.

rpmsg_eptdev_open()

  • rpmsg_create_eptforwarded to core and then to the virtio backend’svirtio_rpmsg_create_ept→__rpmsg_create_ept（virtio_rpmsg_bus.c:). The latter allocates a local address in idr (ifchinfo.src == RPMSG_ADDR_ANYthen dynamically allocate from 1024), put the callbackrpmsg_ept_cbandpriv=eptdevbound to that address.
  • After the callback is registered, messages sent remotely with dst equal to this local address will triggerrpmsg_ept_cb (see backendrpmsg_recv_singlebymsg->dstlook up idr and callept->cb）。
  • get_device/ open failedput_device: Ensure that the device will not be released while the file is open.

And the callback of ept is as follows:

  • The backend calls ept->cb in the virtqueue callback (softirq context), so mutex cannot be used here, cannot sleep, use GFP_ATOMIC and spinlock.
  • Here a memory copy (buf → skb) is performed. When the backend receives, it gets the vring buffer with zero copy, but in order to hand the data to user space and not let the buffer be held by the user for a long time (the backend needs to return the buffer to rvq as soon as possible), a copy is made into skb.rpmsg_recv_singleImmediately after calling cb,virtqueue_add_inbufreturn the original buffer.

rpmsg_eptdev_read_iter()

  • Check simultaneously in the wait condition!ept: In this way,When destroy sets ept to NULL and calls wake_up, the blocked read can immediately wake up and return -EPIPE, and will not be stuck forever.. This is the standard way to use “resource disappearance” as the wake-up condition.
  • One read retrieves one message: rpmsg is a protocol with clear message boundaries,One skb = one rpmsg message。use = min_t(size_t, iov_iter_count(to), skb->len);It means that if the user buffer is smaller than the message, the excess part is discarded (skb is directly freed). This is a pitfall for beginners: the buffer must be large enough (the backend buffer is 512 bytes minus 16 bytes header = 496 bytes payload).
  • Usespin_lock_irqsaveinstead ofspin_lock: Because the read may be in normal process context, but the lock may be held by the cb in interrupt context, disabling interrupts is safer.

rpmsg_eptdev_write_iter()

  • Firstcopy_from_iterthen acquire the lock, becauseUser-space copy may cause page fault and sleep, should not be done while holding ept_lock — otherwise it will block the destroy path for a long time.

  • rpmsg_sendThe blocking version will go all the way to the backendrpmsg_send_offchannel_raw(..., wait=true): If there is no TX buffer,wait_event_interruptible_timeout(..., 15000)。rpmsg_trysendiswait=false, if no buffer, immediately-ENOMEM。

  • On success, return len (number of bytes written), not ret (backend ret is 0). Note that a single message is limited by the backend buffer:len > buf_size - sizeof(hdr)That is, 496 bytes will-EMSGSIZE。

  • Sending is also one write = one message, no streaming concatenation.
