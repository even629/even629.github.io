Yocto Project / OpenEmbedded build systemavailable indistribution configuration

  1. poky-master

    • Pokyis the reference distribution of the Yocto Project.

    • -masterindicates usingthe latest development main branchcode, i.e.,a version that has not been released and is under development。

  2. oe-nodistro-master

    • **OpenEmbedded (OE)**is the underlying build framework of the Yocto Project.

    • nodistroindicates “no distribution” mode —not bound to any specific distribution strategy, providing only the most basic build capabilities.

    • -masteralso indicates usingDevelopment mainline。

  3. oe-nodistro-whinlatter

    • Based onOpenEmbedded’snodistroconfiguration;

    • Using**Yocto 5.3 version codenamed “whinlatter”**stable branch;

    • It is a long-term support version;

    • nodistroindicates “no distribution” mode.

  4. poky-whinlatter

    • Poky distribution’s5.3 “whinlatter” stable version；

    • Includes complete default configuration (e.g., using systemd, optional RPM/deb package formats, default toolchain, etc.);

    • It is a long-term support version;

    • It is the official test and certification reference platform for the Yocto Project.

“Whinlatter” is the codename for Yocto Project 5.3(Yocto versions since 4.0 are named after bird species; 5.3 = Whinchat + Lark → “Whinlatter”).

BitBake build configuration (layers / templates)

  1. poky

    • This isthe standard Poky build configuration。

    • All packages (recipes) are built completely from source (unless you already have a local cache).

    • Secure, reliable, and predictable behavior.

  2. poky-with-sstate

    • Enabledremote shared state (sstate) cache。
    • During the build, it attempts todownload precompiled intermediate artifacts (such as compiled libraries, kernel modules, etc.) from the official Yocto or specified sstate mirror server,therebysignificantly speeding up the build。
    • but with strict prerequisites：
    • YourThe local network must be very stable and have sufficient bandwidth.；
    • Must be able to access external sstate servers (usually requiring internet).
    • If the network is interrupted or the mirror does not match, it may cause build failures or inconsistencies.
    • “Use with caution” is precisely a reminder of this.

**sstate is Yocto’s caching mechanism.**It saves the output of tasks (such asdo_compileresults), which can be directly reused the next time the same content is built, without redoing.

Target Machine

  1. machine/qemux86-64

    • Simulate a PC with x86_64 architecture,usingQEMU virtual machine,。

    • based on standard PC hardware models (such as Intel/AMD 64-bit CPUs, common peripherals).

  2. machine/qemuarm64

    • Simulate ARM64 (AArch64) architecturevirtual devices (such as systems based on Cortex-A57).

    • Also run via QEMU, without needing a real ARM board.

  3. machine/qemuriscv64

    • Simulated RISC-V 64-bit architecturevirtual machine.

  4. machine/genericarm64

    • Generic ARM64 real hardwarereference configuration (not an emulator).

    • It is not tied to a specific board (e.g., Raspberry Pi, BeagleBone), but provides a “generic” ARM64 BSP.

    • The generated image needs to be flashed toa real ARM64 development boardto run.

    • It may lack specific board-level drivers (e.g., GPU, WiFi), requiring self-adaptation.

  5. machine/genericx86-64

    • Generic x86_64 real hardwarereference configuration (e.g., ordinary PC, Intel NUC, industrial PC).

    • The generated image can be directly written to a USB drive and booted on a real x86_64 computer.

Distribution configuration variants

  1. distro/poky

    • Standard Poky distribution configuration. Includes a complete Linux system: glibc, systemd, package manager (RPM or IPK), common tools (bash, coreutils, networking, etc.).
    • By default, systemd is used as the init system.
    • Suitable for general development, learning, and most embedded application scenarios.
    • The image is larger (hundreds of MB) but fully functional.

  2. distro/poky-altcfg

    • An alternative configuration of Poky.

    • Typically used to test different combinations of underlying components, for example:

      • Usingmusl libcinstead of glibc;
      • Usingbusybox + sysvinitinstead of systemd;
      • Smaller default images or different security policies.

    • Specific behavior depends on the Yocto version,Less documentation and less stable than standard Poky。

    • Primarily used forYocto internal testing or advanced customization scenarios。

  3. distro/poky-tiny

    • Minimal version of Poky, designed specifically for resource-constrained devices.
    • Features:
      • Usesmusl libc(smaller, faster);
      • Usesbusybox + sysvinit(without systemd);
      • Removed a large number of non-essential packages;
      • The final root filesystem can be as small as10–20 MB。
    • Limited functionality: may lack network tools, simplified shell features, no package management.
    • Suitable for: microcontroller-level applications, minimal environment after bootloader, safety-critical systems.
