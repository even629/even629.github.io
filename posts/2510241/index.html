This article introduces the basic concepts and paging mechanisms of ARM memory management, discusses the evolution from single-level to multi-level page tables, and the address translation principles of the MMU hardware unit under the VMSA architecture. This article summarizes the page table format, virtual address space division, and the types and memory attributes of the four-level page table descriptors under the ARMv8 architecture.

Reference documents:

Arm Architecture Reference Manual Armv8

https://developer.arm.com/documentation/ddi0487/fc/

Basic knowledge and concepts of memory management

Disadvantages of using physical memory directly

  • Process address space protection issues. All user processes can access the entire physical memory, so malicious programs can modify the memory data of other programs
  • Low memory usage efficiency. If the memory space required by the upcoming process is insufficient, a process must be selected for overall swapping out. This mechanism results in a large amount of data needing to be swapped in and out, which is extremely inefficient
  • Program runtime address relocation issues

Basic concepts of the paging mechanism

  • Virtual Memory
  • Virtual Address Space (Virtual Address)
  • Physical Memory
  • Page Frame
  • Virtual Page Frame Number
  • Physical Frame Number
  • Page Table (PT)
  • Page Table Entry (PTE)

The process of mapping virtual addresses to physical addresses

Single-level page table

Disadvantages of using a single-level page table

  • The processor uses a single-level page table, the virtual address space has a bit width of 32 bits with an addressing range of 4GB, the physical address space bit width is also 32 bits, supporting a maximum of 4GB of physical memory, and the page size is 4KB. To map the entire 4GB address space, 4GB/4KB = 1M page table entries are required, each page table entry occupies 4 bytes, requiring a total of 4MB of physical memory to store this page table
  • **Each process has its own set of page tables, and the page table base address needs to be switched during process switching.**As with the single-level page table mentioned above, each process needs to be allocated 4MB of contiguous physical memory to store the page table, which is unacceptable because it wastes too much memory.
  • Multi-level page table: mapping level by level on demand, without needing to map the entire address space all at once

Level 2 page table

VMSA

Virtual Memory System Architecture

  • VMSA provides the MMU hardware unit
    • Virtual address to physical address translation
    • Access permissions
    • Memory attribute check
  • The MMU hardware unit is used to implement VA to PA translation
    • Hardware page table walking
    • The TTBR register holds the page table base address
    • The TLB holds the most recently used VA to PA translation results

In the absence of virtualization, translation has only one stage, mapping from VA to PA

In the presence of virtualization, translation needs to first convert to IPA

TTBR0_ELx Used for the address space of each process

TTBR1_ELx Used for kernel space, shared by all processes

ARMv8 page table

  • AArch64 only supports the Long Descriptor page table format

  • AArch32 supports two page table formats

    • Armv7-A Short Descriptor format
    • Armv7-A (LPAE) Long Descriptor format

  • AArch64 supports three different page sizes: 4KB, 16KB, 64KB

    • Larger page size can reduce the size of the page table

    • Address bus width supports 48-bit or 52-bit

    • 52-bit width: ARMv8.2-LVA is implemented and the 64 KB translation granule

    • Taking 48-bit bus width as an example

    • The virtual address VA is divided into two spaces，Each space supports a maximum of 256TB

      • The lower virtual address space is located from 0x0000_0000_0000_0000 to 0x0000_FFFF_FFFF_FFFF
      • The higher virtual address space is located from 0xFFFF_0000_0000_0000 to 0xFFFF_FFFF_FFFF_FFFF

      （Fault is a non-canonical region, which the CPU cannot access）

Only points to Final physical page (block/page) entries require page table attributes

ARM ARM D5.3.3Chapter

    • 0= Invalid (Fault entry)
    • 1= Valid (Valid entry)
  • bit[1] → Type (only meaningful when valid):
    • 0= Block entry (block mapping, large page mapping)
    • 1= Table entry (points to the next-level page table; becomes a Page entry at the last level)

Share Domain

Non-shareable

This represents memory accessible only by a single processor or other agent, so memory accesses never need to be synchronized with other processors. This domain is not typically used in SMP systems.

Inner shareable

This represents a shareability domain that can be shared by multiple processors, but not necessarily all of the agents in the system. A system might have multiple Inner Shareable domains. An operation that affects one Inner Shareable domain does not affect other Inner Shareable domains in the system. An example of such a domain might be a quad-core Cortex-A57 cluster.

Outer shareable

An outer shareable (OSH) domain re-order is shared by multiple agents and can consist of one or more inner shareable domains. An operation that affects an outer shareable domain also implicitly affects all inner shareable domains inside it.
However, it does not otherwise behave as an inner shareable operation.

Full system

An operation on the full system (SY) affects all observers in the system.

Contiguous Block entries

  • ARMv8An optimization using TLB: usingone TLB entry to complete the VA-to-PA translation for multiple consecutive pages
  • Conditions for using the Contiguous bit
    • The VA corresponding to the pages must be contiguous
    • For 4KB pages, 16 consecutive pages
    • For 16KB pages, 32 or 128 consecutive pages
    • For 64KB pages, 32 consecutive pages
    • Consecutive pages must have the same attributes
    • The starting address must be page-aligned

4KB page table

  • 4-level page table
  • 48-bit virtual address
  • Each level of the page table uses 9 bits for indexing (512 entries)

16KB page table

  • 4-level page table

  • 48-bit virtual address

  • L0 page table has only two entries

  • L1, L2, and L3 page tables use 11 bits for indexing (2048 entries)

64KB page table

  • 3-level page table
  • 48-bit virtual address
  • L1 page table has only 64 entries
  • L2 and L3 page tables use 13 bits

Separate dual page table design

  • User space (EL0) and kernel space (EL1) adopt a dual separate page table base address design
    • TTBR1_EL1 is selected when the high 16 bits of the virtual address are 1
    • TTBR0_EL1 is selected when the high 16 bits of the virtual address are 0

System registers related to the page table

TCR_EL1

Translation Control Register

Configure address space size and page table granularity

Configure address space size and page table granularity

  • IPS: Intermediate Physical Address Size, used to configure the physical address size, e.g., 48bit, 256TB physical space
  • TG1andTG0: Configure the size of the page table granularity, e.g., 4KB, 16KB, 64KB
  • T1SZ: Used to configure the size managed by the TTBR1_EL1 page table, calculated as 2^(64-T1SZ) bytes
  • T0SZ: Used to configure the size managed by the TTBR0_EL1 page table, calculated as 2^(64-T0SZ) bytes

In ARM64, the high bits of the virtual address are not used arbitrarily, but are restricted by TCR_EL1.T0SZ / T1SZ restrictions.

Fields related to Cache

  • SH1: Set memory-related cache attributes, this memory is accessed via the TTBR1_EL1 page table. E.g., Non-shareable, Outer Shareable, Inner Shareable
  • SH0: Set memory-related cache attributes, this memory is accessed via the TTBR0_EL1 page table

  • ORGN1: Set Outer Shareable related attributes
  • ORGN0: Set Outer Shareable related attributes
  • IRGN1: Set Inner Shareable related attributes
  • IRGN0: Set Inner Shareable related attributes

TCR_The fields SH0/SH1, IRGN0/IRGN1, ORGN0/ORGN1 in EL1, they are only used for memory regions without address translation (i.e., when the MMU is disabled), or as “default” attributes in certain special contexts. Once the MMU is enabled, memory attributes are completely determined by page table entries + MAIR_EL1.

SCTLR_EL1

System Control Register (EL1)

  • M: Enable and disable MMU
  • I: Enable instruction cache
  • C: Enable data cache

TTBR0_EL1

Points to the base address of the TTBR0 page table, usually used for EL1/EL0 page table mapping

TTBR1_EL1

Base address pointing to the TTBR1 page table, typically used for page table mapping at EL1/EL0

CPACR_EL1

Architectural Feature Access Control Register

FPEN field (CPACR_EL1[21:20])

FPEN = Floating-point Enable controls, controlling Whether EL0/EL1 access to SVE, Advanced SIMD, and floating-point registers is trapped by EL1/EL2。

  • Register impact：
    • AArch64:
      • FPCR、FPSR
      • SIMD/Floating-point registersV0-V31(including D0-D31 / S0-S31 views)
    • AArch32 / Advanced SIMD:
      • FPSCR
      • Q0-Q15 (including D0-D31 / S0-S31 views)
  • Exception reporting：
    • EL0/EL1 trap → EC syndrome =0x07
    • EL2 trap → EC syndrome =0x00(When EL2 is enabled andHCR_EL2.TGE = 1）
FPENBehavior description
0b00Instructions at both EL0 and EL1 are trapped, unless CPACR_EL1.ZEN has already trapped them
0b01Only EL0 instructions are trapped, EL1 is not trapped
0b10Instructions at both EL0 and EL1 are trapped(Same as 0b00)
0b11No instructions are trapped(Register is freely accessible)

Simply put：

  • FPEN controls whether user state (EL0) or kernel state (EL1) can directly use floating-point/SIMD/SVE registers, usually configured before enabling the MMU

  • Combined with CPACR_EL1.ZEN, fine-grained control over access at different levels can be achieved.

  • Common configurations:

    • 0b11 → No trapping, allowing all EL0/EL1 instructions to access FP/SIMD.

    • 0b00/0b10 → Trapping, typically used in security or virtualization scenarios.

MDSCR_EL1

TDCC

  • TDCC = 0 → User state EL0 can directly read and write DCC registers.

  • TDCC = 1 → EL0 access to DCC registers is trapped to EL1/EL2, commonly used for security/virtualization/debug control。

DCC is the Debug Communication Channel, which provides a data transfer interface between the CPU and the debugger (Debug Host), JTAG can only be used after it is enabled

Enable MMU

TBI（Top Byte Ignore）

The TBI (Top Byte Ignore) feature in the ARMv8-A architecture, also known asTop Byte Ignore, which allows you to usethe top 8 bits ([63:56]) of a virtual address to store additional information without affecting memory access. This mechanism is also called Tagged Pointers, a feature provided by ARM to programming languages and runtime systems.

Normally, addresses must be valid

In a 64-bit system, registers are 64-bit, but virtual addresses are not full 64-bit—ARMv8 typically supports 48-bit or 52-bit virtual addresses. So the virtual addressthe top 16 bits must be sign-extended：

Virtual address typeThe top 16 bits must be
User-space memory0x0000
Kernel high address0xFFFF

If you write[63:48]as any other value, the CPU will triggeran address size fault。

In Enabling TBI in TCR_EL1 (there are respectivelyTBI0andTBI1controls), the CPU will ignore the top 8 bits of the address[63:56], which means you can freely write tag data into these 8 bits, tostore additional information, without affecting memory access**!

Note: Kernel/user addresses are still determined by VA[55], where 1 indicates a kernel address and 0 indicates the user address space.

InTCR_EL1contains two fields:

  • TBI0→ Controls EL0 accessing addresses using TTBR0 whether pointer tagging is enabled
  • TBI1→ Controls EL1 accessing addresses using TTBR1 whether pointer tagging is enabled

Memory attributes

Memory attributes defined by ARMv8

  • ARMv8 architecture processors provide two types of memory attributes

    • Normal Memory

      Normal memory is weakly ordered, with no other additional constraints, providing the highest memory access performance

    • Device Memory

      There are many restrictions when the processor accesses device memory, such as no speculative access, etc. Device memory is executed strictly in instruction order. The ARMv8 architecture defines various attributes for device memory

      • Device-nGnRnE (Does not support coalescing, does not support instruction reordering, does not support early write acknowledge)
      • Device-nGnRE (Does not support coalescing, does not support instruction reordering, supports early write acknowledge)
      • Device-nGRE (Does not support coalescing, supports instruction reordering, supports early write acknowledge)
      • Device-GRE (Supports coalescing, supports instruction reordering, supports early write acknowledge)

  • Gathering or non Gathering (G or nG)

    This attribute determines whether it is possible tocombine multiple accesses to this memory region into a single bus transaction。

    • If an address is marked as non-coalescing (nG), thenthe number and size of accesses performed on the memory bus to that location must exactly match the number and size of explicit accesses in the code。
    • If an address is marked as Gathering (G), the processor canfor example, merge two byte writes into a single half-word write. For regions marked as Gathering, it is also possible tomerge multiple memory accesses to the same memory location。

    For example, if a program reads the same location twice, the core only needs to perform one read and can return the same result for both instructions. For reads from regions marked as non-Gathering, the data value must come from the terminal device. It cannot be snooped from the write buffer or other locations.

  • Re-ordering (R or nR)

    This determineswhether accesses to the same device can be reordered with respect to each other。If an address is marked as non-Reordering (nR), accesses within the same block always appear on the bus in program order. The size of this block is IMPLEMENTATION DEFINED. If the block size is large, it can span several table entries. In this case, the ordering rule is observed for any other accesses also marked as nR.

  • Early Write Acknowledgement (E or nE)

    This determineswhether an intermediate write buffer between the processor and the slave device being accessed is allowed to send a write completion acknowledgement. In modern SoCs (System on Chips), processors (such as CPUs) access peripherals or memory through an Interconnect. To improve performance, the Interconnect typically containsa Write Bufferto temporarily store data that has not yet been actually written to the target device.

    However, in some scenarios,it must be ensured that the write operation is considered complete only after the data has actually reached the target device(such as writing a register to trigger a hardware action), while in other scenarios, it is possible toacknowledge the write completion earlyto improve throughput (such as writing to normal memory).

    This leads to the distinction between E (Early) and nE (not Early) .

    • If an address is marked as non-Early Write Acknowledge (nE), the write response must come from the peripheral, meaningearly acknowledgement is prohibitedthe write completion signalmust be issued by the target peripheral itself, i.e., the datais actually received by the peripheral, before notifying the processor that the write operation is complete.

    • If the address is marked as Early Write Acknowledgement (E), thenthe buffer in the interconnect logic is allowed to issue a write accept signal before the endpoint device actually receives the write. This essentially allows the write buffer in the interconnect, for the external storage system, tobefore the data has actually reached the target peripheral, send a “write complete” acknowledgement signal to the processor.

Defined in the Linux kernel

Memory attributes are not stored in the page table entries of the page table, but are stored inMAIR_ELnregisters (Memory Attribute Indirection Register）。

there is a 3-bit index value (AttrInx[2:0]) in the page table entry to look up the MAIR_ELn register。

Memory attributes defined in the Linux kernel

  • The operating system (Linux system) defines a series of attributes based on the memory attributes defined by armv8, as well as memory read/write attributes, etc.:
    • PAGE_KERNEL: the most common memory page in memory
    • PAGE_KERNEL_RO: read-only normal memory page in the kernel
    • PAGE_KERNEL_ROX: read-only executable normal page in the kernel
    • PAGE_KERNEL_EXEC: executable normal page in the kernel
    • PAGE_KERNEL_EXEC_CONT: executable normal page in the kernel, and is multiple physically contiguous pages

  • Upper-level directory entry PUD （Page Upper Directory) corresponds to arm64 L1 page table

  • Page table entry (Page Table Entry) corresponds to arm64 L3 page table

Experiment 1: Establish identity mapping

ARM64 uses 4-level page table(PGD → PUD → PMD → PTE), each level uses 9-bit index(512 entries), page size is 4KB：

LevelNameIndex bitsMapping rangeNumber of entriesTable size
L0PGD[47:39]512 GB5124KB
L1PUD[38:30]1 GB5124KB
L2PMD[29:21]2 MB5124KB
L3PTE[20:12]4 KB5124KB

Note: If a page table entry at a certain level is marked as SECTION (block mapping), it skips the next level and directly maps a large block of memory.

Why identity mapping?

To ensure no problems when turning on the MMU:

  1. When the MMU is off, the addresses accessed by the processor are all physical addresses.When the MMU is turned on, the addresses accessed by the processor become virtual addresses.
  2. Modern processors all havemulti-level pipeline architectures.，The processor prefetches multiple instructions into the pipeline in advance.When the MMU is turned on, the processor has already prefetched multiple instructions in advance, and these instructions were prefetched using physical addresses. When the instruction to turn on the MMU finishes executing, the processor’s MMU function takes effect. Therefore, this is toensure that the processor can continuously fetch instructions before and after turning on the MMU.。
