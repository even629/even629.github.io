Concepts of Concurrency and Race Conditions

Hereafter, concurrency and parallelism are collectively referred to as concurrency.

In a concurrently executing environment, multiple programs may access the same shared resource simultaneously. When multiple tasks attempt to operate on such resources at the same time, execution anomalies or data errors may occur, which are known as race conditions.

Common causes of race conditions include:

  1. Multithreaded access. Since Linux is a multitasking operating system, multiple threads may access the same shared resource simultaneously, which is the fundamental cause of race conditions.
  2. Interrupt access. When a process is accessing a shared resource and an interrupt interrupts the executing process, a race condition may also occur between the interrupting process and the interrupted process.
  3. Preemptive access. Linux 2.6 and later versions introduced a preemptive kernel, where high-priority tasks can interrupt low-priority tasks. If a task accessing a shared resource is interrupted and another task subsequently accesses the same resource, a race condition may be triggered.
  4. Multi-core concurrent access (SMP). In multi-core processor systems, different CPU cores may concurrently access the same shared resource, forming inter-core race conditions.

The Linux kernel provides various mechanisms to address this issue, commonly includingAtomic operations、Spinlock、mutex、Semaphoreetc.

Atomic operations

Atomic operations treat a single read/write operation on an integer variable as a whole, ensuring it is indivisible, thereby avoiding race conditions.

Atomic operations can be further subdivided into “integer atomic operations” and “bit atomic operation”, and here we will first explain integer atomic operations.

Spinlock deadlock

Spinlock deadlock refers to the phenomenon where multiple tasks or processes wait for each other to release resources, resulting in none of them being able to continue execution.

For example, suppose there are two processes, Process A and Process B. Process A holds a spinlock on Resource 1, and at the same time, Process A also wants to acquire Resource 2. Meanwhile, Process B holds a spinlock on Resource 2, and at the same time, Process B also wants to acquire Resource 1. In this case, both Process A and Process B are waiting for each other to release resources, thus causing a deadlock.

Another example is when an interrupt occurs while Process A holds a spinlock, and the CPU turns to execute the interrupt handler, which also needs to acquire the same spinlock. At this point, since the lock is already occupied by A, the interrupt handler can only wait and spin, causing the system to enter a deadlock state.

  • such asWhen a spinlock also needs to be acquired in an interrupt handler, the driver needs to disable interrupts while holding the spinlock (spin_lock_irqsave, spin_lock_irqrestore) and enable interrupts when releasing the spinlock.。

  • At the same time, try to hold the spinlock for as short a time as possible; holding a spinlock for a long time may exhaust system resources and cause a deadlock.

  • Finally, alsoavoid a function that has acquired a spinlock calling another function that also tries to acquire this lock, otherwise it will also cause a deadlock.

  • No functions that can cause sleeping or blocking can be called in the critical section

Semaphore

Spinlocks handle concurrency and competition by “busy waiting”, so the protected critical section cannot be too long to avoid wasting CPU resources. However, in some cases we inevitablyneed to protect some resources for a long time. In this case, semaphores can be used.

Semaphores cause the caller to sleep, so semaphores are also called sleeping locks.

Semaphores have P operationand V operation. The P operation decrements the semaphore value by one. If the value is less than or equal to 0, it means the resources are all occupied. If other processes need to use them, the process will be added to the waiting queue. If it is greater than 0, it means there are n available resources

Reader-writer semaphore

The difference from reader-writer locks is that reader-writer semaphores are sleeping locks, while reader-writer locks are spinlocks. Choose:

Can the critical section sleep?
├─ Yes → Use reader-writer semaphore (rw_semaphore) or mutex
│ └─ Read operations far outnumber write operations? → Yes → Reader-writer semaphore
│ └─ Read operations do not significantly outnumber write operations? → Mutex is simpler and has better performance
│
└─ No → Is the critical section very brief (a few integer operations)?
├─ Yes → A regular spinlock (spinlock_t) might be better than a reader-writer spinlock
└─ No → Reads far outnumber writes? → Yes → Reader-writer spinlock (rwlock_t)
└─ No → Regular spinlock

Functions that may cause sleep cannot be used in interrupt context:

Because interrupt context does not belong to any process, it has no process control block (task_struct), and therefore fundamentally lacks the infrastructure required for “sleep”. When a process wants to enter a sleep state, it actively marks itself as “sleeping”, removes itself from the run queue, and tells the scheduler: “Please wake me up when the lock is available.” This operation relies on the process context.

RCU

RCU（Read-Copy-Update, Read-Copy-Update) as a high-performance concurrency control mechanism, specifically designed to solve the “read-mostly” scenario. It separates the synchronization logic of read and write operations, makingRead operations have almost no overhead(no locking, no atomic instructions, no blocking), while write operations ensure safety through “copy-update” and “deferred release” methods.

RCU has become an indispensable fundamental component in the Linux kernel, widely used in core modules such as process scheduling, memory management, and file systems.

Applicable Scenarios of RCUIt mainly has the following characteristics:

  1. Data StructureMainly accessed through pointers;
  2. Read operations far outnumber update operations;
  3. Reader-side code cannot tolerate the overhead of locking;
  4. Update operations are relatively infrequent;

The core idea of RCU can be summarized as**“Read directly when reading, copy and modify when writing”**

Consider a life scenario: There is a bulletin board at home that records the family members’ schedules. When mom needs to update the schedule, she doesn’t stop others from checking the board. Instead, she first copies the current content onto a new piece of paper, modifies it on the new paper, and then, at a moment when no one is paying attention, replaces the old paper on the board with the new one. After the replacement, she doesn’t immediately destroy the old paper; instead, she waits for a period of time to ensure that everyone who might have been reading the old board has finished, before discarding the old paper.

In this analogy, “the moment of replacement” corresponds toPublishing operation, “waiting to ensure” Corresponds toGrace Period, while**“discarding the old paper”** corresponds toReclamation. These three concepts form the core triangle of RCU

Terminology

  • Read-Side Critical Section: The code segment where read operations access RCU-protected data needs to usercu_read_lock()andrcu_read_unlock()markers. Within this interval, read operations can safely access the old data (even if the write operation has updated the data).

  • Grace Period: The time period from when the write operation starts updating the data until all read-side critical sections (for the old data) end. The end of the grace period means “all read operations that might access the old data have completed”, at which point the old data can be safely freed.

  • Write-Side Operation: consists of three steps:

    1. Copy: Create a copy of the data (or new data).
    2. Update: Atomically switch the pointer from the old data to the new data.
    3. Deferred Free: Wait for the grace period to end, then free the old data.

How RCU Works

Read-Side

The lock-free access of read-side operations is the core source of RCU’s performance advantage. The process is as follows:

  1. Through thercu_read_lock()Enter the read-side critical section (essentially disabling preemption, or tracking read-side state in Preemptible RCU).
  2. Usercu_dereference(p)Acquire a pointer to the shared data (includes a memory barrier, ensuring atomicity and visibility of the pointer read).
  3. Access the data pointed to by the pointer (no locking required, read directly).
  4. Through thercu_read_unlock()Exit the read-side critical section.

Herercu_read_lock()andrcu_read_unlock()do not perform actual lock and unlock operations like traditional locks. In a non-preemptible kernel, they might be completely no-ops. Their true purpose is totell the compiler not to reorder instructions, and mark the beginning and end of the critical section in a preemptible kernel

rcu_dereference()is a critical macro that ensures read operations proceed in the expected order on processors with weak memory ordering, such as Alpha. It can be viewed as an encapsulation of a memory barrier, ensuring that the pointer read occurs before the actual data is fetched.

The key feature of the read side lies inzero overhead - No atomic operations, no memory barriers (on most architectures), no lock contention. This makes the RCU read path extremely efficient, and performance remains stable even under high-concurrency read scenarios.

Write-Side

The write side operation must ensure that ‘old data is released after all read operations are complete’, the process is as follows:

  1. Copy/Create new data: For example, allocate new memory for a linked list node and initialize it.
  2. Atomically update the pointer: usercu_assign_pointer(p, new_ptr)Switch the shared pointer from the old data to the new data (includes a memory barrier to ensure the new pointer is visible to all CPUs).
  3. Wait for the grace period: viasynchronize_rcu()(blocking wait) orcall_rcu()(asynchronous callback) wait for the grace period to end.
  4. Release old data: After the grace period ends, safely release the old data (such askfree(old_ptr)）。

Grace Period

How to ensure safe release?

The grace period is the core mechanism of RCU, its role is toensure that all read-side critical sections that started before the ‘pointer update’ have ended. How does RCU detect the end of the grace period?

  • Core idea: Each CPU maintains an ‘RCU state’ (such asrcu_seq), recording whether there is currently an active read-side critical section. When a write operation requests a grace period, RCU waits for all CPUs to undergo a “context switch” (such as scheduling, interrupt return, etc.) — this means the read-side critical section on that CPU has ended (because the read-side critical section disables preemption and cannot be interrupted by scheduling; a context switch indicates that the Read-Side has ended).
  • Implementation details: The Linux kernel tracks the RCU state of each CPU through arcu_nodetree structure,synchronize_rcu()and will block until all CPUs have completed a “grace period confirmation” (rcu_qs()）。

A grace period can last hundreds of milliseconds (depending on system load); the write side should not assume it ends quickly, and avoid designing logic that depends on the duration of the grace period.

The Linux kernel usescontext switches、user-mode executionandidle loopsas indicators of read-side critical section exit. After a CPU experiences one of these states, the kernel considers all RCU read-side critical sections on that CPU to be complete

The implementation of RCU heavily relies onmemory barriersto ensure sequential consistency in multi-core environments, preventing the compiler and CPU from reordering instructions for optimization.

  • On the read side,rcu_dereference()it includes a read memory barrier, ensuring that operations after acquiring a pointer are not reordered before the pointer acquisition.

  • On the update side,rcu_assign_pointer()it includes a write memory barrier, ensuring that the pointer update itself is not visible to other CPUs until all store operations before the pointer update are visible to other CPUs.

This ordering guarantee is critical for the correctness of RCU. Consider the scenario of data publication: the updater must first initialize the new data, and then publish the pointer. Without memory barriers, the CPU or compiler might reorder these two steps, causing the reader to see incompletely initialized data.

RCU best practices

Minimize Read-Side critical section length

The longer the read-side critical section, the longer the grace period may be (write-side needs to wait longer), leading to delayed release of old data and increased memory pressure.

Principle: Only enter the critical section when accessing RCU data, and avoid performing time-consuming operations (such as I/O, complex calculations) within the critical section.

RCU variants

The Linux kernel provides multiple RCU variants, suitable for different scenarios:

  • Classic RCU（CONFIG_RCU_FANOUT）: Default variant, does not support read-side preemption, suitable for non-real-time kernels.
  • Preemptible RCU（CONFIG_PREEMPT_RCU）: Supports read-side preemption, read-side critical section can be scheduled, suitable for real-time kernels.
  • Sleepable RCU（SRCU，CONFIG_SRCU）: Read-side can block (e.g., callingmsleep()), need to usesrcu_read_lock()/srcu_read_unlock(), applicable to scenarios where the read side needs to sleep.
