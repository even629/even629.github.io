This article introduces the basic concepts, characteristics, and topology of the AXI bus in the ARM AMBA architecture. It discusses the five independent transmission channels of the AXI bus, the master-slave mechanism, the VALID/READY handshake process, and the read/write transaction flow. It also summarizes key features such as transaction structure, access permissions, cache support, and QoS, demonstrating its advantages of high performance and flexible scalability.

AXI Basic Concepts: Channels, Transfers, Transactions

What is a Bus

  • A bus consists of a single or a group of signals that transmit data or control information

  • On the bus, the sender encodes and transmits information according to the high/low level combinations specified by the protocol, and the receiver then decodes it

  • External Bus and Internal Bus

    • External buses connect peripheral devices, which may be subject to external interference, connector and PCB losses, and poor signal integrity.

    I2C，UART，SPI，PCIe，USB

    • Internal buses are used inside the SOC, routed within the chip with extremely short distances, so many interference and signal integrity issues do not need to be considered.

    AXI，UPI（intel）

  • Parallel bus and serial bus

    • Parallel: Multiple data are transmitted simultaneously, requiring consideration of data coordination, which results in the frequency of parallel transmission not being able to be made very high and being prone to interference.
    • Serial: There is only one link, the frequency is made very high to increase transmission speed.

Characteristics of AXI bus

  • AXI bus is a synchronous serial bus inside the chip
  • Advantages of AXI bus
    • High performance: high-bandwidth, low-latency, High-frequency
    • Flexible expansion of bus bit width and topological connections
    • Compatible with AHB and APB buses
  • Characteristics of AXI bus
    • Separate address/control and data channels (separate read & write channel)
    • Supports burst transmission
    • Support multiple outstanding addresses
    • No strict timing between address and data phases
    • Support out-of-order transaction completion
    • Support unaligned data transfer

AXI topology

AXI uses a master-slave mechanism

  • Requests are initiated by the master
  • The slave responds to the request

AXI channels

AXI defines 5 independent transmission channels, improving bandwidth

  1. Read address channel: AR
  2. Read data channel: R
  3. Write address channel: AW
  4. Write data channel: W
  5. Write response channel: B

Each channel has more than just one signal line, but a group of signal lines

Write transaction

  • The address channel contains transfer control information
  • Transaction steps:
    • Master initiates a write transfer through the write address channel, including address and control information
    • Master writes data to the slave through the write data channel
    • Slave responds through the write response channel

Read transaction

  • Master sends address and control information to the slave through the read address channel
  • Slave returns data through the read data channel, and the response information is included in the returned data

Handshake signal (Handshake process)

  • All 5 channels use the same handshake signal: VALID/READY handshake process
  • The source generates a VALID signal indicating that the address, data, control information, etc. are ready.
  • The destination generates a READY signal indicating that it is ready to accept information.

Handshake occurs only when both READY and VALID signals are valid simultaneously.

The difference between transfer and transaction

Transfer

Only one handshake process, transferring one data item.

transaction

Has multiple handshake signals, composed of multiple transfers.

Taking write as an example, address and control information are passed through the write address channel. Then data is transferred through the write data channel. Finally, a response is given through the write response channel.

There are 3 transfers in total.

Transaction structure
  • How many transfers are included in one burst
    • AXI3 contains up to 16 transfers: Burst_Length = AxLEN[3:0] + 1
    • AXI4 contains up to 256 transfers: Burst_Length = AxLEN[7:0] + 1
  • AxSIZE: indicates how many bytes are transferred in one transfer, up to 128 bytes

  • AxBURST: indicates the burst type

    • FIXED: Fixed address mode, applied to FIFO

    • INCR: Address increment mode, applied to RAM

      Slave increment address, supports 1~256 transfers, supports unaligned transfer

    • WRAP: Address increment, wraps around after reaching the upper limit, applied to Cache

Access permissions
  • ARPROT[2:0]: Indicates the access permissions for read transactions
  • AWPROT[2:0]: Indicates the access permissions for write transactions

Cache support
  • Various caches can be utilized during transmission
    • System-level caches at all levels, such as L2/L3 cache
    • Cache inside the system bus (with the interconnect)
  • AxCACHE[3:0]: Indicates cache attributes

Response packet (response structure)

  • RRESP[1:0]，for read transfers

  • BRESP[1:0]，for write transfers

  • OKAY: Indicates normal access success or exclusive access has failed

  • EXOKAY: indicates successful exclusive access

  • SLVERR：slave error

  • DECERR: decode error

Write strobes
  • WSTRB[n:0]: indicates whether the data on WDATA is valid, one bit represents one byte
  • Mainly to support unaligned access

QoS signal
  • AXI bus provides additional signal lines to support quality of service
  • AWQOS: 4-bit QoS, in the write address channel of each write transaction
  • ARQOS: 4-bit QoS, in the read address channel of each read transaction
  • 0 indicates the lowest priority, F indicates the highest priority
  • Generally, system bus IP provides registers to configure the QoS of each master
Channel dependencies
  • Dependency 1: WVALID is set valid before AWVALID is valid
    • AWVALID indicates that the write address channel is valid
    • WVALID indicates that the write data channel is valid
    • Data can be sent before the write address is valid
  • Dependency 2: WLAST must be sent before BVALID is valid
    • BVALID indicates that the write response channel is valid
    • WLAST indicates the last transfer in a transaction
    • Before the write response is sent, all write data and addresses must have been sent
  • Dependency 3: RVALID cannot be valid before ARADDR has been sent
    • ARADDR indicates the address of the first transfer in a read transaction
    • RVALID indicates that the read data channel is valid
    • If the address has not been sent, no read data should be returned

lock access and exclusive access

Locked accesses

  • Locked access is only in the AXI 3 protocol; AXI 4 has deprecated it
  • AXI 3 and AXI 4 have the AxLOCK signal
  • In AXI 3, Locked access is similar to locking the bus

Exclusive access

  • Exclusive access is more efficient than Locked access; it does not need to lock the bus, and other masters can access the bus simultaneously

  • An exclusive monitor needs to be implemented on the slave side to coordinate the completion of exclusive access

  • ARID ID of the read transaction

  • AWID Write transaction ID

  • Access flow

    1. Master initiates an exclusive read. The slave’s exclusive monitor records the ARID, address, and data in the table.
    2. Master initiates an exclusive write operation to the same address. The exclusive monitor looks up the table and compares whether the AWID and ARID match.
    3. Response
      • EXOKAY (success): if no other master writes to this address during this period, the exclusive write operation succeeds.
      • OKAY (failure): if another master writes to this address during this period, the exclusive write fails.

Example: Exclusive access failure

Operation sequence:

  1. Master initiates the 1st exclusive read operation to address 0x8000.
  2. Master initiates the 2nd exclusive read operation to address 0x8000.
  3. Master exclusively writes 0x3 to address 0x8000 -> Success
  4. Master exclusively writes 0x5 to address 0x8000 -> Failure

First three steps of operation:

When there is a read transaction with the same ID during an exclusive write, the write succeeds

After an exclusive write succeeds, the exclusive monitor deletes all entries related to this address and ID from this table

Exclusive Write fails because there is no read transaction with the same ID when looking up the table

Transaction ordering

AXI transaction ID

  • AXI has an independent transaction ID for each transaction channel
  • All transfers must have an ID
  • Transfers in the same transaction have the same ID
  • Transaction ID is for out-of-order

Write transaction ordering rules

  • Write transaction ordering rule 1: The order of writing data in the write data channel must be consistent with the order of address transfers in the write address channel (data must follow the same order as address transfers)

In the write address channel, address A is sent first, and then address B, so in the write data channel, data A is sent first, and then data B

  • Write Transaction Rule 2: Write transactions with different IDs can complete in any order (data for different transaction IDs can be interleaved)

Transaction B completes before Transaction A, even though Transaction A was executed before Transaction B

  • Write Transaction Rule 3: Write transactions with the same ID must be executed and completed in order (data with the same ID must follow in the order as issued)

Transaction B has a different ID from Transactions A and C, so Transaction B can complete in any order. Transactions A and C use the same ID, so Transaction A must complete first, followed by Transaction C

Read Transaction Ordering Rules

  • Read Transaction Ordering Rule 1: Transactions with different IDs on the read data channel can be in any order (data for different IDs has no ordering restrictions)

Even though Transaction B sends its address later than Transaction A on the read address channel, Transaction B can read data before Transaction A

  • Read Transaction Rule 2: Transfers of transactions with different IDs can be interleaved on the read data channel (data with different transaction IDs can be interleaved)

Transaction A and Transaction B interleave reading data

  • Read transaction rule 3: Read transactions with the same ID must complete in the order they were issued (data with same ID must follow in the order as issued)

Transaction A and C have the same ID. Transaction A was issued before Transaction C, so Transaction A must also complete before Transaction C.

Unaligned address access

  • AXI bus supports unaligned address access, using the byte strobes mechanism

Starting from address 0x1, 3 bytes are transferred first, and then the address becomes aligned.

AXI-Lite bus

AXI4-Lite bus introduction

  • AXI4-Lite does not support burst mode, or the burst length is 1
  • AXI4-Lite supports data widths of 32-bit or 64-bit
  • All AXI4-Lite accesses are Non-modifiable, Non-bufferable
  • Does not support exclusive access

AXI5-Lite Bus Introduction

  • AXI5-Lite relaxes the requirements for data bit width and transaction ordering based on AXI4-Lite
  • Features of AXI5-Lite
    • Does not support burst mode, or burst length is 1
    • All accesses are Device and Non-bufferable
    • Does not support exclusive access
    • When requests have different IDs, reordering of responses is allowed

New features in AXI5/ACE5

Summary

  • The AXI bus defines 5 independent channels, each consisting of a set of signal lines
  • The AXI bus defines the data structure and attributes of transfer transactions
    • Number of Bursts
    • Transfer size
    • Burst type
    • Response packet
    • Cache support
    • Access permissions
    • QoS signal
  • Supports exclusive access
  • Supports out-of-order transfer via transaction ID
  • Supports unaligned access
