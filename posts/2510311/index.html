This article introduces the basic principles of ARM ACE (AXI Coherency Extensions) as a hardware cache coherency solution based on the AXI bus, elaborating in detail on its method of achieving system-level cache coherency among multiple Masters through the snoop mechanism, newly added transfer channels and signal lines, and a five-state cache state machine. Furthermore, this article discusses the division of shareability domains, specific scenarios of snoop transactions, the critical role of RACK/WACK signals in ensuring transaction ordering and coherency, as well as extended mechanisms such as exclusive access and DVM.

Reference documents:

Arm Architecture Reference Manual Armv8

https://developer.arm.com/documentation/ddi0487/fc/

ACE

  • ACE (AXI Coherency Extensions) is a hardware cache coherency solution based on the AXI bus.
  • Focuses on system-level cache coherency.

It was first supported starting with the Cortex-A15/A7 in the big.LITTLE architecture.

Cortex-A9 does not support the ACE interface and is connected to L2 and Memory via the AXI interface.

Cortex-A15 supports the ACE interface; in the big.LITTLE architecture, the big cluster and the little cluster are connected to the CCI bus via ACE.

  • Three Masters, each with a local cache. ACE allows all three Masters to have the same cache copy for the same memory address.
  • Here, Master generally refers to a CPU cluster or a controller with a cache.
  • ACE ensures that for a given address, all Masters can access the correct data.

ACE focuses on the system cache coherency issues between Masters with caches, such as the Cortex-A72 family and the Cortex-A53 family.

ACE-Lite is used to connect hardware IO devices without caches, but these devices need to access system cache-coherent memory, such as GPU, SMMU, etc. The DVM interface is used to broadcast TLB invalidation.

Basic idea of ACE implementation.

  • RequiresImplement a snoop mechanism to ensure cache coherency among multiple Masters.
    • Add new signal lines and new transfer transactions on the basis of AXI to implement snooping.
    • Add new transfer channels on the basis of AXI.
    • Implement a 5-state cache state transition mechanism.(Because some Masters use MESI and some use MOESI, etc.)
  • ACE supports memory barriers (removed in ACE5).
  • ACE supports exclusive access.
  • ACE supports DVM.

ACE state machine

  • valid: cache line is valid

  • invalid: cache line is invalid

  • unique: indicates that this cache line is exclusive, only the current master has it

  • shared: multiple masters have a copy of this cache line

  • clean: indicates that the content of the cache line is consistent with memory

  • dirty: the content of the cache line is inconsistent with memory and needs to be written back to memory later

Similar to MOESI protocol

New channel added in ACE

New signals added in ACE

  • AxDOMAIN: used to indicate the shareability domain:
    • Non-shareable，Inner Shareable，Outer Shareable，System
  • AxSNOOP: used to indicate the snoop transfer transaction type
  • AxBAR: indicates initiating a barrier transaction (removed in ACE5)
  • AWUNIQUE: used to optimize cache state transitions for write transfer transactions

AxDOMAIN signal line

AxSNOOP signal line

  • AxSNOOP: used to indicate the snoop transfer transaction type
    • Non-snooping
    • Coherent
    • Memory update
    • Cache maintenance
    • DVM
    • barrier

shareability domains

  • Before initiating coherency or barrier transfer transactions, the master uses it to determine which masters these transaction transfers should be sent to

    • Coherent transaction: determines which masters might have a copy of this data, used to send snoop transactions
    • Barrier transactions: determining which masters will establish ordering relationships, and how far the barrier transaction should propagate

  • Supports 4 types of shareability domains:

    • Non-shareable: contains only one master
    • Inner Shareable
    • Outer Shareable
    • System

  • Inner share: usually refers to the integrated cache inside the CPU, which is closest to the processor core. For example, the Cortex-A72 core can integrate L1 and L2 caches internally.
  • Outer share: cache extended through the system bus, such as an expanded L3 cache connected to the system bus

Example 5: Simultaneously initiating ReadUnique write operations

  1. At T0, the cache line states of Master0 and Master1 are both I, and the initial value of address A is 0x11223344
  2. At time T, Master0 and Master1 simultaneously initiate write operations to address A
    • Master0 wants to write to address A: 0x555667788
    • Master1 wants to write to address: 0xaabbccdd

The bus performs an arbitration, processing one request before handling the other

RACK and WACK signal lines

  • In Example 5, the RACK signal line is used: indicating that the ReadUnique transaction is complete
  • On the AXI bus, multiple outstanding transfers and out-of-order transfers are supported, but in ACE, cache coherency issues are very likely to occur
  • The RACK signal line is used to indicate that a read transaction is complete
  • The WACK signal line is used to indicate that a write transaction is complete
  • The interconnect bus IP uses the RACK/WACK signal lines to ensure thatthe previous transfer transaction to an address must be completed before sending snooping for other transactions to other masters
  • For read transactions, after the RLAST signal is pulled high indicating the last read transfer is complete, the master can then send a read acknowledge signal
  • For write transactions, after the write response channel sends a response handshake, the master can then send the WACK signal

Example 6: Usage of RACK signal line - Problem introduction

At T0: The cache line states of Master0 and Master1 are both SC

T1 moment: Master0 wants to rewrite the data at address A to: 0xAABBCCDD

T2 moment: Master1 performs a read operation on address A

Root Cause

When Master1 initiates a ReadShared transaction, master1 thinks that the previous master0’s MakeUnique transaction has been completed, but in fact it is blocking in some intermediate state and has not been completed yet

Example 6: Usage of RACK signal line - Problem resolution

Solution: Master0 uses the RACK signal to tell the bus IP: the MakeUnique transaction has been completed, and only then can the bus initiate other snoop-related transactions

exclusive access

  • Flow of Exclusive access

    • Execute exclusive load
    • Calculate
    • Execute exclusive store
      • If another master writes to this address, fail
      • If no other master writes to this address, success

  • For Non-shareable and System Shareable memory addresses, the behavior of exclusive access is the same as AXI

  • For inner share and outer share memory

    • Master exclusive monitor ensures that no other master writes to this address after the exclusive load
    • The PoS exclusive monitor of the bus interconnect implements access serialization

DVM（Distributed Virtual Memory）

  • Send broadcast to other masters:
    • TLB invalidate
    • Branch Predictor Invalidate
    • Physical Instruction Cache Invalidate
    • Virtual Instruction Cache Invalidate
    • Synchronization

DVM transport transactions - DVM message transactions

DVM message transaction processing flow:

  1. Master sends DVM message request through read address channel
  2. After receiving it, the bus sends snoop to other masters
  3. After receiving it, other masters respond through snoop response channel
  4. After collecting all responses, the bus sends a response packet to the initiating master through read data channel

DVM transfer transactions - DVM Synchronization and DVM Complete transactions

DVM sync and complete transaction processing flow:

  1. Initiating master issues DVM synchronization request through read address channel
  2. After receiving it, the bus sends snoop broadcast to other masters
  3. After receiving it, other masters send responses through snoop response channel.
  4. After collecting all response packets, the bus sends a response packet to the initiating master through read data channel.
  5. Each participating master, after completing TLB invalidation, sends a DVM complete transaction through read address channel.
  6. After receiving DVM complete, the bus immediately responds to the master.
  7. After the bus collects DVM complete transactions from all participating masters, it sends DVM complete to the initiating master through snoop address channel.
  8. Initiating master responds to the bus through snoop response channel.

ACE-Lite

  • ACE-Lite is applied to hardware devices that do not need to participate in system cache coherence, such as hardware devices without local cache
    • GPU
    • SMMU

  • ACE-Lite supported transactions
    • Non-coherent transactions: ReadNoSnoop, WriteNoSnoop
    • Partial coherent transactions: ReadOnce, WriteUnique, WriteLineUnique
    • Cache maintenance transactions: CleanShared, CleanInvalid, MakeInvalid
