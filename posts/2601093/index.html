Introduction to CAN Bus

CAN(Controller Area Network) is a serial communication protocol introduced by Bosch in 1986, specifically designed for automotive electronic systems, aiming to reduce the number of wire harnesses in vehicles and achieve efficient communication among multiple Electronic Control Units (ECUs).

Core Features

  • The CAN bus supports multi-master communication, where all nodes (ECUs) can actively send data without relying on a central controller.
  • It employs a non-destructive arbitration mechanism to resolve bus conflicts through ID priority, ensuring that high-priority data is transmitted first.
  • The bus features high reliability, with built-in CRC checks, error detection and recovery mechanisms, and uses differential signals to enhance electromagnetic interference resistance.
  • The CAN bus achieves the advantage of low cost by simplifying wiring and reducing system complexity.

Application Fields

  • Automotive Electronics: engine control, body modules (lighting/windows), diagnostic systems (OBD-II), etc.
  • Industrial Automation: PLC, sensor networks, robot control.
  • Other Fields: medical equipment, aerospace, smart buildings.

Development History

  • 1983: Bosch initiated research on automotive network technology.
  • 1986: Official release of the CAN protocol.
  • 1987: Intel and Philips launched the first CAN controller chips.
  • 1991: CAN 2.0 specification released, first applied in the Mercedes-Benz S-Class.
  • 1993: Became an international standard (ISO 11898).
  • 2015: CAN FD (Flexible Data Rate) standardized (supporting higher speeds and 64-byte data)
  • 2020: Next-generation CAN XL released, further increasing bandwidth.

CAN Electrical Properties

The CAN bus uses differential signals to transmit data.

  1. CAN Signal Levels

The CAN bus uses two signal lines for data transmission: CAN_H (High level) and CAN_L (Low level).

CAN_H and CAN_The voltage difference between L determines the state of the logical signal. This enables the CAN bus to have better anti-interference capability, making it particularly suitable for working in noisy environments.

  • CAN_H: High-level signal line, typically with a voltage range of 2.5V to 3.5V.
  • CAN_L: Low-level signal line, typically with a voltage range of 1.5V to 2.5V.
  1. Differential Signals and Anti-interference

The CAN bus transmits data via differential signals, meaning that the transmission of information relies on CAN__H and CAN__L, rather than individual voltage values. The characteristics of differential signals give the CAN bus extremely strong anti-interference capability in noisy environments.

Advantages of Differential Signals：

  1. Noise typically affects CAN__H and CAN__L simultaneously, but because differential signals rely on the voltage difference between the two,the effects of noise on both will cancel each other out, thereby maintaining signal integrity.
  2. This characteristic makes the CAN bus perform excellently in noisy environments such as automotive and industrial control applications.

Electrical Properties and Operating States

The electrical properties of the CAN bus determine its operating states, specifically manifested as the recessive state and the dominant state.

  • Recessive state (logical “1”)：When CAN_H and CAN_L voltages are equal（typically 2.5V）, representingthe bus is in an idle state. At this time, there is no data transmission on the bus, and all devices are waiting for communication. The recessive state is also known as “idle level” or “logical 1”.
  • Dominant state (logical “0”)：When CAN_H and CAN_L the voltage difference is 2V when, representingdata is being transmitted. Specifically, CAN_H is 3.5V, CAN_L is 1.5V, representing logic “0”. The dominant state indicates that the bus is transmitting data, andhas a higher priority。

CAN protocol frame structure

Types of frames

Frames are the basic unit of data transmission in CAN communication. Five different types of frames are used in CAN communication to achieve different functions. Below is a brief description and purpose of each frame:

  • Data Frame: The data frame is the most commonly used frame type in CAN communication, used for the sending unit to transmit data to the receiving unit. There are two formats for data frames:
    • Standard format (11-bit identifier) is suitable for simpler networks
    • The extended format (29-bit identifier) is suitable for more complex networks.
  • Remote Frame: A remote frame is used by a receiving unit to request data from a sending unit. It does not contain a data field; instead, it uses the identifier (ID) to tell the sending unit, ‘I need the data corresponding to this ID.’ Remote frames also come in standard and extended formats. Their structure is similar to that of a data frame, but they lack the data field.
  • Error Frame: An error frame is used to notify other nodes when a node detects an error. It is a special frame used to interrupt the current communication and restart it. The error frame consists of an error flag (6 consecutive dominant or recessive bits) and an error delimiter. Trigger conditions include bit errors, stuff errors, CRC errors, etc.
  • Overload Frame: An overload frame is used by a receiving unit to notify a sending unit that it is not yet ready to receive. It typically occurs when a receiving node needs more time to process the previous frame. Overload frames can only be sent during the interframe space. Their structure is similar to that of an error frame, consisting of an overload flag and an overload delimiter.
  • Interframe Space: The interframe space is used to separate data frames or remote frames from the preceding frames. It is a time interval that ensures nodes on the network can correctly identify the start of the next frame. For data frames and remote frames, the interframe space is typically 3 bit times; for error frames and overload frames, the interframe space may differ.

Data Frame

The data frame is the most core frame type, responsible for carrying actual data content and enabling efficient information transfer between sending and receiving units.

In early CAN communication,standard data frames used an 11-bit identifier, which could meet the needs of the simpler network architectures at the time. However, as vehicle functions became more complex (such as advanced driver-assistance systems, powertrain control, body electronics, etc.), the address space of the 11-bit identifier gradually became insufficient to support larger-scale nodes and more refined communication needs. To this end, the CAN protocol introducedExtended Data Frame, via 29-bit IdentifierProvides a larger address space, significantly enhancing the flexibility and scalability of the network.

    • ACK slot (1 bit), when the receiving device successfully receives the data frame, it returns a dominant bit in the ACK slot as an acknowledgment signal, indicating that the data frame has been received correctly
    • ACK delimiter (1 bit), the ACK delimiter is fixed at recessive 1, used to identify the end of the ACK field.

  • End of Frame（End of Frame）

    • Consists of 7 consecutive recessive bits (logic 1), used to clearly identify the end of the data frame.
    • It not only tells the receiving device that the data frame has been transmitted completely, but also provides extra time for signal synchronization to ensure the correct reception of the next frame.

SRR bit function

In the CAN protocol, the SRR bit is used to replace the RTR bit of the standard frame, located at the position of the original RTR bit in the arbitration field of the extended frame.

Its function is to be forced to a recessive level (1) to ensure that when a standard frame and an extended frame compete for the bus simultaneously, the standard frame has a higher priority due to its dominant RTR bit (0). When a standard frame and an extended frame are sent simultaneously, the bus detects the dominant level (0) through arbitration logic, thus allowing the standard frame to win.

IDE bit
The IDE bit is the identifier extension flag, used to distinguish frame types:Dominant (0) indicates the current frame is a standard frame, recessive (1) indicates an extended frame。

In extended frames, the IDE bit not only exists in the arbitration field to identify the frame type, but also appears in the control field, further clarifying the frame structure and enabling the distinction between extended frames and standard frames.

Error frame

The error frame is a special frame in the CAN bus used to identify errors that occur during communication. Its structure differs from other frames (such as data frames or remote frames) and is primarily used to notify all nodes in the network that an error exists in the current transmission.

  • Error Flag
    • 6 consecutive dominant bits (logic 0) or recessive bits (logic 1), divided into active error flags and passive error flags.
    • Active error flagSent by the node that detected the error, forcibly interrupts the transmission of the current frame, and notifies all nodes in the network of the error;
    • Passive error flagSent by a node in a passive error state to avoid interfering with the normal communication of other nodes.
  • Error Delimiter
    • Composed of 8 consecutive recessive bits (logic 1), used to mark the end of the error frame. It ensures that all nodes can clearly identify the boundary of the error frame and resume normal communication after error handling is complete.

Overload frame

The overload frame is a special frame in the CAN bus used to notify other nodes in the network that the current receiving device needs more time to process data. It is usually sent when the receiving node cannot process the received data in time, to request the sending node to delay the transmission of the next frame. The composition of the overload frame is shown below:

  • Overload Flag
    • Consists of 6 consecutive dominant bits (logic 0), sent by the receiving node to notify other nodes in the network that the current receiving device needs more time to process data, thereby requesting a delay in the transmission of the next frame.
    • Multiple nodes can send overload flags simultaneously, and these flags will superimpose to form a longer sequence of dominant bits.
  • Overload Delimiter
    • Consists of 8 consecutive recessive bits (logic 1), used to mark the end of the overload frame.
    • It ensures that all nodes can clearly identify the boundaries of the overload frame and resume normal communication after the overload processing is complete.

The design of the overload frame ensures that nodes in the network can coordinate communication under high load conditions, avoiding data loss or errors.

Bit Stuffing

Bit stuffing is an important mechanism in the CAN protocol used to ensure signal synchronization during data transmission.

Bit stuffing ensures that there is always a level change on the bus by inserting a stuff bit of opposite polarity after 5 consecutive bits of the same polarity, thereby maintaining synchronization between the transmitter and receiver.

Stuffing is performed only between the start of frame and the CRC field to avoid interference with other parts. The receiver automatically removes the stuffing bits upon detecting them, restoring the original data. This mechanism is crucial for preventing signal synchronization issues and improving the reliability of CAN bus communication.

Core Purpose

By inserting a stuffing bit of opposite polarity after 5 consecutive bits of the same polarity, it prevents bus synchronization issues caused by prolonged “0” or “1” levels, thereby improving communication reliability.

Bit Stuffing Rules

When 5 consecutive bits of the same polarity appear in the data, a stuffing bit of opposite polarity is inserted at the 6th bit.

This mechanism ensures that there are always level changes on the bus by breaking up potentially long strings of identical levels, thereby maintaining synchronization between the transmitter and receiver.

Stuffing Range

Performed from the start of frame to the CRC field，CRC Data after the field（such as ACK Field and end of frame）No stuffing is performed. This design limits the stuffing range to avoid interfering with parts that do not require synchronization, while ensuring the synchronization of critical data segments.

CAN Bus Core Mechanisms and Communication Principles

Bit time

In CAN bus communication,**bit time refers to the time required to transmit one logical bit,**which is one of the core concepts of the CAN protocol for achieving reliable data transmission, directly affecting the communication baud rate and signal synchronization accuracy. As the basis of asynchronous communication, all nodes maintain synchronization through the agreed-upon bit time.

The CAN protocol divides the bit time into multiple time segments, which together determine the signal sampling and synchronization mechanism. Specifically, it includes the following parts:

  • Synchronization segment (Sync Segment)

The synchronization segment is the first part of the bit time,Fixed at 1 time quantum（TQ (Time Quantum) is the basic unit of bit time, and all time segments are divided based on the time quantum), mainly used to detect signal edges and achieve hard synchronization.

During the Start of Frame (SOF) phase, the receiving node forcibly adjusts its own clock based on the falling edge of the bus signal to align with the bus signal, thereby ensuring that all nodes remain synchronized during the initial stage of communication.

  • Propagation Segment

The propagation segment is used to compensate for the propagation delay of the signal on the bus and the response time of the input comparator. Its length is usually 1 to 8 time quanta (TQ), with the specific value determined by configuration. This

period of time allows the signal to propagate on the physical medium and ensures that all nodes can receive a stable signal, thereby avoiding sampling errors caused by propagation delay.

  • Phase Segment 1

Phase Segment 1 is a configurable part of the bit time, usually 1 to 8 time quanta (TQ),mainly used to adjust the position of the sampling point to ensure that the sampling point is located in the stable region of the signal。

If a signal edge is detected to arrive early，it can be achieved by shortening the phase segment 1 to achieve resynchronization，thereby dynamically compensating for the effects of clock deviation or signal jitter。
  • Phase Segment 2

Phase Segment 2 is similar to Phase Segment 1, also being a configurable part of the bit time, usually 1 to 8 time quanta (TQ).

It is mainly used to handle the situation where the signal edge arrives late。If the signal edge is delayed，it can be achieved by extending the phase segment 2 to achieve resynchronization，thereby ensuring that the receiving node can sample the signal at the correct time，improving the reliability of data transmission。

The role of bit time

  1. Ensuring synchronization: By dividing time segments, CAN nodes can dynamically adjust the sampling point when a signal edge occurs, compensating for clock deviation and signal delay, thereby achieving synchronization among multiple nodes.
  2. Improving anti-interference capability: Setting the sampling point in the stable region of the signal (such as at 87.5% of the bit time) avoids misjudgment caused by signal edge jitter or interference.
  3. Supports flexible configuration: by adjusting the length of each time segment (such as propagation segment, phase buffer segment, etc.), it can adapt to different baud rates and network environments.

Hard synchronization and re-synchronization

In CAN communication, due to the adoption of asynchronous communication and the absence of an independent clock line, each node relies on a local crystal oscillator for data transmission. However, clock drift of the crystal oscillator can cause the timing between nodes to gradually shift, eventually leading to sampling errors. To solve this problem, the CAN protocol achieves dynamic timing adjustment through two major mechanisms: hard synchronization and re-synchronization:

  1. Hard synchronization forces the clocks of all nodes to align at the start of a frame, eliminating initial phase deviation.；
  2. Re-synchronization dynamically adjusts the length of the phase buffer segment based on real-time detection of signal edges during data transmission to compensate for clock drift.。

These two mechanisms work together, combining bit time division (synchronization segment, propagation segment, phase segment I/II) and quantitative management of time quanta (Tq), to ensure that all nodes accurately read the bus level at the sampling point, thereby achieving highly reliable data transmission in asynchronous communication without an independent clock line. In this section, hard synchronization and re-synchronization will be explained in detail.

Hard synchronization

In CAN communication, hard synchronization is an important mechanism to ensure consistent initial timing of all nodes. Its core function is toforce the timing of each node to align at the start of frame (SOF) stage, thereby eliminating the initial phase difference caused by local crystal oscillator discrepancies.。

The trigger condition for hard synchronization iswhen the bus detects a falling edge of a dominant level (logic 0) from the idle state (i.e., the edge of the SOF bit), the node immediately adjusts its own timing to align the synchronization segment of the current bit to this edge.。

This process can be divided into the following steps:

  • The nodecontinuously monitors level changes during the bus idle period.；
  • WhenWhen the SOF falling edge is detected and this edge falls within the synchronization segment, hard synchronization is triggered.；
  • The node forcibly adjusts its timing, aligning the start of the synchronization segment of the current bit to this edge, and restarts the counting of subsequent bit times.。

However,the scope of hard synchronization is limited; it only takes effect at the start of a message frame and can only solve the initial phase deviation problem.。

For dynamic phase errors caused by clock drift or signal interference during communication,，hard synchronization is powerless.，These problems need to be handled by relying on the subsequent re-synchronization mechanism.

Re-synchronization

In CAN communication,re-synchronization is the key mechanism to solve dynamic phase errors caused by clock drift or signal interference during communication.。

Unlike hard synchronization, resynchronization is not limited to the start of a frame, butcontinues to function throughout the entire data transmission process。Its core function is to dynamically adjust the position of the sampling point，to ensure that the receiving node can read the bus level at the correct time，thereby maintaining the reliability of communication。

Specifically,The trigger condition for resynchronization is when a deviation is detected between the signal edge (such as a transition from dominant to recessive level) and its expected position。

  • If the signal edge arrives early, it means the transmitting node’s clock is faster than the receiving node’s; in this case, the receiving node will advance the sampling point by shortening phase buffer segment 1;
  • Conversely, if the signal edge arrives late, it means the transmitting node’s clock is slower than the receiving node’s, and the receiving node will delay the sampling point by lengthening phase buffer segment 2.

This dynamic adjustment process relies onthe Synchronization Jump Width (SJW), which definesthe maximum range of each resynchronization adjustmentto prevent over-adjustment from causing new errors.

The advantage of resynchronization lies in its flexibility and real-time capability, effectively addressing challenges caused by clock drift or signal propagation delays during communication.

However,the adjustment range of resynchronization is limited by SJW the configuration of，If the phase error exceeds SJW the limit of，it may lead to sampling errors，thereby affecting the reliability of communication。

Therefore, resynchronization and hard synchronization work together to form the complete synchronization mechanism in the CAN protocol: hard synchronization provides the initial alignment for communication, while resynchronization dynamically compensates for errors during subsequent transmissions, ensuring the accuracy and stability of data transmission.

Arbitration Rules

In CAN bus communication, since multiple nodes share the same bus and only one data frame can be transmitted at a time, an efficient arbitration mechanism is required to ensure the orderly transmission of data.

The CAN protocol achieves efficient, conflict-free communication among multiple nodes through the collaborative operation of carrier sensing, non-destructive arbitration, and readback mechanisms.

Carrier Sensing

InBefore sending data, each node first listens to the state of the bus, and is only allowed to start sending data when the bus is idle (i.e., no other nodes are sending data).

This mechanism is similar to ‘Carrier Sense Multiple Access (CSMA)’ in Ethernet, which can effectively prevent multiple nodes from simultaneously seizing the bus.

If two or more nodes detect the bus is idle almost simultaneously and start sending data，they enter the non-destructive arbitration phase。

Non-destructive Arbitration

Non-destructive arbitration is the core of the CAN bus arbitration mechanism, and its core principle isbased on the wired-AND mechanism

  • When all nodes transmit a recessive level (logic 1), the bus appears recessive;
  • as long as one node transmits a dominant level (logic 0), the bus appears dominant.

This mechanism ensures that even if multiple nodes send data simultaneously, the signal on the bus will not be destroyed but dominated by the dominant level. The arbitration process is as follows:

  • The identifier (ID) field of each data frame is used for arbitration priority,the smaller the ID, the higher the priority。
  • When multiple nodes send data simultaneously, they compare their transmitted ID bit by bit with the actual level on the bus.
    • If a node transmits a recessive level (1) but detects a dominant level (0) on the bus, it means a node with higher priority is sending data. The node will immediately stop transmitting and enter listening mode, waiting for the next bus idle to attempt retransmission.
    • If a node transmits a dominant level (0) and the bus also shows a dominant level (0), the node continues to transmit, indicating it currently has the highest priority.

Readback Mechanism

To ensure the correctness of arbitration and the reliability of data transmission, CAN nodescontinuously read back the signal on the bus while sending data，and compare it with the data they are sending

If the sending node finds that the readback data is inconsistent with the data it sent（For example, transmitting a recessive level but reading back a dominant level）, it indicates that an arbitration conflict has occurred, and the node will immediately**stop transmitting and switch to listening mode**。

This mechanism ensures that low-priority nodes can exit transmission in time to avoid interfering with the communication of high-priority nodes

Error States

In the CAN protocol, the error state of a node is dynamically adjusted based on the**Transmit Error Counter (TEC)andReceive Error Counter (REC)**values.

Through three different error states—active error state, passive error state, and bus-off state—

the CAN protocol achieves effective management and isolation of faulty nodes while ensuring the reliability and stability of communication.

Active Error State

When both the Transmit Error Counter (TEC) and Receive Error Counter (REC) of a node are less than or equal to 127, the node is in the active error state.

In this state, the node can normally participate in bus communication and actively transmit an active error flag (dominant bit) when an error is detected. This dominant flag forcibly overrides the bus signal to notify other nodes that an error currently exists.

Since the dominant bit has a higher priority, this mechanism can quickly propagate error information, ensuring that errors are handled promptly and maintaining the reliability of bus communication.

Passive Error State

If the Transmit Error Counter (TEC) or Receive Error Counter (REC) of a node exceeds 127 (usually triggered by TEC), the node will enter the passive error state.

In this state, the node can still participate in communication, but its behavior is restricted: it cannot actively transmit a dominant error flag; instead, it transmits a passive error flag (recessive bit) to avoid interfering with the bus. Additionally, after each transmission, a delay of 8 recessive bits must be inserted before attempting communication again.

It is worth noting that if only a node in the passive error state detects an error while other active nodes do not, the bus is considered error-free and communication continues.

This design both limits the impact of low-reliability nodes on the bus and ensures the communication efficiency of high-reliability nodes.

Bus-Off State

When the Transmit Error Counter (TEC) of a node exceeds 255, the node enters the bus-off state.

In this state, the node is completely prohibited from communication—it can neither transmit nor receive data—to prevent further interference with the bus.

To resume communication, the node needs to detect 128 consecutive occurrences of 11 recessive bits (i.e., the bus is idle for a sufficiently long time), then reset TEC and REC to 0, thereby recovering to the active error state and rejoining bus communication.

This mechanism aims to isolate severely faulty nodes while providing them with an opportunity to rejoin the bus.

Error Detection Mechanism

The CAN (Controller Area Network) protocol effectively improves communication reliability through various error detection methods.

Error detection mechanisms can be divided into the following five types: bit error, stuff error, CRC error, form error, and ACK error.

Bit error

Bit error refers toan error detected when the transmitting unit compares its own output level with the bus level (excluding stuff bits) during communication and finds that the two are inconsistent。

This error usually indicates a conflict or interference between the transmitting node and the signal on the bus.

Bit errors can occur in data frames,、remote frames,、or any segment of error frames and overload frames.（from SOF to EOF）。

Since the bit error is detected by the transmitting unit itself, its detection unit is the transmitting unit.

Stuff error

Stuff error refers toan error triggered when the receiving unit continuously detects 6 bits of the same level within a segment that requires bit stuffing.。

The CAN protocol stipulates that in the bit-stuffed segment, a stuff bit of the opposite level must be inserted after every 5 consecutive bits of the same level to ensure the DC balance of the signal.

If the receiving unit detects 6 consecutive bits of the same level, a stuff error is considered to have occurred.

Stuff errors mainly occur in any segment of data frames and remote frames.（from SOF to EOF），Its detection unit is the receiving unit.。

CRC error

CRC error refers tothe error detected when the CRC check value calculated by the receiving unit based on the received data is inconsistent with the CRC sequence actually received in the framewhen the error is detected.

CRC check is a key mechanism in the CAN protocol used to verify data integrity. If the check fails, it indicates that the data may have been corrupted or lost during transmission.

CRC Errors usually occur in data frames and remote frames' CRC segment，Its detection unit is the receiving unit.。

Form error

Form error refers toThe error triggered when the receiving unit detects a format opposite to the fixed-format bit field while parsing a frame. For example, illegal values of the CRC delimiter, ACK delimiter, or EOF may cause a format error.

This error usually indicates that the frame structure does not conform to the CAN protocol specification.

Format errors can occur in the CRC delimiter、ACK delimiter、EOF segment，as well as in the error delimiter and overload delimiter，Its detection unit is the receiving unit.。

ACK Error

ACK error refers tothe error triggered when the sending unit detects a recessive level in the ACK Slot。

In the CAN protocol,after the sending unit completes the transmission of a data frame or remote frame, it waits for the acknowledgment signal (dominant level) from other nodes in the ACK Slot。

If the sending unit does not detect a dominant level, it considers that no acknowledgment signal has been received, thereby triggering an ACK error.

ACK The error mainly occurs in the ACK slot segment，and its detection unit is the sending unit。

CAN Bus Protocol and Standard Specifications

In modern communication systems, the ISO seven-layer network model is a classic framework for describing and standardizing network communication processes. It divides network communication into seven layers, from the physical signal transmission at the bottom to the application services at the top, with each layer having specific functions and roles.

Although the design of the CAN protocol does not fully follow the ISO seven-layer network model, its functions can be mapped to some layers of the model.

The CAN protocol mainly focuses on thePhysical Layer、Data Link LayerandTransport layerpartial functions.

ISO seven-layer network model

  1. Physical Layer
    The Physical Layer is responsible for the transmission of physical signals between devices, defining the electrical signals, hardware specifications, and communication media required for communication. It covers communication modes (such as simplex, half-duplex, full-duplex), signal levels (such as high and low levels representing logic 0 and 1), hardware specifications (such as cables, connectors, transceivers), and bit timing conditions (such as bit synchronization, sampling points), thereby ensuring reliable data transmission over the physical medium.

  2. Data Link Layer
    The Data Link Layer organizes the bit sequences received from the Physical Layer into meaningful data frames and provides error detection and control functions. Its specific items include data frame formats (such as frame start, identifier, data field, CRC check), access control methods (such as CSMA/CD, CSMA/CA), and error detection and correction mechanisms (such as CRC check, ACK response), to ensure the accuracy and integrity of data transmission over the link.

  3. Network Layer
    The Network Layer is responsible for the routing and relaying of data packets, ensuring efficient data transmission from the source device to the destination device. It achieves cross-network data transmission through functions such as routing (such as static routing, dynamic routing), address management (such as IP address assignment), and packet fragmentation and reassembly, providing support for communication in complex network environments.

  4. Transport Layer
    The Transport Layer ensures the reliability of data transmission, controlling data sequence, error recovery, and flow control. It guarantees the consistency, integrity, and efficiency of data during end-to-end transmission through mechanisms such as data transmission sequence control (such as sequence numbers, acknowledgment mechanisms), error recovery (such as retransmission mechanisms), and flow control (such as sliding window protocol).

  5. Session Layer
    The Session Layer manages the establishment, maintenance, and termination of communication sessions, ensuring correct data transmission during the session. It coordinates interactions between applications and improves the stability and efficiency of communication through functions such as session establishment and termination (such as handshake protocols), session synchronization (such as checkpoint mechanisms), and session recovery (such as breakpoint resume).

  6. Presentation Layer
    The Presentation Layer is responsible for the conversion of data representation formats, ensuring data compatibility between different systems. It enables seamless exchange and processing of data across different platforms and applications through functions such as data format conversion (such as text encoding, image formats), data encryption and decryption, and data compression and decompression.

  7. Application Layer
    The Application Layer provides network service interfaces for applications, implementing specific application functions. It directly serves end-users and meets diverse network application needs through functions such as application protocols (such as HTTP, FTP, SMTP), user interfaces (such as browsers, email clients), and data services (such as file transfer, email).

CAN Protocol Physical Layer and Data Link Layer

The CAN protocol primarily focuses on the Physical Layer and Data Link Layer.

  • Physical Layer in the CAN Protocol

The Physical Layer is responsible for defining the actual method of signal transmission, using NRZ (Non-Return-to-Zero) encoding and inserting a stuff bit after every 6 bits.

It also specifies the bit timing and the number of samples, which users can select according to their needs.

Synchronization is achieved through the Synchronization Segment (SS) and features resynchronization. However, specific details such as signal levels, communication speed, sampling points, electrical characteristics of drivers and the bus, and the form of connectors are not defined in the protocol and need to be determined by the user based on system requirements.

  • Data Link Layer in the CAN Protocol
    • **MAC Sublayer (Medium Access Control Sublayer)**It is responsible for organizing data into frames, including data frames, remote frames, error frames, and overload frames. It also handles arbitration during data conflicts, deciding which message continues to be sent based on priority, and responds with ACK (Acknowledgment) and NACK (Negative Acknowledgment). In addition, the MAC sublayer detects and reports CRC errors, stuff errors, bit errors, ACK errors, and format errors, and features fault confinement, which can automatically distinguish between temporary and permanent errors and exclude faulty nodes.
    • **LLC Sublayer (Logical Link Control Sublayer)**is responsible for the selection of received messages, supporting point-to-point connections, broadcasting, and multicasting. It also provides error recovery functionality, performing retransmission control when an error is detected to ensure reliable data transmission

After the CAN protocol was standardized by ISO, it was divided into two standards: ISO11898 and ISO11519-2.

The definitions of these two standards at the data link layer are the same, but they differ at the physical layer:

  • ISO11898 is the CAN high-speed communication standard with a communication speed of 125kbps-1Mbps
  • ISO11519-2 is the CAN low-speed communication standard with a communication speed below 125kbps.

struct can_frame

CAN data frames are usually represented bystruct can_framea struct. This struct is defined in the<linux/can.h>header file, used to encapsulate the CAN data to be sent. The contents of this struct are as follows:

  • can_id: can_id is a 32-bit unsigned integer field used to represent the CAN identifier (ID).
    • For standard frames, the lower 11 bits of can_id represent the identifier;
    • For extended frames, the lower 29 bits represent the identifier.
    • The 30th bit is used to identify whether it is a remote frame (via the CAN_RTR_FLAG).
    • The highest bit (31st bit) is used to identify whether it is an extended frame (via the CAN_EFF_FLAG),
  • can_dlc: can_dlc is an 8-bit unsigned integer field representing the length of the data field in the CAN data frame. Its value ranges from 0 to 8, corresponding to the maximum supported data length of 8 bytes in the classic CAN protocol.
    • For CAN FD (Flexible Data-rate CAN), can_dlc can support larger data lengths (e.g., 64 bytes). This field determines the actual number of data bytes sent or received, and unused parts are ignored.
  • data: data is an 8-byte array field used to store the actual data content to be sent or received.
    • The actual length of the data is determined by the can_dlc field, with a maximum of 8 bytes (classic CAN) or larger (CAN FD).
    • The unused part of the array will not be transmitted, so only valid bytes need to be filled when encapsulating data.
