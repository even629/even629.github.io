CAN Bus Introduction

CAN(Controller Area Network) is a serial communication protocol introduced by Bosch in 1986, specifically designed for automotive electronic systems, aiming to reduce the number of wire harnesses in vehicles and achieve efficient communication among multiple Electronic Control Units (ECUs).

Core Features

  • The CAN bus supports multi-master communication, where all nodes (ECUs) can actively send data without relying on a central controller.
  • It employs a non-destructive arbitration mechanism to resolve bus conflicts through ID priority, ensuring that high-priority data is transmitted first.
  • The bus features high reliability, with built-in CRC checks, error detection and recovery mechanisms, and uses differential signaling to enhance electromagnetic interference resistance.
  • The CAN bus achieves the advantage of low cost by simplifying wiring and reducing system complexity.

Application Areas

  • Automotive electronics: engine control, body modules (lighting/windows/doors), diagnostic systems (OBD-II), etc.
  • Industrial automation: PLC, sensor networks, robot control.
  • Other fields: medical equipment, aerospace, smart buildings.

Development history

  • 1983: Bosch initiated research on automotive network technology.
  • 1986: Official release of the CAN protocol.
  • 1987: Intel and Philips launched the first CAN controller chip.
  • 1991: CAN 2.0 specification released, first applied in the Mercedes-Benz S-Class sedan.
  • 1993: Became an international standard (ISO 11898).
  • 2015: CAN FD (Flexible Data Rate) standardized (supporting higher speeds and 64-byte data)
  • 2020: Next-generation CAN XL released, further increasing bandwidth.

CAN Electrical Properties

The CAN bus uses differential signals to transmit data.

  1. CAN Signal Levels

The CAN bus uses two signal lines for data transmission: CAN_H (High level) and CAN_L (Low level).

CAN_H and CAN_The voltage difference between H and CAN L determines the state of the logical signal. This gives the CAN bus better anti-interference capability, making it particularly suitable for working in noisy environments.

  • CAN_H: High-level signal line, typically with a voltage range of 2.5V to 3.5V.
  • CAN_L: Low-level signal line, typically with a voltage range of 1.5V to 2.5V.
  1. Differential Signals and Anti-interference

The CAN bus transmits data through differential signals, which means that the transmission of information relies on the CAN_H and CAN_voltage difference between H and CAN L, rather than individual voltage values. The characteristics of differential signals give the CAN bus extremely strong anti-interference capability in noisy environments.

Advantages of differential signals：

  1. Noise usually affects both CAN_H and CAN_L simultaneously, but since the differential signal relies on the voltage difference between the two,the effect of noise on both will cancel each other out, thereby maintaining signal integrity.
  2. This characteristic makes the CAN bus perform excellently in noisy environments such as automotive and industrial control.

Electrical properties and operating states

The electrical properties of the CAN bus determine its operating states, specifically manifested as the recessive state and the dominant state.

  • Recessive state (logic “1”)：When CAN_H and CAN_L have equal voltages（usually at 2.5V）, representingThe bus is in an idle state. At this time, there is no data transmission on the bus, and all devices are waiting for communication. The recessive state is also known as the ‘idle level’ or ‘logic 1’.
  • Dominant state (logic ‘0’)：When CAN_H and CAN_L the voltage difference is 2V ,, it indicatesdata is being transmitted. Specifically, CAN_H is 3.5V, CAN_L is 1.5V, representing logic ‘0’. The dominant state indicates that the bus is transmitting data, andhas a higher priority。

CAN protocol frame structure

Frame types

A frame is the basic unit of data transmission in CAN communication. Five different types of frames are used in CAN communication to achieve different functions. Below is a brief description and purpose of each frame:

  • Data Frame: The data frame is the most commonly used frame type in CAN communication, used by a sending unit to transmit data to a receiving unit. There are two formats for data frames:
    • Standard format (11-bit identifier) is suitable for simpler networks
    • Extended format (29-bit identifier) is suitable for more complex networks.
  • Remote Frame: The remote frame is used by a receiving unit to request data from a sending unit. It does not contain a data segment, but instead tells the sending unit ‘I need the data corresponding to this ID’ through the identifier (ID). The remote frame also has two formats: standard and extended. Its structure is similar to the data frame, but lacks the data segment.
  • Error Frame: The error frame is used to notify other nodes of an error when a node detects one. It is a special frame used to interrupt the current communication and restart it. The error frame consists of an error flag (6 consecutive dominant or recessive bits) and an error delimiter. Trigger conditions include bit errors, stuff errors, CRC errors, etc.
  • Overload Frame: The overload frame is used by a receiving unit to notify a sending unit that it is not yet ready to receive. It typically occurs when the receiving node needs more time to process the previous frame. The overload frame can only be sent during the interframe space, and its structure is similar to the error frame, consisting of an overload flag and an overload delimiter.
  • Interframe Space: The interframe space is used to separate data frames or remote frames from the preceding frames. It is a time interval that ensures nodes in the network can correctly identify the start of the next frame. For data frames and remote frames, the interframe space is typically 3 bit times; for error frames and overload frames, the interframe space may differ.

Data Frame

Data frames are the most core frame type, responsible for carrying actual data content and achieving efficient information transfer between the sending unit and the receiving unit.

In early CAN communication,standard data frames used an 11-bit identifier, which could meet the needs of the relatively simple network architecture at the time. However, with the increasing complexity of vehicle functions (such as advanced driver-assistance systems, powertrain control, body electronics, etc.), the address space of the 11-bit identifier gradually became insufficient, making it difficult to support larger-scale nodes and more refined communication requirements. To this end, the CAN protocol introducedextended data frames, which, through 29-bit identifiers, provide a larger address space, significantly enhancing the flexibility and scalability of the network.

    • CRC delimiter (1 bit), fixed at recessive 1, used to clearly define the end position of the CRC field.

    • ACK slot (1 bit), when the receiving device successfully receives the data frame, it returns a dominant bit in the ACK slot as an acknowledgment signal, indicating that the data frame has been correctly received
    • ACK delimiter (1 bit), the ACK delimiter is fixed at recessive 1, used to mark the end of the ACK field.

  • End of Frame（End of Frame）

    • Composed of 7 consecutive recessive bits (logic 1), used to clearly mark the end of the data frame.
    • It not only tells the receiving device that the data frame has been completely transmitted, but also provides extra time for signal synchronization to ensure the correct reception of the next frame.

SRR bit function

In the CAN protocol, the SRR bit is used to replace the RTR bit of the standard frame, located at the position of the original RTR bit in the arbitration field of the extended frame.

Its function is to be forced to a recessive level (1) to ensure that when a standard frame and an extended frame compete for the bus simultaneously, the standard frame has a higher priority due to its dominant RTR bit (0). When a standard frame and an extended frame are sent simultaneously, the bus detects the dominant level (0) through arbitration logic, allowing the standard frame to win.

IDE bit
The IDE bit is the Identifier Extension flag, used to distinguish frame types:Dominant (0) indicates the current frame is a standard frame, and recessive (1) indicates an extended frame.。

In the extended frame, the IDE bit not only exists in the arbitration field to identify the frame type but also appears in the control field, further clarifying the frame structure and achieving the distinction between extended and standard frames.

Remote frame (Remote Frame)is in the CAN protocola special frame type used to actively request data。

When a node needs to obtain data from another node, it can send a remote frame to initiate a request on the bus, and the target node returns the corresponding data frame upon receiving the request.

Error Frame

An error frame is a special frame in the CAN bus used to indicate errors occurring during communication. Its structure differs from other frames (such as data frames or remote frames) and is primarily used to notify all nodes in the network that an error has occurred in the current transmission.

  • Error Flag
    • 6 consecutive dominant bits (logic 0) or recessive bits (logic 1), which are divided into active error flags and passive error flags.
    • Active Error FlagSent by the node that detects the error, it forcibly interrupts the transmission of the current frame and notifies all nodes in the network of the error;
    • Passive Error FlagSent by a node in the passive error state to avoid interfering with the normal communication of other nodes.
  • Error Delimiter
    • Composed of 8 consecutive recessive bits (logic 1), it is used to mark the end of the error frame. It ensures that all nodes can clearly identify the boundary of the error frame and resume normal communication after error handling is completed.

Overload Frame

The overload frame is a special frame in the CAN bus used to notify other nodes in the network that the current receiving device needs more time to process data. It is usually sent when the receiving node cannot process the received data in time, to request the sending node to delay the transmission of the next frame. The composition of the overload frame is shown below:

  • Overload Flag
    • Composed of 6 consecutive dominant bits (logic 0), sent by the receiving node to notify other nodes in the network that the current receiving device needs more time to process data, thereby requesting a delay in the transmission of the next frame.
    • Multiple nodes can send overload flags simultaneously, and these flags will overlap to form a longer sequence of dominant bits.
  • Overload Delimiter
    • Composed of 8 consecutive recessive bits (logic 1), used to mark the end of the overload frame.
    • It ensures that all nodes can clearly identify the boundaries of the overload frame and resume normal communication after the overload processing is completed.

The design of the overload frame ensures that nodes in the network can coordinate communication under high load conditions, avoiding data loss or errors.

Bit stuffing

Bit stuffing is an important mechanism in the CAN protocol used to ensure signal synchronization during data transmission.

Bit stuffing ensures that there is always a level change on the bus by inserting a stuff bit of opposite polarity after 5 consecutive bits of the same polarity, thereby maintaining synchronization between the transmitter and receiver.

Stuffing is only performed between the start of frame and the CRC field to avoid interference with other parts. The receiver automatically removes the stuff bits after detecting them, restoring the original data. This mechanism is crucial for preventing signal synchronization issues and improving the reliability of CAN bus communication.

Core purpose

By inserting a stuff bit of opposite polarity after 5 consecutive bits of the same polarity, it avoids bus synchronization problems caused by prolonged “0” or “1” levels, thereby improving communication reliability.

Bit stuffing rules

When 5 consecutive bits of the same polarity appear in the data, a stuff bit of opposite polarity is inserted at the 6th bit.

This mechanism ensures that there is always a level change on the bus by breaking up possible long strings of the same level, thereby maintaining synchronization between the transmitter and receiver.

Stuffing range

Performed from the start of frame to the CRC field，CRC Data after the field（Like ACK field and frame end）no padding is performed. This design limits the padding range, avoids interference with parts that do not need synchronization, and ensures the synchronization of key data segments.

    • Original data: 110000011
    • Analysis: No 5 consecutive bits of the same polarity appear, so no padding is required.
    • Actual transmitted data: 1100000111

  • Example 2:

CAN Bus Core Mechanism and Communication Principle

Bit time

In CAN bus communication,**bit time refers to the time required to transmit one logic bit,**which is one of the core concepts for the CAN protocol to achieve reliable data transmission, directly affecting the communication baud rate and signal synchronization accuracy. As the basis of asynchronous communication, all nodes maintain synchronization through the agreed-upon bit time.

The CAN protocol divides the bit time into multiple time segments, which together determine the signal sampling and synchronization mechanism. Specifically, it includes the following parts:

  • Sync Segment

The sync segment is the first part of the bit time,fixed at 1 time quantum（TQ (Time Quantum) is the basic unit of bit time, and all time segments are divided based on the time quantum), which is mainly used to detect signal edges and achieve hard synchronization.

During the Start of Frame (SOF) phase, the receiving node forcibly adjusts its own clock based on the falling edge of the bus signal to align with the bus signal, thereby ensuring that all nodes remain synchronized during the initial stage of communication.

  • Propagation Segment

The propagation segment is used to compensate for the signal propagation delay on the bus and the response time of the input comparator, and its length is usually 1 to 8 time quanta (TQ), with the specific value determined by the configuration. This

period of time allows the signal to propagate on the physical medium and ensures that all nodes can receive a stable signal, thereby avoiding sampling errors caused by propagation delay.

  • Phase Segment 1

Phase Segment 1 is a configurable part of the bit time, usually 1 to 8 time quanta (TQ),which is mainly used to adjust the position of the sampling point to ensure that the sampling point is located in the stable region of the signal。

If a signal edge is detected to arrive early,，resynchronization can be achieved by shortening the phase segment 1 to achieve resynchronization，thereby dynamically compensating for the effects of clock deviation or signal jitter。
  • Phase Segment 2 (Phase Segment 2)

Phase Segment 2 is similar to Phase Segment 1, also being a configurable part of the bit time, typically ranging from 1 to 8 time quanta (TQ).

It is mainly used to handle the situation where signal edges arrive late。If the signal edge is delayed，it can be resolved by extending the phase buffer segment 2 to achieve resynchronization，thereby ensuring that the receiving node can sample the signal at the correct time，improving the reliability of data transmission。

The role of bit time

  1. Ensuring synchronization: By dividing time segments, CAN nodes can dynamically adjust the sampling point when signal edges appear, compensating for clock deviation and signal delay, thereby achieving synchronization among multiple nodes.
  2. Improving anti-interference capability: Setting the sampling point in the stable region of the signal (such as at 87.5% of the bit time) avoids misjudgment caused by signal edge jitter or interference.
  3. Supporting flexible configuration: By adjusting the length of each time segment (such as propagation segment, phase buffer segment, etc.), it can adapt to different baud rates and network environments.

Hard synchronization and resynchronization

In CAN communication, due to the adoption of asynchronous communication and the absence of an independent clock line, each node relies on a local crystal oscillator for data transmission. However, the clock drift of the crystal oscillator causes the timing between nodes to gradually shift, eventually leading to sampling errors. To solve this problem, the CAN protocol achieves dynamic timing adjustment through two major mechanisms: hard synchronization and resynchronization:

  1. Hard synchronization forces the alignment of all node clocks at the start of a frame, eliminating initial phase deviations；
  2. Resynchronization dynamically adjusts the length of the phase buffer segment based on real-time detection of signal edges during data transmission to compensate for clock drift。

These two mechanisms work in synergy, combining the bit time division (synchronization segment, propagation segment, phase segment I/II) and the quantitative management of the time quantum (Tq), to ensure that all nodes accurately read the bus level at the sampling point, thereby achieving highly reliable data transmission in asynchronous communication without an independent clock line. This section will provide a detailed explanation of hard synchronization and resynchronization

Hard synchronization

In CAN communication, hard synchronization is a crucial mechanism for ensuring consistent initial timing across all nodes. Its core function isto force the alignment of each node’s timing at the Start of Frame (SOF) stage, thereby eliminating initial phase differences caused by local oscillator variations。

The trigger condition for hard synchronization isWhen the bus detects the falling edge of a dominant level (logic 0) from the idle state (i.e., the edge of the SOF bit), the node immediately adjusts its own timing, aligning the synchronization segment of the current bit to this edge。

This process can be divided into the following steps:

  • The node, duringcontinuously monitors level changes during bus idle periods；
  • Whena SOF falling edge is detected and this edge falls within the synchronization segment, hard synchronization is triggered；
  • The node forcibly adjusts its timing, aligning the start of the synchronization segment of the current bit to this edge, and restarts the counting of subsequent bit times。

However,The scope of hard synchronization is limited; it only takes effect at the start of a message frame and can only resolve initial phase deviation issues,。

For dynamic phase errors caused by clock drift or signal interference during communication,，hard synchronization is powerless.，These issues need to be handled by subsequent resynchronization mechanisms.

Resynchronization

In CAN communication,resynchronization is a key mechanism for resolving dynamic phase errors caused by clock drift or signal interference during communication.。

Unlike hard synchronization, resynchronization is not limited to the start of the frame, butcontinues to function throughout the entire data transmission process.。Its core function is to dynamically adjust the position of the sampling point，to ensure that the receiving node can read the bus level at the correct time,，thereby maintaining the reliability of communication.。

Specifically,the trigger condition for resynchronization is when a deviation is detected between a signal edge (such as a transition from dominant to recessive level) and its expected position.。

  • If the signal edge arrives early, it indicates that the transmitting node’s clock is faster than the receiving node’s. In this case, the receiving node will advance the sampling point by shortening Phase Buffer Segment 1;
  • Conversely, if the signal edge arrives late, it indicates that the transmitting node’s clock is slower than the receiving node’s. The receiving node will delay the sampling point by lengthening Phase Buffer Segment 2.

This dynamic adjustment process relies onSynchronization Jump Width (SJW), which definesthe maximum range of each resynchronization adjustment, to prevent over-adjustment from causing new errors.

The advantage of resynchronization lies in its flexibility and real-time capability, which can effectively address challenges caused by clock drift or signal propagation delay during communication.

However,the adjustment range of resynchronization is limited by SJW the configuration of，If the phase error exceeds SJW the limit of，may lead to sampling errors，thereby affecting communication reliability。

Therefore, resynchronization and hard synchronization work together to form the complete synchronization mechanism in the CAN protocol: hard synchronization provides initial alignment for communication, while resynchronization dynamically compensates for errors during subsequent transmission, ensuring the accuracy and stability of data transmission.

Arbitration Rules

In CAN bus communication, since multiple nodes share the same bus, only one data frame can be transmitted at a time, thus requiring an efficient arbitration mechanism to ensure the orderly transmission of data.

The CAN protocol achieves efficient, conflict-free communication among multiple nodes through the collaborative operation of carrier sensing, non-destructive arbitration, and readback mechanisms.

Carrier Sensing

Beforesending data, each node first listens to the bus status, and the node is only allowed to start sending data when the bus is idle (i.e., no other node is sending data).

This mechanism is similar to “Carrier Sense Multiple Access (CSMA)” in Ethernet, which can effectively avoid multiple nodes seizing the bus simultaneously.

If two or more nodes detect that the bus is idle almost simultaneously and start sending data，they enter the non-destructive arbitration phase。

Non-destructive Arbitration

Non-destructive arbitration is the core of the CAN bus arbitration mechanism, and its core principle isWired-AND mechanism

  • When all nodes transmit a recessive level (logic 1), the bus is recessive;
  • As long as one node transmits a dominant level (logic 0), the bus becomes dominant.

This mechanism ensures that even if multiple nodes transmit data simultaneously, the signal on the bus will not be corrupted, but rather dominated by the dominant level. The arbitration process is as follows:

  • The identifier (ID) field of each data frame is used for arbitration priority,The smaller the ID, the higher the priority.。
  • When multiple nodes transmit data simultaneously, they compare their transmitted ID bit by bit with the actual level on the bus.
    • If a node transmits a recessive level (1) but detects a dominant level (0) on the bus, it means a node with higher priority is transmitting data. The node will immediately stop transmitting and enter listening mode, waiting to retry when the bus is idle next time.
    • If a node transmits a dominant level (0) and the bus is also at a dominant level (0), the node continues to transmit, indicating that it currently has the highest priority.

Readback mechanism

To ensure the correctness of arbitration and the reliability of data transmission, CAN nodescontinuously read back the signal on the bus while transmitting data，and compare it with the data they are transmitting.

If the transmitting node finds that the readback data is inconsistent with the data it transmitted,（for example, transmitting a recessive level but reading back a dominant level,）, it indicates that an arbitration conflict has occurred, and the node will immediately**stop transmitting and switch to listening mode**。

This mechanism ensures that low-priority nodes can exit transmission in time to avoid interfering with the communication of high-priority nodes

Error states

In the CAN protocol, the error state of a node is dynamically adjusted based on the**Transmit Error Counter (TEC)andReceive Error Counter (REC)**values.

Through three different error states: Error Active, Error Passive, and Bus Off,

the CAN protocol achieves effective management and isolation of faulty nodes while ensuring the reliability and stability of communication.

Error Active

When both the Transmit Error Counter (TEC) and Receive Error Counter (REC) of a node are less than or equal to 127, the node is in the Error Active state.

In this state, the node can normally participate in bus communication and actively transmit an Active Error Flag (dominant bit) when an error is detected. This dominant flag forcibly overrides the bus signal to notify other nodes that an error currently exists.

Since the dominant bit has a higher priority, this mechanism can quickly propagate error information, ensuring that errors are handled in a timely manner and maintaining the reliability of bus communication.

Error Passive State

If a node’s Transmit Error Counter (TEC) or Receive Error Counter (REC) exceeds 127 (usually triggered by TEC), the node will enter the Error Passive state.

In this state, the node can still participate in communication, but its behavior is restricted: it cannot actively send a dominant error flag; instead, it sends a passive error flag (recessive bits) to avoid interfering with the bus. Additionally, after each transmission, it must insert a delay of 8 recessive bits before attempting to communicate again.

It is worth noting that if only a node in the Error Passive state detects an error, while other active nodes do not, the bus is considered error-free and communication continues.

This design both limits the impact of low-reliability nodes on the bus and ensures the communication efficiency of high-reliability nodes.

Bus-Off State

When a node’s Transmit Error Counter (TEC) exceeds 255, the node enters the Bus-Off state.

In this state, the node is completely prohibited from communication; it can neither send nor receive data to prevent further interference with the bus.

To recover communication, the node needs to detect 128 occurrences of 11 consecutive recessive bits (i.e., the bus is idle for a sufficiently long time), then reset the TEC and REC to 0, thereby recovering to the Error Active state and rejoining bus communication.

This mechanism aims to isolate severely faulty nodes while providing them with an opportunity to rejoin the bus.

Error detection mechanism

The CAN (Controller Area Network) protocol effectively improves communication reliability through a variety of error detection methods.

The error detection mechanism can be divided into the following five types: bit error, stuff error, CRC error, form error, and ACK error.

Bit error

A bit error refers tothe error detected when the transmitting unit compares its own output level with the bus level (excluding stuff bits) during communication and finds them inconsistent.。

This error usually indicates a conflict or interference between the transmitting node and the signal on the bus.

Bit errors can occur in data frames,、remote frames,、or any segment of error frames and overload frames.（from SOF to EOF）。

Since the bit error is detected by the transmitting unit itself, the detection unit is the transmitting unit.

Stuff Error

A stuff error refers toan error triggered when a receiving unit detects 6 consecutive bits of the same level within a segment requiring bit stuffing.。

The CAN protocol stipulates that in the bit-stuffing segment, a stuff bit of the opposite level must be inserted after every 5 consecutive bits of the same level to ensure the DC balance of the signal.

If the receiving unit detects 6 consecutive bits of the same level, a stuff error is considered to have occurred.

Stuff errors mainly occur in any segment of data frames and remote frames.（from SOF to EOF），Its detection unit is the receiving unit.。

CRC Error

A CRC error refers tothe CRC check value calculated by the receiving unit based on the received data is inconsistent with the CRC sequence actually received in the framedetected at that time.

CRC check is a key mechanism in the CAN protocol used to verify data integrity. If the check fails, it indicates that the data may have been corrupted or lost during transmission.

CRC Errors usually occur in the data frames and remote frames' CRC Within the segment，Its detection unit is the receiving unit。

Form error

Form error refers toAn error triggered when the receiving unit detects a format opposite to the fixed-format bit field while parsing a frame. For example, illegal values of the CRC delimiter, ACK delimiter, or EOF may cause a form error.

This error usually indicates that the frame structure does not comply with the CAN protocol specification.

Form errors can occur in the CRC Delimiter、ACK Delimiter、EOF Segment，as well as in the error delimiter and overload delimiter，Its detection unit is the receiving unit。

ACK error

ACK error refers toThe error triggered when the transmitting unit detects a recessive level in the ACK Slot。

In the CAN protocol,After the transmitting unit completes the transmission of a data frame or remote frame, it waits for the acknowledgment signal (dominant level) from other nodes in the ACK Slot。

If the transmitting unit does not detect a dominant level, it considers that no acknowledgment signal has been received, thereby triggering an ACK error.

ACK The error mainly occurs in the ACK slot segment of data frames and remote frames，Its detection unit is the transmitting unit。

CAN bus protocol and standard specifications

In modern communication systems, the ISO seven-layer network model is a classic framework for describing and standardizing network communication processes. It divides network communication into seven layers, ranging from the physical signal transmission at the bottom to the application services at the top, with each layer having specific functions and roles.

Although the design of the CAN protocol does not fully follow the ISO seven-layer network model, its functions can be mapped to some layers of the model.

The CAN protocol is mainly concentrated in thephysical layer、data link layerandtransport layersome of the functions.

ISO seven-layer network model

  1. Physical Layer
    The Physical Layer is responsible for the transmission of physical signals between devices, defining the electrical signals, hardware specifications, and communication media required for communication. It covers communication methods (such as simplex, half-duplex, and full-duplex), signal levels (such as high and low levels representing logic 0 and 1), hardware specifications (such as cables, connectors, and transceivers), and bit timing conditions (such as bit synchronization and sampling points), thereby ensuring reliable data transmission over the physical medium.

  2. Data Link Layer
    The Data Link Layer organizes the bit sequences received from the Physical Layer into meaningful data frames and provides error detection and control functions. Its specific items include data frame formats (such as frame start, identifier, data field, and CRC check), access control methods (such as CSMA/CD and CSMA/CA), and error detection and correction mechanisms (such as CRC check and ACK response), to ensure the accuracy and integrity of data transmission over the link.

  3. Network Layer
    The Network Layer is responsible for the routing and relaying of data packets, ensuring efficient data transmission from the source device to the destination device. It achieves cross-network data transmission through functions such as routing (such as static routing and dynamic routing), address management (such as IP address assignment), and packet segmentation and reassembly, providing support for communication in complex network environments.

  4. Transport Layer
    The Transport Layer ensures the reliability of data transmission, controlling data sequence, error recovery, and flow control. It guarantees the consistency, integrity, and efficiency of data during end-to-end transmission through mechanisms such as data transmission sequence control (such as sequence numbers and acknowledgment mechanisms), error recovery (such as retransmission mechanisms), and flow control (such as the sliding window protocol).

  5. Session Layer
    The Session Layer manages the establishment, maintenance, and termination of communication sessions, ensuring correct data transmission during the session. It coordinates interactions between applications and improves the stability and efficiency of communication through functions such as session establishment and termination (such as handshake protocols), session synchronization (such as checkpoint mechanisms), and session recovery (such as breakpoint resume).

  6. Presentation Layer
    The Presentation Layer is responsible for the conversion of data representation formats, ensuring data compatibility between different systems. It enables seamless exchange and processing of data across different platforms and applications through functions such as data format conversion (such as text encoding and image formats), data encryption and decryption, and data compression and decompression.

  7. Application Layer
    The application layer provides network service interfaces for applications and implements specific application functions. Through functions such as application protocols (e.g., HTTP, FTP, SMTP), user interfaces (e.g., browsers, email clients), and data services (e.g., file transfer, email), it directly serves end users and meets diverse network application needs.

CAN Protocol Physical Layer and Data Link Layer

The CAN protocol primarily focuses on the physical layer and the data link layer.

  • Physical Layer in the CAN Protocol

The physical layer is responsible for defining the actual method of signal transmission, using NRZ (Non-Return-to-Zero) encoding and inserting a stuff bit after every 6 bits.

It also specifies bit timing and the number of samples, which users can select according to their needs.

Synchronization is achieved through the Synchronization Segment (SS) and features resynchronization capabilities. However, specific details such as signal levels, communication speed, sampling points, electrical characteristics of drivers and the bus, and connector shapes are not defined in the protocol and must be determined by the user based on system requirements.

  • Data Link Layer in the CAN Protocol
    • **MAC Sublayer (Medium Access Control Sublayer)**It is responsible for organizing data into frames, including data frames, remote frames, error frames, and overload frames. It also handles arbitration during data conflicts, determining which message continues to be sent based on priority, and responds via ACK (Acknowledgment) and NACK (Negative Acknowledgment). Additionally, the MAC sublayer detects and reports CRC errors, stuff bit errors, bit errors, ACK errors, and format errors. It also features fault confinement suppression, capable of automatically distinguishing between temporary and permanent errors and removing faulty nodes.
    • **LLC Sublayer (Logical Link Control Sublayer)**It is responsible for the selection of received messages, supporting point-to-point connections, broadcasting, and multicasting. It also provides error recovery functions, performing retransmission control when errors are detected to ensure reliable data transmission.

After ISO standardization, the CAN protocol is divided into two standards: ISO11898 and ISO11519-2.

These two standards have the same definition in the data link layer, but differ in the physical layer:

  • ISO11898 is the CAN high-speed communication standard with a communication speed of 125kbps-1Mbps.
  • ISO11519-2 is the CAN low-speed communication standard for communication speeds below 125kbps.

  • Meaning: Exclude extended frames, and the CAN ID exactly matches 0x12345678.
  • Analysis:
    • can id = 0x12345678。
    • mask = 0x1FFFFFFF (matches all 29 bits, and requires it to be an extended frame). Use ~ to indicate exclusion matching

Multiple matching rules

If you want to apply multiple matching rules, you just need to append them with commas, for example, only receive standard frames, and the CAN ID exactly matches 0x123 and 0x456

    • Command format:cansend <interface> <frame ID>#<data>

struct can_frame

CAN data frames are usually represented bystruct can_framea struct. This struct is defined in the<linux/can.h>header file, used to encapsulate the CAN data to be sent. The contents of this struct are as follows:

  • can_id: can_id is a 32-bit unsigned integer field used to represent the CAN identifier (ID).
    • For standard frames, the lower 11 bits of can_id represent the identifier;
    • For extended frames, the lower 29 bits represent the identifier.
    • Bit 30 is used to identify whether it is a remote frame (via the CAN_RTR_FLAG flag).
    • The highest bit (bit 31) is used to identify whether it is an extended frame (via the CAN_EFF_FLAG flag),
  • can_dlc: can_dlc is an 8-bit unsigned integer field representing the length of the data field in a CAN data frame. Its value ranges from 0 to 8, corresponding to the maximum supported data length of 8 bytes in the classic CAN protocol.
    • For CAN FD (Flexible Data-rate CAN), can_dlc can support larger data lengths (such as 64 bytes). This field determines the number of data bytes actually sent or received, and unused parts are ignored.
  • data: data is an 8-byte array field used to store the actual data content to be sent or received.
    • The actual length of the data is determined by the can_dlc field, with a maximum of 8 bytes (classic CAN) or larger (CAN FD).
    • Unused parts of the array are not transmitted, so only valid bytes need to be filled when encapsulating data.
