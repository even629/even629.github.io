Linux driver abstraction

Linux divides memory and peripherals into 3 basic categories:character device driver，block device driver，network device driver。

Linux source code directory structure

Table of ContentsDescription
archArchitecture-related directory, storing adaptation code for various CPU architectures (such as arm, X86, MIPS, etc.)
blockBlock device related code directory, in Linux, storage devices such as hard drives and SD cards are managed in the form of block devices
cryptoEncryption algorithm directory, storing various encryption-related implementation code
DocumentationOfficial Linux kernel documentation directory, containing detailed descriptions of kernel features, interfaces, etc.
driversDrivers directory, storing driver code for various hardware devices supported by the Linux system
firmwareFirmware directory, storing firmware files required by hardware devices
fsFilesystems directory, storing implementation code for filesystems such as ext2, ext3, fat, etc.
includePublic headers directory, providing header files shared by various kernel modules
initKernel boot initialization directory, storing initialization code for the Linux kernel boot phase
ipcInter-process communication directory, storing implementation code for IPC mechanisms such as pipes, message queues, shared memory, etc.
kernelKernel core directory, storing the core functionality code of the kernel itself
libLibrary directory, storing various library functions used by the kernel
mmMemory management directory (mm stands for memory management), responsible for kernel memory management functions
netNetworking directory, storing implementation code for network functions such as the TCP/IP protocol stack
scriptsScripts directory, storing script files used in processes such as kernel compilation and testing
securitySecurity directory, storing implementation code for kernel security mechanisms
soundSound directory, storing audio device drivers and audio processing code
toolsTools directory, storing tool programs used for Linux kernel development and debugging
usrCode directory related to Linux kernel boot
virtKernel virtual machine directory, storing implementation code for kernel-level virtualization features

Analysis of the simplest Linux driver structure

  • Components
    1. Header files (required): Drivers need to include kernel-related header files, among which<linux/module.h>and<linux/init.h>is essential.
    2. Driver loading function (required): When loading the driver, this function will be automatically executed by the kernel.
    3. Driver unloading function (required): When the driver is unloaded, this function will be automatically executed by the kernel.
    4. License declaration (required): Because the Linux kernel follows the GPL protocol, the driver must also comply with relevant protocols when loaded. Common ones include various license types such as GPL v2.
    5. Module parameters (optional): are the values passed to the kernel module when the module is loaded.
    6. Author and version information (optional): Used to declare the driver’s author and code version information.
  • Example

The module will be loaded successfully if and only if the init function in module_init returns a value greater than or equal to 0

Compiling Linux drivers

  • Put the driver inside the Linux kernel, and then compile the Linux kernel. Compile the driver into the Linux kernel.
  • Compile the driverCompile into a kernel module, independent of the Linux kernel
    • Kernel modules are a special mechanism in the Linux system that cancompile some rarely used or temporarily unused functions into kernel modules, and dynamically load it into the kernel when needed.
    • Using kernel modules can reduce the kernel size and speed up the boot process. And drivers can be inserted or unloaded while the system is running without rebooting. The suffix for kernel modules is .ko

obj-<X>kbuild variable

This actually corresponds toobj- <X>mode, where<X>should be y, m, blank, or n. In general, the makefile located at the head of the kernel build system uses it.

This tells kbuild that there is an object named mymodule.o in the current directory. mymodule.o will be built from mymodule.c or mymodule.S.

How and whether to build or link mymodule.o depends on<X>the value of

  • If<X>If set to m, the variable obj-m is used, and mymodule.o is built as a module.
  • If<X>If set to y, the variable obj-y is used, and mymodule.o will be built as part of the kernel. It can also be called a built-in module.
  • If<X>If set to n, it means the module is not compiled.

Module-related commands

Module loading commands

  • insmod
    • Function: Load Linux kernel module
    • Syntax: insmod module_name
    • Example: insmod hello_world.ko
  • modprobe
    • Function: Load a kernel module, and the modules it depends on are also loaded simultaneously
    • Syntax: modprobe module_name
    • Example: modprobe hello_world.ko

System administrators or production systems commonly use modprobe. modprobe is smarter; it parses the modules.dep file before loading the specified module to load dependencies first. It automatically handles module dependencies, just like a package manager does.

But**modprobecannot directly load.kofiles under arbitrary paths** — it only looks for modules from standard kernel module directories (such as/lib/modules/$(uname -r)/), and relies on themodules.depindex.

Module unloading commands

  • rmmod

    • Function: Remove a kernel module already loaded into Linux
    • Syntax: rmmod module_name
    • Example: rmmod hello_world.ko

  • modprobe -r

    • It will not only attempt to unloadmymodule, but will also automatically check and unload those dependency modules that are only depended on bymymoduleand are not currently used by other modules/processes.
    • Syntax:modprobe -r mymodule

The enabling or disabling of the kernel module unloading feature is determined by the CONFIG_MODULE_UNLOAD configuration option. Without this option, no modules can be unloaded.

At runtime, if unloading a module would cause other adverse effects, the kernel will prevent it even if someone requests it. This is becausethe kernel records the usage count of a module through reference counting, so it knows whether the module is in use. If the kernel considers it unsafe to remove a module, it will not remove it. However, the following setting can change this behavior: MODULE_FORCE_UNLOAD=y

Setting up module loading at boot time

If you want to load some modules at boot time, you just need to create a file/etc/modules-load.d/<filename>.conf, and add the names of the modules that should be loaded (one per line). People usually use modules:/etc/modules-load.d/modules.conf. Of course, you can also create multiple .conf files as needed.

Command to view module information

  • lsmodCommand
    • Function: List the kernel modules loaded into Linux
    • You can also use the command cat /proc/modules to check if the module is loaded successfully
  • modinfoCommand
    • Function: View kernel module information
    • Syntax: modinfo module_name
    • Example: modinfo hello_world.ko

Configuration file

menuconfigConfigure driver option status operation:

Driver status:

  1. Compile the driver as a kernel module, represented by M
  2. Compile the driver into the kernel, represented by *
  3. Do not compile

Use the spacebar to switch between these three different states.

The states of the option are:

  • []: indicates that there are two states, which can only be set to selected or unselected
  • <>: indicates that there are three states, which can be set to selected, unselected, and compiled as a module
  • () : indicates that it is used to store strings or hexadecimal numbers

Kconfig file

The Kconfig file is the source file for the graphical configuration interface, and the options in the graphical configuration interface are determined by the Kconfig file. When we execute the commandmake menuconfigcommand, the kernel’s configuration tool will read thearch/xxx/Kconfig. xxx is the value of ARCH, such as arm64, and then generate the corresponding configuration interface for developers to use.

config file and .config file

Both the config file and the .config file are configuration files for the Linux kernel.

  • The config file is located in the arch/$(ARCH)/configs directory of the Linux kernel source code, and is thedefault configuration file for the Linux system。

  • The .config file is located in the top-level directory of the Linux kernel source code. When compiling the Linux kernel, the configuration in the .config file is used to compile the kernel image.

    • If .config exists, the default configuration of the make menuconfig interface is the configuration of the current .config file. If the settings of the graphical configuration interface are modified and saved, the .config file will be updated.

    • If the .config file does not exist, the default configuration of the make menuconfig interface is the default configuration in the Kconfig file.

Using the make xxx_defconfig command will generate a .config file based on the default file in the arch/$(ARCH)/configs directory.

Driver module parameters

Significance of driver parameters:

Advantages:

  1. By passing parameters to the driver, the driver program can be made more flexible and have stronger compatibility.
  2. Security checks can be set through driver parameters to prevent the driver from being misused

Disadvantages

  1. Complicates the driver code
  2. Increased driver resource usage

Parameter types that can be passed to the driver

The kernel supports most commonly used C language data types for driver parameter passing. Here, the parameter types supported by the kernel for driver passing are divided into three categories:

  • Basic types：char, bool, int, long, short, byte, ushort, uint
  • Array：array
  • String：string

Kernel symbol table import and export

Device drivers can be compiled into kernel modules, which are KO files. Each KO file is independent, meaning modules cannot access each other. However, in some use cases, they need to access each other, such as when Module B needs to use a function from Module A. (Module B depends on Module A)

Symbol table

A “symbol” is a function name, global variable name, etc. in the kernel. A symbol table is a file used to record these “symbols”.

Module dependencies

Modules in the Linux kernel can provide functions or variables, usingEXPORT_SYMBOLmacros to export them for use by other modules; these are called symbols.

Module B’s dependency on Module A means that Module B uses symbols exported from Module A.

depmodis a user-space tool (usually provided bykmodpackage), run after kernel installation:

  • It scans/lib/modules/<kernel_release>/all.komodule files:
    • Usemodinfo -F dependsor directly parses the ELF symbol table;
    • determines for each module which symbols it needs (imports) and which symbols it provides (exports)；
    • and builds the dependency graph between modules.

The generated dependency file

FileFunction
modules.deptext format, each line format:module.ko: dependent_module1.ko dependent_module2.ko ...
modules.dep.binbinary format, formodprobequick loading (avoiding text parsing every time)
modules.symbols/modules.symbols.binrecords all exported symbols and their owning modules (for reverse lookup)

depmod also processes module files to extract and collect this information, and generates the modules.alias file in/lib/modules/<kernel_release>/modules.alias, which maps devices to their corresponding drivers.

modprobe parses the modules.alias file.
