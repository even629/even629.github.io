This article introduces the working principle and maintenance mechanism of TLB under the ARMv8 architecture, discussing in detail the background knowledge, multi-level design, entry attributes, and solutions to the homonym and synonym problems of TLB. In addition, this article summarizes the application of ASID technology in process switching and the Linux kernel, the contiguous block cache function, the usage scenarios and format of TLBI instructions, and explains the correct process of TLB flushing after modifying page tables and the related control registers.

TLB background knowledge

  • TLB iscachea kind of
  • TLB entries:records the conversion results from the most recently used VA to PA
  • The TLB base address is specified in the translation base register (TTBR0_EL1) and (TTBR1_EL1). When the upper bits of VA are all 0, TTBR0
    The pointed-to translation table is selected. When the upper bits of the VA are all set to 1, TTBR1 is selected.
  • EL2 and EL3 only have TTBR0, not TTBR1. By default (TCR_ELx.T0SZ = 16) the addressable range is 0x0 to 0x0000FFFF_FFFFFFFF, but this range can be adjusted via the T0SZ field.

  • TLB also supports three mapping methods: fully associative, direct-mapped, and set-associative.
  • Cortex-A72 adopts a two-level TLB design, similar to the design concept of a 2-level cache.
    • L1 instruction and data TLBs (fully associative)
      • 48-entry fully-associative L1 I-TLB
      • 32-entry fully-associative L1 D-TLB
    • L2 unified TLB. 4-way set-associative of 1024-entry L2 TLB (set-associative)

Each TLB entry typically contains not only physical and virtual addresses, but also attributes such as memory type, cache policy, access permissions, and Address Space ID
(ASID) and Virtual Machine ID (VMID).

TLB aliasing (synonym) problem

Although VP1 and VP2 cache two different TLB entries in the TLB, the PFNs in the TLB entries both point to Phys Page, so,no aliasing problem will occur

TLB homonym problem

When process A switches to process B, the TLB entries used by the old process are useless data for the new process, and may cause homonym problems. If the TLB is directly invalidated during process switching, there will be a significant performance loss. The solution is to use**ASID（Address Space Identifier）**technology

ASID（Address Space Identifier）

  • global type TLB：Kernel space is a space shared by all processes

  • process-specific type TLB: user address space is an independent address space for each process

  • The ASID mechanism is used to implement process-specific type TLB

  • The ASID of ARMv8 is stored in TTBR0_EL1 or TTBR1_EL1. The A1 field of the TCR register can be used to select

  • ASID supports 8-bit or 16-bit

    • 8-bit wide ASID, supporting up to 256 IDs
    • 16-bit wide ASID, supporting 65536 IDs

TCR_EL1 Defined in:

The TLB can identify which TLB entry belongs to which process, this is the core idea for solving the TLB homonym problem. This way, when switching processes, only the TLB entries of the switched-out process need to be flushed

Operating systems often use a bitmap to manage ASIDs, and generally do not use the process’s PID

In addition, the ARMv8-A architecture provides thread ID registers for use by operating system software. These registershave no hardware significance, typically used by thread libraries as
a base pointer for per-thread data. This is commonly known as Thread Local Storage (TLS). For example, the pthreads library uses this feature, incl
uding the following registers:

  • User Read/Write Thread ID Register (TPIDR_EL0).
  • User Read-Only Thread ID Register (TPIDRRO_EL0).
  • Thread ID Register, privileged access only (TPIDR_EL1)

ASID in the Linux kernel

  • In the Linux kernel, two ASIDs are allocated for each process, consisting of a pair of odd and even numbers

    • When a process runs in user mode, the odd ASID is used to look up the TLB
    • When a program traps into kernel mode, the even ASID is used to look up the TLB

    Allocating two ASIDs for each process is mainly to solve the Meltdown vulnerability issue

  • The allocation of hardware ASIDs is managed through a bitmap

  • During process switching, the hardware ASID held by the process needs to be written into the TTBR1_EL1 register

  • When the total number of hardware ASIDs in the system exceeds the hardware maximum, an overflow occurs, requiring a full TLB flush and reallocation of hardware ASIDs

nG attribute in page table entries

Bit 11 nG:

  • 1：Indicates that the TLB page table entry corresponding to this page is process-specific

  • 0：Indicates that a global TLB is used

TLB caches a block

The ARMv8-A architecture provides a feature called contiguous block entries to efficiently use TLB space.

Each mapping table block entry contains a contiguous bit. When set, this bit signals to the TLB that it can cache a single entry covering a mapping of multiple blocks. A lookup can index anywhere within the address range covered by a contiguous block. Therefore, the TLB can cache an entry for a defined address range, making it possible to store a larger range of virtual addresses in the TLB.

To use a contiguous bit, the contiguous blocks must be adjacent, meaning they must correspond to a contiguous virtual address range. They must start from an aligned boundary, have consistent attributes, and point to a contiguous output address range of the same level of translation.

TLB instructions

If the operating system modifies mapping entries that may be cached in the TLB, it is the OS’s responsibility to invalidate these stale TLB entries, which requires the use of the TLBI instruction.

  • ARMv8 provides the TLBI instruction.
  • Instruction format:
  • Type:
    • ALL Entire TLB
    • VMALL All TLB entries (stage 1, for current guest OS)
    • VMALLS12 All TLB entries (stage 1&2 for current guest OS)
    • ASID TLB entries matching the ASID, where xt specifies the virtual address and ASID
    • VA TLB entries specified by virtual address, where xt specifies the virtual address and ASID
    • VAA TLB entries specified by virtual address, where xt specifies the virtual address but excludes ASID
  • Level: En = Exception level (n can be 3, 2, or 1)
  • IS: indicates inner shareable
  • Xt: A parameter composed of a virtual address and an ASID.
    • Bit[63:48]：ASID
    • Bit[47:44]: TTL, used to indicate which level of page table’s saved address is to be invalidated. If it is 0, it means that all levels of page tables need to be invalidated (usually set to 0).
    • Bit[43:0]: Bit[55:12] of the virtual address (i.e., VA[55:12]).

Example: Flushing the TLB after modifying the page table.

Execution scenario: You modified the page table mapping (e.g., modifying the mapping from virtual address to physical address), the CPU may still use the old TLB translation cache, so:

  1. MustFirst ensure the page table write is complete →DSB ISHST
  2. Invalidate the old TLB →TLBI ALLE1
  3. Ensure the invalidation action is truly complete →DSB ISH
  4. Flush the CPU instruction execution context →ISB
