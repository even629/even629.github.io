USB Basics

USB (Universal Serial Bus) is a widely used general interface standard between computers and electronic devices, capable of transmitting data and supplying power. USB is a standard.

  1. Concept Proposal and Initial Standard Formulation (1994 - 1996)
    In 1994, companies such as Compaq, IBM, and Intel jointly developed USB technology to simplify the connection between computers and peripherals. In 1996, the USB 1.0 standard was released, supporting transfer rates of 1.5Mbps (Low Speed) and 12Mbps (Full Speed), meeting the needs of low-speed devices like mice and keyboards, and also implementing hot-swapping, which was convenient for users.
  2. Technical Improvement and Popularization (1998 - 2000)
    In 1998, USB 1.1 optimized electrical characteristics and compatibility, improving reliability and driving the popularization of devices. In 2000, USB 2.0 was a major breakthrough, introducing a 480Mbps High Speed transfer mode, meeting the needs of high-speed devices like digital cameras and external hard drives, and becoming the mainstream interface standard.
  3. Rapid Development and Performance Improvement (2008 - 2013)
    In 2008, USB 3.0 (later called USB 3.1 Gen 1) was introduced, increasing the rate to 5Gbps, optimizing power management and signal transmission, supporting high-power device charging, and meeting needs such as high-definition video transmission. In 2013, USB 3.1 Gen 2 was released, with a rate of 10Gbps, making it possible for devices like high-performance external SSDs.
  4. Continuous Evolution and Diversification (2014 - Present)
    In 2014, the USB Type-C interface appeared; it is small in size, reversible, capable of high-power transmission (up to 100W), and high-speed data transmission, gradually becoming mainstream. In 2017, USB 3.2 defined different rate identifiers, enriching rate options. In 2019, USB4, based on Thunderbolt 3 technology, reached a maximum rate of 40Gbps, enhancing features like multi-monitor support and expanding application scenarios.

USB continues to evolve, continuously improving in data transmission, device compatibility, and power supply capabilities, becoming an indispensable connection method for modern electronic devices. There are different types of USB interfaces, and interface standards can be found on the official website:

USB Interface Classification

USB interfaces can be mainly divided into three categories based on appearance: Type-A, Type-B, and Type-C, among which Type-B has also derived specifications such as Mini-B, Micro-USB, and Micro-B SuperSpeed. A detailed introduction will follow.

  1. The Type-A interface, as shown in the figure below, is a rectangular interface widely used in host devices such as computers and chargers, as well as traditional peripherals like USB flash drives, mice, and keyboards.

  1. The Type-B interface, as shown in the figure below, is a square interface, mostly used for devices such as printers, microphones, and scanners.

  1. The Mini-B interface, as shown in the figure below, is mostly used for dashcam devices.

  1. The Micro-USB interface, as shown in the figure below, is a small interface that was once used for portable devices like older mobile phones and power banks.

  1. The Micro-B SuperSpeed interface, as shown in the figure below, is commonly used for external hard drives.

  1. The Type-C interface, as shown in the figure below, is reversible. Currently, the Type-C interface is the mainstream interface for modern devices such as mobile phones, laptops, and headphones.

Summary:

Although the different types of USB interfaces in the figure differ in appearance and purpose, they all follow the USB standard protocol, ensuring compatibility and interoperability between devices.

It should be noted here that one cannot assume a device has high-speed data transfer capabilities just because it uses a Type-C interface. Type-C is just an appearance specification; the actual data transfer speed depends on the USB protocol version it uses. For example, if the USB 2.0 protocol is used, even if the interface is Type-C, its speed is relatively limited, while USB 3.0 and higher protocol versions are required to achieve high-speed transfer.

USB Version

Initially, USB 3.0 was named SuperSpeed USB, emphasizing its high-speed data transfer capabilities. Later, the USB 3.1 version was released, further improving its transfer rate.

However, to simplify the naming:

  • USB 3.0 was renamed USB 3.1 Gen 1
  • USB 3.1 was renamed USB 3.1 Gen 2.

USB-IF introduced the latest USB naming convention:

  • USB 3.1 Gen 1 was renamed USB 3.2 Gen 1
  • USB 3.1 Gen 2 was renamed USB 3.2 Gen 2
  • USB 3.2 Gen 2*2 is USB 3.2

After two name changes:

  • USB 3.0 is named USB 3.2 Gen 1
  • USB 3.1 is named USB 3.2 Gen 2
  • USB 3.2 is named USB 3.2 Gen 2x2

The version of a USB port can generally be inferred from its color

  • Black/White: USB 1.0/2.0
  • Dark Blue: USB 3.0
  • Light Blue: USB 2.0
  • Red: USB 3.1

USB devices are divided into Host (master device) and Slave (slave device), and data transfer can only be achieved when a Host is connected to a Slave.

  • USB Host: USB Host refers to a device with USB host functionality. A USB host device is a device that controls and manages the USB bus. USB host devices are usually computers or other host devices, such as tablets, laptops, desktops, etc.

  • USB Slave: USB Slave refers to a device with USB slave device functionality. A USB slave device is a device controlled by a USB host. USB slave devices rely on USB host devices for data transfer and communication. USB slave devices can be various peripheral devices, such as keyboards, mice, USB flash drives, etc.

USB OTG（USB On-The-Go）

Refers to devices that support USB OTG functionality (plug and play). USB OTG allows devices to switch between master and slave devices, enabling them to communicate directly with other USB devices without the need for a traditional USB host device. For example, OTG can be used to directly connect a camera and a printer.

The RK3568 chip has two USB 2.0 host controllers. Each USB 2.0 host controller fully supports USB 2.0 functionality through an Enhanced Host Controller Interface (EHCI) host controller and an Open Host Controller Interface (OHCI) host controller, and each host controller has one USB port.

  • The OHCI host controller only supports full-speed and low-speed modes, and is used to connect full-speed devices and low-speed devices.

  • The EHCI host controller only supports high-speed mode, and is used to connect high-speed devices.

The OHCI host controller and the EHCI host controller share the same USB port. The EHCI host controller automatically selects the ownership of the USB port (OHCI or EHCI) based on the speed mode of the connected device.

  • When OHCI is selected as the host controller, the OHCI host controller will provide services for the connected device;
  • When EHCI is selected as the host controller, the EHCI host controller will provide services for the connected device.

USB Bus Architecture and Device Interaction Mechanism

USB Topology

USB adopts a tree topology, which is a master-slave structure, meaningall devices are connected to the host through hubs, forming a tree structure, as shown in the figure below. USB data communication can only occur between the host and devices; communication between devices is not possible.

Although USB hubs can expand the number of interfaces, their expansion capability is strictly limited. According to the USB2.0 protocol specification, the entire system supports a maximum of 7 levels of cascading expansion (counting from the root hub), and the total number of connected devices must not exceed 127 (including the hubs themselves, as address 0 has a special purpose). This pyramid-shaped topology design (as shown in the figure below) ensures interface expansion flexibility while avoiding signal attenuation and system overload issues through hierarchical limitations.

USB Device State Transition

**From the moment a USB device is not yet connected, to being fully recognized by the USB Host and ensuring its functions are normal,**the USB device will sequentially go through the following stages.

  1. Attached state: This is the stage where the device is physically connected to the USB host but has not yet been powered on. This stage is primarily guaranteed by hardware.
  2. Powered state: This is the second stage, corresponding to a device that is connected to the USB host and has just been powered on. This stage is primarily guaranteed by hardware.
  3. Default state: When a USB device is first connected to the host, it enters the default state. In the default state, the device waits for the host to send a reset signal.
  4. Address state: After receiving the reset signal, the device enters the address state. In this state, the host assigns a unique address to the device.
  5. Configured State (Configured): Once the device receives its address, it enters the configured state. In this state, the device can select its configuration descriptor, which determines how it communicates with the host.
  6. Suspend State (Suspend): When the USB device is inactive or the host enters a low-power mode, the device can enter the suspend state. In the suspend state, the device consumes no or very little power.

USB Device Hardware Identification

How does the USB HOST detect the insertion of a USB device?

USB 2.0 is backward compatible with USB 1.0 and USB 1.1, and is divided into three modes: Low-speed, Full-speed, and High-speed.

  • Full-speed Device

Full-speed devices are connected in the following way, as shown in the figure below. The left side is the USB host side, and the right side is the USB device side.

As shown in the figure above, on the left hub side, the data lines D+ and D- both have 15k pull-down resistors Rpd (Pull-down Resistors), and on the right device side, there is a 1.5k pull-up resistor Rpu (Pull-up Resistors) on the data line D+.

When the D+ signal line changes from low level to high level, the USB host side can determine that a full-speed device has been inserted.

  • Low-speed Device

Low-speed devices are connected in the following way, as shown in the figure below. The left side is the USB host side, and the right side is the USB device side.

As shown in the figure above, on the left hub side, the data lines D+ and D- both have 15k pull-down resistors Rpd (Pull-down Resistors), and on the right device side, there is a 1.5k pull-up resistor Rpu (Pull-up Resistors) on the data line D-.

When the D- signal line changes from low level to high level, the USB host side can determine that a low-speed device has been inserted.

  • High-speed Device

When a high-speed device is connected to the system, it is initially recognized as a full-speed device. Subsequently, the HOST will detect the DEVICE to determine if it is a high-speed device. During this process, the HOST and DEVICE need to confirm with each other. After the confirmation is complete, the system will switch to high-speed mode. In high-speed mode, current transmission mode is used, and the pull-up resistor on D+ needs to be disconnected at this time.

When the device is disconnected, its differential termination resistors disappear immediately, but high-speed data packets will continue to be transmitted from the port originally connected to the device. When these data packets reach the unloaded path endpoint, strong reflections occur. The reflected signal returns to the hub interface, causing the differential voltage at the hub’s connection port to rise. When the differential signal level on D+ and D- of the high-speed device is greater than 625mV, it can be determined that the USB device has been disconnected.

After the USB host and USB device are connected, on the USB host and device’sSSRX+/-an equivalent pull-down resistor is generatedR_Term, with a range of 18~30 ohms, formed bySSRX+/-a 50-ohm equivalent pull-down resistor on each side connected in parallel.

  • Configuration Descriptor
  • Interface Descriptor

The USB communication data format takes the packet as the basic unit, achieving reliable communication between the device and the host through a hierarchical structure (field -> packet -> transaction -> transfer).

Packet

USB is like a conveyor belt for express delivery that can only queue up (serial communication), where data (fields) must be queued up and sent bit by bit.

To prevent data from getting lost, USB first packs the data into small parcels (data packets), and then distinguishes their purposes through four differently shaped “express boxes” (packet structures)—such as those for files, commands, data, etc.—ensuring each parcel reaches its destination accurately.

These four differently shaped “express boxes” correspond to four different packet structures, which areToken packet、Data packet、Handshake packetandSpecial packet。

A complete data packet is composed of multiple different fields.

All packets start with a Synchronization field (SYNC), followed by a Packet Identifier (PID), and end with an End of Packet (EOP) signal. Different types of packets have different bit fields in the middle. Common ones include Address (ADDR), Endpoint (ENDP), Data, Frame Index, and CRC. The specific structure of each packet needs to be analyzed based on the actual situation.

Transaction

The basic unit of USB transmission is the Packet, and the type of packet is represented by the PID. A single packet cannot transmit complete data. For example, to output data, an OUT token packet can be sent because the OUT token packet can specify the destination. However, an OUT token packet alone is not enough; data transmission also requires sending a DATA0 or DATA1 data packet.

After the device receives the data, it also needs to reply with an ACK handshake packet. Therefore, complete data transmission involves multiple packets, including token packets, data packets, and handshake packets. This complete data transmission process is called a Transaction. There are different types of transactions; some require handshake packets, while others do not; some can transmit large amounts of data, while others can only transmit small amounts of data.

There are four types of transactions, which are:

  • Bulk Transaction

Bulk transactions are used to transmit large amounts of data; data correctness is guaranteed, but timeliness is not.

  • Interrupt Transaction

Interrupt transactions are used to transmit periodic, small amounts of data; both data correctness and timeliness are guaranteed.

  • Real-time Transaction

Real-time transactions are used to transmit real-time data; data correctness is not guaranteed, but timeliness is.

  • Setup Transaction

Setup transactions are similar to bulk transactions, except that the token packet is a SETUP token packet.

USB Transfer Types

  1. Control TransferIt is the most basic data transfer type in the USB protocol, mainly used for querying, configuring, and sending general commands to USB devices. Control transfer is bidirectional, and the data volume is usually small. Control transfers consist of setup transactions and bulk transactions, and all USB devices must support control transfers.
  2. Isochronous TransferUsed to transmit large amounts of constant-rate data that require a specific service interval. Generally, isochronous transfer is commonly used for audio and video devices because these devices require high real-time performance. Isochronous transfer uses real-time transactions to achieve data transmission.
  3. Interrupt TransferSuitable for transmitting small or medium amounts of data that require a fixed transaction processing period. Generally, USB interrupt transfer is often applied to HID human interface devices such as USB mice and USB keyboards, because these devices require fast response and have a fixed transaction processing period, but have low data demand, which is exactly the advantage of USB interrupt transfer. Interrupt transfer uses interrupt transactions to achieve data transmission.
  4. Bulk TransferAlso known as USB block transfer. Bulk transfer is suitable for transmitting large amounts of data, requiring correct transmission, but having no requirements for transmission time, transmission rate, or real-time performance. Generally, bulk transfer is used for storage devices such as USB flash drives. Bulk transfer uses bulk transactions to achieve data transmission.

Bits make up Fields, Fields make up Packets, Packets make up Transactions, and Transactions make up Transfers.

USB Enumeration

After a USB device is connected to a USB host, the host uses the bus enumeration process to identify and manage the connected device.USB enumeration is essentially an information exchange process between the USB Host and Device. During this process, the Device reports its own parameters to the Host, and the Host obtains key information based on these parameters, thereby clarifying the specific type of the device and the way to communicate with it.

Based on the obtained information, the host can load the driver suitable for the device. The USB device enumeration process is described as follows:

  1. After the USB device is inserted into the USB port, the host detectsD+/D-the voltage on the line, confirms that a device is connected, and the USB hub reports to the host via the interrupt IN channel that a USB device is connected.
  2. After the host detects a USB device insertion, it resets the USB device. After reset, the USB device’s address is 0, so the host can communicate with the USB device using address 0.
  3. The host sends a standard request to get the device descriptor to device endpoint 0 (the default endpoint) at address 0 (i.e., the newly inserted USB device). After receiving the request, the USB device sends the device descriptor to the host. After receiving the device descriptor, the host returns a zero-length data acknowledgment (ACK).
  4. The host resets the device again. After reset, the host sends a Set Address request packet to device endpoint 0 at address 0, and the new device address is in this packet. The host sends a request status return, and the device returns a zero-length status packet.
  5. After receiving the status packet, the host sends an ACK packet to the device. After receiving the ACK, the device enables the new device address, and thereafter the host accesses the device through the new address.
  6. The host gets the device descriptor again. This time it’s a bit different from the first time; this time it needs to get the full 18-byte device descriptor.
  7. Next, the host will get the configuration descriptor. After obtaining the configuration descriptor, based on the total length of the configuration set inside, it then gets the configuration set. The configuration set includes configuration descriptor, interface descriptor, endpoint descriptor, etc. If there are string descriptors, the system will also get the string descriptors.

Introduction to libusb library

libusb is a library written in C that provides a general method for accessing USB devices. Through it, applications can conveniently access USB devices without needing to write USB device drivers. The libusb library has three characteristics:

  1. Good portability, supporting Linux, macOS, and Windows systems.
  2. Simple and easy to use; applications do not need privileged mode, nor do they need to elevate their own permissions to access USB devices.
  3. Supports all USB protocols, from USB 1.0 to USB 3.1, and the API interface remains unchanged, facilitating development.

USB Driver Development Framework

In the Linux kernel, USB driver development is mainly divided into two categories:

  • Host-side driver
  • and Device-side driver.

The host-side driver is responsible for managing USB devices plugged into the host; the device-side driver is responsible for how the device communicates with the host as a USB device.

Since the term “USB devices drivers” is prone to confusion, people typically use “usb gadget driver (USB device driver)” to describe USB device drivers.

At the hardware level, the USB controllers on the host side and device side are called Host Controller and USB Device Controller (UDC), respectively, as shown in the figure below

  • In the Linux system, USB drivers can be viewed from two perspectives: the host side and the device side. From the host side, the USB host controller driver is at the bottom of the USB driver stack, responsible for controlling the hardware. Above it runs the USB core layer, and above that is the USB device driver.

  • From the device side, the USB device-side driver program has 3 layers: the UDC driver, the Gadget Function API, and the Gadget Function driver.

The USB driver architecture is shown in the figure below

The Linux USB protocol stack is a layered architecture. On the left is the USB Device driver, on the right is the USB Host driver, the top layer is the application layer, the bottom layer is the driver for different USB controllers and PHYs of Rockchip series chips, and the middle is the USB core layer.

USB Request Block

URB, which stands for USB Request Block, is the basic carrier used to describe communication with USB devices in USB device drivers. It is also the core data structure, similar to that in network device driverssk_buffstructure. The URB structure is shown as follows:

Each endpoint in a USB device handles a URB queue. The USB communication steps are described as follows:

USB Communication Steps

Step 1: Useusb_alloc_urbfunction to create the URB structure,usb_alloc_urbThe function is introduced as follows:

usb_alloc_urbThe function is used to create a USB Request Block (URB) for managing data transfer requests of USB devices. URB is the core structure for data transfer in USB driver development, supporting the following transfer types:

  • Control Transfer: Device initialization, configuration.
  • Interrupt Transfer: Devices with high real-time requirements (such as keyboards, mice).
  • Bulk Transfer: Large-capacity data transfer (such as USB drives).
  • Isochronous Transfer: Real-time streaming data (such as cameras, audio devices).

Function parameters:

  • int iso_packets: Number of packets for isochronous transfer (only used for isochronous transfer, set to 0 for other transfer types)
  • gfp_t mem_flags: Memory allocation flags, specifying the memory allocation method (such asGFP_KERNEL、GFP_ATOMIC）

Returns the allocated URB pointer on success, and NULL on failure.

Step 2: URB initialization, designated for a specific endpoint of a specific USB device. For interrupt URB, useusb_fill_int_urbfunction to initialize the URB,usb_fill_int_urbThe function is introduced as follows:

Function parameters:

  • struct usb_device *dev: Pointer to the USB device to which this URB is to be sent
  • unsigned int pipe: The specific endpoint of the USB device to which this URB is to be sent

  • int buffer_length: Istransfer_bufferThe size of the buffer pointed to by the pointer
