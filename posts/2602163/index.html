USB Basics

USB (Universal Serial Bus) It is a widely used general interface standard between computers and electronic devices, capable of transmitting data and supplying power. USB is a standard.

  1. Concept Proposal and Initial Standard Formulation (1994 - 1996)
    In 1994, companies such as Compaq, IBM, and Intel jointly developed USB technology to simplify the connection between computers and peripherals. In 1996, the USB 1.0 standard was released, supporting transfer rates of 1.5Mbps (Low Speed) and 12Mbps (Full Speed), meeting the needs of low-speed devices like mice and keyboards, and also implementing hot-swapping, which was convenient for users.
  2. Technical Improvement and Popularization (1998 - 2000)
    In 1998, USB 1.1 optimized electrical characteristics and compatibility, improving reliability and driving the popularization of devices. In 2000, USB 2.0 was a major breakthrough, introducing a 480Mbps High Speed transfer mode to meet the needs of high-speed devices like digital cameras and external hard drives, becoming the mainstream interface standard.
  3. Rapid Development and Performance Improvement (2008 - 2013)
    In 2008, USB 3.0 (later renamed USB 3.1 Gen 1) was introduced, increasing the speed to 5Gbps, optimizing power management and signal transmission, supporting high-power device charging, and meeting the needs of high-definition video transmission. In 2013, USB 3.1 Gen 2 was released, reaching a speed of 10Gbps, making it possible for high-performance external SSDs and other devices.
  4. Continuous Evolution and Diversification (2014 - Present)
    In 2014, the USB Type-C connector emerged. It is small, reversible, capable of high-power delivery (up to 100W), and high-speed data transmission, gradually becoming mainstream. In 2017, USB 3.2 defined different speed identifiers, enriching speed options. In 2019, USB4, based on Thunderbolt 3 technology, reached speeds of up to 40Gbps, enhancing features like multi-monitor support and expanding application scenarios.

USB continues to evolve, continuously improving in data transmission, device compatibility, and power delivery capabilities, becoming an indispensable connection method for modern electronic devices. There are different types of USB interfaces, and interface standards can be referenced on the official website:

USB Interface Classification

USB interfaces can be mainly divided into three major categories based on physical appearance: Type-A, Type-B, and Type-C. Among them, Type-B has also derived specifications such as Mini-B, Micro-USB, and Micro-B SuperSpeed. A detailed introduction will follow.

  1. The Type-A interface, as shown in the figure below, is a rectangular interface widely used in host devices such as computers and chargers, as well as traditional peripherals like USB flash drives, mice, and keyboards.

  1. The Type-B interface, as shown in the figure below, is a square interface, mostly used for devices such as printers, microphones, and scanners.

  1. The Mini-B interface, as shown in the figure below, is mostly used for dashcam devices.

  1. The Micro-USB interface, as shown in the figure below, is a small interface that was previously used in portable devices such as older mobile phones and power banks.

  1. The Micro-B SuperSpeed interface, as shown in the figure below, is commonly used for external hard drives.

  1. The Type-C interface, as shown in the figure below, is reversible. Currently, the Type-C interface is the mainstream interface for modern devices such as mobile phones, laptops, and headphones.

Summary:

Although the different types of USB interfaces in the figure differ in appearance and purpose, they all follow the USB standard protocol, ensuring compatibility and interoperability between devices.

It is important to note that you cannot assume a device has high-speed data transfer capabilities just because it uses a Type-C port. Type-C is merely a physical form factor; the actual data transfer speed depends on the USB protocol version it supports. For example, if it uses the USB 2.0 protocol, even with a Type-C port, the speed is relatively limited, whereas USB 3.0 and higher protocol versions are required to achieve high-speed transfer.

USB Version

Initially, USB 3.0 was named SuperSpeed USB, emphasizing its high-speed transfer characteristics. Later, the USB 3.1 version was released, further improving its transfer rate.

However, to simplify the naming:

  • USB 3.0 was renamed USB 3.1 Gen 1
  • USB 3.1 was renamed USB 3.1 Gen 2.

USB-IF introduced the latest USB naming convention:

  • USB 3.1 Gen 1 was further renamed USB 3.2 Gen 1
  • USB 3.1 Gen 2 was further renamed USB 3.2 Gen 2
  • USB 3.2 Gen 2x2 is USB 3.2

Having undergone two name changes:

  • USB 3.0 is named USB 3.2 Gen 1
  • USB 3.1 is named USB 3.2 Gen 2
  • USB 3.2 is named USB 3.2 Gen 2x2

Generally, the USB version can be inferred from the color of the USB port.

  • Black/White: USB 1.0/2.0
  • Dark Blue: USB 3.0
  • Light Blue: USB 2.0
  • Red: USB 3.1

USB devices are divided into Host (master device) and Slave (slave device); data transmission can only be achieved when a Host is connected to a Slave.

  • USB Host: USB Host refers to a device with USB host functionality. A USB host device is a device that controls and manages the USB bus. USB host devices are usually computers or other host devices, such as tablets, laptops, desktops, etc.

  • USB Slave: USB Slave refers to a device with USB slave functionality. A USB slave device is a device controlled by a USB host. USB slave devices rely on USB host devices for data transmission and communication. USB slave devices can be various peripheral devices, such as keyboards, mice, USB flash drives, etc.

USB OTG（USB On-The-Go）

Refers to devices that support USB OTG functionality (plug and play). USB OTG allows devices to switch between host and slave, enabling direct communication with other USB devices without the need for a traditional USB host device. For example, OTG can be used to directly connect a camera and a printer.

The RK3568 chip has two USB 2.0 host controllers. Each USB 2.0 host controller fully supports USB 2.0 functionality through an Enhanced Host Controller Interface (EHCI) host controller and an Open Host Controller Interface (OHCI) host controller, and each host controller has a USB port.

  • The OHCI host controller only supports full-speed and low-speed modes, used to connect full-speed and low-speed devices.

  • The EHCI host controller only supports high-speed mode, used to connect high-speed devices.

The OHCI host controller and the EHCI host controller share the same USB port. The EHCI host controller automatically selects the control ownership of the USB port (OHCI or EHCI) based on the speed mode of the connected device.

  • When OHCI is selected as the host controller, the OHCI host controller will provide services for the connected devices;
  • When EHCI is selected as the host controller, the EHCI host controller will provide services for the connected devices.

USB Bus Architecture and Device Interaction Mechanism

USB Topology

USB adopts a tree topology, which is a master-slave structure, namelyAll devices are connected to the host through a hub, forming a tree structure, as shown in the figure below. USB can only perform data communication between the host and devices; communication between devices is not allowed

Although USB hubs can expand the number of interfaces, their expansion capability is strictly limited. According to the USB 2.0 protocol specification, the entire system supports a maximum of 7 levels of cascading expansion (counting from the root hub), and the total number of connected devices must not exceed 127 (including the hubs themselves, as address 0 has a special purpose). This pyramid-shaped topological structure design (as shown in the figure below) ensures interface expansion flexibility while avoiding signal attenuation and system overload issues through level restrictions.

USB Device State Transition

From when the USB device is not yet connected, to when it is fully recognized by the USB Host and its functionality is ensured to be normal, the USB device will go through the following stages in sequence.

  1. Attached State: This is the stage where the device is physically connected to the USB host but has not yet been powered on. This stage is mainly guaranteed by hardware.
  2. Powered State: This is the second stage, corresponding to a device that is connected to the USB host and has just been powered on. This stage is mainly guaranteed by hardware.
  3. Default State: When a USB device is first connected to the host, it enters the default state, in which the device waits for the host to send a reset signal.
  4. Address State (Address): After receiving the reset signal, the device enters the Address State. In this state, the host assigns a unique address to the device.
  5. Configured State (Configured): Once the device receives its address, it enters the Configured State. In this state, the device can select its configuration descriptor, which determines how it communicates with the host.
  6. Suspend State (Suspend): When the USB device is inactive or the host enters a low-power mode, the device can enter the Suspend State. In the Suspend State, the device consumes no or very little power.

USB Device Hardware Identification

How does the USB HOST detect the insertion of a USB device?

USB 2.0 is backward compatible with USB 1.0 and USB 1.1, and is divided into three modes: Low-speed, Full-speed, and High-speed.

  • Full-speed Device

Full-speed devices are connected in the following way, as shown in the figure below. The left side is the USB host side, and the right side is the USB device side.

As shown in the figure above, on the left hub side, the data lines D+ and D- both have 15k pull-down resistors Rpd (Pull-down Resistors), and on the right device side, the data line D+ has a 1.5k pull-up resistor Rpu (Pull-up Resistors).

When the D+ signal line changes from low level to high level, the USB host side can determine that a full-speed device has been inserted.

  • Low-speed Device

Low-speed devices are connected in the following way, as shown in the figure below. The left side is the USB host side, and the right side is the USB device side.

As shown in the figure above, on the left hub side, the data lines D+ and D- both have 15k pull-down resistors Rpd (Pull-down Resistors), and on the right device side, there is a 1.5k pull-up resistor Rpu (Pull-up Resistors) on the data line D-.

When the D- signal line changes from low level to high level, the USB host can determine that a low-speed device has been connected.

  • High-speed device

When a high-speed device is connected to the system, it is initially recognized as a full-speed device. Subsequently, the HOST will detect the DEVICE to determine if it is a high-speed device. During this process, the HOST and DEVICE need to confirm with each other. After the confirmation is completed, the system will switch to high-speed mode. In high-speed mode, the current transmission mode is used, and the pull-up resistor on D+ needs to be disconnected at this time.

When the device is disconnected, its differential termination resistor disappears immediately, but high-speed data packets will continue to be transmitted from the port originally connected to the device. When these data packets reach the unloaded path endpoint, strong reflections occur, and the reflected signal returns to the hub interface, causing the differential voltage at the hub’s connection port to rise. When the differential signal level on D+ and D- of the high-speed device is greater than 625mV, it can be determined that the USB device has been disconnected.

When the USB host and USB device are connected, an equivalent pull-down resistor is generated on the USB host and the device’sSSRX+/-generating an equivalent pull-down resistorR_Term, with a range of 18~30 ohms, formed bySSRX+/-each having a 50-ohm equivalent pull-down resistor in parallel.

USB protocol

USB descriptor

A USB descriptor is a structure that describes USB device information. The host reads these descriptors to identify device type, function, configuration, and other information, thereby correctly loading drivers and communicating. The main USB descriptors are:

  • Device Descriptor
  • Configuration Descriptor
  • Interface Descriptor
  • Endpoint Descriptor

Device Descriptor

The device descriptor contains basic information about the device, such as the device’s vendor ID, product ID, etc. The device descriptor is the first information requested and returned when the device is connected to the host, providing the basic characteristics of the device. The USB communication data format takes the packet as the basic unit, achieving reliable communication between the device and the host through a hierarchical structure (field -> packet -> transaction -> transfer).

Packet

USB is like a conveyor belt for express delivery that can only queue up (serial communication), where data (fields) must line up and be sent forward bit by bit.

To prevent data from getting lost, USB first packs the data into small parcels (data packets), and then uses four differently shaped “express boxes” (packet structures) to distinguish their purposes—such as for files, commands, data, etc.—ensuring each parcel reaches its destination accurately.

These four differently shaped “express boxes” correspond to four different packet structures, which areToken packet、Data packet、Handshake packetandSpecial packet。

A complete data packet consists of multiple different fields.

All data packets start with a synchronization field (SYNC), followed by a packet identifier (PID), and end with an end-of-packet (EOP) signal. Different types of data packets have different bit fields in the middle, common ones include packet target address (ADDR), packet target endpoint (ENDP), data, frame index, and CRC, etc. The specific structure of each data packet needs to be analyzed based on the actual situation.

Transaction

The basic unit of USB transmission is the packet, and the type of packet is represented by the PID. A single packet cannot transmit complete data. For example, to output data, an OUT token packet can be sent because the OUT token packet can specify the destination. However, an OUT token packet alone is not enough; data transmission also requires sending a DATA0 or DATA1 data packet.

After the device receives the data, it also needs to reply with an ACK handshake packet. Therefore, complete data transmission involves multiple packets, including token packets, data packets, and handshake packets. This complete data transmission process is called a transaction. There are different types of transactions; some require handshake packets, while others do not; some can transmit large amounts of data, while others can only transmit small amounts of data.

There are four types of transactions, which are:

  • Bulk transaction

Bulk transactions are used to transmit large amounts of data; the correctness of the data is guaranteed, but the timeliness is not.

  • Interrupt transaction

Interrupt transactions are used to transmit periodic, small amounts of data; both the correctness and timeliness of the data are guaranteed.

  • Isochronous transaction

Real-time transactions are used to transmit real-time data, where the correctness of the data is not guaranteed, but the timeliness is guaranteed.

  • Setup Transaction

The setup transaction is similar to the bulk transaction, except that the token packet is a SETUP token packet.

USB Transfer Types

  1. Control TransferIt is the most basic data transfer method in the USB protocol, mainly used for querying, configuration, and sending general commands to USB devices. Control transfer is bidirectional, and the data volume is usually small. Control transfer consists of setup transactions and bulk transactions, and all USB devices must support control transfer.
  2. Isochronous TransferUsed to transmit large amounts of data with a constant rate and requirements for the service period. Generally speaking, isochronous transfer is often used for audio and video devices because these devices require high real-time performance. Isochronous transfer uses real-time transactions to achieve data transmission.
  3. Interrupt TransferSuitable for transmitting small or medium amounts of data, requiring a fixed transaction processing period. Generally speaking, USB interrupt transfer is often used in HID human-computer interaction devices such as USB mice and USB keyboards, because these devices require fast response, have a fixed transaction processing period, but have low data requirements, which is exactly the advantage of USB interrupt transfer. Interrupt transfer uses interrupt transactions to achieve data transmission.
  4. Bulk TransferAlso known as USB block transfer. Bulk transfer is suitable for transmitting large amounts of data, requiring correct transmission, but having no requirements for transmission time, transmission rate, or real-time performance. Generally speaking, bulk transfer is used for storage devices such as USB flash drives. Bulk transfer uses bulk transactions to achieve data transmission.

Bits form Fields, Fields form Packets, Packets form Transactions, and Transactions form Transfers.

USB Enumeration

After a USB device is connected to a USB host, the host uses the bus enumeration process to identify and manage the connected device.**USB enumeration is essentially the process of information exchange between the USB Host and Device.**During this process, the Device reports its own parameters to the Host. Based on these parameters, the Host obtains key information to determine the specific type of the device and how to communicate with it.

Based on the obtained information, the host can then load the driver suitable for the device. The USB device enumeration process is described as follows:

  1. After a USB device is inserted into a USB port, the host detects theD+/D-voltage on the line, confirming that a device is connected. The USB hub reports the connection of the USB device to the host through the interrupt IN channel.
  2. After the host detects the insertion of the USB device, it resets the USB device. After the reset, the address of the USB device is 0, so the host can use address 0 to communicate with the USB device.
  3. The host sends a standard request to get the device descriptor to device endpoint 0 (the default endpoint) at address 0 (i.e., the just-inserted USB device). After receiving the request, the USB device sends the device descriptor to the host. After receiving the device descriptor, the host returns a zero-length data acknowledgment (ACK).
  4. The host resets the device again. After the reset, the host sends a Set Address request packet to device endpoint 0 at address 0, and the new device address is in this packet. The host sends a request for status return, and the device returns a zero-length status packet.
  5. After receiving the status packet, the host sends an ACK packet to the device. After receiving the ACK, the device enables the new device address, and from then on, the host accesses the device through the new address.
  6. The host gets the device descriptor again. This time it is a little different from the first time; this time it needs to get the full 18-byte device descriptor.
  7. Next, the host will get the configuration descriptor. After obtaining the configuration descriptor, the host gets the configuration set based on the total length of the configuration set inside. The configuration set includes configuration descriptors, interface descriptors, endpoint descriptors, etc. If there are string descriptors, the system will also get the string descriptors.

Introduction to the libusb library

libusb is a library written in C that provides a general method for accessing USB devices. Through it, applications can easily access USB devices without needing to write USB device drivers. The libusb library has three characteristics:

  1. Good portability, supporting Linux, macOS, and Windows systems.
  2. Simple and easy to use, applications do not need privileged mode or elevated permissions to access USB devices
  3. Supports all USB protocols, from USB 1.0 to USB 3.1, and the API interface remains unchanged, facilitating development

USB driver development framework

In the Linux kernel, USB driver development is mainly divided into two categories:

  • Host-side driver
  • and device-side driver.

The host-side driver is responsible for managing USB devices plugged into the host; the device-side driver is responsible for how the device communicates with the host as a USB device.

Since the term “USB devices drivers” is easily confused, people usually use “usb gadget driver” to describe USB device drivers.

At the hardware level, the USB controllers on the host side and device side are called Host Controller and USB Device Controller (UDC) respectively, as shown in the figure below

  • In the Linux system, USB drivers can be observed from two perspectives: the host side and the device side. From the host side, the USB host controller driver is at the lowest level of the USB driver stack, responsible for controlling the hardware. Running above it is the USB core layer, and above that is the USB device driver.

  • From the device side, the USB device-side driver program has 3 layers: the UDC driver, the Gadget Function API, and the Gadget Function driver.

The USB driver architecture is shown in the figure below

The Linux USB protocol stack is a layered architecture. On the left is the USB Device driver, on the right is the USB Host driver, the top layer is the application layer, the bottom layer is the driver for different USB controllers and PHYs of Rockchip series chips, and the middle is the USB core layer.

URB, namely USB Request Block, is the basic carrier used to describe communication with USB devices in USB device drivers. It is also a core data structure, similar to that in network device drivers,sk_buffstructure. The URB structure is as follows:

Each endpoint in a USB device processes a URB queue. The USB communication steps are as follows:

USB Communication Steps

The first step uses theusb_alloc_urbfunction to create the URB structure,usb_alloc_urbThe function is described as follows:

usb_alloc_urbThe function is used to create a USB Request Block (URB) for managing data transfer requests of USB devices. URB is the core structure for data transfer in USB driver development, supporting the following transfer types:

  • Control Transfer: Device initialization and configuration.
  • Interrupt Transfer: Devices with high real-time requirements (e.g., keyboards and mice).
  • Bulk Transfer: Large capacity data transfer (e.g., USB drive).
  • Isochronous Transfer: Real-time streaming data (e.g., cameras, audio devices).

Function parameters:

  • int iso_packets: Number of packets for isochronous transfer (only used for isochronous transfer, set to 0 for other transfer types)
  • gfp_t mem_flags: Memory allocation flag, specifying the memory allocation method (e.g.,GFP_KERNEL、GFP_ATOMIC）

Returns the allocated URB pointer on success, NULL on failure.
