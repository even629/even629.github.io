    Docker Image.

    What is an image?

    Docker images are actually composed of layers of file systems; this layered file system is UnionFS.

    UnionFS (Union File System).

    Union File System (UnionFS) is a layered, lightweight, and high-performance file system that supports modifications to the file system as a commit to be stacked layer by layer, while also being able to mount different directories under the same virtual filesystem (unite several directories into a single virtual filesystem). Union File System is the foundation of Docker images. Images can be inherited through layering; based on a base image (with no parent image), various specific application images can be created.

    Docker Image Loading Principle.

    • bootfs (boot file system) mainly contains the bootloader and kernel.
      The bootloader mainly loads the kernel. When Linux just starts, it loads the bootfs file system. The bottom layer of a Docker image is the boot file system bootfs. This layer is the same as our typical Linux/Unix file system, containing the boot loader and kernel. After the boot loading is complete, the entire kernel is in memory, at which point the right to use memory has been transferred from bootfs to the kernel, and the system will also unload bootfs.
    • rootfs(root file system)
      Above bootfs. It contains standard directories and files in a typical Linux system, such as /dev, /proc, /bin, /etc, etc. rootfs is the distribution of various operating systems, such as Ubuntu, CentOS, etc.

    For a streamlined OS, rootfs can be very small, only needing to include the most basic commands, tools, and program libraries, because the underlying layer directly uses the Host’s kernel, and it only needs to provide the rootfs itself. It can be seen that for different Linux distributions, bootfs is basically the same, while rootfs will differ, so different distributions can share bootfs.

    One of the biggest benefits of image layering is sharing resources, facilitating replication and migration, which is for reuse.

    For example, if multiple images are built from the same base image, then the Docker Host only needs to save one base image on the disk; at the same time, it only needs to load one base image in memory to serve all containers. Moreover, each layer of the image can be shared.

    Key point to understand

    Docker’s image layers are read-only, while the container layer is writable.
    When a container starts, a new writable layer is loaded on top of the image. This layer is usually called the “container layer”, and everything below the container layer is called the image layer.

    Docker container data volumes

    If you encounter “cannot open directory: Permission denied” when accessing a mounted host directory in Docker, just add a --privileged=true parameter after the mounted directory.
    Using this parameter, the root inside the container has true root permissions; otherwise, the root inside the container is just an ordinary user permission externally.

    By default, the repository is created in the /var/lib/registry directory of the container. It is recommended to map it with a container volume yourself to facilitate joint debugging with the host machine.

    A volume is a directory or file that exists in one or more containers, mounted to the container by docker, but does not belong to the Union File System, so it can bypass the Union File System to provide some features for persistent storage or shared data for users.

    The design purpose of volumes isData persistence, completely independent of the container’s lifecycle, so Docker will not delete its mounted data volumes when the container is deleted.

    Note: Changes to container volumes will not be included in image updates, and the lifecycle of data volumes continues until no container uses them.

    Docker File analysis

    What is Docker File

    Dockerfile is a text file used to build Docker images, consisting of a script of instructions and parameters required to build the image.

    Basic knowledge of Docker File content

    1. Each reserved word instruction must be in uppercase letters and must be followed by at least one parameter
    2. Instructions are executed in order from top to bottom
    3. # represents a comment
    4. Each instruction creates a new image layer and commits the image

    The general process of Docker executing a Dockerfile

    1. docker runs a container from the base image
    2. Execute an instruction and modify the container
    3. Execute an operation similar to docker commit to submit a new image layer
    4. docker then runs a new container based on the newly submitted image
    5. Execute the next instruction in the dockerfile until all instructions are completed

    Docker File reserved words

    FROM

    Base image, which image the current image is based on, specify an existing image as a template, the first line must be from

    MAINTAINER

    Name and email address of the image maintainer

    RUN

    Commands to run when building the container
    Two formats: shell format and exec format
    RUN runs during docker build

    EXPOSE

    Ports exposed by the current container to the outside

    WORKDIR

    Specifies the default working directory for the terminal after the container is created, a landing point

    USER

    Specifies the user under which the image runs; if not specified, it defaults to root

    ENV

    Used to set environment variables during the image build process

    ADD

    Copies files from the host directory into the image and automatically handles URLs and decompresses tar archives

    COPY

    Similar to ADD, copies files and directories into the image. Copies files/directories from in the build context directory to in the new layer of the image
    COPY src dest
    COPY [“src”,“dest”]

    VOLUME

    Container data volume, used for data saving and persistence

    The difference between VOLUME andrun -v, and when to use VOLUME

    During container runtime, write operations to the container storage layer should be minimized as much as possible. For applications like databases that need to save dynamic data, their database files should be stored in a volume. To prevent users from forgetting to mount the directory where dynamic files are saved as a volume during runtime, we can specify in advance in the Dockerfile that certain directories are mounted as anonymous volumes. This way, if the user does not specify a mount during runtime, the application can still run normally and will not write a large amount of data to the container storage layer.

    So, in actual use, is the VOLUME instruction in a Dockerfile the same as the-vparameter indocker run, which binds a host directory to a container directory to achieve directory sharing?

    Not exactly, actuallyThe VOLUME instruction only declares a directory in the container as an anonymous volume, but does not have the function of binding the anonymous volume to a specified directory on the host。

    When an anonymous volume is declared in the Dockerfile of the generated image, and we run a container with this image, Docker will generate a directory under a specific directory in the installation directory to bind the container’s anonymous volume (this specific directory varies across different versions of Docker). My current directory is: /var/lib/docker/volumes/{container ID}.

    SummaryVOLUME just specifies a directory to ensure the container runs normally even if the user forgets to specify the-vparameter at startup. For example, with MySQL, you can’t have a situation where the user doesn’t specify-vat startup, then deletes the container, and all the MySQL data files are deleted; that would cause a major accident in production. Therefore, MySQL’s Dockerfile needs to configure VOLUME, so that even if the user doesn’t specify-v, the data files won’t be lost after the container is deleted. They can still be recovered.

    Will the data files at the location specified by VOLUME be deleted after the container is deleted?

    Just like the-vinstruction, the files mapped on the host will not be deleted after the container is deleted.

    What happens if-vand VOLUME specify different locations?

    The directory set by-vwill take precedence. In fact, the purpose of the VOLUME instruction is to prevent data loss when users forget to specify-v. So if the user specifies-v, naturally the location specified by VOLUME is not needed.

    In fact, general Dockerfiles do not need to specify VOLUME if they are not for applications like databases that need to persist data to disk. Specifying VOLUME is only to prevent all data from residing in the container when the user forgets to specify-v, which would result in all data being lost once the container is deleted.

    **So why doesn’t the Dockerfile provide an instruction that can map a host directory to a container directory?**In fact, this design makes sense. If a host directory is specified in the Dockerfile, the Dockerfile would lose its portability. After all, the directories that different people need to map may vary. Therefore, the best approach is to give this right to everyone who runs this Dockerfile, which is why there is therun -v host_directory:container_directoryinstruction.

    Docker Network

    After starting docker, a virtual bridge named docker0 is generated

    The 3 major network modes created by default in docker:

    • bridge
    • host
    • none

    The 5 major network modes of docker:

    • bridge
      Assign, set IP, etc. for each container, and connect the container to a docker0
      Virtual bridge, this mode is the default
      Specify using --network bridge, docker0 is used by default
    • host
      The container will not virtualize its own network card, configure its own IP, etc., but will use the host’s IP and ports
      Specify using --network host
    • none
      The container has an independent Network namespace, but no network settings are made to it, such as assigning veth pairs and bridge connections, IP, etc.
      Specify using --network none
    • container
      The newly created container will not create its own network card and configure its own IP, but will share the IP and port range with a specified container
      Specify using --network container:NAME or container ID
    • …

    What can Docker networking do:

    • Interconnection and communication between containers, as well as port mapping
    • When the container IP changes, network communication can be done directly through the service name without being affected

    The bridge docker0 creates a pair of peer virtual device interfaces, one called veth and the other called eth0, matched in pairs

    In host mode, the same network segment as the host is used

    If using host mode, the port number will be based on the host port number, and will increment when duplicated

    none mode:
    In this mode, no network configuration is performed for the Docker container
    In other words, this Docker container has no network card, IP, routing, etc., only a lo
    We need to add a network card, configure IP, etc. for the Docker container ourselves

    container mode

    It is equivalent to tomcat86 and tomcat85 sharing the same IP and the same port, causing port conflicts. Because tomcat needs port 8080, the container mode is not very suitable in this case

    docker-compose

    Compose is a tool software launched by Docker that can manage multiple Docker containers to form an application. It requires defining a YAML format configuration file**docker-compose.yml，**Write the calling relationships between multiple containers. Then, with just one command, you can start/stop all these containers simultaneously.

    Compose core concepts

    • One file

    docker-compose.yml

    • Two elements
    1. Service

    Individual application container instances, such as order microservice, inventory microservice, MySQL container, Nginx container, or Redis container

    1. Project

    A complete business unit composed of a group of related application containers, defined in the docker-compose.yml file

    3 steps to use Compose

    1. Write Dockerfiles to define each microservice application and build the corresponding image files
    2. Use docker-compose.yml to define a complete business unit and arrange the various container services in the overall application
    3. Finally, execute the docker-compose up command to start and run the entire application, completing one-click deployment

    Docker container monitoring with CAdvisor+InfluxDB+Grafana(CIG)

    Native command

    It is very convenient to see the CPU, memory, and network traffic data of all containers on the current host, but the docker stats statistics can only be for all containers on the current host, the data is real-time, there is no place to store it, and there is no health indicator threshold warning function

    CAdvisor monitoring collection + InfluxDB data storage + Grafana chart display

    1. CAdvisor

    CAdvisor is a container resource monitoring tool, including monitoring of container memory, CPU, network IO, disk IO, etc., and provides a WEB page for viewing the real-time running status of containers. CAdvisor stores 2 minutes of data by default, and is only for a single physical machine. However, CAdvisor provides many data integration interfaces, supporting integration with InfluxDB, Redis, Kafka, etc., and can send monitoring data to these databases for storage by adding corresponding configurations

    CAdvisor functions mainly have two points:

    • Display monitoring data at two levels: Host and container
    • Display historical change data
    1. InfluxDB

    InfluxDB is an open-source distributed time-series, event, and metrics database written in Go without external dependencies. CAdvisor itself already provides an integration method for InfluxDB; you just need to specify the configuration when starting the container.

    Main features of InfluxDB:

    • Based on time series, supports event-related functions (such as max, min, sum, etc.)
    • Measurability: You can compute large amounts of data in real-time
    • Event-based: It supports arbitrary event data
    1. Grafana

    Grafana is an open-source data analysis and visualization platform that supports multiple data source configurations (including InfluxDB, MYSQL, Elasticsearch, OpenTSDB, Graphite, etc.) and rich plugin and template features, supporting chart permission control and alerts.

    Main features of Grafana:

    • Flexible and rich graphical options
    • Can mix multiple styles
    • Supports day and night modes
    • Multiple data sources
