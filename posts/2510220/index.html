GIC Interrupt Controller

  • Traditional interrupt controllers, such as the legacy interrupt controller on Raspberry Pi 4B
    • Interrupt enable register
    • Interrupt disable register
    • Interrupt status register
  • The traditional way of managing interrupts using simple status registers is becoming increasingly difficult to manage
    • The number of interrupt sources is increasing
    • Different types of interrupts, such as inter-core interrupts, interrupt priorities, software-defined interrupts, etc.,

GIC has two main functional blocks：

  • Distributor: All interrupt sources in the system are connected to the distributor. The distributor provides registers to control the attributes of each interrupt, such as priority, status, security, routing information, and enable status. The distributor determines which interrupt to forward to the core through the attached CPU interface.
  • CPU Interface: The CPU receives interrupts through it. The CPU interface provides registers to mask, identify, and control the interrupt state forwarded to that core. Each core in the system has a separate CPU interface.

Interrupt types supported by GIC

  • SGI （Software Generated Interrupt), software-generated interrupt, software interrupt, used to send interrupt signals to other CPU cores

  • PPI （Private Peripheral Interrupt), private peripheral interrupt, this interrupt is exclusive to a specified CPU

  • SPI（Shared Peripheral Interrupt), shared peripheral interrupt, all CPUs can access this interrupt

  • LPI （Locality-specific Peripheral Interrupt), local specific peripheral interrupt,New interrupt types in GICv3。Message-based interrupt type

Interrupt trigger types

Each interrupt type is eitherEdge-triggeredorLevel-sensitive:

Controlled byGICD_ICFGRnregister

  • Edge-triggered: Considered valid when the GIC detects a rising edge on the relevant input, and remains valid until cleared
  • Level-sensitive: Valid only when the GIC’s relevant input is at a high level

Interrupt number

banked interrupt means that for SGI and PPI, although their IDs are the same throughout the system (e.g., Timer PPI is ID30in all cores), but they are private, each CPU has its own copy, independent of each other.

  • SGI (0–15): If a core triggers SGI0, it will not affect the SGI0 of other cores.
  • PPI (16–31): The Timer PPI of core 0 (e.g., ID30) and the Timer PPI of core 1 (ID30) are independent; they are not shared.

Interrupt numbers 1020~1023 are reserved

Interrupt priority

Each interrupt priority is set inGICD_IPRIORITYRnin the register

Width of the priority field

  • Each interrupt priority field is 8 bits
  • Number of priority levels actually supported by the implementation = 2ⁿ, n ∈ [4, 8] (i.e., 16 ~ 256 levels）。
  • If the hardware implementation is less than 8, for example, only implementing 5 bits, then writing to the lower 3 bits is RAZ/WI(read as 0, writes ignored).

Numeric value and priority

  • The smaller the value, the higher the priority
  • The maximum priority value is implementation-dependent
  • By default during initialization, all interrupts should be given a medium priority to avoid disrupting system scheduling, commonly 0xa0 or 0x80

Interrupt states

  • inactive (Inactive state):The interrupt is in an inactive state

  • pending (Pending state):The interrupt is in a valid state, but is waiting for the CPU to respond to it

  • active (Active state):The CPU has responded to the interrupt

  • active and pending (Active and pending state):The CPU is responding to the interrupt, but the interrupt source sends another interrupt

GICV2 interrupt controller

  • The Distributor registers (GICD_) containsinterrupt setup and configuration
  • The CPU Interface registers (GICC_) containsCPU-related special settings

The priority and target core for interrupt delivery are both configured in the distributor.

Interrupts sent by the peripheral distributor are all in a pending state. The distributor determines the highest priority pending interrupt, which can be passed to the core and forwarded to the CPU interface. At the CPU interface, the interrupt is sequentially sent to the core, at which point the core accepts the FIQ or IRQ exception. The core executes the exception handler in response. The handler queries the interrupt ID from the CPU interface register and starts executing the interrupt service routine. Upon completion, the handler must write to the CPU interface register to report the end of processing.

The Distributor provides registers that report the current status of different interrupt IDs. In multi-core/multi-processor systems, a single GIC can be shared by multiple cores (up to 8 in GICv2). The GIC provides registers to control the cores corresponding to SPIs. This mechanism enables the operating system to share and distribute interrupts across cores and coordinate work.

Configuration

The GIC registers are all implemented asexternal memory-mapped form. All cores can access the public Distributor, but the CPU interface is banked, meaning each core uses the same address to access its own private CPU interface. One core cannot access the CPU interface of another core.

The Distributor contains many registers that you can use to configure the attributes of individual interrupts. These configurable attributes are:

  • Interrupt priority（GICD_IPRIORITY), which the distributor uses to determine which interrupt is forwarded to the CPU interface next.
  • Interrupt configuration（GICD_ICFGR). This determines whether the interrupt is level or edge-sensitive. Not applicable to SGI.
  • An interrupt target core（GICD_ITARGETSR). This determines which cores the interrupt can be routed to. Only applicable to SPI.
  • Interrupt enable or disable state（GICD_ISENABLERGICD_ICENABLER). Only those interrupts that are enabled in the distributor. When they are in the pending state, they are eligible to be routed to the CPU interface.
  • Interrupt security (GICD_IGROUPR) determines whether the interrupt is assigned to Secure or Non-secure.
  • Interrupt state。

The distributor also provides priority masking, which prevents interrupts below a certain priority from reaching the core. The distributor uses it when determining whether a pending interrupt can be forwarded to a specific core. The CPU interface on each core helps fine-tune interrupt control and processing.

Initialization：

The Distributor and CPU interface are both disabled on reset. The GIC must be initialized after reset before it can deliver interrupts to the core.

In the Distributor, software must configure the priority, target core, security, and enable individual interrupts. The Distributor must then be enabled via its control register (GICD_CTLR) Enable distributor。

For each CPU interface, software must set the priority mask and priority preemption. Each CPU interface itself must be enabled via its control register (GICD_CTLR).

Before the CPU processes interrupts, software prepares the CPU to accept valid interrupt vectors from the interrupt vector table, clears the interrupt mask bits in PSTATE, and sets the routing control.

The entire interrupt mechanism in the system can be disabled by disabling the Distributor. Interrupt delivery to individual cores can also be disabled by disabling their CPU interfaces. Individual interrupts can also be disabled (or enabled) in the distributor.

For an interrupt to reach a core, the individual interrupt, the distributor, and the CPU interface must all be enabled. The interrupt must also have sufficient priority, i.e., higher than the core’s priority mask.

Interrupt handling

When a core accepts an interrupt, it jumps to the top-level interrupt vector table and begins execution. The top-levelinterrupt handler reads a register from the CPU interface to get the interrupt ID。

Reading the register, in addition to returning the interrupt ID, also causes the interrupt to be marked as active in the distributor. Once the interrupt ID is known (identifying the interrupt source), the top-level handler can dispatch the device-specific interrupt handler to service the interrupt.

When the interrupt handler finishes execution, the top-level handler writes the same interrupt ID to the End of Interrupt (EoI) register in the CPU interface block, indicating the end of interrupt processing.

In addition to removing the active state, making the final interrupt state Inactive or pending (if the state was active and pending), this allows the CPU interface to forward more pending interrupts to the core. This ends the processing of the individual interrupt.

There may be multiple interrupts waiting to be serviced on the same core, but the CPU interface can only signal one interrupt at a time. The top-level interrupt handler can repeat the above sequence until it reads thespecial interrupt ID value 1023，indicating that there are no more interrupts pending on this core. This special interrupt ID is known asSpurious interrupt identifier. A spurious interrupt ID is a reserved value and cannot be assigned to the system.After the top-level handler reads the spurious interrupt ID, it can complete its execution and prepare the kernel to restore the task it was executing before the interrupt.

GICv3 Interrupt Controller

What improvements does GICv3 have over GICv2

  • GICv3 is compatible with GICv2
  • Supports more CPUs, >8
  • Supports message-based interrupts (Message Signaled Interrupt，MSI）
  • SupportsITS（Interrupt Translation Service) service
  • Supports more hardware interrupt numbers, >1020
  • To better support the ARMv8 exception model, supportsInterrupt groups（Interrupt grouping）
  • To optimize access latency,Provides a system register method to access the CPU Interface

Interrupt types supported by GICv3

  • Private Peripheral Interrupt（PPI）
    • PPI refers to interrupts specific to the local CPU, such as the internal timer of the CPU. Different CPUs can use the same PPI interrupt number.
    • PPI can be in group0 and group1
    • Can be edge-triggered or level-sensitive
  • Shared Peripheral Interrupt（SPI）
    • SPI is usually used for peripheral interrupts, and it can be routed to any CPU
    • SPI can be in group0 and group1
    • Can be edge-triggered or level-sensitive
  • Software Generated Interrupt（SGI）
    • SGI is usually a software-triggered interrupt, used for inter-core communication, such as IPI (Inter-Processor Interrupts)
    • SGI can only be edge-triggered
  • Added:Locality-specific Peripheral Interrupt（LPI）
    • In non-secure interrupt group 1
    • Edge-triggered
    • Using ITS service
    • No active state
    • message-based interrupt

Wired interrupts and message-based interrupts

  • Both SPI and LPI support message-based interrupts
    • SPI’s message-based interrupt does not need to go through ITS, write to theGICD_SETSPI_NSRWrite the interrupt number to the register to trigger the interrupt
    • LPI’s message-based interrupt needs ITS

  • Pending The interrupt is triggered, but has not been acknowledged by the CPU yet
  • Active Interrupt is responded to and processed by the CPU
  • Active & Pending Currently, an interrupt is being responded to and processed by the CPU. At this time, an identical interrupt is triggered, and this new interrupt is set to active & pending
  • LPINo active and active & pending states

Question: If the GIC is in the process of responding to an interrupt, and another identical interrupt is triggered, what should be done?

There are two cases to consider here:

  • If the GIC is responding to the first interrupt, meaning the state of the first interrupt is active, and at this time a second identical interrupt is triggered, then the state of the second interrupt will become active & pending, thus avoiding the loss of the interrupt
  • If the first interrupt is still in the pending state, and another identical interrupt arrives at this time, then they will merge into one

Interrupt affinity routing levels (Affinity routing)

  • GICv3 supports 4-level routing
  • Level 0 faces the redistributor

Interrupt affinity routing of GIC-500

  • GIC-500 supports two-level affinity routing
    • Level 0 is core
    • Level 1 is cluster
  • GIC-500 supports a maximum of 128 cores and 32 clusters

0.0.0.1 represents the 1st core in the 0th cluster

0.0.1.1 represents the 1st core in the 1st cluster

  • Interrupt groups and security modes
    • ARMv8 supports secure mode and non-secure mode
    • GICv3 supports EL0~EL3, so each interrupt source needs to be configured with the corresponding interrupt group and security mode
      • Group0 is used for EL3
      • Secure mode Group1: used for Trusted OS on EL2
      • Non-secure mode Group1: used for VMM or OS

Group0 uses FIQ, Group1 uses IRQ or FIQ depending on the situation

    • Non-secure group1 interrupts are directly responded to by Rich OS
    • Secure group1 and Group0 FIQ interrupts are routed to EL3
  • In secure mode

Special interrupt number

Usage of interrupt number 1021 1

  1. The CPU is running the Trusted OS in secure mode, and an interrupt from the OS in non-secure mode occurs. It needs to trap to the FIQ of EL3 to handle it.
  2. The CPU traps to the secure monitor of EL3. The secure monitor reads the IAR register and reads 1021, indicating that this interrupt is expected to be handled in non-secure mode. Switch to the Rich OS in non-secure mode.
  3. The CPU switches to the Rich OS in non-secure mode to handle this interrupt.

Usage of interrupt number 1021 2

  1. The CPU is running the Trusted OS in secure mode, and an interrupt from the OS in non-secure mode occurs. It needs to trap to the FIQ of EL3 to handle it. However, because SCR_EL3.FIQ=0, it can only be handled in EL2.
  2. The Trusted OS in EL2 executes an SMC system call to trap to EL3.
  3. The secure monitor reads the IAR register and reads 1021, indicating that this interrupt is expected to be handled in non-secure mode. Switch to the Rich OS in non-secure mode.
  4. The CPU switches to the Rich OS in non-secure mode to handle this interrupt.

GICC and ICC

**ICC（Interrupt Controller CPU interface）**Specifically refers to in the GIC the interface part that directly interacts with the CPU, via System Registers accessing

Why are the CPU interface registers of GICv2 calledGICC_*, while those of GICv3 are calledICC_*？

The core of the answer lies in:GICv2 and GICv3 have undergone fundamental changes in the implementation of the CPU Interface — from Memory-Mapped I/O (MMIO) to System Registers. ARM clearly distinguishes between these two architectures through naming differences.

Interrupt priority

  • GICv3 supports 8-bit priority, up to 256 levels
    • When supporting 2 security modes, it supports a minimum of 32 interrupt priorities and a maximum of 256
    • When supporting 1 security mode, it supports a minimum of 16 interrupt priorities
  • Interrupt priority
    • The smaller the value, the higher the interrupt priority; 0 indicates the highest priority, 255 indicates the lowest priority, idle priority
    • GICR_IPRIORITYR<n>Setting PPI and SGI interrupt priorities
    • GICD_IPRIORITYR<n>Setting SPI interrupt priority
    • The LPI configuration table stores the interrupt priorities of the LPIs

Interrupt Priority Grouping Register (ICC_BPR0_EL1/ICC_BPR1_EL1）

  • Each interrupt in the GIC has a 8-bit priority value (priority field), the smaller the value, the higher the priority (e.g., 0x00 is the highest priority, 0xFF is the lowest).
  • The CPU Interface (CPU Interface) when deciding whether to preempt the currently processing interrupt , compares the priority of the new interrupt and the current interrupt.

But the GIC does not simply compare the entire 8-bit value — it divides Binary Point Register（BPR） the priority into two parts:

FieldMeaning
Group PriorityThe high-order part, used to determine whether preemption can occur
SubpriorityThe low-order part, used for sorting only when group priorities are the same
  • ICC_BPR0_EL1: used for Group 0 interrupt
  • ICC_BPR1_EL1: used for Group 1 interrupt

Note: The grouping of group1 and group0 is slightly different

Interrupt Priority Threshold and Running Interrupt Priority

  • Interrupt Threshold
    • Register:ICC_PMR_EL1（Priority Mask Register）
    • PMR determines the priority threshold of the target CPU. The GIC will only send an interrupt to the CPU when the priority of the pending interrupt is higher than this interrupt priority threshold.
    • A PMR of 0 means all interrupts are masked from being sent to the CPU.
  • Running priority
    • Register:ICC_RPR_EL1（Running Priority Register）
    • Read-only register, returns the Group Priority of the highest priority interrupt currently being processed on the current CPUGroup Priority of the highest priority interrupt currently being processed。

Preemption of interrupt priority

  • GICv3Supports interrupt preemption, when an interrupt priority simultaneously satisfies the following conditions
    • Priority is higher than the priority mask register (PMR) of the CPU interface
    • Group priority is higher than the running priority being processed

GICv3 internal architecture

  • Distributor: priority queuing, dispatching SPI and SGI to the redistributor
  • Redistributor: connects to the CPU interface. One redistributor per CPU
  • CPU interface: sends interrupts to the CPU, acknowledges interrupts, etc.
  • ITS: Interrupt Translation Service, converts LPI interrupt requests to interrupt numbers and sends them to the redistributor

ITS service (Interrupt Translation Service)

  • ITS function: converts the device (device_id)'s Event_ID into:
    • Hardware interrupt number (INTID)
    • Target redistributor
  • ITS translation process
    • Uses Device_ID to query the device table
    • Uses Event_ID to query the Interrupt Translation Table
      • Physical interrupt number INTID
      • interrupt collection number
    • Uses ICID to query the Collection Table to get the target redistributor
  • ITS five tables
    • Interrupt configuration table (configure table)
    • Interrupt pending table
    • device table
    • Interrupt translation table
    • Collection table

Configuration table and pending table

  • Interrupt Configuration TableUsed to store the priority and enable bit of each LPI interrupt

    • Each table entry occupies 8 bits
    • Configuration table base address: GICR_PROPBASER.Physical_Address
    • Number of entries: 2^(GICR_PROPBASER.IDbits)

  • Interrupt pending Used to indicate the pending state of each LPI interrupt
    • Each table entry occupies 1 bit
    • Each redistributor has an interrupt pending table

Creation of Device Table

  • Device Table is created by OS software

    • Allocate memory, create Table
    • Set the base address of the table to GITS_BASER.Physical_Address

  • Important parameters of Device Table are inGITS_BASER\<n\>in the register

    • Table type: GITS_BASER.Type
    • Entry size: GITS_BASER.Entry_Size (8 bytes)
    • Size of each page in Table: GITS_BASER.Page_Size（eg, 64KB）
    • Whether a secondary page table is needed: GITS_BASER_Indirect
    • Set table base address: GITS_BASER.Physical_Address
    • Number of entries needed: 2^(device_id bit width), device_id bit width in GITS_TYPER.devbits

  • Device Table supports 1-level or 2-level tables

  • The content of the Device Table entry is determined by:

    • Software sends commands to hardware via the command queue, and the hardware fills the table entry
    • Through the MAPD command, map the deviceID to the ITT table

  • The entry of the Device Table is called DTE, which is used to point to the ITT table

  • The entry of the ITT is called ITE, which is used to describe the relationship between the EventID and the final physical ID number

Creation of the Interrupt translation table

  • ITT entry is used to map EventID -> physical interrupt number INTID and ICID
  • ITT important parameters
    • ITT entry size: GITS_TYPER.ITT_entry_size (e.g.: 16 bytes)
    • Number of ITT entries: the number of vectors requested when applying for interrupts
    • Base address of the ITT table: allocated by the OS
  • The content of the ITT entry is determined by:
    • Software sends commands to hardware via the command queue, and the hardware completes it
    • Through the MAPD command, map the deviceID to the ITT table

Creation of the Collection Table

  • Collection Table entry is used to map:ICID -> redistributor

  • This table does not need to allocate memory in memory

  • The content of the Collection Table entry is composed of:

    • Software sends commands to the hardware through the command queue, and the hardware completes them.
    • Software informs: the physical address of the redistributor or GICR_TYPER.Processor_Number
    • Map ICID -> redistributor through the MAPC command
    • During GIC initialization, traverse all redistributors and call the MAPC command for initialization

  • Collection table entry CTE

ITS Command Queue

  • Three registers related to the command queue
    • GITS_CBASER: Specifies the size and base address of the command queue. The base address must be 64KB aligned, and the size must be a multiple of 4KB.
    • GITS_CREADR: The next command to be processed by the ITS
    • GITS_CWRITER: The next command to be written
  • Commonly used Commands
    • MAPD: Maps device ID to the ITT table
      • MAPD <DeviceID>, <ITT_addr>, <size>
    • MAPI: Maps eventid and deviceid to the ITT
      • MAPI <DeviceID>, <EventID>, <Collection ID>
    • MAPTI: Maps eventid, deviceid, and hardware interrupt number to the ITT
      • MAPTI <DeviceID>, <EventID>, <INTID>, <Collection ID>
    • MAPC: Maps collection id to the target redistributor
      • MAPC <Collection ID>, <Target Redistributor>

Example

Assume a device has a Device ID of 5, and you want to map Event ID=0 to physical interrupt number 8192. The base address of the ITT table is 0x850000. The corresponding Collection ID is 3, and the physical address of the corresponding target redistributor is 0x78400000.

The Linux kernel prefers to use vector to represent eventid

How does an IO device trigger an interrupt?

For GICv3 interrupt control, an IO device only needs to write an event ID to the GITS_TRANSLATER register to trigger an MSI interrupt

The Linux kernel encapsulates a struct msi_msg data structure, which includes the address of this register and the data to be written

In irq_chip’s ops has an its_irq_compose_msi_msg callback function, used to fill this msi_msg, which is to write the GITS_TRANSLATER register’s physical address into the msi_msg->address field

When the device driver uses platform_msi_domain_alloc_irqs() to register MSI, it will get the GITS from the msi_msg data structure_TRANSLATER register’s physical address and eventID, and then write them into the IO device’s own registers (e.g., SMMU’s SMMU_EVENTQ_IRQ_CFGO and CFG1).

When a device triggers an MSI, it automatically writes the eventid into its own register to trigger an interrupt.
