Floating-point number

  • ARMv8 supports the IEEE 754 standard
  • The ARM64 processor supports single-precision and double-precision floating-point numbers. In the ARM64 processor, single-precision floating-point numbers are represented using 32-bit Sn registers, and double-precision floating-point numbers are represented using 64-bit Dn registers.
  • In the C language, the float type can be used to represent single-precision floating-point numbers, and the double type to represent double-precision floating-point numbers.
  • A floating-point number consists of three parts: the sign bit S, the exponent, and the mantissa.
    • A sign bit of 0 indicates a positive number, and 1 indicates a negative number.
    • The exponent has a fixed bias of 127.
    • Single-precision floating-point numbers use 32 bits of space, with 8 bits for the exponent and 24 bits for the mantissa.
    • Double-precision floating-point numbers use 64 bits of space, with 11 bits for the exponent and 53 bits for the mantissa.

Binary representation of floating-point numbers

  • Convert the decimal number (5.25) to a single-precision floating-point number, what is its binary storage format?

  • Steps

    1. Convert a decimal number to a binary number: the integer part is directly converted to a binary number, and the fractional part is multiplied by 2 to take the integer

    The integer part directly converts 5 to binary, becoming the binary number: 101

    The fractional part requires multiplying the decimal fraction by 2, and the digit (0 or 1) to the left of the decimal point in the resulting product is used as the digit in the binary representation, until the required precision is met

    0.25 * 2 = 0.5 The digit to the left of the decimal point is 0

    0.5 * 2 = 1.0 The digit to the left of the decimal point is 1

    The decimal number (5.25) converted to a binary number is 101.01

    1. Normalize the binary number, adjust the exponent so that there is only one significant digit before the decimal point

    The binary number (101.01) after normalization becomes: 1.0101*2^2, where the mantissa is 0101 and the exponent is 2

    1. Calculate the exponent. For single-precision floating-point numbers, an offset of 7F (127) needs to be added, and for double-precision floating-point numbers, an offset of 3FF (1023) needs to be added.

    In this example, the final exponent is 129

    1. Combining the sign bit, exponent, and mantissa gives the floating-point storage format

    In this example, the sign bit is 0, the exponent is: 1000 0001, the mantissa is: 0101, represented in hexadecimal as: 0x40a80000

FPCR register

Floating-point Control Register

AHP (bit 26) — Alternative half-precision control bit
This flag is used to control whether half-precision floating-point uses the IEEE 754-2008 standard or an alternative implementation designed by ARM. When this flag is 0, the IEEE 754-2008 standard half-precision representation is used; when it is 1, the alternative representation designed by ARM is used.

It should be noted here that the FP16 instruction extension introduced from ARMv8.2 directly uses IEEE format half-precision floating-point and ignores this flag.

Below is a brief introduction to the half-precision floating-point representation designed by ARM. It is described in detail in section A1.4.2 (Half-precision floating-point formats) of the ARMv8 Reference Guide. Like the IEEE 754-2008 specification, it has 1 sign bit, 5 exponent bits, and 10 mantissa bits. However, there are differences in the representation of infinity and NaN (Not a Number).

When the exponent part is all 1s (i.e., 0x1F):

  • For IEEE 754-2008: This half-precision floating-point number may be positive or negative infinity (±∞, ±INF), or it may be a NaN (Not a Number). It depends on the mantissa part—if the mantissa part is all zeros, then when the sign bit S=0, the floating-point number is +INF; when the sign bit S=1, it is -INF. If the mantissa part is not zero, then look at the most significant bit of the mantissa (bit 9); if it is 0, the NaN is an SNaN, otherwise it is a QNaN.
  • For the ARM alternative implementation of half-precision floating-point: The floating-point value is treated as a normalized number, which is equivalent to (-1)^S × 2^16 × (1.fraction). This means the maximum positive normalized number is (2 - 2^-10) × 2^16 = 131008.

DN (bit 25) — Default NaN mode control bit
This flag controls whether a NaN operand is propagated to the output of a floating-point operation, meaning that even if one of the operands is NaN, the calculation is still fully executed. If this flag is set to 1, then for a floating-point arithmetic instruction, as long as one of the operands is NaN, a default NaN value set by the ARM processor is returned immediately. This flag can also improve floating-point execution performance.

If the DN flag is 1, then the default NaN values for the three floating-point types are as follows:

  • 16-bit half-precision floating-point: 0x7E00
  • 32-bit single-precision floating-point: 0x7FC0’0000
  • 64-bit double-precision floating-point: 0x7FF8’0000’0000’0000

FZ (bit 24) — Flush-to-zero mode control bit
The behavior of this mode is consistent with the DAZ flag in x86 (rather than FTZ) — if this flag is set to 1, then for a floating-point calculation instruction, all denormalized numbers in the operands are rounded to zero. In addition, this flag only controls 32-bit single-precision floating-point and 64-bit double-precision floating-point, and does not affect the 16-bit half-precision floating-point introduced by ARMv8.2-FP16.

RMode (bits 23:22) — Rounding Mode control field
Regarding this field, it is related to the rounding mode of floating-point numbers.

FZ16 (bit 19) — Flush-to-zero mode control bit on half-precision data-processing instructions
This flag is only effective when the ARM architecture supports the ARMv8.2-FP16 instruction extension. This flag controls the flush-to-zero operation for half-precision floating-point, with the same semantics as the FZ above. However, this flag is only used to control 16-bit half-precision floating-point and does not affect other floating-point types.

Architectural Feature Access Control Register CPACR_EL1

Field that controls whether access to FP/NEON registers traps to EL1:FPEN

  • When FPEN is 0b01, it means that accessing SVE, Advanced SIMD, and floating-point unit registers in EL0 will trap to EL1 for processing, with an exception type code of 0x7
  • When FPEN is 0b00 or 0b10, it means that accessing SVE, Advanced SIMD, and floating-point unit registers in EL0 or EL1 will trap to EL1 for processing, with an exception type code of 0x7
  • When FPEN is 0b11, it means it will not trap to EL1

Reverse instruction

  • REV16 instruction

Indicates that 16-bit data elements in a vector register form a container. In this 16-bit container, the order of the 8-bit data elements is reversed, i.e., the order between B[0] and B[1] is reversed.

  • REV32 instruction

Indicates that 32-bit data elements in a vector register form a container. In this container, the order of the 8-bit or 16-bit data elements is reversed.

  • REV64 instruction

Indicates that 64-bit data elements in a vector register form a container. In this container, the order of the 8-bit, 16-bit, or 32-bit data elements is reversed.

Auto-vectorization

  • There are three ways to optimize code using the NEON instruction set:
    • Hand-written NEON assembly code
    • Using NEON intrinsics provided by the compiler
    • Using the auto-vectorization option provided by the compiler to automatically generate NEON instructions for optimization
  • The GCC assembler has built-in automatic vectorization optimization features. GCC provides the following compilation options:
    • -ftree-vectorize: Performs vectorization optimization. This enables “-ftree-loop-vectorize” and “-ftree-slp-vectorize” by default.
    • -ftree-loop-vectorize: Performs loop vectorization optimization. Unrolls loops to reduce the number of iterations while performing more operations in each iteration.
    • -ftree-slp-vectorize: Bundles scalar operations together to utilize the bandwidth of vector registers. SLP stands for Superword-Level Parallelism.
    • GCC’s “O3” optimization option automatically enables “-ftree-vectorize”, which enables the automatic vectorization optimization feature.

Automatic Vectorization Optimization Constraints

  • GCC’s automatic vectorization optimization feature may not work in some cases.
    • In iterations of different loops with interdependencies
    • Loops with break clauses
    • Loops with complex conditions
